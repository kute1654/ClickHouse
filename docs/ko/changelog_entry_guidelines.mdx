<div id="changelog-entry-guidelines">
  # 변경 로그 항목 작성 가이드라인
</div>

좋은 변경 로그 항목은 사용자에게 무엇이 새로 추가되었는지와 그로 인해 어떤 영향이 있는지를 빠르게 이해하는 데 도움이 됩니다. 기여자께서는 각 릴리스의 변경 로그에 포함될, 사용자가 이해하기 쉬운 변경 로그 항목을 작성해 주시기 바랍니다.

아래에서는 좋은 변경 로그 항목을 작성하기 위한 몇 가지 간단한 가이드라인을 설명합니다.

<div id="write-with-the-user-in-mind-not-the-developer">
  ## 개발자가 아니라 사용자를 염두에 두고 작성하세요
</div>

변경 로그 항목은 &#95;개발자&#95;만이 아니라 &#95;사용자&#95;에게 변경 사항을 전달하기 위한 것입니다.
항목을 작성할 때는, 적절한 경우 단순히 무엇이 바뀌었는지만이 아니라 이 변경이 사용자에게 왜 유용한지, 또는 사용자에게 어떤 영향을 주는지도 함께 전달하세요.

예를 들어, 다음과 같이 쓰는 대신:

> `system.iceberg_history` 테이블 추가

다음과 같이 작성하세요:

> 이제 사용자는 새 `system.iceberg_history` 테이블을 사용해 Iceberg 테이블의 과거 스냅샷을 확인할 수 있습니다.

다음과 같이 쓰는 대신:

> `stringBytesUniq` 및 `stringBytesEntropy` 함수를 추가하여 무작위 데이터 또는 암호화된 데이터일 가능성이 있는 값을 검색합니다.&quot;

다음과 같이 작성하세요:

> 이제 새 `stringBytesUniq` 및 `stringBytesEntropy` 함수를 사용해 문자열에서 암호화되었거나 무작위 데이터일 가능성이 있는 값을 감지할 수 있으므로, 데이터 품질 문제나 보안 우려를 식별하는 데 도움이 됩니다.

<div id="keep-it-simple">
  ## 간단하게 유지하세요
</div>

설명이 없으면 사용자가 이해하기 어려운 기술 용어는 피하십시오. 1~5문장 정도로 작성하고,
오타나 문법 오류를 잡아내거나 항목을 더 사용자 친화적으로 다듬기 위해 LLM을 활용하는 것도
주저하지 마십시오(꼼수가 아닙니다. 정말입니다!)

다음 대신:

> `EXISTS` 표현식의 인수로 상관 서브쿼리 지원

다음과 같이 작성하십시오:

> 이제 `EXISTS` 절에서 외부 쿼리 컬럼을 참조하는 서브쿼리를 사용할 수 있습니다.

명확하고 간단한 항목의 예시:

> 쿼리별로 페이지 캐시 설정을 조정할 수 있습니다. 이렇게 하면 더 빠르게 실험할 수 있고, 높은 처리량과 낮은 지연 시간이 필요한 쿼리에 맞춰 세부 조정도 가능합니다.

<div id="follow-a-few-simple-formatting-guidelines">
  ## 몇 가지 간단한 서식 지침을 따르세요
</div>

<div id="write-in-full-sentences-and-in-the-present-tense">
  ### 완전한 문장으로 작성하고 현재 시제를 사용합니다
</div>

다음과 같이 쓰지 마십시오:

> 임시 파일을 제거하려고 시도할 때 예외가 발생하면 충돌 수정

다음과 같이 쓰십시오:

> 임시 파일을 제거하려고 시도할 때 예외가 발생하는 경우의 충돌을 수정합니다.

<div id="use-backticks-where-necessary">
  ### 필요한 경우 백틱을 사용합니다
</div>

설정, 함수 이름, SQL 문, 포맷 이름, 데이터 타입 등 코드 요소에는 백틱을 사용하십시오. 일반적으로
`clickhouse-client`에 입력하는 모든 내용은 백틱으로 감싸야 합니다. 이렇게 하면 변경 로그 항목을 더 읽기 쉽게 만들 수 있습니다.

다음 대신:

> Settings use&#95;skip&#95;indexes&#95;if&#95;final and use&#95;skip&#95;indexes&#95;if&#95;final&#95;exact&#95;mode now default to True

다음과 같이 작성하십시오:

> Settings `use_skip_indexes_if_final` and `use_skip_indexes_if_final_exact_mode` now default to `True`

<div id="try-to-follow-a-consistent-format">
  ### 일관된 형식을 유지하세요
</div>

가급적 동일한 형식을 따르세요: 무엇을 하는지 → 왜 사용자에게 중요한지 → 사용 방법(필요한 경우). 이렇게 하면 독자가 항목을 빠르게 훑어보기 쉽고, 내용 전개도 예측하기 쉬워집니다.

예시:

> 이제 벡터 검색 결과를 검색 작업 전이나 후에 filter할 수 있으므로, 성능과 정확도 간의 절충점을 더 잘 제어할 수 있습니다. 새 `vector_search_filter_mode` 설정을 사용해 원하는 방식을 선택하세요.
