이 문서는 ClickHouse Private 클러스터의 컴퓨트 분리 기능인 Hydra를 설명합니다.

<div id="what-is-hydra">
  ## Hydra란 무엇입니까?
</div>

Hydra를 사용하면 **하나의 논리적 ClickHouse 클러스터 내에 여러 노드 그룹을 생성**할 수 있으며, 각 노드 그룹은 다음과 같은 구성을 가질 수 있습니다.

* 서로 다른 수의 노드
* 서로 다른 리소스 할당(CPU, 메모리)
* 별도의 네트워크 엔드포인트

이를 통해 동일한 데이터셋에 대해 서로 다른 워크로드(예: 고부하 분석과 경량 대시보드)를 실행하면서도, 이들 워크로드가 동일한 컴퓨트 리소스를 두고 경쟁하지 않도록 할 수 있습니다.

<div id="how-it-works">
  ## 작동 방식
</div>

Hydra는 통합된 데이터 계층을 유지하면서 컴퓨트를 분리합니다:

* **공유 ClickHouse Keeper** -- Hydra 클러스터의 모든 노드 그룹은 분산 조정을 위해 동일한 Keeper 앙상블을 사용합니다. 노드 그룹마다 별도의 Keeper 인스턴스는 필요하지 않습니다.
* **공유 S3 데이터** -- 모든 노드 그룹은 동일한 S3 데이터 프리픽스에서 읽고, 경우에 따라 여기에 쓰기도 합니다. ClickHouse는 테이블 데이터에 공유 객체 스토리지를 사용하므로 여러 컴퓨트 그룹이 중복 없이 동일한 기반 데이터에 접근할 수 있습니다.
* **분리된 컴퓨트** -- 각 노드 그룹은 독립적인 리소스 구성과 자체 Kubernetes 서비스 엔드포인트를 갖춘 자체 ClickHouse 서버 파드 집합을 실행합니다.

<div id="parent-child-relationship">
  ## 부모-자식 관계
</div>

Hydra는 **부모-자식 모델**을 사용합니다.

* **부모**는 데이터와 Keeper 앙상블을 보유하는 기존 ClickHouse 클러스터입니다.
* **자식**은 부모의 이름과 네임스페이스를 참조해 프로비저닝되는 추가 노드 그룹입니다. 자식을 생성할 때는 `parentCluster.name` 및 `parentCluster.namespace`를 사용하여 부모를 지정합니다.

부모 하나에 여러 자식이 있을 수 있으며, 각 자식은 서로 다른 용량 산정과 액세스 특성을 가질 수 있습니다.

<div id="read-only-vs-read-write-children">
  ## 읽기 전용과 읽기-쓰기 하위 인스턴스
</div>

기본적으로 하위 인스턴스는 **읽기-쓰기** 모드로, 공유 스토리지의 데이터를 읽고 쓸 수 있습니다.

하위 인스턴스는 필요에 따라 **읽기 전용**으로 구성할 수도 있습니다(생성 시 `isReadonly=true` 설정). 읽기 전용 하위 인스턴스는 데이터를 쿼리할 수는 있지만 수정할 수는 없으므로, 쓰기 권한이 필요 없고 쓰기 작업과의 격리가 필요한 보고, 대시보드, 분석과 같은 워크로드에 적합합니다.

<div id="limitations">
  ## 제한 사항
</div>

Hydra를 사용할 때 알아두어야 할 중요한 제약 사항은 2가지입니다.

1. **부모 인스턴스가 중지, 종료 또는 idled 상태이면 하위 인스턴스를 프로비저닝할 수 없습니다.** 하위 인스턴스는 부모의 Keeper 앙상블과 공유 데이터 구성에 의존하므로, 하위 인스턴스를 생성하려면 부모가 실행 중 상태여야 합니다.

2. **부모 클러스터를 삭제하기 전에 모든 하위 인스턴스를 먼저 삭제해야 합니다.** 하위 인스턴스가 여전히 부모를 참조하고 있으면 부모를 삭제할 수 없습니다. 활성 상태의 하위 인스턴스가 있는 부모를 삭제하려고 하면 실패합니다.
