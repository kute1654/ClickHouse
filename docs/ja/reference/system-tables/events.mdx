<div id="description">
  ## 説明
</div>

システム内で発生したイベント数に関する情報が含まれています。たとえば、このテーブルでは、ClickHouseサーバーの起動以降に処理された `SELECT` クエリの数を確認できます。

<div id="columns">
  ## カラム
</div>

{/*AUTOGENERATED_START*/}

* `event` ([String](/ja/reference/data-types/index)) — イベント名。
* `value` ([UInt64](/ja/reference/data-types/index)) — 発生したイベント数。
* `description` ([String](/ja/reference/data-types/index)) — イベントの説明。

{/*AUTOGENERATED_END*/}

<div id="example">
  ## 例
</div>

```sql
SELECT * FROM system.events LIMIT 5
```

Does not include queries that failed to parse or were rejected due to AST size limits, quota limits or limits on the number of simultaneously running queries. <div id="dataaftermergediffersfromreplica">
  ### DataAfterMergeDiffersFromReplica
</div>

マージ後のデータが、別のレプリカ上のデータとバイト単位で一致しなかった回数です。原因としては、次のようなものが考えられます。

1. サーバーの更新後に、新しいバージョンの圧縮ライブラリを使用している。
2. 別の圧縮方式を使用している。
3. 圧縮アルゴリズムが決定論的でない (可能性は極めて低い) 。
4. コード内の論理エラーにより、マージアルゴリズムが決定論的でなくなっている。
5. コードのバグにより、メモリ内のデータが破損している。
6. ハードウェアの問題により、メモリ内のデータが破損している。
7. サーバーの起動後に、元データが手動で変更された。
8. ZooKeeper に保存されているチェックサムが手動で変更された。
9. &#39;enable&#95;mixed&#95;granularity&#95;parts&#39; のようなパーツのフォーマットに関する設定が、レプリカごとに異なる。
   サーバーはこの状況を正常に検出しており、バイト単位で同一の結果にするため、レプリカからマージ済みパーツをダウンロードします。

<div id="readerexecutormodeledcostmicroseconds">
  ### ReaderExecutorModeledCostMicroseconds
</div>

ReaderExecutor の読み取りにおけるモデル化された I/O コスト。読み取りパスの最適性を表す合成的な代理 KPI であり、実測レイテンシではありません。上記のカウンターの加重和で、経験則に基づく S3 の重みを用います: ソースへのリクエスト 1 件あたり 30ms + 未完了の接続 1 件あたり 5ms + ソースから転送された 1 MiB あたり 20ms (有効なペイロードと過剰読み取りを含む) + cache put 1 回あたり 0.1ms + cache get 1 回あたり 0.05ms。負荷に依存しないバイトあたりコストを得るには、これを ReaderExecutorRequestedBytes で割ります。実験的機能であり、実験的な ReaderExecutor に対応しています。
