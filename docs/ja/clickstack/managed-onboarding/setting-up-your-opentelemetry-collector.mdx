collector は **ゲートウェイ** として動作します。つまり、アプリケーション、SDK、agent collector の送信先となる単一の OTLP endpoint です。ゲートウェイはイベントをバッチ化し、設定した処理を適用したうえで、[ClickHouse exporter](https://github.com/open-telemetry/opentelemetry-collector-contrib/tree/main/exporter/clickhouseexporter) を介して ClickHouse に書き込みます。このパターンにより、収集ロジックを application code から切り離し、データを生成するワークロードとは独立してインジェストをスケールできます。ゲートウェイ ロールと agent ロールの違いについては、[Collector roles](/ja/clickstack/ingesting-data/collector#collector-roles) を参照してください。

<Info>
  **既存の collector**

  既存の OpenTelemetry collector を使用している場合は、すでに **ゲートウェイ** ロールとして構成されていることを前提としています。**agent** ロールの collector を再構成する用途には、この手順は推奨しません。
</Info>

        ## collector をデプロイする

        Managed ClickStack 向けにあらかじめ設定された、**ClickStack 版 OpenTelemetry collector**をデプロイします。以下の例では、簡単にするため、collector をローカルで実行し、同じマシンからダミーのテレメトリーを生成します。

        <Note>
          本番環境では、通常、collector は Kubernetes クラスター内、または OpenTelemetry SDKs、agents、その他の collectors からアクセス可能な仮想マシン上にデプロイします。これにより、環境全体のテレメトリーを一元的に収集し、ClickStack に転送できます。
        </Note>

        collector にデータを送信するクライアントを認証するための共有シークレットを選択し、それを接続情報および `hyperdx_ingest` ユーザー用に選択したパスワードとあわせてエクスポートします:

        ```shell
        export CLICKHOUSE_ENDPOINT=<HTTPS_ENDPOINT>
        export CLICKHOUSE_USER=hyperdx_ingest
        export CLICKHOUSE_PASSWORD=ClickH0u3eRocks123!
        export OTLP_AUTH_TOKEN="a-strong-shared-secret"
        ```

        ClickStack OTel collectorを実行します:

        ```shell
        docker run -d \
          -e OTLP_AUTH_TOKEN=${OTLP_AUTH_TOKEN} \
          -e CLICKHOUSE_ENDPOINT=${CLICKHOUSE_ENDPOINT} \
          -e CLICKHOUSE_USER=${CLICKHOUSE_USER} \
          -e CLICKHOUSE_PASSWORD=${CLICKHOUSE_PASSWORD} \
          -e HYPERDX_OTEL_EXPORTER_CLICKHOUSE_DATABASE=otel \
          -p 4317:4317 \
          -p 4318:4318 \
          clickhouse/clickstack-otel-collector:latest
        ```

        collector は現在、`4317` で OTLP gRPC を、`4318` で OTLP HTTP を公開しています。アプリケーション、SDKs、および agent collector は、リクエストヘッダーに `authorization: $OTLP_AUTH_TOKEN` を指定して、これらのポート宛てに送信してください。

        <Info>
          **本番環境でのデプロイメント**

          本番環境では、OTLP endpoint で TLS を有効にすることを推奨します。[collector の保護](/ja/clickstack/ingesting-data/collector#securing-the-collector) を参照してください。
        </Info>
      </Step>

      <Step>
        ## エンドポイントの確認

        パイプライン全体が正常に動作することを確認するため、collectorに対して合成トラフィックを生成します。ここでは、OpenTelemetry Collector Contribのジェネレーターである[`telemetrygen`](https://github.com/open-telemetry/opentelemetry-collector-contrib/tree/main/cmd/telemetrygen)を使用します。このツールはOTLPのlogs、traces、およびメトリクスを出力し、サービス、重大度、spanのステータス、メトリクスの種別にわたってデータの形式を制御するflagを提供しています。

        Docker イメージから実行します (インストール不要) 。以下のコマンドを読みやすくするために小さなラッパーを定義します。`--add-host` を指定すると、コンテナーがホスト上でリッスンしている collector に接続できます。

        ```shell
        telemetrygen() {
          docker run --rm --add-host=host.docker.internal:host-gateway \
            ghcr.io/open-telemetry/opentelemetry-collector-contrib/telemetrygen:latest "$@"
        }
        export OTEL_ENDPOINT=host.docker.internal:4317
        ```

        または、Goを使用してバイナリをインストールし、`localhost`を対象にする方法もあります：

        ```shell
        go install github.com/open-telemetry/opentelemetry-collector-contrib/cmd/telemetrygen@latest
        export OTEL_ENDPOINT=localhost:4317
        ```

        <Info>
          **想定される動作**

          各 `tg` の実行は約20秒間 (`--duration 20s` で設定) 続き、その間ずっと verbose ログがストリーミングされますが、これは正常です。各実行は20秒経過すると自動的に終了します。以下のログだけでパイプラインの確認には十分です。より充実したオプションのセットでは、さらに数回実行されるため、数分かかります。
        </Info>

        小さな `tg` ヘルパーを定義して、各コマンドで変化する部分 (service、severity、status、属性) のみを指定できるようにします：

        ```shell
        tg() { local signal=$1; shift; telemetrygen "$signal" \
          --otlp-endpoint ${OTEL_ENDPOINT} --otlp-insecure \
          --otlp-header "authorization=\"${OTLP_AUTH_TOKEN}\"" \
          --rate 5 --duration 20s "$@"; }
        ```

        均一なエラーストリームではなく、サービス全体にわたって現実的な重大度の組み合わせ (主に情報レベルで、警告とエラーを含む) でログを送信します。

        ```shell
        tg logs --service frontend --severity-text Info  --severity-number 9  --body "GET /api/products 200" \
          --otlp-attributes 'deployment.environment="production"' \
          --telemetry-attributes 'http.method="GET"' --telemetry-attributes 'http.status_code="200"'
        tg logs --service checkout --severity-text Warn  --severity-number 13 --body "retrying payment authorization" \
          --otlp-attributes 'deployment.environment="production"' \
          --telemetry-attributes 'http.method="POST"'
        tg logs --service payment  --severity-text Error --severity-number 17 --body "payment gateway timeout" \
          --otlp-attributes 'deployment.environment="production"' \
          --telemetry-attributes 'http.status_code="500"'
        ```

        ログだけでもエンドポイントが正常に動作していることを確認できます。より充実したデモデータセット、複数サービスにわたる Service Map、および各種メトリクスタイプのチャートを確認するには、以下のコマンドも展開して実行してください。これらのコマンドは `tg` ヘルパーを再利用するため、同じシェル上で実行してください。

        <Accordion title="より詳細なテレメトリーを生成する（任意）">
          複数の正常なサービスと、障害が発生している 1 つの依存先から、複数のスパンを含むトレースを送信します。これにより、Service Map は大半が正常で 1 つのサービスのみでエラーが発生している、より現実的な構成となり、同時にエラービューにもデータが入ります。

          ```shell
          # Healthy services: the bulk of the traffic, all spans Ok
          for svc in frontend checkout cart; do
            tg traces --service "$svc" --child-spans 3 --span-duration 80ms --status-code Ok \
              --otlp-attributes 'deployment.environment="production"' \
              --telemetry-attributes "http.route=\"/$svc\""
          done

          # One slow dependency returning errors
          tg traces --service payment --child-spans 3 --span-duration 450ms --span-links 1 --status-code Error \
            --otlp-attributes 'deployment.environment="production"' \
            --telemetry-attributes 'http.route="/charge"'
          ```

          グラフにカウンター、Gauge、ディストリビューションを表示できるよう、一般的な 3 種類のメトリクスを送信します:

          ```shell
          tg metrics --service frontend --metric-type Sum       --otlp-metric-name http.server.requests --aggregation-temporality cumulative
          tg metrics --service frontend --metric-type Gauge     --otlp-metric-name system.memory.usage
          tg metrics --service payment  --metric-type Histogram --otlp-metric-name http.server.duration
          ```

          さらにバリエーションを増やすために、サービスをいくつか追加し、指数ヒストグラムも追加します。

          ```shell
          tg traces  --service catalog   --child-spans 2 --span-duration 60ms  --status-code Ok \
            --otlp-attributes 'deployment.environment="production"' --telemetry-attributes 'http.route="/catalog"'
          tg traces  --service inventory --child-spans 2 --span-duration 220ms --status-code Ok \
            --otlp-attributes 'deployment.environment="staging"'   --telemetry-attributes 'http.route="/inventory"'
          tg metrics --service payment   --metric-type ExponentialHistogram --otlp-metric-name db.query.duration
          ```
        </Accordion>

        フラグの全一覧とその他のバリエーションについては、[telemetrygen を使った合成データ](/ja/clickstack/example-datasets/telemetrygen)を参照してください。
      </Step>

        ## collector の設定を調整する

        既存の collector 設定を拡張し、[ClickHouse exporter](https://github.com/open-telemetry/opentelemetry-collector-contrib/tree/main/exporter/clickhouseexporter) 経由で Managed ClickStack に書き込むようにします。

        <Info>
          **ClickHouse exporter が必要です**

          独自のディストリビューションを使用している場合は、ClickHouse exporter が含まれていることを確認してください。アップストリームの [contrib image](https://github.com/open-telemetry/opentelemetry-collector-contrib) には、すでに含まれています。
        </Info>

        以下は、ClickStack UI が想定する receivers、プロセッサ、pipelines で ClickHouse exporter を使用する設定例です。Session Replay (`rrweb`) のルーティング経路を含め、ClickStack ディストリビューションの動作に合わせています。`<clickhouse_cloud_endpoint>` と `<your_password_here>` は、前述で作成した `hyperdx_ingest` ユーザーの認証情報に置き換えてください。

        ```yaml
        receivers:
          otlp/hyperdx:
            protocols:
              grpc:
                include_metadata: true
                endpoint: "0.0.0.0:4317"
              http:
                cors:
                  allowed_origins: ["*"]
                  allowed_headers: ["*"]
                include_metadata: true
                endpoint: "0.0.0.0:4318"

        processors:
          batch:
          memory_limiter:
            # 80% of maximum memory up to 2G, adjust for low memory environments
            limit_mib: 1500
            # 25% of limit up to 2G, adjust for low memory environments
            spike_limit_mib: 512
            check_interval: 5s

        connectors:
          routing/logs:
            default_pipelines: [logs/out-default]
            error_mode: ignore
            table:
              - context: log
                statement: route() where IsMatch(attributes["rr-web.event"], ".*")
                pipelines: [logs/out-rrweb]

        exporters:
          clickhouse:
            database: otel
            endpoint: <clickhouse_cloud_endpoint>
            username: hyperdx_ingest
            password: <your_password_here>
            ttl: 720h
            timeout: 5s
            retry_on_failure:
              enabled: true
              initial_interval: 5s
              max_interval: 30s
              max_elapsed_time: 300s
          clickhouse/rrweb:
            database: otel
            endpoint: <clickhouse_cloud_endpoint>
            username: hyperdx_ingest
            password: <your_password_here>
            ttl: 720h
            logs_table_name: hyperdx_sessions
            timeout: 5s
            retry_on_failure:
              enabled: true
              initial_interval: 5s
              max_interval: 30s
              max_elapsed_time: 300s

        service:
          pipelines:
            traces:
              receivers: [otlp/hyperdx]
              processors: [memory_limiter, batch]
              exporters: [clickhouse]
            metrics:
              receivers: [otlp/hyperdx]
              processors: [memory_limiter, batch]
              exporters: [clickhouse]
            logs/in:
              receivers: [otlp/hyperdx]
              exporters: [routing/logs]
            logs/out-default:
              receivers: [routing/logs]
              processors: [memory_limiter, batch]
              exporters: [clickhouse]
            logs/out-rrweb:
              receivers: [routing/logs]
              processors: [memory_limiter, batch]
              exporters: [clickhouse/rrweb]
        ```

        注意すべき点がいくつかあります。

        * `otlp/hyperdx` receiver は gRPC (`4317`) と HTTP (`4318`) の両方で待ち受けます。アプリケーションとエージェントは、collector ホスト上のこれらのポートを送信先にしてください。
        * `clickhouse` エクスポーターは、ログ、トレース、メトリクスを `otel` データベースに書き込み、ClickStack UI が想定するレイアウトに合わせます。`clickhouse/rrweb` エクスポーターは、`routing/logs` コネクタによって `otel.hyperdx_sessions` にルーティングされる Session Replay イベントを処理します。
        * OTLP receiver の認証は既存の構成に委ねられています。インジェストトークンを必須にしたい場合は、collector の[拡張機能](https://opentelemetry.io/docs/collector/configuration/#extensions) (たとえば `bearertokenauth`) または TLS 対応の reverse proxy を使って設定してください。

        新しい設定で collector をリロードしてください。その後、アプリケーション、SDKs、エージェント collector は、構成で必要な認証ヘッダーを付けて、collector が公開する OTLP エンドポイントに送信するようにしてください。

        Managed ClickStack 向けの OpenTelemetry collector の設定について詳しくは、[OpenTelemetry によるインジェスト](/ja/clickstack/ingesting-data/opentelemetry)を参照してください。
      </Step>

      <Step>
        ## エンドポイントの確認

        パイプライン全体が正常に動作することを確認するために、collectorに対して合成トラフィックを生成します。ここでは、[`telemetrygen`](https://github.com/open-telemetry/opentelemetry-collector-contrib/tree/main/cmd/telemetrygen) (OpenTelemetry Collector Contrib のジェネレーター) を使用します。このツールは、OTLPのログ、トレース、およびメトリクスを出力し、サービス、重大度、スパンのステータス、メトリクスの種類ごとにデータを制御するフラグを提供します。

        Docker imageから実行します (インストール不要) 。`<your-collector-host>` をcollectorがリッスンするhostに置き換え、`authorization` header (または代替認証方式) をcollectorが要求する値に設定してください：

        ```shell
        telemetrygen() {
          docker run --rm --add-host=host.docker.internal:host-gateway \
            ghcr.io/open-telemetry/opentelemetry-collector-contrib/telemetrygen:latest "$@"
        }
        export OTEL_ENDPOINT=<your-collector-host>:4317
        export OTLP_AUTH_TOKEN=<your-auth-token>
        ```

        または、Goを使用してバイナリをインストールする方法もあります：

        ```shell
        go install github.com/open-telemetry/opentelemetry-collector-contrib/cmd/telemetrygen@latest
        ```

        <Info>
          **想定される動作**

          各 `tg` の実行は約20秒間 (`--duration 20s` で設定) 続き、その間はずっと verbose ログがストリーミングされますが、これは正常な動作です。各実行は20秒経過すると自動的に終了します。パイプラインの確認には、以下のログだけで十分です。任意のより充実した一式では、さらに数回の実行が追加されるため、数分かかります。

          --otlp-attributes 'deployment.environment="production"' \

          --otlp-attributes 'deployment.environment="production"' \

        ログだけでもエンドポイントが正常に動作していることを確認できます。より充実したデモデータセット、複数サービスにわたる Service Map、および各種メトリクスタイプのチャートを確認するには、以下のコマンドも展開して実行してください。これらのコマンドは `tg` ヘルパーを再利用するため、同じシェル上で実行してください。

        <Accordion title="より詳細なテレメトリーを生成する（任意）">

<div id="next-steps">
  ## 次のステップ: 独自のデータを送信する
</div>

上記の合成バーストで確認できるのは、パイプラインが動作していることだけです。実際のテレメトリーの送信を開始するには、[ClickStack SDKs](/ja/clickstack/ingesting-data/sdks/index) を使って独自のサービスをインストルメントしてください。これらの SDK は、Node.js、Python、Go、Java など、先ほど確認した collector エンドポイントに OTLP をエクスポートする各種言語向けのインストルメンテーションを提供します。完全な実例については、[アプリケーションをインストルメントする](/ja/clickstack/managed-onboarding/instrument-application) を参照してください。

アプリケーションコードではなくインフラストラクチャから収集するには:

* [Kubernetes の監視](/ja/clickstack/managed-onboarding/monitoring-kubernetes): クラスターからログ、インフラストラクチャのメトリクス、Kubernetes イベントを収集します。
* [AWS CloudWatch Logs の監視](/ja/clickstack/managed-onboarding/monitoring-aws-cloudwatch-logs): OpenTelemetry CloudWatch receiver を介して CloudWatch Logs を転送します。
