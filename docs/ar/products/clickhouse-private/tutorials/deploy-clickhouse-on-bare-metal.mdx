يرشدك هذا الدليل إلى كيفية نشر ClickHouse على عقد الخوادم الفعلية باستخدام Terraform وAnsible على مثيلات AWS EC2 مع RKE2 كتوزيعة Kubernetes. وفي النهاية، سيكون لديك cluster ‏RKE2 قيد التشغيل مع نشر ClickHouse باستخدام operator.

للاطلاع على مواصفات البنية التحتية بالتفصيل، راجع [reference/infrastructure-requirements.md](/ar/products/clickhouse-private/reference/infrastructure-requirements-reference).

***

</div>

* `helm` v3.x
* بيانات اعتماد AWS IAM مع أذونات EC2 وVPC وS3

    ## المرحلة 1: تثبيت Terraform


  <Step>
    ## المرحلة 2: تنزيل البرامج النصية الخاصة بالأمثلة

    نزّل واستخرج البرامج النصية الخاصة بالأمثلة التي يوفّرها ClickHouse:

    ```text
    unzip example-scripts.zip
    cd ./bare-metal
    ```

    يحتوي الأرشيف على وحدات Terraform، وملفات playbook الخاصة بـ Ansible، وقوالب الجرد، وملفات values الخاصة بـ Helm.

    ***

  <Step>
    ## المرحلة 3: إعداد بيانات اعتماد AWS

    أنشئ ملفًا لبيانات الاعتماد يتضمن مفتاح الوصول والمفتاح السري لـ AWS IAM:

    ```bash
    cat > aws-creds.sh <<EOF
    export AWS_ACCESS_KEY_ID=<Your AWS ACCESS KEY ID>
    export AWS_SECRET_ACCESS_KEY=<Your AWS SECRET ACCESS KEY>
    export AWS_REGION=us-east-2
    EOF
    ```

    استبدل `<Your AWS ACCESS KEY ID>` و`<Your AWS SECRET ACCESS KEY>` بالقيم الموجودة في وحدة تحكم AWS IAM.

    حمّل بيانات الاعتماد:

    ```bash
    source ./aws-creds.sh
    ```

    ***
  </Step>

  <Step>
    ## المرحلة 4: تهيئة Terraform وتطبيقه

    هيِّئ وحدة Terraform الخاصة بـ RKE2. أضف `-migrate-state` إذا كنت تنقل ملف الحالة من التخزين المحلي إلى S3.

    ```text
    terraform -chdir=modules/rke2 init \
    -backend-config="encrypt=true" \
    -backend-config="region=us-east-2" \
    -backend-config="bucket=chb-tfstate-359774249275" \
    -backend-config="use_lockfile=true" \
    -backend-config="key=chb/rke2/terraform.tfstate"
    ```

    طبّق تهيئة Terraform لإنشاء البنية التحتية:

    ```text
    terraform -chdir=modules/rke2 apply \
    -var-file=../../vars/common.tfvars \
    -var-file=../../vars/instances.tfvars \
    -var-file=../../vars/rke2.tfvars
    ```

    يؤدي هذا إلى إنشاء مثيلات EC2 وإعدادات الشبكة ومجموعات الأمان اللازمة لعنقود RKE2.

    ***
  </Step>

  <Step>
    ## المرحلة 5: تثبيت تبعيات Ansible

    ```bash
    cd ~/bare-metal/ansible
    ```

    ```text
    ansible-galaxy role install lablabs.rke2
    ```

    ```text
    ansible-galaxy collection install kubernetes.core
    ```

    ***
  </Step>

  <Step>
    ## المرحلة 6: إعداد RKE2

    ### تحقّق من جاهزية العقد

    ```text
    ansible all -i inventory.yml -a 'cloud-init status'
    ```

    ### إعداد أقراص NVMe

    ```text
    ansible-playbook -i inventory.yml disks_nvme.yml
    ```

    ### شغّل ملف Playbook لـ RKE2

    ```text
    ansible-playbook -i inventory.yml rke2_playbook.yml
    ```

    يقوم دليل التشغيل هذا بتثبيت RKE2 وتهيئته على جميع العقد. ويجهّز أول عقدة خادم، ثم ينتظر حتى تصبح جاهزة، وبعد ذلك يضمّ بقية عقد الخادم والعقد العاملة إلى العنقود.

    سيُظهر خرج دليل التشغيل المهام قيد التشغيل على جميع العقد (على سبيل المثال، من `chb-rke2s101` إلى `chb-rke2s302`). وعند اكتمال العملية، ينبغي أن ترى ملخصًا مثل هذا:

    ```text
    PLAY RECAP *********************************************************************
    chb-rke2s101               : ok=40   changed=12   unreachable=0    failed=0    skipped=45   rescued=0    ignored=0
    chb-rke2s102               : ok=23   changed=7    unreachable=0    failed=0    skipped=38   rescued=0    ignored=0
    chb-rke2s201               : ok=23   changed=7    unreachable=0    failed=0    skipped=38   rescued=0    ignored=0
    chb-rke2s202               : ok=23   changed=7    unreachable=0    failed=0    skipped=38   rescued=0    ignored=0
    chb-rke2s301               : ok=23   changed=7    unreachable=0    failed=0    skipped=38   rescued=0    ignored=0
    chb-rke2s302               : ok=23   changed=7    unreachable=0    failed=0    skipped=38   rescued=0    ignored=0
    ```

    ستطبع مهمة `K8s nodes state` أيضًا ملخصًا يوضح جميع العُقد التي حالتها `Ready`، مع أدوارها وإصداراتها.

    ***
  </Step>

  <Step>
    ## المرحلة 7: إعداد cert-manager والشهادات

    ثبّت cert-manager وtrust-manager، وأنشئ شهادات لـ ClickHouse باستخدام ملف Ansible playbook:

    ```bash
    cd ~/bare-metal/ansible
    ansible-playbook -i 127.0.0.1, --connection=local manage_helm_applications.yml
    ```

    ينفّذ هذا الدليل الإجرائي ما يلي:

    1. تثبيت مخطط Helm الخاص بـ cert-manager
    2. تثبيت مخطط Helm الخاص بـ trust-manager
    3. إنشاء جهة إصدار موقَّعة ذاتيًا وشهادة CA
    4. إنشاء جهة إصدار لشهادة CA
    5. إعداد Rancher (اختياري)
    6. إنشاء مساحة اسم لعنقود ClickHouse
    7. إنشاء شهادات خادم ClickHouse وKeeper
    8. نسخ الشهادات إلى أسرار Kubernetes المناسبة

    النتيجة المتوقعة:

    ```text
    PLAY RECAP *****************************************************************
    127.0.0.1                  : ok=18   changed=13   unreachable=0    failed=0    skipped=0    rescued=0    ignored=0
    ```

    ***
  </Step>

  <Step>
    ## المرحلة 8: تثبيت وحدة التخزين المحلية

    ### تثبيت local-path-provisioner

    ```bash
    kubectl apply -f https://raw.githubusercontent.com/rancher/local-path-provisioner/v0.0.32/deploy/local-path-storage.yaml
    ```

    ### إنشاء StorageClass

    ```bash
    cd ~/bare-metal/ch-cluster-deployment
    kubectl apply -f local-nvme-sc.yaml
    ```

    يجب أن يكون ملف YAML الخاص بـ StorageClass على النحو التالي (حدّث قيم topology بحيث تطابق أسماء العُقد لديك):

    ```yaml
    apiVersion: storage.k8s.io/v1
    kind: StorageClass
    metadata:
      name: local-nvme-sc
    allowVolumeExpansion: true
    parameters:
      path: /nvme/disk
    provisioner: rancher.io/local-path
    reclaimPolicy: Delete
    volumeBindingMode: WaitForFirstConsumer
    allowedTopologies:
    - matchLabelExpressions:
        - key: topology.kubernetes.io/zone
          values:
            - <your_cluster_name>-keeper
            - <your_host_1>
            - <your_host_2>
            - <your_host_3>
            - <your_host_4>
            - <your_host_5>
            - <your_host_6>
        - key: directpv.min.io/region
          values:
            - default
    ```

  </Step>

  <Step>
    ## المرحلة 9: تسمية العقد

    أضِف إلى العقد تسميات مجموعة ClickHouse وتسميات منطقة الطوبولوجيا التي يتطلبها المشغّل:

    ```bash
    kubectl get nodes |awk '{ print $1}'|grep '01'|xargs -I {} -n 1 kubectl label nodes '{}' clickhouseGroup=keeper-amd64  topology.kubernetes.io/zone=c-baremetal-ch-01-keeper --overwrite
    kubectl get nodes |awk '{ print $1}'|grep '02'|xargs -I {} -n 1 kubectl label nodes '{}' clickhouseGroup=server-amd64  --overwrite
    kubectl label nodes chb-rke2s102 topology.kubernetes.io/zone=chb-rke2s102 --overwrite
    kubectl label nodes chb-rke2s202 topology.kubernetes.io/zone=chb-rke2s202 --overwrite
    kubectl label nodes chb-rke2s302 topology.kubernetes.io/zone=chb-rke2s302 --overwrite
    ```

    عدّل أسماء العُقد والوسوم بما يتناسب مع بيئتك.

  </Step>

  <Step>
    ## المرحلة 10: تثبيت ClickHouse Operator والكلاستر

    ### ضبط متغيرات البيئة

    أنشئ سكريبت متغيرات Helm واستورده. حدِّث جميع قيم العناصر النائبة:

    ```bash
    #!/bin/bash
    # AWS/S3 creds
    export AWS_ACCESS_KEY_ID=<your_aws_key>
    export AWS_SECRET_ACCESS_KEY=<your_aws_secret>
    export AWS_REGION=<your_aws_region>
    export S3_ACCESS_KEY_ID=<s3_or_minio_key>
    export S3_SECRET_ACCESS_KEY=<s3_or_minio_secret>
    export S3_REGION=us-east-1
    export S3_BUCKET_NAME=<your_s3_bucket>
    export S3_KEY_PREFIX=<ch-s3-unique-uuid>
    export S3_ENDPOINT=https://s3.us-east-1.amazonaws.com
    export ECR_HOST=<ecr_host>
    # Set AZs as determined by VPC subnets. If not relevant then use your k8s host names.
    export AZ_LIST='["<your_host_1>", "<your_host_2>", "<your_host_3>","<your_host_4>", "<your_host_5>", "<your_host_6>"]'

    PASSWORD='My super secret p@$$w0rd'
    export HASHED_PASSWORD=$(echo -n "$PASSWORD" | sha256sum | awk '{printf $1}' | base64 -w 0)
    export CLUSTER_NAME="baremetal-ch-01"
    export CHART_VERSION="1.1.311"

    export SERVER_NG_GROUP="server-amd64"
    export KEEPER_NG_GROUP="keeper-amd64"
    export KEEPER_CPU="4"
    export KEEPER_MEMORY="16Gi"
    export SERVER_CPU="64"
    export SERVER_MEMORY="128Gi"
    export CACHE_DISK_SIZE_GI=379Gi
    ```

    قم بتنفيذ السكريبت:

    ```bash
    source ./set_helm_vars.sh
    ```

    ### تثبيت المشغّل

    سجّل الدخول إلى سجل Helm وثبّت المشغّل:

    ```text
    # Log into the Helm registry
    aws ecr get-login-password | helm registry login --username AWS --password-stdin $ECR_HOST

    # Create image pull secret
    kubectl create secret docker-registry all-icr-io \
      --docker-server=$ECR_HOST \
      --docker-username=AWS \
      --docker-password=$(aws ecr get-login-password) \
      -n clickhouse-operator-system

    OPERATOR_VERSION=1.18587.1

    helm upgrade clickhouse-operator \
      oci://$ECR_HOST/helm/clickhouse-operator-helm \
      --install --reuse-values \
      --version="$OPERATOR_VERSION" \
      --create-namespace \
      -n clickhouse-operator-system \
      --set-json="image.repository=\"$ECR_HOST/clickhouse-operator\"" \
      --set-json='cilium.enabled=false' \
      --set-json='idleScalerEnabled=false' \
      --set-json='webhooks.enabled=false' \
      --set-json='operator.debug=true' \
      --set-json='operator.metricsScraper.enabled=false' \
      --set-json='operator.featureFlags.backupOnASeparatePod=true' \
      --set-json='operator.featureFlags.serverCMEKEnabled=true' \
      --set-json='imagePullSecrets=[{"name":"all-icr-io"}]' \
      --set-json="operator.availabilityZones=$AZ_LIST"
    ```

    ### نشر مجموعة ClickHouse

    أنشئ ملف قيم Helm (`clickhouse-values.yaml`) بإعدادات الكلاستر الخاص بك:

    ```yaml expandable
    imagePullSecrets:
    - name: all-icr-io

    account:
      hashedPassword: "$HASHED_PASSWORD"
    server:
      ssdCacheConfiguration:
        cacheDiskSize: "$CACHE_DISK_SIZE_GI"
      openSSL:
        enabled: true
        required: true
      replicaCount: 3
      arm64: false
      podPolicy:
        nodeSelector:
          clickhouseGroup: "$SERVER_NG_GROUP"
        resources:
          limits:
            cpu: "$SERVER_CPU"
            memory: "$SERVER_MEMORY"
          requests:
            cpu: "$SERVER_CPU"
            memory: "$SERVER_MEMORY"
      tolerations:
        - effect: NoSchedule
          key: "clickhouse.com/do-not-schedule"
          operator: Exists
      storage:
        storageClassName: local-nvme-sc
        resources:
          requests: "300Gi"
        s3:
          bucketName: "$S3_BUCKET_NAME"
          keyPrefix: "$S3_KEY_PREFIX"
          endpoint: "$S3_ENDPOINT"
          region: "$S3_REGION"
      image:
        repository: "$ECR_HOST/clickhouse-server"
        tag: "25.12.1.1497"
      additionalEnvVars:
        - name: AWS_ACCESS_KEY_ID
          value: "${S3_ACCESS_KEY_ID}"
        - name: AWS_SECRET_ACCESS_KEY
          value: "${S3_SECRET_ACCESS_KEY}"

    keeper:
      openSSL:
        enabled: true
        required: true
      image:
        repository: "$ECR_HOST/clickhouse-keeper"
        tag: "26.2.1.258"
      arm64: false
      podPolicy:
        nodeSelector:
          clickhouseGroup: "$KEEPER_NG_GROUP"
        resources:
          limits:
            cpu: "$KEEPER_CPU"
            memory: "$KEEPER_MEMORY"
          requests:
            cpu: "$KEEPER_CPU"
            memory: "$KEEPER_MEMORY"
      tolerations:
        - effect: NoSchedule
          key: "clickhouse.com/do-not-schedule"
          operator: Exists
      storage:
        storageClassName: local-nvme-sc
        resources:
          requests: 30Gi

    serviceAccount:
      create: false
      name: default

    storageClass:
      create: false
      name: local-nvme-sc
      provisioner: rancher.io/local-path
    ```

    ثبِّت الكلستر باستخدام ملف القيم:

    ```text
    # Create namespace
    kubectl create namespace ns-$CLUSTER_NAME

    # Create image pull secret in the cluster namespace
    kubectl create secret docker-registry all-icr-io \
      --docker-server=$ECR_HOST \
      --docker-username=AWS \
      --docker-password=$(aws ecr get-login-password) \
      -n ns-$CLUSTER_NAME

    # Expand variables and install
    TEMP_VALUES_FILE=$(mktemp)
    envsubst < clickhouse-values.yaml > "$TEMP_VALUES_FILE"

    helm upgrade "$CLUSTER_NAME" \
      oci://$ECR_HOST/helm/onprem-clickhouse-cluster \
      --install --reuse-values \
      --version="$CHART_VERSION" \
      -n ns-$CLUSTER_NAME \
      --create-namespace \
      --values "$TEMP_VALUES_FILE" \
      --timeout 10m0s

    rm "$TEMP_VALUES_FILE"
    ```

    ### إصلاحات ما بعد التثبيت

    بعد نشر الكلستر، قد تحتاج إلى تصحيح StatefulSet الخاص بـ Keeper لإضافة image pull secrets وبيانات اعتماد S3:

    ```text
    # Copy pull secret to cluster namespace
    kubectl get secret all-icr-io -n default -o yaml | \
      sed "s/namespace: default/namespace: ns-$CLUSTER_NAME/" | \
      kubectl apply -n ns-$CLUSTER_NAME --force -f -

    # Patch keeper StatefulSet (use sts_keeper_patch_template.yaml from example scripts)
    kubectl patch sts c-$CLUSTER_NAME-keeper -n ns-$CLUSTER_NAME \
      --patch "$(envsubst < sts_keeper_patch_template.yaml)"

    # Restart keeper pods
    kubectl delete pod c-$CLUSTER_NAME-keeper-{0,1,2} -n ns-$CLUSTER_NAME
    ```

    انتظر حتى تُعاد نشر حاويات Keeper، ثم أصلح وحدات التخزين الثابتة للخادم إذا لزم الأمر:

    ```text
    ./sts_server_patch.sh "ns-$CLUSTER_NAME" all-icr-io
    kubectl delete pod -l kind=clickhouse-server
    ```

  </Step>

<div id="verify-installation">
  ## التحقق من التثبيت
</div>

بمجرد أن تكون جميع الـpodات قيد التشغيل، نفّذ إعادة توجيه للمنفذ ثم تحقّق. وبما أن هذا النشر يستخدم TLS (مع تمكين OpenSSL)، فاستخدم المنفذ 9440:

```bash
kubectl port-forward svc/c-$CLUSTER_NAME-server-any 9440:9440 -n ns-$CLUSTER_NAME
```

اتصل بعميل ClickHouse:

```bash
clickhouse client --host localhost --port 9440 --password "$PASSWORD" --secure --accept-invalid-certificate
```

نفّذ استعلامًا تجريبيًا:

```sql
SELECT 1;
```
