<div id="description">
  ## DescripciĂłn
</div>

El formato `RowBinary` analiza los datos fila por fila en formato binario.
Las filas y los valores se muestran consecutivamente, sin separadores.
Como los datos estĂĄn en formato binario, el delimitador despuĂŠs de `FORMAT RowBinary` se especifica estrictamente de la siguiente manera:

* Cualquier cantidad de espacios en blanco:
  * `' '` (espacio - cĂłdigo `0x20`)
  * `'\t'` (tabulaciĂłn - cĂłdigo `0x09`)
  * `'\f'` (salto de pĂĄgina - cĂłdigo `0x0C`)
* Seguido de exactamente una secuencia de salto de lĂ­nea:
  * estilo Windows `"\r\n"`
  * o estilo Unix `'\n'`
* Seguido inmediatamente de datos binarios.

<Note>
  Este formato es menos eficiente que el formato [Native](/es/reference/formats/Native), ya que estĂĄ basado en filas.
</Note>

<div id="unsigned-leb128">
  ### LEB128 sin signo (Base 128 little-endian)
</div>

Una codificaciĂłn de enteros sin signo de longitud variable en formato **little-endian** que se usa para codificar la longitud de tipos de datos de tamaĂąo variable, como `String`, `Array` y `Map`. Se puede encontrar una implementaciĂłn de ejemplo en la [pĂĄgina de Wikipedia sobre LEB128](https://en.wikipedia.org/wiki/LEB128#Decode_unsigned_integer).

<div id="integer-types">
  ### (U)Int8, (U)Int16, (U)Int32, (U)Int64, (U)Int128, (U)Int256
</div>

Todos los tipos enteros se codifican con el nĂşmero de bytes correspondiente en formato **little-endian**. Los tipos con signo (`Int8` a `Int256`) usan la representaciĂłn en **complemento a dos**. La mayorĂ­a de los lenguajes permiten extraer estos enteros de arrays de bytes, ya sea con herramientas integradas o con bibliotecas conocidas. Para `Int128`/`Int256` y `UInt128`/`UInt256`, que superan el tamaĂąo de entero nativo de la mayorĂ­a de los lenguajes, puede ser necesaria una deserializaciĂłn personalizada.

<div id="bool">
  ### Bool
</div>

Los valores booleanos se codifican en un solo byte y pueden deserializarse de forma similar a `UInt8`.

* `0` es `false`
* `1` es `true`

<div id="float32-float64">
  ### Float32, Float64
</div>

NĂşmeros de coma flotante en formato **little-endian**, codificados en 4 bytes para `Float32` y en 8 bytes para `Float64`. Al igual que con los enteros, la mayorĂ­a de los lenguajes ofrecen herramientas adecuadas para deserializar estos valores.

<div id="bfloat16">
  ### BFloat16
</div>

[BFloat16](/es/reference/data-types/float#bfloat16) (Brain Floating Point) es un formato de punto flotante de 16 bits con el mismo rango que Float32, pero con menor precisiĂłn, lo que lo hace Ăştil para cargas de trabajo de aprendizaje automĂĄtico. El formato de transmisiĂłn es, bĂĄsicamente, los 16 bits mĂĄs altos de un valor Float32. Si tu lenguaje no lo admite de forma nativa, la forma mĂĄs sencilla de manejarlo es leerlo y escribirlo como UInt16, convirtiĂŠndolo desde y hacia Float32:

Para convertir BFloat16 a Float32 (pseudocĂłdigo):

```text
// Leer 2 bytes como UInt16 little-endian
// Desplazar a la izquierda 16 bits para obtener los bits de Float32
bfloat16Bits = readUInt16()
float32Bits = bfloat16Bits << 16
floatValue = reinterpretAsFloat32(float32Bits)
```

Para convertir Float32 en BFloat16 (pseudocĂłdigo):

```text
// Desplazar a la derecha los bits de Float32 en 16 posiciones para truncar a BFloat16
float32Bits = reinterpretAsUInt32(floatValue)
bfloat16Bits = float32Bits >> 16
writeUInt16(bfloat16Bits)
```

Ejemplos de valores subyacentes para `BFloat16`:

```sql
SELECT CAST(1.25, 'BFloat16')
```

```text
0xA0, 0x3F, // 1.25 como BFloat16
```

<div id="decimal">
  ### Decimal32, Decimal64, Decimal128, Decimal256
</div>

Los tipos Decimal se representan como enteros **little-endian** con el ancho en bits correspondiente.

* `Decimal32` - 4 bytes, o `Int32`.
* `Decimal64` - 8 bytes, o `Int64`.
* `Decimal128` - 16 bytes, o `Int128`.
* `Decimal256` - 32 bytes, o `Int256`.

Al deserializar un valor Decimal, las partes entera y fraccionaria se pueden obtener usando el siguiente pseudocĂłdigo:

```text
let scale_multiplier = 10 ** scale
let whole_part = trunc(value / scale_multiplier)  // truncar hacia cero
let fractional_part = value % scale_multiplier
let result = Decimal(whole_part, fractional_part)
```

Donde `trunc` realiza el truncado hacia cero (no la divisiĂłn con redondeo hacia abajo, que da un resultado distinto para valores negativos), y `scale` es el nĂşmero de dĂ­gitos despuĂŠs del punto decimal. Por ejemplo, para `Decimal(10, 2)` (equivalente a `Decimal32(2)`), la escala es `2`, y el valor `12345` se representarĂĄ como `(123, 45)`.

La serializaciĂłn requiere la operaciĂłn inversa:

```text
let scale_multiplier = 10 ** scale
let result = whole_part * scale_multiplier + fractional_part
```

Consulta mĂĄs informaciĂłn en la [documentaciĂłn de ClickHouse sobre los tipos Decimal](/es/reference/data-types/decimal).

<div id="string">
  ### String
</div>

Las cadenas de ClickHouse son **secuencias arbitrarias de bytes**. No es necesario que sean UTF-8 vĂĄlidas. El prefijo de longitud es la **longitud en bytes**, no el nĂşmero de caracteres.

Se codifican en dos partes:

1. Un entero de longitud variable (LEB128) que indica la longitud de la cadena en bytes.
2. Los bytes en bruto de la cadena.

Por ejemplo, una cadena `foobar` se codificarĂĄ usando *siete* bytes de la siguiente manera:

```text
0x06, // Longitud LEB128 de la cadena (6)
0x66, // 'f'
0x6f, // 'o'
0x6f, // 'o'
0x62, // 'b'
0x61, // 'a'
0x72, // 'r'
```

<div id="fixedstring">
  ### FixedString
</div>

A diferencia de `String`, `FixedString` tiene una longitud fija, definida en el esquema. Se codifica como una secuencia de bytes, rellenada con bytes cero al final si el valor es mĂĄs corto que `N`.

<Note>
  Al leer un `FixedString`, los bytes cero finales pueden ser relleno o caracteres `\0` reales de los datos; no se pueden distinguir en la transmisiĂłn. El propio ClickHouse conserva los `N` bytes tal cual.
</Note>

Un `FixedString(3)` vacĂ­o contiene Ăşnicamente ceros de relleno:

```text
0x00, 0x00, 0x00
```

`FixedString(3)` no vacĂ­a que contiene la cadena `hi`:

```text
0x68, // 'h'
0x69, // 'i'
0x00, // cero de relleno
```

`FixedString(3)` no vacĂ­o que contiene la cadena `bar`:

```text
0x62, // 'b'
0x61, // 'a'
0x72, // 'r'
```

No se necesita relleno en el Ăşltimo ejemplo, ya que se usan los *tres* bytes.

<div id="datetime">
  ### DateTime
</div>

Se almacena como `UInt32` (cuatro bytes), que representa el nĂşmero de segundos transcurridos ***desde*** `1970-01-01 00:00:00 UTC`.

Sintaxis:

```text

```

Por ejemplo, `DateTime` o `DateTime('UTC')`.

<Note>
  El valor binario siempre es un desplazamiento de ĂŠpoca en UTC. La zona horaria no cambia la codificaciĂłn. Sin embargo, la zona horaria **sĂ­** afecta a cĂłmo se interpretan los valores de cadena al insertar: insertar `'2024-01-15 10:30:00'` en una columna `DateTime('America/New_York')` almacena un valor de ĂŠpoca distinto que insertar la misma cadena en una columna `DateTime('UTC')`, porque la cadena se interpreta como hora local en la zona horaria de la columna. En la transmisiĂłn, ambos son simplemente segundos de ĂŠpoca `UInt32`.
</Note>

Rango de valores admitido: `[1970-01-01 00:00:00, 2106-02-07 06:28:15]`.

Valores subyacentes de ejemplo para `DateTime`:

```sql
SELECT CAST('2024-01-15 10:30:00', 'DateTime(\'UTC\')') AS d
```

```text
0x28, 0x09, 0xA5, 0x65, // 1705314600 como UInt32 (little-endian)
```

<div id="datetime64">
  ### DateTime64
</div>

Se almacena como `Int64` (ocho bytes), que representa el nĂşmero de **ticks** ***anteriores o posteriores*** a `1970-01-01 00:00:00 UTC`. La resoluciĂłn del tick se define mediante el parĂĄmetro `precision`; consulte la sintaxis a continuaciĂłn:

```text

```

Donde `precision` es un entero de `0` a `9`. Por lo general, solo se usan los siguientes: `3` (milisegundos), `6` (microsegundos),
`9` (nanosegundos).

Ejemplos de definiciones vĂĄlidas de `DateTime64`: `DateTime64(0)`, `DateTime64(3)`, `DateTime64(6, 'UTC')` o `DateTime64(9, 'Europe/Amsterdam')`.

<Note>
  Al igual que con `DateTime`, el valor binario siempre es un desplazamiento respecto de la ĂŠpoca UTC. La zona horaria afecta a cĂłmo se interpretan los valores de cadena al insertarlos (consulta la nota de [DateTime](#datetime)), pero la codificaciĂłn en sĂ­ siempre son ticks `Int64` desde la ĂŠpoca UTC.
</Note>

El valor `Int64` subyacente del tipo `DateTime64` puede interpretarse como la cantidad de las siguientes unidades con respecto a la ĂŠpoca UNIX, ya sea antes o despuĂŠs de ella:

* `DateTime64(0)` - segundos.
* `DateTime64(3)` - milisegundos.
* `DateTime64(6)` - microsegundos.
* `DateTime64(9)` - nanosegundos.

Rango de valores admitido: `[1900-01-01 00:00:00, 2299-12-31 23:59:59.99999999]`.

Valores subyacentes de ejemplo para `DateTime64`:

* `DateTime64(3)`: el valor `1546300800000` representa `2019-01-01 00:00:00 UTC`.
* `DateTime64(6)`: el valor `1705314600123456` representa `2024-01-15 10:30:00.123456 UTC`.
* `DateTime64(9)`: el valor `1705314600123456789` representa `2024-01-15 10:30:00.123456789 UTC`.

<Note>
  La precisiĂłn del valor mĂĄximo es 8. Si se usa la precisiĂłn mĂĄxima de 9 dĂ­gitos (nanosegundos), el valor mĂĄximo admitido es 2262-04-11 23:47:16 en UTC.
</Note>

<div id="time64">
  ### Time64
</div>

Se almacena internamente como un `Decimal64` (que a su vez se almacena como `Int64`) y representa un valor de tiempo con segundos fraccionarios y precisiĂłn configurable. Se permiten valores negativos.

Sintaxis:

```text

```

Donde `precision` es un entero entre `0` y `9`. Valores comunes: `3` (milisegundos), `6` (microsegundos), `9` (nanosegundos).

Rango de valores admitidos: `[-999:59:59.xxxxxxxxx, 999:59:59.xxxxxxxxx]`.

<Note>
  En este momento, la configuraciĂłn `enable_time_time64_type` debe establecerse en `1` para usar `Time` o `Time64`.
</Note>

El valor subyacente de `Int64` representa segundos fraccionarios escalados por `10^precision`.

Valores subyacentes de ejemplo para `Time64`:

```sql

SELECT CAST('15:32:16.123456', 'Time64(6)') AS t
```

```text
0x40, 0x82, 0x0D, 0x06,
0x0D, 0x00, 0x00, 0x00, // 55936123456 como Int64
// 55936123456 / 10^6 = 55936.123456 Seconds = 15:32:16.123456
```

<div id="interval-types">
  ### Tipos de intervalos
</div>

Todos los tipos de intervalos se almacenan como `Int64` (ocho bytes, little-endian). El valor representa la cantidad de la unidad de tiempo correspondiente. Los valores negativos son vĂĄlidos.

<Note>

</Note>

Ejemplos de valores subyacentes:

     INTERVAL -7 DAY     AS c,

<div id="uuid">
  ### UUID
</div>

Se representa como una secuencia de 16 bytes. El UUID se almacena como **dos valores `UInt64` little-endian**: los primeros 8 bytes de la representaciĂłn estĂĄndar del UUID se almacenan con el orden de bytes invertido, y los segundos 8 bytes se invierten de forma independiente.

Por ejemplo, dado el UUID `61f0c404-5cb3-11e7-907b-a6006ad3dba0`:

* RepresentaciĂłn estĂĄndar de bytes: `61 f0 c4 04 5c b3 11 e7` | `90 7b a6 00 6a d3 db a0`
* Primera mitad invertida (LE UInt64): `e7 11 b3 5c 04 c4 f0 61`
* Segunda mitad invertida (LE UInt64): `a0 db d3 6a 00 a6 7b 90`

Valores subyacentes de ejemplo para `UUID`:

* `61f0c404-5cb3-11e7-907b-a6006ad3dba0` se representa como:

```text

0xA0, 0xDB, 0xD3, 0x6A, 0x00, 0xA6, 0x7B, 0x90,
```

* El UUID predeterminado `00000000-0000-0000-0000-000000000000` se representa como 16 bytes en cero:

```text

0x00, 0x00, 0x00, 0x00, 0x00, 0x00, 0x00, 0x00,
```

Se puede usar cuando se ha insertado un registro nuevo, pero no se ha especificado un valor UUID.

<div id="ipv4">
  ### IPv4
</div>

Se almacena en cuatro bytes como `UInt32` en orden de bytes **little-endian**. Tenga en cuenta que esto difiere del orden de bytes de red tradicional (big-endian), usado habitualmente para las direcciones IP. Valores internos de ejemplo para `IPv4`:

```sql

  CAST('0.0.0.0',         'IPv4') AS a,
  CAST('127.0.0.1',       'IPv4') AS b,
  CAST('192.168.0.1',     'IPv4') AS c,
  CAST('255.255.255.255', 'IPv4') AS d,
  CAST('168.212.226.204', 'IPv4') AS e
```

```text

0x01, 0x00, 0x00, 0x7f, // 127.0.0.1
0x01, 0x00, 0xa8, 0xc0, // 192.168.0.1
0xff, 0xff, 0xff, 0xff, // 255.255.255.255
0xcc, 0xe2, 0xd4, 0xa8, // 168.212.226.204
```

<div id="nullable">
  ### Nullable
</div>

Un tipo de dato Nullable se codifica de la siguiente manera:

1. Un Ăşnico byte que indica si el valor es `NULL` o no:
   * `0x00` significa que el valor no es `NULL`.
   * `0x01` significa que el valor es `NULL`.
2. Si el valor no es `NULL`, el tipo de dato subyacente se codifica como de costumbre. Si el valor es `NULL`, **no se escriben bytes adicionales** para el tipo subyacente.

Por ejemplo, un valor `Nullable(UInt32)`:

```sql
SELECT    
   CAST(42,   'Nullable(UInt32)') AS a,
   CAST(NULL, 'Nullable(UInt32)') AS b
```

```text
0x00,                   // No es NULL - el valor sigue a continuaciĂłn
0x2A, 0x00, 0x00, 0x00, // UInt32(42)
0x01,                   // NULL - no hay nada a continuaciĂłn
```

<div id="lowcardinality">
  ### LowCardinality
</div>

En el formato RowBinary, el marcador de baja cardinalidad no afecta al wire format. Por ejemplo, `LowCardinality(String)` se codifica de la misma manera que un `String` normal.

<Warning>
  Esto solo se aplica a RowBinary. En el formato Native, `LowCardinality` usa una codificaciĂłn diferente basada en diccionario.
</Warning>

<Note>
  Una columna puede definirse como `LowCardinality(Nullable(T))`, pero no es posible definirla como `Nullable(LowCardinality(T))`: siempre producirĂĄ un error del servidor.
</Note>

Al hacer pruebas, [allow&#95;suspicious&#95;low&#95;cardinality&#95;types](/es/reference/settings/session-settings#allow_suspicious_low_cardinality_types) puede establecerse en `1` para permitir la mayorĂ­a de los tipos de datos dentro de `LowCardinality` y conseguir una mejor cobertura.

<div id="array">
  ### Array
</div>

Un array se codifica de la siguiente manera:

1. Un [entero de longitud variable (LEB128)](#unsigned-leb128) que indica el nĂşmero de elementos del array.
2. Los elementos del array, codificados de la misma manera que el tipo de datos subyacente.

<div id="tuple">
  ### Tuple
</div>

Una Tuple se codifica como todos sus elementos, uno a continuaciĂłn del otro, en el wire format correspondiente, sin metainformaciĂłn ni delimitadores adicionales.

<div id="dynamic">

</div>

El tipo `Dynamic` puede contener valores de cualquier tipo, determinados en tiempo de ejecuciĂłn. <div id="json">
  ### JSON
</div>

El tipo JSON codifica los datos en dos categorĂ­as distintas:

1. **Rutas tipadas** - Rutas declaradas con tipos explĂ­citos en el esquema (p. ej., `JSON(user_id UInt32, name String)`)
2. **Rutas dinĂĄmicas/rutas de desbordamiento cuando se supera el lĂ­mite de rutas dinĂĄmicas** - Rutas detectadas en tiempo de ejecuciĂłn almacenadas con el tipo `Dynamic`. La codificaciĂłn del valor estĂĄ precedida por la definiciĂłn del tipo.

El formato de transmisiĂłn y las reglas son diferentes para estas dos categorĂ­as.

| CategorĂ­a de ruta   | Incluido en la serializaciĂłn | CodificaciĂłn del valor              | Se admiten Variant/Nullable |
| ------------------- | ---------------------------- | ----------------------------------- | --------------------------- |
| **Rutas tipadas**   | Siempre (incluso si es NULL) | Formato binario especĂ­fico del tipo | SĂ­                          |
| **Rutas dinĂĄmicas** | Solo si no es nulo           | DinĂĄmico                            | No                          |

Las rutas se serializan en tres grupos escritos de forma secuencial: rutas tipadas, rutas dinĂĄmicas y rutas de datos compartidos (desbordamiento). Las rutas tipadas y dinĂĄmicas se escriben en un orden definido por la implementaciĂłn (determinado por la iteraciĂłn interna del mapa hash), mientras que las rutas de datos compartidos se escriben en orden alfabĂŠtico. Cada fila JSON en formato RowBinary se serializa de la siguiente manera:

```text

[String: path_1][value_1]
[String: path_2][value_2]
...
```

<div id="aggregatefunction">
  ### AggregateFunction
</div>

`AggregateFunction(func, T)` almacena el estado intermedio completo de una funciĂłn de agregaciĂłn. El formato interno varĂ­a segĂşn la funciĂłn. Algunos ejemplos sencillos:
