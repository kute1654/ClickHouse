La clГЎusula `ORDER BY` contiene:

* una lista de expresiones, por ejemplo, `ORDER BY visits, search_phrase`;
* una lista de nГәmeros que hacen referencia a las columnas de la clГЎusula `SELECT`, por ejemplo, `ORDER BY 2, 1`; o
* `ALL`, que significa todas las columnas de la clГЎusula `SELECT`, por ejemplo, `ORDER BY ALL`.

Para desactivar la ordenaciГіn por nГәmeros de columna, establezca el ajuste [enable&#95;positional&#95;arguments](/es/reference/settings/session-settings#enable_positional_arguments) = 0.
Para desactivar la ordenaciГіn por `ALL`, establezca el ajuste [enable&#95;order&#95;by&#95;all](/es/reference/settings/session-settings#enable_order_by_all) = 0.

La clГЎusula `ORDER BY` puede incluir un modificador `DESC` (descendente) o `ASC` (ascendente), que determina la direcciГіn de ordenaciГіn.
A menos que se especifique explГӯcitamente un criterio de ordenaciГіn, se usa `ASC` de forma predeterminada.
La direcciГіn de ordenaciГіn se aplica a una sola expresiГіn, no a toda la lista; por ejemplo, `ORDER BY Visits DESC, SearchPhrase`.
AdemГЎs, la ordenaciГіn distingue entre mayГәsculas y minГәsculas.

Las filas con valores idГ©nticos en las expresiones de ordenaciГіn se devuelven en un orden arbitrario y no determinista.
Si se omite la clГЎusula `ORDER BY` en una sentencia `SELECT`, el orden de las filas tambiГ©n es arbitrario y no determinista.

<div id="sorting-of-special-values">
  ## OrdenaciГіn de valores especiales
</div>

Hay dos opciones para el orden de ordenaciГіn de `NaN` y `NULL`:

* De forma predeterminada o con el modificador `NULLS LAST`: primero los valores, luego `NaN` y despuГ©s `NULL`.
* Con el modificador `NULLS FIRST`: primero `NULL`, luego `NaN` y despuГ©s los demГЎs valores.

<div id="collation-support">
  ## Compatibilidad con la intercalaciГіn
</div>

Para ordenar valores [String](/es/reference/data-types/string), puede especificar una intercalaciГіn (comparaciГіn). Ejemplo: `ORDER BY SearchPhrase COLLATE 'tr'`: para ordenar por palabra clave en orden ascendente, usando el alfabeto turco, sin distinguir entre mayГәsculas y minГәsculas, asumiendo que las cadenas estГЎn codificadas en UTF-8. `COLLATE` puede especificarse, o no, de forma independiente para cada expresiГіn en ORDER BY. Si se especifica `ASC` o `DESC`, `COLLATE` se coloca despuГ©s. Al usar `COLLATE`, la ordenaciГіn siempre distingue entre mayГәsculas y minГәsculas.

La intercalaciГіn es compatible con [LowCardinality](/es/reference/data-types/lowcardinality), [Nullable](/es/reference/data-types/nullable), [Array](/es/reference/data-types/array) y [Tuple](/es/reference/data-types/tuple).

Solo recomendamos usar `COLLATE` para la ordenaciГіn final de un nГәmero reducido de filas, ya que ordenar con `COLLATE` es menos eficiente que la ordenaciГіn normal por bytes.

<div id="implementation-details">
  ## Detalles de implementaciГіn
</div>

Se utiliza menos RAM si, ademГЎs de `ORDER BY`, se especifica un [LIMIT](/es/reference/statements/select/limit) lo bastante pequeГұo. De lo contrario, la cantidad de memoria utilizada es proporcional al volumen de datos que se va a ordenar. En el procesamiento distribuido de consultas, si se omite [GROUP BY](/es/reference/statements/select/group-by), la ordenaciГіn se realiza parcialmente en los servidores remotos y los resultados se fusionan en el servidor que realiza la solicitud. Esto significa que, en la ordenaciГіn distribuida, el volumen de datos que se debe ordenar puede ser mayor que la cantidad de memoria disponible en un solo servidor.

Si no hay suficiente RAM, es posible realizar la ordenaciГіn en memoria externa (creando archivos temporales en disco). Para ello, use el ajuste `max_bytes_before_external_sort`. Si se establece en 0 (el valor predeterminado), la ordenaciГіn externa se desactiva. Si estГЎ habilitada, cuando el volumen de datos que se debe ordenar alcanza la cantidad de bytes especificada, los datos recopilados se ordenan y se vuelcan en un archivo temporal. DespuГ©s de leer todos los datos, todos los archivos ordenados se fusionan y se generan los resultados. Los archivos se escriben en el directorio `/var/lib/clickhouse/tmp/` configurado (de forma predeterminada, aunque puede usar el parГЎmetro `tmp_path` para cambiar este ajuste). TambiГ©n puede usar spilling to disk solo si la consulta supera los lГӯmites de memoria; es decir, `max_bytes_ratio_before_external_sort=0.6` habilitarГЎ el spilling to disk solo cuando la consulta alcance el `60%` del lГӯmite de memoria (usuario/servidor).

La ejecuciГіn de una consulta puede usar mГЎs memoria que `max_bytes_before_external_sort`. Por este motivo, este ajuste debe tener un valor significativamente menor que `max_memory_usage`. Por ejemplo, si su servidor tiene 128 GB de RAM y necesita ejecutar una sola consulta, establezca `max_memory_usage` en 100 GB y `max_bytes_before_external_sort` en 80 GB.

La ordenaciГіn externa es mucho menos eficaz que la ordenaciГіn en RAM.

<div id="optimization-of-data-reading">
  ## OptimizaciГіn de la lectura de datos
</div>

Si la expresiГіn `ORDER BY` tiene un prefijo que coincide con la clave de ordenaciГіn de la tabla, puede optimizar la consulta mediante la configuraciГіn [optimize&#95;read&#95;in&#95;order](/es/reference/settings/session-settings#optimize_read_in_order).

Cuando la configuraciГіn `optimize_read_in_order` estГЎ habilitada, el servidor de ClickHouse usa el Гӯndice de la tabla y lee los datos en el orden de la clave `ORDER BY`. Esto permite evitar leer todos los datos cuando se especifica [LIMIT](/es/reference/statements/select/limit). Por tanto, las consultas sobre grandes volГәmenes de datos con un lГӯmite pequeГұo se procesan mГЎs rГЎpido.

La optimizaciГіn funciona tanto con `ASC` como con `DESC`, y no funciona junto con la clГЎusula [GROUP BY](/es/reference/statements/select/group-by) ni con el modificador [FINAL](/es/reference/statements/select/from#final-modifier).

Cuando la configuraciГіn `optimize_read_in_order` estГЎ deshabilitada, el servidor de ClickHouse no usa el Гӯndice de la tabla al procesar consultas `SELECT`.

Considere deshabilitar manualmente `optimize_read_in_order` al ejecutar consultas que tengan una clГЎusula `ORDER BY`, un `LIMIT` grande y una condiciГіn [WHERE](/es/reference/statements/select/where) que requiera leer una gran cantidad de registros antes de encontrar los datos consultados.

La optimizaciГіn es compatible con los siguientes motores de tabla:

* [MergeTree](/es/reference/engines/table-engines/mergetree-family/mergetree) (incluidas las [vistas materializadas](/es/reference/statements/create/view#materialized-view)),
* [Merge](/es/reference/engines/table-engines/special/merge),
* [Buffer](/es/reference/engines/table-engines/special/buffer)

En las tablas con motor `MaterializedView`, la optimizaciГіn funciona con vistas como `SELECT ... FROM merge_tree_table ORDER BY pk`. Pero no es compatible con consultas como `SELECT ... FROM view ORDER BY pk` si la consulta de la vista no incluye la clГЎusula `ORDER BY`.

<div id="order-by-expr-with-fill-modifier">
  ## Modificador ORDER BY Expr WITH FILL
</div>

Este modificador tambiГ©n se puede combinar con el [modificador LIMIT ... WITH TIES](/es/reference/statements/select/limit#limit--with-ties-modifier).

El modificador `WITH FILL` puede especificarse despuГ©s de `ORDER BY expr`, con los parГЎmetros opcionales `FROM expr`, `TO expr` y `STEP expr`.
Todos los valores ausentes de la columna `expr` se rellenarГЎn secuencialmente y las demГЎs columnas se rellenarГЎn con sus valores predeterminados.

Para rellenar varias columnas, agregue el modificador `WITH FILL`, con parГЎmetros opcionales, despuГ©s de cada nombre de campo en la secciГіn `ORDER BY`.

```sql title="Query"
ORDER BY expr [WITH FILL] [FROM const_expr] [TO const_expr] [STEP const_numeric_expr] [STALENESS const_numeric_expr], ... exprN [WITH FILL] [FROM expr] [TO expr] [STEP numeric_expr] [STALENESS numeric_expr]
[INTERPOLATE [(col [AS expr], ... colN [AS exprN])]]
```

`WITH FILL` se puede aplicar a campos con tipos numГ©ricos (cualquier tipo de float, decimal o int) o tipos Date/DateTime. Cuando se aplica a campos `String`, los valores que faltan se rellenan con cadenas vacГӯas.
Cuando no se define `FROM const_expr`, la secuencia de relleno usa el valor mГӯnimo del campo `expr` de `ORDER BY`.
Cuando no se define `TO const_expr`, la secuencia de relleno usa el valor mГЎximo del campo `expr` de `ORDER BY`.
Cuando se define `STEP const_numeric_expr`, `const_numeric_expr` se interpreta `as is` para los tipos numГ©ricos, como `days` para el tipo Date y como `seconds` para el tipo DateTime. TambiГ©n admite el tipo de dato [INTERVAL](/es/reference/data-types/special-data-types/interval), que representa intervalos de fecha y hora.
Cuando se omite `STEP const_numeric_expr`, la secuencia de relleno usa `1.0` para el tipo numГ©rico, `1 day` para el tipo Date y `1 second` para el tipo DateTime.
Cuando se define `STALENESS const_numeric_expr`, la consulta generarГЎ filas hasta que la diferencia con la fila anterior en los datos originales supere `const_numeric_expr`.
`INTERPOLATE` se puede aplicar a columnas que no participan en `ORDER BY WITH FILL`. Estas columnas se rellenan en funciГіn de los valores de los campos anteriores al aplicar `expr`. Si `expr` no estГЎ presente, se repetirГЎ el valor anterior. Si se omite la lista, se incluirГЎn todas las columnas permitidas.
