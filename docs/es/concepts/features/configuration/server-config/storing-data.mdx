Los datos procesados en ClickHouse suelen almacenarse en el sistema de archivos local de la
mГЎquina en la que se ejecuta el servidor de ClickHouse. Esto requiere discos de gran capacidad,
que pueden resultar costosos. Para evitar almacenar los datos localmente, se admiten varias opciones de almacenamiento:

1. Almacenamiento de objetos [Amazon S3](https://aws.amazon.com/s3/).
2. [Azure Blob Storage](https://azure.microsoft.com/en-us/products/storage/blobs).
3. No compatible: Hadoop Distributed File System ([HDFS](https://hadoop.apache.org/docs/current/hadoop-project-dist/hadoop-hdfs/HdfsDesign.html))

<br />

<Note>
  ClickHouse tambiГ©n admite motores de tabla externos, que son distintos de
  la opciГіn de almacenamiento externo descrita en esta pГЎgina, ya que permiten leer datos
  almacenados en formatos de archivo genГ©ricos (como Parquet). En esta pГЎgina describimos
  la configuraciГіn de almacenamiento para las tablas de la familia `MergeTree` o de la familia `Log`.

  1. para trabajar con datos almacenados en discos `Amazon S3`, use el motor de tabla [S3](/es/reference/engines/table-engines/integrations/s3).
  2. para trabajar con datos almacenados en Azure Blob Storage, use el motor de tabla [AzureBlobStorage](/es/reference/engines/table-engines/integrations/azureBlobStorage).
  3. para trabajar con datos en Hadoop Distributed File System (no compatible), use el motor de tabla [HDFS](/es/reference/engines/table-engines/integrations/hdfs).
</Note>

<div id="configuring-external-storage">
  ## Configurar almacenamiento externo
</div>

Los motores de tabla de la familia [`MergeTree`](/es/reference/engines/table-engines/mergetree-family/mergetree) y [`Log`](/es/reference/engines/table-engines/log-family/log)
pueden almacenar datos en `S3`, `AzureBlobStorage` y `HDFS` (no compatible) usando un disco de tipo `s3`,
`azure_blob_storage` o `hdfs` (no compatible), respectivamente.

La configuraciГіn del disco requiere:

1. Una secciГіn `type`, igual a uno de `s3`, `azure_blob_storage`, `hdfs` (no compatible), `local_blob_storage`, `web`.
2. La configuraciГіn de un tipo especГӯfico de almacenamiento externo.

A partir de la versiГіn 24.1 de ClickHouse, es posible usar una nueva opciГіn de configuraciГіn.
Requiere especificar:

1. Un `type` igual a `object_storage`
2. `object_storage_type`, igual a uno de `s3`, `azure_blob_storage` (o simplemente `azure` a partir de `24.3`), `hdfs` (no compatible), `local_blob_storage` (o simplemente `local` a partir de `24.3`), `web`.

<br />

Opcionalmente, se puede especificar `metadata_type` (su valor predeterminado es `local`), pero tambiГ©n puede establecerse en `plain`, `web` y, a partir de `24.4`, `plain_rewritable`.
El uso del tipo de metadatos `plain` se describe en la [secciГіn sobre almacenamiento plain](/es/concepts/features/configuration/server-config/storing-data#plain-storage); el tipo de metadatos `web` solo puede usarse con el tipo de almacenamiento de objetos `web`; el tipo de metadatos `local` almacena los archivos de metadatos localmente (cada archivo de metadatos contiene la asignaciГіn a archivos en el almacenamiento de objetos y cierta metainformaciГіn adicional sobre ellos).

Por ejemplo:

```xml
<s3>
    <type>s3</type>
    <endpoint>https://s3.eu-west-1.amazonaws.com/clickhouse-eu-west-1.clickhouse.com/data/</endpoint>
    <use_environment_credentials>1</use_environment_credentials>
</s3>
```

equivale a la siguiente configuraciГіn (a partir de la versiГіn `24.1`):

```xml
<s3>
    <type>object_storage</type>
    <object_storage_type>s3</object_storage_type>
    <metadata_type>local</metadata_type>
    <endpoint>https://s3.eu-west-1.amazonaws.com/clickhouse-eu-west-1.clickhouse.com/data/</endpoint>
    <use_environment_credentials>1</use_environment_credentials>
</s3>
```

La siguiente configuraciГіn:

```xml
<s3_plain>
    <type>s3_plain</type>
    <endpoint>https://s3.eu-west-1.amazonaws.com/clickhouse-eu-west-1.clickhouse.com/data/</endpoint>
    <use_environment_credentials>1</use_environment_credentials>
</s3_plain>
```

es igual a:

```xml
<s3_plain>
    <type>object_storage</type>
    <object_storage_type>s3</object_storage_type>
    <metadata_type>plain</metadata_type>
    <endpoint>https://s3.eu-west-1.amazonaws.com/clickhouse-eu-west-1.clickhouse.com/data/</endpoint>
    <use_environment_credentials>1</use_environment_credentials>
</s3_plain>
```

Un ejemplo de una configuraciГіn de almacenamiento completa serГӯa:

```xml
<clickhouse>
    <storage_configuration>
        <disks>
            <s3>
                <type>s3</type>
                <endpoint>https://s3.eu-west-1.amazonaws.com/clickhouse-eu-west-1.clickhouse.com/data/</endpoint>
                <use_environment_credentials>1</use_environment_credentials>
            </s3>
        </disks>
        <policies>
            <s3>
                <volumes>
                    <main>
                        <disk>s3</disk>
                    </main>
                </volumes>
            </s3>
        </policies>
    </storage_configuration>
</clickhouse>
```

A partir de la versiГіn 24.1, tambiГ©n puede tener este aspecto:

```xml
<clickhouse>
    <storage_configuration>
        <disks>
            <s3>
                <type>object_storage</type>
                <object_storage_type>s3</object_storage_type>
                <metadata_type>local</metadata_type>
                <endpoint>https://s3.eu-west-1.amazonaws.com/clickhouse-eu-west-1.clickhouse.com/data/</endpoint>
                <use_environment_credentials>1</use_environment_credentials>
            </s3>
        </disks>
        <policies>
            <s3>
                <volumes>
                    <main>
                        <disk>s3</disk>
                    </main>
                </volumes>
            </s3>
        </policies>
    </storage_configuration>
</clickhouse>
```

Para establecer como opciГіn predeterminada un tipo especГӯfico de almacenamiento para todas las tablas `MergeTree`,
agregue la siguiente secciГіn al archivo de configuraciГіn:

```xml
<clickhouse>
    <merge_tree>
        <storage_policy>s3</storage_policy>
    </merge_tree>
</clickhouse>
```

Si quieres configurar una polГӯtica de almacenamiento especГӯfica para una tabla concreta,
puedes definirla en la clГЎusula SETTINGS al crear la tabla:

```sql
CREATE TABLE test (a Int32, b String)
ENGINE = MergeTree() ORDER BY a
SETTINGS storage_policy = 's3';
```

TambiГ©n puede usar `disk` en lugar de `storage_policy`. En este caso, no es necesario
tener la secciГіn `storage_policy` en el archivo de configuraciГіn; basta con una secciГіn `disk`.

```sql
CREATE TABLE test (a Int32, b String)
ENGINE = MergeTree() ORDER BY a
SETTINGS disk = 's3';
```

<div id="refresh-parts-interval-and-table-disk">
  ## refresh_parts_interval y table_disk
</div>

Esta configuraciГіn estГЎ pensada para tablas MergeTree no replicadas en las que las partes pueden escribirse externamente y la detecciГіn de metadatos debe actualizarse desde el almacenamiento.

La configuraciГіn de MergeTree `refresh_parts_interval` permite actualizar periГіdicamente la lista de partes de datos desde el almacenamiento subyacente (por ejemplo, para detectar partes escritas externamente). La distinciГіn importante es entre **metadatos compartidos entre rГ©plicas** y **metadatos locales de cada rГ©plica** (por ejemplo, S3 con metadatos locales por rГ©plica): solo cuando los metadatos se comparten las partes nuevas serГЎn visibles para todas las rГ©plicas. Usar solo almacenamiento de objetos no implica metadatos compartidos.

* **Almacenamiento de objetos (por ejemplo, `disk = 's3'`) no implica metadatos compartidos.** Cuando los metadatos se almacenan localmente en cada rГ©plica (el comportamiento predeterminado), cada rГ©plica gestiona de forma independiente sus punteros a blobs en almacenamiento de objetos. Los cambios realizados en una rГ©plica no son visibles para las demГЎs. En ese caso, `refresh_parts_interval` no hace que las partes nuevas sean visibles entre rГ©plicas, porque los metadatos que lee cada rГ©plica son locales a esa rГ©plica.

* **La actualizaciГіn automГЎtica de partes requiere que los metadatos del filesystem se compartan** (o que la tabla use metadatos de solo lectura propiedad de la propia tabla para que la actualizaciГіn sea aplicable). Configurar `table_disk = true` junto con un disco local de la tabla (por ejemplo, `SETTINGS disk = disk(type=object_storage, ...), table_disk = true`) es una forma de obtener la semГЎntica correcta: la tabla controla el ciclo de vida de los metadatos y el almacenamiento se trata como de solo lectura, por lo que `refresh_parts_interval` se ejecuta y pueden detectarse las partes aГұadidas externamente.

* **Con un disco definido globalmente** (por ejemplo, `disk = 's3'` en `storage_configuration`) y los metadatos locales predeterminados, cada rГ©plica tiene su propio estado de metadatos. Aunque los blobs puedan estar en S3, el almacenamiento no se considera compartido a efectos de `refresh_parts_interval`, y las partes nuevas creadas fuera de ClickHouse o en otra rГ©plica no se detectarГЎn.

Para la actualizaciГіn automГЎtica de partes, asegГәrese de que los metadatos se compartan o use un disco a nivel de tabla con `table_disk = true`, como se indicГі anteriormente. Si se confГӯa solo en `refresh_parts_interval` con metadatos locales de la rГ©plica, las partes no se actualizarГЎn como se espera.

<Note>
  `refresh_parts_interval` no se usa para tablas ReplicatedMergeTree.
  Las tablas replicadas ya sincronizan las partes mediante el mecanismo de replicaciГіn.
  Esta configuraciГіn solo se aplica a tablas MergeTree no replicadas en las que las partes se escriben externamente y se requiere actualizar los metadatos.
</Note>

<div id="dynamic-configuration">
  ## ConfiguraciГіn dinГЎmica
</div>

TambiГ©n es posible especificar la configuraciГіn de almacenamiento sin definir previamente un
disco en el archivo de configuraciГіn, ya que puede configurarse en los
ajustes de la consulta `CREATE`/`ATTACH`.

La siguiente consulta de ejemplo se basa en la configuraciГіn dinГЎmica de disco anterior y
muestra cГіmo usar un disco local para almacenar en cachГ© datos de una tabla almacenada en una URL.

```sql highlight={20-23}
ATTACH TABLE uk_price_paid UUID 'cf712b4f-2ca8-435c-ac23-c4393efe52f7'
(
    price UInt32,
    date Date,
    postcode1 LowCardinality(String),
    postcode2 LowCardinality(String),
    type Enum8('other' = 0, 'terraced' = 1, 'semi-detached' = 2, 'detached' = 3, 'flat' = 4),
    is_new UInt8,
    duration Enum8('unknown' = 0, 'freehold' = 1, 'leasehold' = 2),
    addr1 String,
    addr2 String,
    street LowCardinality(String),
    locality LowCardinality(String),
    town LowCardinality(String),
    district LowCardinality(String),
    county LowCardinality(String)
)
ENGINE = MergeTree
ORDER BY (postcode1, postcode2, addr1, addr2)
  SETTINGS disk = disk(
    type=web,
    endpoint='https://raw.githubusercontent.com/ClickHouse/web-tables-demo/main/web/'
  );
```

El siguiente ejemplo aГұade una cachГ© al almacenamiento externo.

```sql highlight={20-28}
ATTACH TABLE uk_price_paid UUID 'cf712b4f-2ca8-435c-ac23-c4393efe52f7'
(

    date Date,

    postcode2 LowCardinality(String),
    type Enum8('other' = 0, 'terraced' = 1, 'semi-detached' = 2, 'detached' = 3, 'flat' = 4),
    is_new UInt8,
    duration Enum8('unknown' = 0, 'freehold' = 1, 'leasehold' = 2),
    addr1 String,
    addr2 String,
    street LowCardinality(String),
```

En la configuraciГіn resaltada a continuaciГіn, observe que el disco de `type=web` estГЎ anidado dentro
del disco de `type=cache`.

<Note>
  El ejemplo usa `type=web`, pero cualquier tipo de disco puede configurarse como dinГЎmico,
  incluido un disco local. Los discos locales requieren que el argumento `path` estГ© dentro del
  parГЎmetro de configuraciГіn del servidor `custom_local_disks_base_directory`, que no tiene
  ningГәn valor predeterminado, asГӯ que establГ©zcalo tambiГ©n cuando use un disco local.
</Note>

<div id="plain-storage">
  ### Uso de Plain Storage
</div>

En `22.10` se introdujo un nuevo tipo de disco, `s3_plain`, que proporciona almacenamiento de escritura Гәnica.
Sus parГЎmetros de configuraciГіn son los mismos que los del tipo de disco `s3`.
A diferencia del tipo de disco `s3`, almacena los datos tal cual. En otras palabras,
en lugar de usar nombres de blob generados aleatoriamente, usa nombres de archivo normales
(de la misma forma en que ClickHouse almacena archivos en el disco local) y no almacena
metadatos localmente. Por ejemplo, estos se derivan de los datos en `s3`.

Este tipo de disco permite mantener una versiГіn estГЎtica de la tabla, ya que no
permite ejecutar merges sobre los datos existentes ni insertar nuevos
datos. Un caso de uso de este tipo de disco es crear backups en Г©l, lo que puede hacerse
mediante `BACKUP TABLE data TO Disk('plain_disk_name', 'backup_name')`. DespuГ©s,
puede hacer `RESTORE TABLE data AS data_restored FROM Disk('plain_disk_name', 'backup_name')`
o usar `ATTACH TABLE data (...) ENGINE = MergeTree() SETTINGS disk = 'plain_disk_name'`.

ConfiguraciГіn:

```xml
<s3_plain>
    <type>s3_plain</type>
    <endpoint>https://s3.eu-west-1.amazonaws.com/clickhouse-eu-west-1.clickhouse.com/data/</endpoint>
    <use_environment_credentials>1</use_environment_credentials>
</s3_plain>
```

A partir de `24.1`, es posible configurar cualquier disco de almacenamiento de objetos (`s3`, `azure`, `hdfs` (no compatible), `local`) usando
el tipo de metadatos `plain`.

ConfiguraciГіn:

```xml
<s3_plain>
    <type>object_storage</type>
```

<div id="s3-plain-rewritable-storage">
  ### Uso de almacenamiento S3 simple reescribible
</div>

Se introdujo un nuevo tipo de disco `s3_plain_rewritable` en `24.4`.
Al igual que el tipo de disco `s3_plain`, no requiere almacenamiento adicional para
los archivos de metadatos. En su lugar, los metadatos se almacenan en S3.
A diferencia del tipo de disco `s3_plain`, `s3_plain_rewritable` permite ejecutar merges
y admite operaciones `INSERT`.
No se admiten las [mutaciones](/es/reference/statements/alter/index#mutations) ni la replicaciГіn de tablas.

Un caso de uso de este tipo de disco es en tablas `MergeTree` no replicadas. Aunque
el tipo de disco `s3` es adecuado para tablas `MergeTree` no replicadas, puede optar
por el tipo de disco `s3_plain_rewritable` si no necesita metadatos locales
para la tabla y estГЎ dispuesto a aceptar un conjunto limitado de operaciones. Esto puede
resultar Гәtil, por ejemplo, para las tablas del sistema.

ConfiguraciГіn:

```xml
<s3_plain_rewritable>
    <type>s3_plain_rewritable</type>
    <endpoint>https://s3.eu-west-1.amazonaws.com/clickhouse-eu-west-1.clickhouse.com/data/</endpoint>
    <use_environment_credentials>1</use_environment_credentials>
</s3_plain_rewritable>
```

es igual a

```xml
<s3_plain_rewritable>
    <type>object_storage</type>
    <object_storage_type>s3</object_storage_type>
    <metadata_type>plain_rewritable</metadata_type>
    <endpoint>https://s3.eu-west-1.amazonaws.com/clickhouse-eu-west-1.clickhouse.com/data/</endpoint>
    <use_environment_credentials>1</use_environment_credentials>
</s3_plain_rewritable>
```

A partir de la versiГіn `24.5`, es posible configurar cualquier disco de almacenamiento de objetos
(`s3`, `azure`, `local`) con el tipo de metadatos `plain_rewritable`.

<div id="encrypted-virtual-file-system">
  ### Uso del cifrado de datos
</div>

Puede cifrar los datos almacenados en discos externos [S3](/es/reference/engines/table-engines/mergetree-family/mergetree#table_engine-mergetree-s3) o [HDFS](#using-hdfs-storage-unsupported) (no compatible), o en un disco local. Para activar el modo de cifrado, en el archivo de configuraciГіn debe definir un disco del tipo `encrypted` y elegir el disco en el que se guardarГЎn los datos. Un disco `encrypted` cifra todos los archivos escritos en tiempo real y, al leer archivos desde un disco `encrypted`, los descifra automГЎticamente. AsГӯ, puede trabajar con un disco `encrypted` como si fuera uno normal.

Ejemplo de configuraciГіn de disco:

```xml
<disks>
  <disk1>
    <type>local</type>
    <path>/path1/</path>
  </disk1>
  <disk2>
    <type>encrypted</type>
    <disk>disk1</disk>
    <path>path2/</path>
    <key>_16_ascii_chars_</key>
  </disk2>
</disks>
```

Por ejemplo, cuando ClickHouse escribe datos de una tabla en un archivo `store/all_1_1_0/data.bin` en `disk1`, en realidad ese archivo se escribirГЎ en el disco fГӯsico en la ruta `/path1/store/all_1_1_0/data.bin`.

Al escribir el mismo archivo en `disk2`, en realidad se escribirГЎ en el disco fГӯsico en la ruta `/path1/path2/store/all_1_1_0/data.bin`, en modo cifrado.

| ParГЎmetro | Tipo   | DescripciГіn                                                                                                                                                     |
| --------- | ------ | --------------------------------------------------------------------------------------------------------------------------------------------------------------- |

| `disk`    | String | Tipo de disco que se utilizarГЎ para el almacenamiento subyacente.                                                                                               |
| `key`     | Uint64 | Clave para el cifrado y descifrado. Puede especificarse en formato hexadecimal mediante `key_hex`. Se pueden especificar varias claves usando el atributo `id`. |

<div id="optional-parameters-encrypted-disk">
  ### ParГЎmetros opcionales
</div>

| ParГЎmetro        | Tipo   | Predeterminado  | DescripciГіn                                                                                                                                                   |
| ---------------- | ------ | --------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------- |

<div id="using-local-cache">
  ### Uso de la cachГ© local
</div>

Es posible configurar una cachГ© local sobre discos en la configuraciГіn de almacenamiento a partir de la versiГіn 22.3.
En las versiones 22.3 - 22.7, la cachГ© solo es compatible con el tipo de disco `s3`. En las versiones &gt;= 22.8, la cachГ© es compatible con cualquier tipo de disco: S3, Azure, Local, Encrypted, etc.
En las versiones &gt;= 23.5, la cachГ© solo es compatible con tipos de disco remotos: S3, Azure, HDFS (sin soporte).
La cachГ© usa la polГӯtica `LRU`.

Ejemplo de configuraciГіn para versiones posteriores o iguales a 22.8:

```xml

    <storage_configuration>
        <disks>
            <s3>
                <type>s3</type>
                <endpoint>...</endpoint>
                ... s3 configuration ...
            </s3>
            <cache>
                <type>cache</type>
                <disk>s3</disk>
                <path>/s3_cache/</path>
                <max_size>10Gi</max_size>
            </cache>
        </disks>
        <policies>
            <s3_cache>
                <volumes>
                    <main>
                        <disk>cache</disk>
                    </main>
                </volumes>
            </s3_cache>
        <policies>
    </storage_configuration>
```

Ejemplo de configuraciГіn para versiones anteriores a la versiГіn 22.8:

```xml
<clickhouse>
    <storage_configuration>
        <disks>
            <s3>
                <type>s3</type>
                <endpoint>...</endpoint>
                ... s3 configuration ...
                <data_cache_enabled>1</data_cache_enabled>
                <data_cache_max_size>10737418240</data_cache_max_size>
            </s3>
        </disks>
        <policies>
            <s3_cache>
                <volumes>
                    <main>
                        <disk>s3</disk>
                    </main>
                </volumes>
            </s3_cache>
        <policies>
    </storage_configuration>
```

<div id="web-storage">
  ### Uso de almacenamiento web estГЎtico (solo lectura)
</div>

Este es un disco de solo lectura. Sus datos solo se leen y nunca se modifican. Una tabla nueva
se carga en este disco mediante la consulta `ATTACH TABLE` (consulte el ejemplo a continuaciГіn). En realidad, el disco local
no se utiliza; cada consulta `SELECT` darГЎ lugar a una solicitud `http` para
obtener los datos necesarios. Cualquier modificaciГіn de los datos de la tabla producirГЎ una
excepciГіn; es decir, no se permiten los siguientes tipos de consultas: [`CREATE TABLE`](/es/reference/statements/create/table),
[`ALTER TABLE`](/es/reference/statements/alter/index), [`RENAME TABLE`](/es/reference/statements/rename#rename-table),
[`DETACH TABLE`](/es/reference/statements/detach) y [`TRUNCATE TABLE`](/es/reference/statements/truncate).
El almacenamiento web puede utilizarse con fines de solo lectura. Un caso de uso
es alojar datos de muestra o migrar datos. Existe una herramienta, `clickhouse-static-files-uploader`,
que prepara un directorio de datos para una tabla determinada (`SELECT data_paths FROM system.tables WHERE name = 'table_name'`).
Para cada tabla que necesite, se obtiene un directorio de archivos. Estos archivos se pueden cargar
en, por ejemplo, un servidor web de archivos estГЎticos. DespuГ©s de esta preparaciГіn,
puede cargar esta tabla en cualquier servidor de ClickHouse mediante `DiskWeb`.

En esta configuraciГіn de ejemplo:

* el disco es de tipo `web`
* los datos estГЎn alojados en `http://nginx:80/test1/`
* se utiliza una cachГ© en el almacenamiento local

```xml
<clickhouse>
    <storage_configuration>
        <disks>
            <web>
                <type>web</type>
                <endpoint>http://nginx:80/test1/</endpoint>
            </web>
            <cached_web>
                <type>cache</type>
                <disk>web</disk>
                <path>cached_web_cache/</path>
                <max_size>100000000</max_size>
            </cached_web>
        </disks>
        <policies>
            <web>
                <volumes>
                    <main>
                        <disk>web</disk>
                    </main>
                </volumes>
            </web>
            <cached_web>
                <volumes>
                    <main>
                        <disk>cached_web</disk>
                    </main>
                </volumes>
            </cached_web>
        </policies>
    </storage_configuration>
</clickhouse>
```

<Tip>
  El almacenamiento tambiГ©n puede configurarse temporalmente dentro de una consulta, si no se
  prevГ© usar un conjunto de datos web de forma rutinaria; consulte la [configuraciГіn dinГЎmica](#dynamic-configuration) y omita
  editar el archivo de configuraciГіn.

  Hay un [conjunto de datos de demostraciГіn](https://github.com/ClickHouse/web-tables-demo) alojado en GitHub. Para preparar sus propias tablas para el
  almacenamiento web, consulte la herramienta [clickhouse-static-files-uploader](/es/concepts/features/tools-and-utilities/static-files-disk-uploader)
</Tip>

<div id="required-parameters-s3">
  #### ParГЎmetros obligatorios
</div>

| ParГЎmetro  | DescripciГіn                                                                                                                  |
| ---------- | ---------------------------------------------------------------------------------------------------------------------------- |
| `type`     | `web`. De lo contrario, el disco no se crea.                                                                                 |
| `endpoint` | La URL del endpoint en formato `path`. La URL del endpoint debe incluir una ruta raГӯz donde se almacenan los datos cargados. |

<div id="optional-parameters-s3">
  #### ParГЎmetros opcionales
</div>

| ParГЎmetro                           | DescripciГіn                                                                                 | Valor predeterminado |
| ----------------------------------- | ------------------------------------------------------------------------------------------- | -------------------- |
| `min_bytes_for_seek`                | El nГәmero mГӯnimo de bytes para usar la operaciГіn de seek en lugar de una lectura secuencial | `1` MB               |
| `remote_fs_read_backoff_threashold` | El tiempo mГЎximo de espera al intentar leer datos desde el disco remoto                     | `10000` segundos     |
| `remote_fs_read_backoff_max_tries`  | El nГәmero mГЎximo de intentos de lectura con backoff                                         | `5`                  |

Si una consulta falla con la excepciГіn `DB:Exception Unreachable URL`, puedes intentar ajustar esta configuraciГіn: [http&#95;connection&#95;timeout](/es/reference/settings/session-settings#http_connection_timeout), [http&#95;receive&#95;timeout](/es/reference/settings/session-settings#http_receive_timeout), [keep&#95;alive&#95;timeout](/es/reference/settings/server-settings/settings#keep_alive_timeout).

Para obtener archivos para la carga, ejecuta:
`clickhouse static-files-disk-uploader --metadata-path <path> --output-dir <dir>` (`--metadata-path` se puede encontrar en la consulta `SELECT data_paths FROM system.tables WHERE name = 'table_name'`).

Al cargar archivos mediante `endpoint`, deben cargarse en la ruta `<endpoint>/store/`, pero la configuraciГіn solo debe contener `endpoint`.

Si la URL no es accesible al cargar el disco mientras el servidor inicia las tablas, se capturan todos los errores. Si en este caso hubo errores, las tablas pueden volver a cargarse (hacerse visibles) mediante `DETACH TABLE table_name` -&gt; `ATTACH TABLE table_name`. Si los metadatos se cargaron correctamente durante el inicio del servidor, las tablas estГЎn disponibles de inmediato.

Usa la configuraciГіn [http&#95;max&#95;single&#95;read&#95;retries](/es/concepts/features/configuration/server-config/storing-data#web-storage) para limitar el nГәmero mГЎximo de reintentos durante una sola lectura HTTP.

<div id="zero-copy">
  ### ReplicaciГіn zero-copy (no lista para producciГіn)
</div>

La replicaciГіn zero-copy es posible, pero no se recomienda, con discos `S3` y `HDFS` (no compatibles). La replicaciГіn zero-copy significa que, si los datos se almacenan de forma remota en varias mГЎquinas y es necesario sincronizarlos, solo se replican los metadatos (las rutas a las partes de datos), pero no los datos en sГӯ.

<Info>
  **La replicaciГіn zero-copy no estГЎ lista para producciГіn**

  La replicaciГіn zero-copy estГЎ deshabilitada de forma predeterminada en ClickHouse 22.8 y versiones posteriores. Esta funcionalidad no se recomienda para uso en producciГіn.
</Info>
