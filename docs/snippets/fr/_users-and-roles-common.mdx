<div id="test-admin-privileges">
  ## Tester les privilèges d’administrateur
</div>

Déconnectez-vous de la session de l’utilisateur `default`, puis reconnectez-vous en tant qu’utilisateur `clickhouse_admin`.

Toutes les opérations suivantes devraient réussir :

```sql
SHOW GRANTS FOR clickhouse_admin;
```

```sql
CREATE DATABASE db1
```

```sql
CREATE TABLE db1.table1 (id UInt64, column1 String) ENGINE = MergeTree() ORDER BY id;
```

```sql
INSERT INTO db1.table1 (id, column1) VALUES (1, 'abc');
```

```sql
SELECT * FROM db1.table1;
```

```sql
DROP TABLE db1.table1;
```

```sql
DROP DATABASE db1;
```

<div id="non-admin-users">
  ## Utilisateurs non administrateurs
</div>

Les utilisateurs doivent disposer des privilèges nécessaires, et tous ne doivent pas être administrateurs. La suite de ce document présente des scénarios d’exemple ainsi que les rôles requis.

<div id="preparation">
  ### Préparation
</div>

Créez ces tables et ces utilisateurs pour les exemples.

<div id="creating-a-sample-database-table-and-rows">
  #### Création d’une base de données, d’une table et de lignes d’exemple
</div>

<Steps>
  <Step>
    ##### Créer une base de données de test

    ```sql
    CREATE DATABASE db1;
    ```
  </Step>

  <Step>
    ##### Créer une table

    ```sql
    CREATE TABLE db1.table1 (
       id UInt64,
       column1 String,
       column2 String
    )
    ENGINE MergeTree
    ORDER BY id;
    ```
  </Step>

  <Step>
    ##### Insérer des lignes d’exemple dans la table

    ```sql
    INSERT INTO db1.table1
       (id, column1, column2)
    VALUES
       (1, 'A', 'abc'),
       (2, 'A', 'def'),
       (3, 'B', 'abc'),
       (4, 'B', 'def');
    ```
  </Step>

  <Step>
    ##### Vérifier la table

    ```sql title="Requête"
    SELECT *
    FROM db1.table1
    ```

    ```response title="Réponse"
    Query id: 475015cc-6f51-4b20-bda2-3c9c41404e49

    ┌─id─┬─column1─┬─column2─┐
    │  1 │ A       │ abc     │
    │  2 │ A       │ def     │
    │  3 │ B       │ abc     │
    │  4 │ B       │ def     │
    └────┴─────────┴─────────┘
    ```
  </Step>

  <Step>
    ##### Créer `column_user`

    Créez un utilisateur standard qui servira à démontrer la restriction de l’accès à certaines colonnes :

    ```sql
    CREATE USER column_user IDENTIFIED BY 'password';
    ```
  </Step>

  <Step>
    ##### Créer `row_user`

    Créez un utilisateur standard qui servira à démontrer la restriction de l’accès aux lignes contenant certaines valeurs :

    ```sql
    CREATE USER row_user IDENTIFIED BY 'password';
    ```
  </Step>
</Steps>

<div id="creating-roles">
  #### Création de rôles
</div>

Avec cet ensemble d’exemples :

* des rôles avec différents privilèges, au niveau des colonnes et des lignes, seront créés
* des privilèges seront accordés aux rôles
* des utilisateurs seront affectés à chaque rôle

Les rôles servent à définir des groupes d’utilisateurs disposant de privilèges spécifiques, au lieu de gérer chaque utilisateur individuellement.

<Steps>
  <Step>
    ##### Créer un rôle pour limiter les utilisateurs de ce rôle à la consultation de `column1` uniquement dans la base de données `db1` et la table `table1` :

    ```sql
    CREATE ROLE column1_users;
    ```
  </Step>

  <Step>
    ##### Définir les privilèges pour autoriser l’accès à `column1`

    ```sql
    GRANT SELECT(id, column1) ON db1.table1 TO column1_users;
    ```
  </Step>

  <Step>
    ##### Ajouter l’utilisateur `column_user` au rôle `column1_users`

    ```sql
    GRANT column1_users TO column_user;
    ```
  </Step>

  <Step>
    ##### Créer un rôle pour limiter les utilisateurs de ce rôle à la consultation de certaines lignes uniquement, en l’occurrence celles contenant `A` dans `column1`

    ```sql
    CREATE ROLE A_rows_users;
    ```
  </Step>

  <Step>
    ##### Ajouter l’utilisateur `row_user` au rôle `A_rows_users`

    ```sql
    GRANT A_rows_users TO row_user;
    ```
  </Step>

  <Step>
    ##### Créer une politique pour autoriser l’accès uniquement aux lignes où `column1` a la valeur `A`

    ```sql
    CREATE ROW POLICY A_row_filter ON db1.table1 FOR SELECT USING column1 = 'A' TO A_rows_users;
    ```
  </Step>

  <Step>
    ##### Définir les privilèges sur la base de données et la table

    ```sql
    GRANT SELECT(id, column1, column2) ON db1.table1 TO A_rows_users;
    ```
  </Step>

  <Step>
    ##### Accorder des permissions explicites aux autres rôles afin qu’ils conservent l’accès à toutes les lignes

    ```sql
    CREATE ROW POLICY allow_other_users_filter 
    ON db1.table1 FOR SELECT USING 1 TO clickhouse_admin, column1_users;
    ```

    <Note>
      Lorsqu’une politique est attachée à une table, le système l’applique, et seuls les utilisateurs et rôles définis peuvent effectuer des opérations sur cette table ; toutes les autres opérations sont refusées. Pour éviter que la politique de lignes restrictive ne s’applique aux autres utilisateurs, une autre politique doit être définie afin de permettre aux autres utilisateurs et rôles de conserver un accès normal ou un autre type d’accès.
    </Note>
  </Step>
</Steps>

<div id="modifying-users-and-roles">
  ## Modifier les utilisateurs et les rôles
</div>

Les utilisateurs peuvent se voir attribuer plusieurs rôles afin d&#39;obtenir la combinaison de privilèges nécessaire. Lorsque plusieurs rôles sont utilisés, le système les combine pour déterminer les privilèges ; au final, les permissions des rôles sont cumulatives.

Par exemple, si `role1` autorise uniquement select sur `column1` et que `role2` autorise select sur `column1` et `column2`, l&#39;utilisateur aura accès aux deux colonnes.

<Steps>
  <Step>
    ##### À l&#39;aide du compte Admin, créez un nouvel utilisateur avec des rôles par défaut afin de restreindre l&#39;accès à la fois par ligne et par colonne

    ```sql
    CREATE USER row_and_column_user IDENTIFIED BY 'password' DEFAULT ROLE A_rows_users;
    ```
  </Step>

  <Step>
    ##### Supprimez les privilèges précédents du rôle `A_rows_users`

    ```sql
    REVOKE SELECT(id, column1, column2) ON db1.table1 FROM A_rows_users;
    ```
  </Step>

  <Step>
    ##### Autorisez le rôle `A_row_users` à sélectionner uniquement `column1`

    ```sql
    GRANT SELECT(id, column1) ON db1.table1 TO A_rows_users;
    ```
  </Step>

  <Step>
    ##### Connectez-vous au client ClickHouse avec `row_and_column_user`

    ```bash
    clickhouse-client --user row_and_column_user --password password;
    ```
  </Step>

  <Step>
    ##### Testez avec toutes les colonnes :

    ```sql
    SELECT *
    FROM db1.table1
    ```

    ```response
    Query id: 8cdf0ff5-e711-4cbe-bd28-3c02e52e8bc4

    0 rows in set. Elapsed: 0.005 sec.

    Received exception from server (version 22.3.2):
    Code: 497. DB::Exception: Received from localhost:9000. 
    DB::Exception: row_and_column_user: Not enough privileges. 
    To execute this query it's necessary to have grant 
    SELECT(id, column1, column2) ON db1.table1. (ACCESS_DENIED)
    ```
  </Step>

  <Step>
    ##### Testez avec les colonnes autorisées uniquement :

    ```sql
    SELECT
        id,
        column1
    FROM db1.table1
    ```

    ```response
    Query id: 5e30b490-507a-49e9-9778-8159799a6ed0

    ┌─id─┬─column1─┐
    │  1 │ A       │
    │  2 │ A       │
    └────┴─────────┘
    ```
  </Step>
</Steps>

<div id="troubleshooting">
  ## Dépannage
</div>

Il arrive que des privilèges se recoupent ou se combinent, produisant des résultats inattendus. Les commandes suivantes peuvent être utilisées pour cerner le problème à l’aide d’un compte administrateur.

<div id="listing-the-grants-and-roles-for-a-user">
  ### Lister les grants et rôles d’un utilisateur
</div>

```sql
SHOW GRANTS FOR row_and_column_user
```

```response
Query id: 6a73a3fe-2659-4aca-95c5-d012c138097b

┌─GRANTS FOR row_and_column_user───────────────────────────┐
│ GRANT A_rows_users, column1_users TO row_and_column_user │
└──────────────────────────────────────────────────────────┘
```

<div id="list-roles-in-clickhouse">
  ### Lister les rôles dans ClickHouse
</div>

```sql
SHOW ROLES
```

```response
Query id: 1e21440a-18d9-4e75-8f0e-66ec9b36470a

┌─name────────────┐
│ A_rows_users    │
│ column1_users   │
└─────────────────┘
```

<div id="display-the-policies">
  ### Afficher les politiques
</div>

```sql
SHOW ROW POLICIES
```

```response

┌─name───────────────────────────────────┐
│ A_row_filter ON db1.table1             │
│ allow_other_users_filter ON db1.table1 │
└────────────────────────────────────────┘
```

<div id="example-commands-to-manage-roles-policies-and-users">
  ## Exemple de commandes pour gérer les rôles, les politiques et les utilisateurs
</div>

Les commandes suivantes peuvent être utilisées pour :

* supprimer des privilèges
* supprimer des politiques
* désattribuer des rôles à des utilisateurs
* supprimer des utilisateurs et des rôles
  <br />

<Tip>
  Exécutez ces commandes en tant qu&#39;utilisateur admin ou `default`
</Tip>

<div id="remove-privilege-from-a-role">
  ### Retirer un privilège d’un rôle
</div>

```sql
REVOKE SELECT(column1, id) ON db1.table1 FROM A_rows_users;
```

<div id="delete-a-policy">
  ### Supprimer une politique
</div>

```sql
DROP ROW POLICY A_row_filter ON db1.table1;
```

<div id="unassign-a-user-from-a-role">
  ### Retirer un utilisateur d’un rôle
</div>

```sql
REVOKE A_rows_users FROM row_user;
```

<div id="delete-a-role">
  ### Supprimer un rôle
</div>

```sql
DROP ROLE A_rows_users;
```

<div id="delete-a-user">
  ### Supprimer un utilisateur
</div>

```sql
DROP USER row_user;
```
