<div id="introduction-introduction">
  ## Introduction [#introduction]
</div>

[Amazon Redshift](https://aws.amazon.com/redshift/) est une solution populaire d&#39;entrepôt de données dans le cloud qui fait partie des offres Amazon Web Services. Ce guide présente différentes approches pour migrer des données d&#39;une instance Redshift vers ClickHouse. Nous couvrirons trois options :

<img src="/images/integrations/data-ingestion/redshift/redshift-to-clickhouse.webp" alt="Options de migration de Redshift vers ClickHouse" />

Du point de vue de l&#39;instance ClickHouse, vous pouvez :

1. **[PUSH](#push-data-from-redshift-to-clickhouse)** envoyer des données vers ClickHouse à l&#39;aide d&#39;un outil ou service ETL/ELT tiers

2. **[PULL](#pull-data-from-redshift-to-clickhouse)** extraire des données de Redshift à l&#39;aide du pont JDBC ClickHouse

3. **[PIVOT](#pivot-data-from-redshift-to-clickhouse-using-s3)** utiliser le stockage d&#39;objets S3 selon une logique « décharger puis charger »

<Note>
  Nous avons utilisé Redshift comme source de données dans ce tutoriel. Toutefois, les approches de migration présentées ici ne sont pas propres à Redshift, et des étapes similaires peuvent être appliquées à toute source de données compatible.
</Note>

<div id="push-data-from-redshift-to-clickhouse-push-data-from-redshift-to-clickhouse">
  ## Envoyer des données de Redshift vers ClickHouse [#push-data-from-redshift-to-clickhouse]
</div>

Dans un scénario push, l’idée est de s’appuyer sur un outil ou un service tiers (qu’il s’agisse de code personnalisé ou d’un [ETL/ELT](https://en.wikipedia.org/wiki/Extract,_transform,_load#ETL_vs._ELT)) pour envoyer vos données vers votre instance ClickHouse. Par exemple, vous pouvez utiliser un logiciel comme [Airbyte](https://www.airbyte.com/) pour transférer des données entre votre instance Redshift (en tant que source) et ClickHouse (en tant que destination) ([consultez notre guide d’intégration pour Airbyte](/fr/integrations/connectors/data-ingestion/etl-tools/airbyte-and-clickhouse))

<img src="/images/integrations/data-ingestion/redshift/push.webp" alt="PUSH de Redshift vers ClickHouse" />

<div id="pros-pros">
  ### Avantages [#pros]
</div>

* Il peut tirer parti du catalogue existant de connecteurs des outils ETL/ELT.
* Fonctionnalités intégrées pour maintenir les données synchronisées (logique d’ajout/écrasement/incrémentielle).
* Permet des scénarios de transformation des données (par exemple, consultez notre [guide d’intégration pour dbt](/fr/integrations/connectors/data-ingestion/etl-tools/dbt/index)).

<div id="cons-cons">
  ### Inconvénients [#cons]
</div>

* Les utilisateurs doivent mettre en place et maintenir une infrastructure ETL/ELT.
* Cela introduit un élément tiers dans l’architecture, qui peut devenir un goulot d’étranglement potentiel en matière de montée en charge.

<div id="pull-data-from-redshift-to-clickhouse-pull-data-from-redshift-to-clickhouse">
  ## Rapatrier des données de Redshift vers ClickHouse [#pull-data-from-redshift-to-clickhouse]
</div>

Dans le scénario Pull, il s&#39;agit d&#39;utiliser le ClickHouse JDBC Bridge pour se connecter directement à un cluster Redshift depuis une instance ClickHouse et exécuter des requêtes `INSERT INTO ... SELECT` :

<img src="/images/integrations/data-ingestion/redshift/pull.webp" alt="PULL de Redshift vers ClickHouse" />

<div id="pros-pros-1">
  ### Avantages [#pros-1]
</div>

* Compatible avec tous les outils JDBC
* Solution élégante pour interroger plusieurs sources de données externes depuis ClickHouse

<div id="cons-cons-1">
  ### Inconvénients [#cons-1]
</div>

* Nécessite une instance ClickHouse JDBC Bridge, qui peut constituer un goulot d&#39;étranglement potentiel en matière de scalabilité

<Note>
  Même si Redshift est basé sur PostgreSQL, il n&#39;est pas possible d&#39;utiliser la table function PostgreSQL ou le table engine ClickHouse, car ClickHouse exige PostgreSQL version 9 ou ultérieure, tandis que l&#39;API Redshift repose sur une version antérieure (8.x).
</Note>

<div id="tutorial-tutorial">
  ### Tutoriel [#tutorial]
</div>

Pour utiliser cette option, vous devez configurer ClickHouse JDBC Bridge. ClickHouse JDBC Bridge est une application Java autonome qui gère la connectivité JDBC et sert de proxy entre l&#39;instance ClickHouse et les sources de données. Pour ce tutoriel, nous avons utilisé une instance Redshift préremplie avec une [base de données d&#39;exemple](https://docs.aws.amazon.com/redshift/latest/dg/c_sampledb.html).

<Steps>
  <Step>
    #### Déployez ClickHouse JDBC Bridge [#deploy-clickhouse-jdbc-bridge]

    Déployez ClickHouse JDBC Bridge. Pour plus de détails, consultez notre guide utilisateur sur [JDBC pour les sources de données externes](/fr/integrations/connectors/data-ingestion/jdbc-with-clickhouse)

    <Note>
      Si vous utilisez ClickHouse Cloud, vous devrez exécuter ClickHouse JDBC Bridge dans un environnement distinct et vous connecter à ClickHouse Cloud à l&#39;aide de la fonction [remoteSecure](/fr/reference/functions/table-functions/remote)
    </Note>
  </Step>

  <Step>
    #### Configurez votre source de données Redshift [#configure-your-redshift-datasource]

    Configurez votre source de données Redshift pour ClickHouse JDBC Bridge. Par exemple, `/etc/clickhouse-jdbc-bridge/config/datasources/redshift.json `

    ```json
    {
     "redshift-server": {
       "aliases": [
         "redshift"
       ],
       "driverUrls": [
       "https://s3.amazonaws.com/redshift-downloads/drivers/jdbc/2.1.0.4/redshift-jdbc42-2.1.0.4.jar"
       ],
       "driverClassName": "com.amazon.redshift.jdbc.Driver",
       "jdbcUrl": "jdbc:redshift://redshift-cluster-1.ckubnplpz1uv.us-east-1.redshift.amazonaws.com:5439/dev",
       "username": "awsuser",
       "password": "<password>",
       "maximumPoolSize": 5
     }
    }
    ```
  </Step>

  <Step>
    #### Interrogez votre instance Redshift depuis ClickHouse [#query-your-redshift-instance-from-clickhouse]

    Une fois ClickHouse JDBC Bridge déployé et en fonctionnement, vous pouvez commencer à interroger votre instance Redshift depuis ClickHouse

    ```sql
    SELECT *
    FROM jdbc('redshift', 'select username, firstname, lastname from users limit 5')
    ```

    ```response
    Query id: 1b7de211-c0f6-4117-86a2-276484f9f4c0

    ┌─username─┬─firstname─┬─lastname─┐
    │ PGL08LJI │ Vladimir  │ Humphrey │
    │ XDZ38RDD │ Barry     │ Roy      │
    │ AEB55QTM │ Reagan    │ Hodge    │
    │ OWY35QYB │ Tamekah   │ Juarez   │
    │ MSD36KVR │ Mufutau   │ Watkins  │
    └──────────┴───────────┴──────────┘

    5 rows in set. Elapsed: 0.438 sec.
    ```

    ```sql
    SELECT *
    FROM jdbc('redshift', 'select count(*) from sales')
    ```

    ```response
    Query id: 2d0f957c-8f4e-43b2-a66a-cc48cc96237b

    ┌──count─┐
    │ 172456 │
    └────────┘

    1 rows in set. Elapsed: 0.304 sec.
    ```
  </Step>

  <Step>
    #### Importez des données de Redshift vers ClickHouse [#import-data-from-redshift-to-clickhouse]

    Ci-dessous, nous montrons comment importer des données à l&#39;aide d&#39;une instruction `INSERT INTO ... SELECT`

    ```sql
    CREATE TABLE users_imported
    (
       `username` String,
       `firstname` String,
       `lastname` String
    )
    ENGINE = MergeTree
    ORDER BY firstname
    ```

    ```response
    Query id: c7c4c44b-cdb2-49cf-b319-4e569976ab05

    Ok.

    0 rows in set. Elapsed: 0.233 sec.
    ```

    ```sql
    INSERT INTO users_imported (*) SELECT *
    FROM jdbc('redshift', 'select username, firstname, lastname from users')
    ```

    ```response
    Query id: 9d3a688d-b45a-40f4-a7c7-97d93d7149f1

    Ok.

    0 rows in set. Elapsed: 4.498 sec. Processed 49.99 thousand rows, 2.49 MB (11.11 thousand rows/s., 554.27 KB/s.)
    ```
  </Step>
</Steps>

<div id="pivot-data-from-redshift-to-clickhouse-using-s3-pivot-data-from-redshift-to-clickhouse-using-s3">
  ## Faire transiter les données de Redshift vers ClickHouse via S3 avec PIVOT [#pivot-data-from-redshift-to-clickhouse-using-s3]
</div>

Dans ce scénario, nous exportons les données vers S3 dans un format PIVOT intermédiaire, puis, dans un second temps, nous chargeons les données depuis S3 dans ClickHouse.

<img src="/images/integrations/data-ingestion/redshift/pivot.webp" alt="PIVOT de Redshift via S3" />

<div id="pros-pros-2">
  ### Avantages [#pros-2]
</div>

* Redshift et ClickHouse offrent tous deux de puissantes fonctionnalités d’intégration avec S3.
* S’appuie sur des fonctionnalités existantes telles que la commande Redshift `UNLOAD` et la table function / le table engine S3 de ClickHouse.
* Passe facilement à l’échelle grâce aux lectures parallèles et au haut débit vers et depuis S3 dans ClickHouse.
* Permet d’utiliser des formats sophistiqués et compressés comme Apache Parquet.

<div id="cons-cons-2">
  ### Inconvénients [#cons-2]
</div>

* Deux étapes dans le processus (déchargement de Redshift, puis chargement dans ClickHouse).

<div id="tutorial-tutorial-1">
  ### Tutoriel [#tutorial-1]
</div>

<Steps>
  <Step>
    #### Exporter les données vers un bucket S3 à l’aide de UNLOAD [#export-data-into-an-s3-bucket-using-unload]

    À l’aide de la fonctionnalité [UNLOAD](https://docs.aws.amazon.com/redshift/latest/dg/r_UNLOAD.html) de Redshift, exportez les données vers un bucket S3 privé existant :

    <img src="/images/integrations/data-ingestion/redshift/s3-1.webp" alt="UNLOAD de Redshift vers S3" />

    Cela génère des fichiers part contenant les données brutes dans S3.

    <img src="/images/integrations/data-ingestion/redshift/s3-2.webp" alt="Données dans S3" />
  </Step>

  <Step>
    #### Créer la table dans ClickHouse [#create-the-table-in-clickhouse]

    Créez la table dans ClickHouse :

    ```sql
    CREATE TABLE users
    (
      username String,
      firstname String,
      lastname String
    )
    ENGINE = MergeTree
    ORDER BY username
    ```

    Sinon, ClickHouse peut essayer d’inférer la structure de la table à l’aide de `CREATE TABLE ... EMPTY AS SELECT` :

    ```sql
    CREATE TABLE users
    ENGINE = MergeTree ORDER BY username
    EMPTY AS
    SELECT * FROM s3('https://your-bucket.s3.amazonaws.com/unload/users/*', '<aws_access_key>', '<aws_secret_access_key>', 'CSV')
    ```

    Cela fonctionne particulièrement bien lorsque les données sont dans un format qui contient des informations sur les types de données, comme Parquet.
  </Step>

  <Step>
    #### Charger les fichiers S3 dans ClickHouse [#load-s3-files-into-clickhouse]

    Chargez les fichiers S3 dans ClickHouse à l’aide d’une instruction `INSERT INTO ... SELECT` :

    ```sql
    INSERT INTO users SELECT *
    FROM s3('https://your-bucket.s3.amazonaws.com/unload/users/*', '<aws_access_key>', '<aws_secret_access_key>', 'CSV')
    ```

    ```response
    Query id: 2e7e219a-6124-461c-8d75-e4f5002c8557

    Ok.

    0 rows in set. Elapsed: 0.545 sec. Processed 49.99 thousand rows, 2.34 MB (91.72 thousand rows/s., 4.30 MB/s.)
    ```

    <Note>
      Cet exemple utilise CSV comme format pivot. Cependant, pour les workloads de production, nous recommandons Apache Parquet comme meilleure option pour les migrations volumineuses, car il intègre la compression et peut réduire les coûts de stockage tout en diminuant les temps de transfert. (Par défaut, chaque row group est compressé avec SNAPPY.) ClickHouse exploite également l’organisation en colonnes de Parquet pour accélérer l’ingestion des données.
    </Note>
  </Step>
</Steps>
