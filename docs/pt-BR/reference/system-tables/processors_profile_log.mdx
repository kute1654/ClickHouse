<div id="description">
  ## Descrição
</div>

Esta tabela contém informações de profiling no nível dos processadores (que você pode encontrar em [`EXPLAIN PIPELINE`](/pt-BR/reference/statements/explain#explain-pipeline)).

<div id="columns">
  ## Colunas
</div>

{/*AUTOGENERATED_START*/}

* `hostname` ([LowCardinality(String)](/pt-BR/reference/data-types/lowcardinality)) — Hostname do servidor que executa a consulta.
* `event_date` ([Date](/pt-BR/reference/data-types/date)) — A data em que o evento aconteceu.
* `event_time` ([DateTime](/pt-BR/reference/data-types/datetime)) — A data e a hora em que o evento aconteceu.
* `event_time_microseconds` ([DateTime64(6)](/pt-BR/reference/data-types/datetime64)) — A data e a hora, com precisão de microssegundos, em que o evento aconteceu.
* `id` ([UInt64](/pt-BR/reference/data-types/int-uint)) — ID do processador.
* `parent_ids` ([Array(UInt64)](/pt-BR/reference/data-types/array)) — IDs dos processadores pais.
* `plan_step` ([UInt64](/pt-BR/reference/data-types/int-uint)) — ID do passo do plano de consulta que criou este processador. O valor é zero se o processador não foi adicionado a partir de nenhum passo.
* `plan_step_name` ([String](/pt-BR/reference/data-types/string)) — Nome do passo do plano de consulta que criou este processador. O valor fica vazio se o processador não foi adicionado a partir de nenhum passo.
* `plan_step_description` ([String](/pt-BR/reference/data-types/string)) — Descrição do passo do plano de consulta que criou este processador. O valor fica vazio se o processador não foi adicionado a partir de nenhum passo.
* `plan_group` ([UInt64](/pt-BR/reference/data-types/int-uint)) — Grupo do processador, se ele foi criado por um passo do plano de consulta. Um grupo é um particionamento lógico de processadores adicionados a partir do mesmo passo do plano de consulta. O grupo é usado apenas para melhorar a apresentação do resultado de EXPLAIN PIPELINE.
* `initial_query_id` ([String](/pt-BR/reference/data-types/string)) — ID da consulta inicial (para execução distribuída de consultas).
* `query_id` ([String](/pt-BR/reference/data-types/string)) — ID da consulta.
* `name` ([LowCardinality(String)](/pt-BR/reference/data-types/lowcardinality)) — Nome do processador.
* `elapsed_us` ([UInt64](/pt-BR/reference/data-types/int-uint)) — Número de microssegundos durante os quais este processador foi executado.
* `input_wait_elapsed_us` ([UInt64](/pt-BR/reference/data-types/int-uint)) — Número de microssegundos durante os quais este processador ficou esperando por dados (de outro processador).
* `output_wait_elapsed_us` ([UInt64](/pt-BR/reference/data-types/int-uint)) — Número de microssegundos durante os quais este processador ficou esperando porque a porta de saída estava cheia.
* `input_rows` ([UInt64](/pt-BR/reference/data-types/int-uint)) — O número de linhas consumidas pelo processador.
* `input_bytes` ([UInt64](/pt-BR/reference/data-types/int-uint)) — O número de bytes consumidos pelo processador.
* `output_rows` ([UInt64](/pt-BR/reference/data-types/int-uint)) — O número de linhas geradas pelo processador.
* `output_bytes` ([UInt64](/pt-BR/reference/data-types/int-uint)) — O número de bytes gerados pelo processador.
* `processor_uniq_id` ([String](/pt-BR/reference/data-types/string)) — O ID único do processador no pipeline.
* `step_uniq_id` ([String](/pt-BR/reference/data-types/string)) — O ID único do passo no plano.

{/*AUTOGENERATED_END*/}

<div id="example">
  ## Exemplo
</div>

```sql title="Query"
EXPLAIN PIPELINE
SELECT sleep(1)
┌─explain─────────────────────────┐
│ (Expression)                    │
│ ExpressionTransform             │
│   (SettingQuotaAndLimits)       │
│     (ReadFromStorage)           │
│     SourceFromSingleChunk 0 → 1 │
└─────────────────────────────────┘

SELECT sleep(1)
SETTINGS log_processors_profiles = 1
Query id: feb5ed16-1c24-4227-aa54-78c02b3b27d4
┌─sleep(1)─┐
│        0 │
└──────────┘
1 rows in set. Elapsed: 1.018 sec.

SELECT
    name,
    elapsed_us,
    input_wait_elapsed_us,
    output_wait_elapsed_us
FROM system.processors_profile_log
WHERE query_id = 'feb5ed16-1c24-4227-aa54-78c02b3b27d4'
ORDER BY name ASC
```

```text title="Response"
┌─name────────────────────┬─elapsed_us─┬─input_wait_elapsed_us─┬─output_wait_elapsed_us─┐
│ ExpressionTransform     │    1000497 │                  2823 │                    197 │
│ LazyOutputFormat        │         36 │               1002188 │                      0 │
│ LimitsCheckingTransform │         10 │               1002994 │                    106 │
│ NullSource              │          5 │               1002074 │                      0 │
│ NullSource              │          1 │               1002084 │                      0 │
│ SourceFromSingleChunk   │         45 │                  4736 │                1000819 │
└─────────────────────────┴────────────┴───────────────────────┴────────────────────────┘
```

Aqui você pode ver:

* `ExpressionTransform` estava executando a função `sleep(1)`, então seu `work` levará 1e6 e, portanto, `elapsed_us` &gt; 1e6.
* `SourceFromSingleChunk` precisa esperar, porque `ExpressionTransform` não aceita nenhum dado durante a execução de `sleep(1)`, então ficará no estado `PortFull` por 1e6 us e, portanto, `output_wait_elapsed_us` &gt; 1e6.
* `LimitsCheckingTransform`/`NullSource`/`LazyOutputFormat` precisam esperar até que `ExpressionTransform` execute `sleep(1)` para processar o resultado, então `input_wait_elapsed_us` &gt; 1e6.
