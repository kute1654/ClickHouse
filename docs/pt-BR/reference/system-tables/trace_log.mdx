<div id="description">
  ## Descrição
</div>

Contém stack traces coletados pelo [profiler de consultas por amostragem](/pt-BR/concepts/features/performance/troubleshoot/sampling-query-profiler).

O ClickHouse cria esta tabela quando a seção de configuração do servidor [trace&#95;log](/pt-BR/reference/settings/server-settings/settings#trace_log) é definida. Veja também as configurações: [query&#95;profiler&#95;real&#95;time&#95;period&#95;ns](/pt-BR/reference/settings/session-settings#query_profiler_real_time_period_ns), [query&#95;profiler&#95;cpu&#95;time&#95;period&#95;ns](/pt-BR/reference/settings/session-settings#query_profiler_cpu_time_period_ns), [memory&#95;profiler&#95;step](/pt-BR/reference/settings/session-settings#memory_profiler_step),
[memory&#95;profiler&#95;sample&#95;probability](/pt-BR/reference/settings/session-settings#memory_profiler_sample_probability), [trace&#95;profile&#95;events](/pt-BR/reference/settings/session-settings#trace_profile_events).

Para analisar logs, use as funções de introspecção `addressToLine`, `addressToLineWithInlines`, `addressToSymbol` e `demangle`.

<div id="columns">
  ## Colunas
</div>

{/*AUTOGENERATED_START*/}

* `hostname` ([LowCardinality(String)](/pt-BR/reference/data-types/lowcardinality)) — Hostname do servidor que executa a consulta.
* `event_date` ([Date](/pt-BR/reference/data-types/date)) — Data do momento em que a amostragem ocorreu.
* `event_time` ([DateTime](/pt-BR/reference/data-types/datetime)) — Timestamp do momento da amostragem.
* `event_time_microseconds` ([DateTime64(6)](/pt-BR/reference/data-types/datetime64)) — Timestamp do momento da amostragem com precisão de microssegundos.
* `timestamp_ns` ([UInt64](/pt-BR/reference/data-types/int-uint)) — Timestamp do momento de amostragem em nanossegundos.
* `revision` ([UInt32](/pt-BR/reference/data-types/int-uint)) — revisão de compilação do servidor ClickHouse. Ao se conectar ao servidor com o `clickhouse-client`, você verá uma string semelhante a `Connected to ClickHouse server version 19.18.1.`. Este campo contém a `revision`, mas não a `version` de um servidor.
* `trace_type` ([Enum8(&#39;Real&#39; = 0, &#39;CPU&#39; = 1, &#39;Memory&#39; = 2, &#39;MemorySample&#39; = 3, &#39;MemoryPeak&#39; = 4, &#39;ProfileEvent&#39; = 5, &#39;JemallocSample&#39; = 6, &#39;MemoryAllocatedWithoutCheck&#39; = 7, &#39;Instrumentation&#39; = 8)](/pt-BR/reference/data-types/enum)) — Tipo de trace: `Real` representa a coleta de stack traces por tempo de relógio. `CPU` representa a coleta de stack traces por tempo de CPU. `Memory` representa a coleta de alocações e desalocações quando a alocação de memória excede a watermark subsequente. `MemorySample` representa a coleta aleatória de alocações e desalocações. `MemoryPeak` representa a coleta de atualizações do pico de uso de memória. `ProfileEvent` representa a coleta de incrementos de eventos de profile. `JemallocSample` representa a coleta de amostras do jemalloc. `MemoryAllocatedWithoutCheck` representa a coleta de alocações significativas (&gt;16MiB), feita sem considerar quaisquer limites de memória (apenas para desenvolvedores do ClickHouse).`Instrumentation` representa traces coletados pela instrumentação realizada por meio do XRay.
* `cpu_id` ([UInt64](/pt-BR/reference/data-types/int-uint)) — identificador da CPU.
* `thread_id` ([UInt64](/pt-BR/reference/data-types/int-uint)) — Identificador da thread.
* `thread_name` ([LowCardinality(String)](/pt-BR/reference/data-types/lowcardinality)) — Nome da thread.
* `query_id` ([String](/pt-BR/reference/data-types/string)) — Identificador da consulta que pode ser usado para obter detalhes sobre uma consulta que estava em execução a partir da tabela de sistema query&#95;log.
* `trace` ([Array(UInt64)](/pt-BR/reference/data-types/array)) — Stack trace no momento da amostragem. Cada elemento é um endereço de memória virtual no processo do servidor ClickHouse.
* `size` ([Int64](/pt-BR/reference/data-types/int-uint)) — Para os tipos de trace Memory, MemorySample, MemoryAllocatedWithoutCheck ou MemoryPeak, é a quantidade de memória alocada; para os demais tipos de trace, é 0.
* `ptr` ([UInt64](/pt-BR/reference/data-types/int-uint)) — O endereço do fragmento alocado.
* `memory_context` ([Enum8(&#39;Unknown&#39; = -1, &#39;Global&#39; = 0, &#39;User&#39; = 1, &#39;Process&#39; = 2, &#39;Thread&#39; = 3, &#39;Max&#39; = 4)](/pt-BR/reference/data-types/enum)) — contexto do memory tracker (apenas para Memory/MemoryPeak): o contexto `Unknown` não está definido para este `trace_type`. `Global` representa o contexto do servidor. `User` representa o contexto do usuário/merge. `Process` representa o contexto do processo (ou seja, da consulta). `Thread` representa o contexto da thread (thread de um processo específico). `Max` é um valor especial que significa que o memory tracker não está bloqueado (para a coluna blocked&#95;context).
* `memory_blocked_context` ([Enum8(&#39;Unknown&#39; = -1, &#39;Global&#39; = 0, &#39;User&#39; = 1, &#39;Process&#39; = 2, &#39;Thread&#39; = 3, &#39;Max&#39; = 4)](/pt-BR/reference/data-types/enum)) — Contexto no qual o memory tracker está bloqueado (somente para desenvolvedores do ClickHouse): o contexto `Unknown` não está definido para este trace&#95;type. `Global` representa o contexto do servidor. `User` representa o contexto de usuário/merge. `Process` representa o contexto do processo (ou seja, da consulta). `Thread` representa o contexto da thread (thread de um processo específico). `Max` é um valor especial que significa que o memory tracker não está bloqueado (para a coluna blocked&#95;context).
* `event` ([LowCardinality(String)](/pt-BR/reference/data-types/lowcardinality)) — Para o tipo de trace `ProfileEvent`, é o nome do evento de perfil atualizado; para os outros tipos de trace, é uma string vazia.
* `increment` ([Int64](/pt-BR/reference/data-types/int-uint)) — Para traces do tipo ProfileEvent, é o valor do incremento do evento de profile; para outros tipos de trace, é 0.
* `symbols` ([Array(LowCardinality(String))](/pt-BR/reference/data-types/array)) — Se a simbolização estiver habilitada, contém nomes de símbolos desmanglados, correspondentes ao `trace`. A simbolização pode ser habilitada ou desabilitada pela configuração `symbolize`, em `trace_log`, no arquivo de configuração do servidor.
* `lines` ([Array(LowCardinality(String))](/pt-BR/reference/data-types/array)) — Se a simbolização estiver habilitada, contém strings com nomes de arquivos e números de linha correspondentes ao `trace`.
* `function_id` ([Nullable(Int32)](/pt-BR/reference/data-types/nullable)) — Para o trace do tipo Instrumentation, ID atribuído à função na seção xray&#95;instr&#95;map do binário ELF.
* `function_name` ([Nullable(String)](/pt-BR/reference/data-types/nullable)) — Para o trace do tipo Instrumentation, nome da função instrumentada.
* `handler` ([Nullable(String)](/pt-BR/reference/data-types/nullable)) — Para o trace do tipo Instrumentation, handler da função instrumentada.
* `entry_type` ([Nullable(Enum8(&#39;Entry&#39; = 0, &#39;Exit&#39; = 1))](/pt-BR/reference/data-types/nullable)) — Para o tipo de trace Instrumentation, indica se é a entrada ou a saída da função instrumentada.
* `duration_nanoseconds` ([Nullable(UInt64)](/pt-BR/reference/data-types/nullable)) — Para o tipo de trace Instrumentation, tempo de execução da função, em nanossegundos.

**Aliases:**

* `build_id` — Alias de `'E2EEB60ED9503BFFA825B628D480CCDC40B2D416'`.

{/*AUTOGENERATED_END*/}

A simbolização pode ser ativada ou desativada em `symbolize`, em `trace_log`, no arquivo de configuração do servidor.

<div id="chrome-event-trace-format">
  ## Convertendo para o Chrome Event Trace Format
</div>

Os dados de profiling podem ser convertidos para o Chrome Event Trace Format com a consulta a seguir. Salve a consulta em um arquivo `chrome_trace.sql`:

```sql
WITH traces AS (
    SELECT * FROM system.trace_log
    WHERE event_date >= today() AND trace_type = 'Instrumentation' AND handler = 'profile'
    ORDER BY event_time, entry_type
)
SELECT
    format(
        '{{"traceEvents": [{}\n]}}',
        arrayStringConcat(
            groupArray(
                format(
                    '\n{{"name": "{}", "cat": "clickhouse", "ph": "{}", "ts": {}, "pid": 1, "tid": {}, "args": {{"query_id": "{}", "cpu_id": {}, "stack": [{}]}}}},',
                    function_name,
                    if(entry_type = 0, 'B', 'E'),
                    timestamp_ns/1000,
                    toString(thread_id),
                    query_id,
                    cpu_id,
                    arrayStringConcat(arrayMap((x, y) -> concat('"', x, ': ', y, '", '), lines, symbols))
                )
            )
        )
    )
FROM traces;
```

E executá-lo com o ClickHouse Client para exportá-lo para um arquivo `trace.json`, que podemos importar no [Perfetto](https://ui.perfetto.dev/) ou no [speedscope](https://www.speedscope.app/).

```bash
echo $(clickhouse client --query "$(cat chrome_trace.sql)") > trace.json
```

Podemos omitir a pilha de chamadas se quisermos um trace mais compacto, mas menos informativo.
