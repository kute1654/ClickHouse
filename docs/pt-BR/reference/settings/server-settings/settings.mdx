Esta seção contém descrições das configurações do servidor. São configurações que
não podem ser alteradas no nível de sessão ou de consulta.

Para mais informações sobre arquivos de configuração no ClickHouse, consulte [&quot;&quot;Arquivos de configuração&quot;&quot;](/pt-BR/concepts/features/configuration/server-config/configuration-files).

Outras configurações são descritas na seção &quot;&quot;[Configurações](/pt-BR/concepts/features/configuration/settings/overview)&quot;&quot;.
Antes de estudar as configurações, recomendamos ler a seção [Arquivos de configuração](/pt-BR/concepts/features/configuration/server-config/configuration-files)
e observar o uso de substituições (os atributos `incl` e `optional`).

<div id="access_control_improvements">
  ## access_control_improvements
</div>

Configurações para melhorias opcionais no sistema de controle de acesso.

| Configuração                                    | Descrição                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                 | Padrão  |
| ----------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ------- |
| `on_cluster_queries_require_cluster_grant`      | Define se consultas `ON CLUSTER` exigem o privilégio `CLUSTER`.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                           | `true`  |
| `role_cache_expiration_time_seconds`            | Define por quantos segundos, desde o último acesso, uma função é armazenada no cache de funções.                                                                                                                                                                                                                                                                                                                                                                                                                                                                          | `600`   |
| `select_from_information_schema_requires_grant` | Define se `SELECT * FROM information_schema.<table>` exige algum privilégio ou pode ser executado por qualquer usuário. Se definido como true, essa consulta exige `GRANT SELECT ON information_schema.<table>`, assim como no caso de tabelas comuns.                                                                                                                                                                                                                                                                                                                    | `true`  |
| `select_from_system_db_requires_grant`          | Define se `SELECT * FROM system.<table>` exige algum privilégio ou pode ser executado por qualquer usuário. Se definido como true, essa consulta exige `GRANT SELECT ON system.<table>`, assim como no caso de tabelas que não são do sistema. Exceções: algumas tabelas do sistema (`tables`, `columns`, `databases` e algumas tabelas constantes, como `one` e `contributors`) continuam acessíveis a todos; e, se um privilégio `SHOW` (por exemplo, `SHOW USERS`) tiver sido concedido, a tabela do sistema correspondente (isto é, `system.users`) ficará acessível. | `true`  |
| `settings_constraints_replace_previous`         | Define se uma restrição em um perfil de configurações para determinada configuração cancelará as ações da restrição anterior (definida em outros perfis) para essa configuração, incluindo campos que não foram definidos pela nova restrição. Também habilita o tipo de restrição `changeable_in_readonly`.                                                                                                                                                                                                                                                              | `true`  |
| `table_engines_require_grant`                   | Define se criar uma tabela com um mecanismo de tabela específico exige um privilégio.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                     | `false` |
| `throw_on_unmatched_row_policies`               | Define se a leitura de uma tabela deve lançar uma exceção caso a tabela tenha políticas de linha, mas nenhuma delas se aplique ao usuário atual                                                                                                                                                                                                                                                                                                                                                                                                                           | `false` |
| `users_without_row_policies_can_read_rows`      | Define se usuários sem políticas de linha permissivas ainda podem ler linhas usando uma consulta `SELECT`. Por exemplo, se houver dois usuários, A e B, e uma política de linha estiver definida apenas para A, então, se esta configuração for true, o usuário B verá todas as linhas. Se esta configuração for false, o usuário B não verá nenhuma linha.                                                                                                                                                                                                               | `true`  |

Exemplo:

```xml
<access_control_improvements>
    <throw_on_unmatched_row_policies>true</throw_on_unmatched_row_policies>
    <users_without_row_policies_can_read_rows>true</users_without_row_policies_can_read_rows>
    <on_cluster_queries_require_cluster_grant>true</on_cluster_queries_require_cluster_grant>
    <select_from_system_db_requires_grant>true</select_from_system_db_requires_grant>
    <select_from_information_schema_requires_grant>true</select_from_information_schema_requires_grant>
    <settings_constraints_replace_previous>true</settings_constraints_replace_previous>
    <table_engines_require_grant>false</table_engines_require_grant>
    <role_cache_expiration_time_seconds>600</role_cache_expiration_time_seconds>
</access_control_improvements>
```

<div id="allow_feature_tier">
  ## allow_feature_tier
</div>

<SettingsInfoBlock type="UInt32" default_value="0" changeable_without_restart="Yes" />

Controla se o usuário pode alterar configurações relacionadas aos diferentes níveis de funcionalidade.

* `0` - Alterações em qualquer configuração são permitidas (experimental, beta, produção).
* `1` - Somente alterações nas configurações de funcionalidades beta e de produção são permitidas. Alterações nas configurações experimentais são rejeitadas.
* `2` - Somente alterações nas configurações de funcionalidades de produção são permitidas. Alterações nas configurações experimentais ou beta são rejeitadas.

Isso é equivalente a definir uma restrição readonly em todas as funcionalidades `EXPERIMENTAL` / `BETA`.

<Note>
  Um valor de `0` significa que todas as configurações podem ser alteradas.
</Note>

<div id="async_load_databases">
  ## async_load_databases
</div>

<SettingsInfoBlock type="Bool" default_value="1" changeable_without_restart="Não" />

Carregamento assíncrono de bancos de dados e tabelas.

* Se `true`, todos os bancos de dados não pertencentes ao sistema com engine `Ordinary`, `Atomic` e `Replicated` serão carregados de forma assíncrona após a inicialização do servidor ClickHouse. Consulte a tabela `system.asynchronous_loader` e as configurações do servidor `tables_loader_background_pool_size` e `tables_loader_foreground_pool_size`. Qualquer consulta que tente acessar uma tabela que ainda não tenha sido carregada aguardará a inicialização dessa tabela específica. Se a tarefa de carregamento falhar, a consulta lançará o erro novamente (em vez de desligar o servidor inteiro no caso de `async_load_databases = false`). A tabela pela qual pelo menos uma consulta estiver aguardando será carregada com prioridade mais alta. Consultas DDL em um banco de dados aguardarão a inicialização desse banco de dados específico. Considere também definir um limite `max_waiting_queries` para o número total de consultas em espera.
* Se `false`, todos os bancos de dados são carregados quando o servidor é iniciado.

**Exemplo**

```xml
<async_load_databases>true</async_load_databases>
```

<div id="async_load_system_database">
  ## async_load_system_database
</div>

<SettingsInfoBlock type="Bool" default_value="0" changeable_without_restart="Não" />

Carregamento assíncrono das tabelas do sistema. Útil quando há uma grande quantidade de tabelas de log e partes no banco de dados `system`. Independe da configuração `async_load_databases`.

* Se definido como `true`, todos os bancos de dados do sistema com motores `Ordinary`, `Atomic` e `Replicated` serão carregados de forma assíncrona após a inicialização do servidor ClickHouse. Consulte a tabela `system.asynchronous_loader` e as configurações de servidor `tables_loader_background_pool_size` e `tables_loader_foreground_pool_size`. Qualquer consulta que tente acessar uma tabela do sistema que ainda não tenha sido carregada aguardará a inicialização dessa tabela específica. A tabela pela qual pelo menos uma consulta estiver aguardando será carregada com prioridade mais alta. Considere também definir a configuração `max_waiting_queries` para limitar o número total de consultas em espera.
* Se definido como `false`, o banco de dados do sistema é carregado antes da inicialização do servidor.

**Exemplo**

```xml
<async_load_system_database>true</async_load_system_database>
```

<div id="asynchronous_metric_log">
  ## asynchronous_metric_log
</div>

Habilitado por padrão nas implantações do ClickHouse Cloud.

Se essa configuração não estiver habilitada por padrão no seu ambiente, dependendo de como o ClickHouse foi instalado, você pode seguir a instrução abaixo para ativá-la ou desativá-la.

**Habilitando**

Para ativar manualmente a coleta do histórico de logs de métricas assíncronas [`system.asynchronous_metric_log`](/pt-BR/reference/system-tables/asynchronous_metric_log), crie `/etc/clickhouse-server/config.d/asynchronous_metric_log.xml` com o seguinte conteúdo:

```xml
     <asynchronous_metric_log>

        <table>asynchronous_metric_log</table>
        <flush_interval_milliseconds>7500</flush_interval_milliseconds>
        <collect_interval_milliseconds>1000</collect_interval_milliseconds>
        <max_size_rows>1048576</max_size_rows>
        <reserved_size_rows>8192</reserved_size_rows>
        <buffer_size_rows_flush_threshold>524288</buffer_size_rows_flush_threshold>
        <flush_on_crash>false</flush_on_crash>
    </asynchronous_metric_log>

```

Para desativar a configuração `asynchronous_metric_log`, crie o arquivo `/etc/clickhouse-server/config.d/disable_asynchronous_metric_log.xml` com o seguinte conteúdo:

```xml
<clickhouse><asynchronous_metric_log remove="1" /></clickhouse>
```

<div id="background_merges_mutations_concurrency_ratio">
  ## background_merges_mutations_concurrency_ratio
</div>

<SettingsInfoBlock type="Float" default_value="2" changeable_without_restart="No" />

Define uma proporção entre o número de threads e o número de merges em segundo plano e mutações que podem ser executados simultaneamente.

Por exemplo, se a proporção for igual a 2 e [`background_pool_size`](/pt-BR/reference/settings/server-settings/settings#background_pool_size) estiver definida como 16, o ClickHouse poderá executar 32 merges em segundo plano simultaneamente. Isso é possível porque as operações em segundo plano podem ser suspensas e adiadas. Isso é necessário para dar maior prioridade de execução a merges menores.

<Note>
  Você só pode aumentar essa proporção em tempo de execução. Para reduzi-la, é necessário reiniciar o servidor.

  Assim como a configuração [`background_pool_size`](/pt-BR/reference/settings/server-settings/settings#background_pool_size), [`background_merges_mutations_concurrency_ratio`](/pt-BR/reference/settings/server-settings/settings#background_merges_mutations_concurrency_ratio) pode ser aplicada a partir do perfil `default` para compatibilidade com versões anteriores.
</Note>

<div id="background_merges_mutations_scheduling_policy">
  ## background_merges_mutations_scheduling_policy
</div>

<SettingsInfoBlock type="String" default_value="round_robin" changeable_without_restart="No" />

A política de agendamento de merges e mutações em segundo plano. Os valores possíveis são: `round_robin` e `shortest_task_first`.

Algoritmo usado para selecionar o próximo merge ou mutação a ser executado pelo pool de threads em segundo plano. A política pode ser alterada em tempo de execução sem reiniciar o servidor.
Pode ser aplicada a partir do perfil `default` para manter a compatibilidade retroativa.

Valores possíveis:

* `round_robin` — Cada merge e mutação concorrente é executado em ordem round-robin para garantir uma operação sem inanição. Merges menores são concluídos mais rapidamente que os maiores simplesmente porque têm menos blocos para mesclar.
* `shortest_task_first` — Sempre executa o menor merge ou mutação. Merges e mutações recebem prioridades com base no tamanho resultante. Merges menores têm preferência estrita sobre os maiores. Essa política garante o merge mais rápido possível de partes pequenas, mas pode levar à inanição indefinida de merges grandes em partições muito sobrecarregadas por `INSERT`s.

<div id="background_pool_size">
  ## background_pool_size
</div>

<SettingsInfoBlock type="UInt64" default_value="16" changeable_without_restart="No" />

Define o número de threads que executam merges e mutações em segundo plano para tabelas com motores MergeTree.

<Note>
  * Essa configuração também pode ser aplicada na inicialização do servidor a partir da configuração do perfil `default`, para manter compatibilidade retroativa durante a inicialização do servidor ClickHouse.
  * Você só pode aumentar o número de threads em tempo de execução.
  * Para reduzir o número de threads, é necessário reiniciar o servidor.
  * Ao ajustar essa configuração, você gerencia a carga de CPU e disco.
</Note>

<Danger>
  Um pool menor usa menos recursos de CPU e disco, mas os processos em segundo plano avançam mais lentamente, o que pode acabar afetando o desempenho das consultas.
</Danger>

Antes de alterá-la, veja também as configurações relacionadas do MergeTree, como:

* [`number_of_free_entries_in_pool_to_lower_max_size_of_merge`](/pt-BR/reference/settings/merge-tree-settings#number_of_free_entries_in_pool_to_lower_max_size_of_merge).
* [`number_of_free_entries_in_pool_to_execute_mutation`](/pt-BR/reference/settings/merge-tree-settings#number_of_free_entries_in_pool_to_execute_mutation).
* [`number_of_free_entries_in_pool_to_execute_optimize_entire_partition`](/pt-BR/reference/settings/merge-tree-settings#number_of_free_entries_in_pool_to_execute_optimize_entire_partition)

**Exemplo**

```xml

```

**Campos de `<case>`**:

* `min_part_size` – O tamanho mínimo de uma parte de dados.

<div id="concurrent_threads_scheduler">
  ## concurrent_threads_scheduler
</div>

<SettingsInfoBlock type="String" default_value="max_min_fair" changeable_without_restart="Sim" />

A política de como realizar o agendamento dos slots de CPU especificados por `concurrent_threads_soft_limit_num` e `concurrent_threads_soft_limit_ratio_to_cores`. É o algoritmo usado para controlar como um número limitado de slots de CPU é distribuído entre consultas concorrentes. O agendador pode ser alterado em tempo de execução sem reiniciar o servidor.

Possíveis valores:

* `round_robin` — Toda consulta com a configuração `use_concurrency_control` = 1 aloca até `max_threads` slots de CPU. Um slot por thread. Em caso de contenção, os slots de CPU são concedidos às consultas em round-robin. Observe que o primeiro slot é concedido incondicionalmente, o que pode levar a falta de equidade e ao aumento da latência de consultas com `max_threads` alto quando há um grande número de consultas com `max_threads` = 1.
* `fair_round_robin` — Toda consulta com a configuração `use_concurrency_control` = 1 aloca até `max_threads - 1` slots de CPU. É uma variação de `round_robin` que não exige um slot de CPU para a primeira thread de cada consulta. Dessa forma, consultas com `max_threads` = 1 não exigem nenhum slot e não podem alocar injustamente todos os slots. Não há slots concedidos incondicionalmente.
* `max_min_fair` — Toda consulta com a configuração `use_concurrency_control` = 1 aloca até `max_threads - 1` slots de CPU. Semelhante a `fair_round_robin`, mas os slots liberados são sempre concedidos à consulta com o menor número de slots alocados no momento. Isso proporciona maior equidade em cenários de alta sobresubscrição, quando muitas consultas competem por slots de CPU limitados. Consultas de curta duração não são penalizadas por consultas de longa duração que acumularam mais slots ao longo do tempo.

<div id="concurrent_threads_soft_limit_num">
  ## concurrent_threads_soft_limit_num

<SettingsInfoBlock type="UInt64" default_value="0" changeable_without_restart="Sim" />

O número máximo de threads de processamento de consultas, excluindo as threads usadas para recuperar dados de servidores remotos, permitido para executar todas as consultas. <div id="cpu_slot_preemption">
  ## cpu_slot_preemption
</div>

<SettingsInfoBlock type="Bool" default_value="1" changeable_without_restart="Sim" />

Define como é feito o agendamento de workloads para recursos de CPU (MASTER THREAD e WORKER THREAD).

* Se `true` (padrão), a contabilização é feita com base no tempo de CPU efetivamente consumido. Uma quantidade justa de tempo de CPU é alocada aos workloads concorrentes. Os slots são alocados por um período limitado e solicitados novamente após expirarem. A solicitação de slots pode bloquear a execução da thread em caso de sobrecarga de recursos de CPU, ou seja, pode haver preempção. Isso garante justiça na alocação de tempo de CPU.
* Se `false`, a contabilização é baseada no número de slots de CPU alocados. Uma quantidade justa de slots de CPU é alocada aos workloads concorrentes. Um slot é alocado quando uma thread é iniciada, mantido continuamente e liberado quando a thread termina a execução. O número de threads alocadas para a execução da consulta só pode aumentar de 1 até `max_threads` e nunca diminuir. Isso é mais favorável a consultas de longa duração e pode levar à inanição de CPU para consultas curtas.

**Exemplo**

```xml
<cpu_slot_preemption>true</cpu_slot_preemption>
```

**Veja também**

* [Agendamento de cargas de trabalho](/pt-BR/concepts/features/configuration/server-config/workload-scheduling)

<div id="cpu_slot_preemption_timeout_ms">
  ## cpu_slot_preemption_timeout_ms
</div>

<SettingsInfoBlock type="UInt64" default_value="1000" changeable_without_restart="Sim" />

Define por quantos milissegundos uma thread de trabalho pode esperar durante a preempção, isto é, enquanto aguarda a concessão de outro CPU slot. Após esse timeout, se a thread não conseguir adquirir um novo CPU slot, ela será encerrada, e a consulta será reduzida dinamicamente para um número menor de threads em execução concorrente. Observe que a thread mestre nunca é reduzida, mas pode ser preemptada indefinidamente. Isso só faz sentido quando `cpu_slot_preemption` está habilitado e o recurso de CPU está definido para WORKER THREAD.

**Exemplo**

```xml

```

<div id="disable_tunneling_for_https_requests_over_http_proxy">

Por padrão, todas as solicitações passarão pelo proxy. Para desativá-lo para hosts específicos, a variável `no_proxy` deve ser definida.

<div id="encryption">
  ## criptografia
</div>

Configura um comando para obter uma chave que será usada pelos [codecs de criptografia](/pt-BR/reference/statements/create/table#encryption-codecs). A chave (ou as chaves) deve ser gravada em variáveis de ambiente ou definida no arquivo de configuração.

As chaves podem ser hexadecimais ou strings com comprimento de 16 bytes.

**Exemplo**

Carregamento a partir da configuração:

```xml
<encryption_codecs>
    <aes_128_gcm_siv>
        <key>1234567812345678</key>
    </aes_128_gcm_siv>
</encryption_codecs>
```

<Note>
  Não é recomendável armazenar chaves no arquivo de configuração. Isso não é seguro. Você pode mover as chaves para um arquivo de configuração separado em um disco seguro e colocar um link simbólico para esse arquivo na pasta `config.d/`.
</Note>

Carregando da configuração, quando a chave está em hexadecimal:

```xml
<encryption_codecs>
    <aes_128_gcm_siv>
        <key_hex>00112233445566778899aabbccddeeff</key_hex>
    </aes_128_gcm_siv>
</encryption_codecs>
```

Carregando a chave a partir da variável de ambiente:

```xml
<encryption_codecs>
    <aes_128_gcm_siv>
        <key_hex from_env="ENVVAR"></key_hex>
    </aes_128_gcm_siv>
</encryption_codecs>
```

Aqui, `current_key_id` define a chave atual para criptografia, e todas as chaves especificadas podem ser usadas para descriptografar.

Cada um desses métodos pode ser aplicado a várias chaves:

```xml
<encryption_codecs>
    <aes_128_gcm_siv>
        <key_hex id="0">00112233445566778899aabbccddeeff</key_hex>
        <key_hex id="1" from_env="ENVVAR"></key_hex>
        <current_key_id>1</current_key_id>
    </aes_128_gcm_siv>
</encryption_codecs>
```

Aqui, `current_key_id` mostra a chave atual usada para criptografia.

Além disso, os usuários podem adicionar um nonce, que deve ter 12 bytes de comprimento (por padrão, os processos de criptografia e descriptografia usam um nonce composto por bytes zerados):

```xml
<encryption_codecs>
    <aes_128_gcm_siv>
        <nonce>012345678910</nonce>
    </aes_128_gcm_siv>
</encryption_codecs>
```

Ou pode ser especificado em hexadecimal:

```xml
<encryption_codecs>
    <aes_128_gcm_siv>
        <nonce_hex>abcdefabcdef</nonce_hex>
    </aes_128_gcm_siv>
</encryption_codecs>
```

<Note>
  Tudo o que foi mencionado acima pode ser aplicado a `aes_256_gcm_siv` (mas a chave deve ter 32 bytes).
</Note>

<div id="http_handlers">
  ## http_handlers
</div>

Permite usar handlers HTTP personalizados.
Para adicionar um novo handler HTTP, basta adicionar uma nova `<rule>`.
As regras são verificadas de cima para baixo, na ordem em que são definidas,
e a primeira correspondência executará o handler.
Uma regra sem condições de correspondência (apenas `handler`) corresponde a todas as requisições; como as regras são verificadas em ordem,
esse tipo de regra só é útil como fallback, colocada por último.

As seguintes configurações podem ser definidas por subtags (todas essas subtags são opcionais, exceto `handler`):

| Sub-tags             | Definition                                                                                                                                                                                                                                                                                                                     |
| -------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| `url`                | Para corresponder ao caminho da URL da requisição. A string de consulta é ignorada na correspondência                                                                                                                                                                                                                          |
| `url_prefix`         | Para corresponder ao caminho da URL da requisição em relação a um caminho base: o próprio caminho ou qualquer item abaixo dele no limite de um segmento do caminho (por exemplo, &#39;/api/v1&#39; corresponde a /api/v1, /api/v1/ e /api/v1/write, mas não a /api/v1beta). A string de consulta é ignorada na correspondência |
| `url_regexp`         | Para corresponder ao caminho da URL da requisição em relação a uma expressão regular. A string de consulta é ignorada na correspondência                                                                                                                                                                                       |
| `full_url`           | Para corresponder à URL completa da requisição `scheme://host:port/path`. A string de consulta é ignorada na correspondência, e o host é o endereço IP da conexão (não o header `Host`)                                                                                                                                        |
| `full_url_prefix`    | Para corresponder à URL completa da requisição `scheme://host:port/path` em relação à base URL `scheme://host:port/base_path`, no limite de um segmento do caminho (consulte `url_prefix`). A string de consulta é ignorada na correspondência                                                                                 |
| `full_url_regexp`    | Para corresponder à URL completa da requisição `scheme://host:port/path` em relação a uma expressão regular. A string de consulta é ignorada na correspondência                                                                                                                                                                |
| `methods`            | Para corresponder aos métodos da requisição, você pode usar vírgulas para separar vários métodos                                                                                                                                                                                                                               |
| `headers`            | Para corresponder aos headers da requisição, faça a correspondência de cada elemento filho (o nome do elemento filho é o nome do header)                                                                                                                                                                                       |
| `headers_regexp`     | Como em `headers`, mas o valor de cada elemento filho é comparado com uma expressão regular                                                                                                                                                                                                                                    |
| `empty_query_string` | Verifica se não há string de consulta na URL                                                                                                                                                                                                                                                                                   |
| `handler`            | O handler da requisição (obrigatório)                                                                                                                                                                                                                                                                                          |

<Note>
  Em vez de `url_regexp`, `full_url_regexp` e `headers_regexp`, você também pode escrever uma expressão regular em `url`, `full_url` ou `headers` usando o prefixo `regex:` (por exemplo, `<url>regex:/api/.*</url>`). Isso ainda é compatível com versões anteriores, mas está obsoleto: prefira as subtags dedicadas `url_regexp`, `full_url_regexp` e `headers_regexp`.
</Note>

`handler` contém as seguintes configurações, que podem ser definidas por subtags:

| Sub-tags           | Definition                                                                                                                                                                                           |
| ------------------ | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `url`              | Um local para redirecionamento                                                                                                                                                                       |
| `type`             | Tipos compatíveis: static, dynamic&#95;query&#95;handler, predefined&#95;query&#95;handler, redirect                                                                                                 |
| `status`           | Usado com o tipo static, código de status da resposta                                                                                                                                                |
| `query_param_name` | Usado com o tipo dynamic&#95;query&#95;handler, extrai e executa o valor correspondente ao valor de `<query_param_name>` nos parâmetros da requisição HTTP                                           |
| `query`            | Usado com o tipo predefined&#95;query&#95;handler, executa a consulta quando o handler é chamado                                                                                                     |
| `content_type`     | Usado com o tipo static, content-type da resposta                                                                                                                                                    |
| `response_content` | Usado com o tipo static, conteúdo da resposta enviado ao cliente; ao usar o prefixo &#39;file://&#39; ou &#39;config://&#39;, localiza o conteúdo no arquivo ou na configuração e o envia ao cliente |

Junto com a lista de regras, você pode especificar `<defaults/>`, que habilita todos os handlers padrão.

Exemplo:

```xml
<http_handlers>
    <rule>
        <url>/</url>
        <methods>POST,GET</methods>
        <headers><pragma>no-cache</pragma></headers>
        <handler>
            <type>dynamic_query_handler</type>
            <query_param_name>query</query_param_name>
        </handler>
    </rule>

    <rule>
        <url>/predefined_query</url>
        <methods>POST,GET</methods>
        <handler>
            <type>predefined_query_handler</type>
            <query>SELECT * FROM system.settings</query>
        </handler>
    </rule>

    <rule>
        <handler>
            <type>static</type>
            <status>200</status>
            <content_type>text/plain; charset=UTF-8</content_type>
            <response_content>config://http_server_default_response</response_content>
        </handler>
    </rule>
</http_handlers>
```

<div id="insert_deduplication_version">
  ## insert_deduplication_version
</div>

<SettingsInfoBlock type="InsertDeduplicationVersions" default_value="compatible_double_hashes" changeable_without_restart="No" />

Essa configuração permite migrar da versão do código em que a desduplicação de insert para inserts síncronos e assíncronos era feita de forma totalmente diferente e não transparente para a versão do código em que os dados inseridos são desduplicados entre inserts síncronos e assíncronos.
O valor `old_separate_hashes` significa que o ClickHouse usará hashes de desduplicação diferentes para inserts síncronos e assíncronos (como antes).
Esse valor deve ser usado como valor padrão se não houver intenção de iniciar a migração.
O valor `compatible_double_hashes` significa que o ClickHouse usará dois hashes de desduplicação: o antigo para inserts síncronos ou assíncronos e outro, o novo, para todos os inserts. Esse valor deve ser usado para migrar instâncias existentes para o novo comportamento de forma segura.
Esse valor deve ficar habilitado por algum tempo (consulte as configurações replicated&#95;deduplication&#95;window e non&#95;replicated&#95;deduplication&#95;window) para garantir que nenhum insert síncrono ou assíncrono seja perdido durante a migração.
Por fim, o valor `new_unified_hash` significa que o ClickHouse usará o novo hash de desduplicação para inserts síncronos e assíncronos. Esse valor pode ser habilitado em novas instâncias do ClickHouse ou em instâncias que já tenham usado o valor `compatible_double_hashes` por algum tempo.
O valor padrão será definido como `compatible_double_hashes` e, mais tarde, como `new_unified_hash` em versões futuras do ClickHouse, para concluir a migração com segurança em duas etapas.

Portanto, `interserver_http_credentials` deve ser o mesmo em todas as réplicas de um cluster.

<Note>
  * Por padrão, se a seção `interserver_http_credentials` for omitida, a autenticação não será usada durante a replicação.
  * As configurações de `interserver_http_credentials` não estão relacionadas à [configuração](/pt-BR/concepts/features/interfaces/client#configuration_files) de credenciais do cliente ClickHouse.
  * Essas credenciais são compartilhadas pela replicação via `HTTP` e `HTTPS`.
</Note>

As configurações a seguir podem ser definidas por subtags:

* `user` — Nome de usuário.
* `password` — Senha.
* `allow_empty` — Se `true`, outras réplicas poderão se conectar sem autenticação mesmo que as credenciais estejam definidas. Se `false`, conexões sem autenticação serão recusadas. Padrão: `false`.
* `old` — Contém os valores antigos de `user` e `password` usados durante a rotação de credenciais. Várias seções `old` podem ser especificadas.

**Rotação de credenciais**

O ClickHouse oferece suporte à rotação dinâmica de credenciais entre servidores sem interromper todas as réplicas ao mesmo tempo para atualizar a configuração delas. As credenciais podem ser alteradas em várias etapas.

Para ativar a autenticação, defina `interserver_http_credentials.allow_empty` como `true` e adicione credenciais. Isso permite conexões com e sem autenticação.

```xml
<interserver_http_credentials>
    <user>admin</user>
    <password>111</password>
    <allow_empty>true</allow_empty>
</interserver_http_credentials>
```

Após configurar todas as réplicas, defina `allow_empty` como `false` ou remova essa configuração. Isso torna obrigatória a autenticação com as novas credenciais.

Para alterar as credenciais existentes, mova o nome de usuário e a senha para a seção `interserver_http_credentials.old` e atualize `user` e `password` com novos valores. Nesse ponto, o servidor usa as novas credenciais para se conectar a outras réplicas e aceita conexões com credenciais novas ou antigas.

```xml
<interserver_http_credentials>
    <user>admin</user>
    <password>222</password>
    <old>
        <user>admin</user>
        <password>111</password>
    </old>
    <old>
        <user>temp</user>
        <password>000</password>
    </old>
</interserver_http_credentials>
```

Quando as novas credenciais forem aplicadas a todas as réplicas, as credenciais antigas poderão ser removidas.

<div id="ldap_servers">
  ## ldap_servers
</div>

Liste aqui os servidores LDAP com seus parâmetros de conexão para:

* usá-los como autenticadores para usuários locais dedicados, que tenham o mecanismo de autenticação `'ldap'` especificado em vez de `'password'`
* usá-los como diretórios remotos de usuários.

As configurações a seguir podem ser definidas por subtags:

| Setting                        | Description                                                                                                                                                                                                                                                                                                                                                                                                                                                                             |
| ------------------------------ | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `bind_dn`                      | Modelo usado para construir o DN de bind. O DN resultante será construído substituindo todas as substrings `\{user_name\}` do modelo pelo nome de usuário real em cada tentativa de autenticação.                                                                                                                                                                                                                                                                                     |
| `enable_tls`                   | Sinalizador para ativar o uso de conexão segura com o servidor LDAP. Especifique `no` para o protocolo em texto simples (`ldap://`) (não recomendado). Especifique `yes` para LDAP sobre SSL/TLS (`ldaps://`) (recomendado e padrão). Especifique `starttls` para o protocolo legado StartTLS (protocolo em texto simples (`ldap://`), atualizado para TLS).                                                                                                                            |
| `host`                         | Hostname ou IP do servidor LDAP. Este parâmetro é obrigatório e não pode ficar vazio.                                                                                                                                                                                                                                                                                                                                                                                                   |
| `port`                         | Porta do servidor LDAP; o padrão é 636 se `enable_tls` estiver definido como true, caso contrário `389`.                                                                                                                                                                                                                                                                                                                                                                                |
| `tls_ca_cert_dir`              | caminho para o diretório que contém certificados de CA.                                                                                                                                                                                                                                                                                                                                                                                                                                 |
| `tls_ca_cert_file`             | caminho para o arquivo de certificado de CA.                                                                                                                                                                                                                                                                                                                                                                                                                                            |
| `tls_cert_file`                | caminho para o arquivo de certificado.                                                                                                                                                                                                                                                                                                                                                                                                                                                  |
| `tls_cipher_suite`             | conjunto de cifras permitido (na notação OpenSSL).                                                                                                                                                                                                                                                                                                                                                                                                                                      |
| `tls_key_file`                 | caminho para o arquivo da chave do certificado.                                                                                                                                                                                                                                                                                                                                                                                                                                         |
| `tls_minimum_protocol_version` | A versão mínima do protocolo SSL/TLS. Os valores aceitos são: `ssl2`, `ssl3`, `tls1.0`, `tls1.1`, `tls1.2` (o padrão).                                                                                                                                                                                                                                                                                                                                                                  |
| `tls_require_cert`             | Comportamento de verificação do certificado do par SSL/TLS. Os valores aceitos são: `never`, `allow`, `try`, `demand` (o padrão).                                                                                                                                                                                                                                                                                                                                                       |
| `user_dn_detection`            | Seção com parâmetros de busca LDAP para detectar o user DN real do usuário autenticado via bind. Isso é usado principalmente em filtros de busca para mapeamento de função adicional quando o servidor é o Active Directory. O user DN resultante será usado ao substituir substrings `\{user_dn\}` onde elas forem permitidas. Por padrão, o user DN é definido como igual ao bind DN, mas, depois que a busca é executada, ele será atualizado com o valor real do user DN detectado. |
| `verification_cooldown`        | Um período, em segundos, após uma tentativa de bind bem-sucedida, durante o qual será considerado que o usuário foi autenticado com sucesso em todas as requisições consecutivas sem contatar o servidor LDAP. Especifique `0` (o padrão) para desabilitar o cache e forçar o contato com o servidor LDAP em cada requisição de autenticação.                                                                                                                                         |

A configuração `user_dn_detection` pode ser definida com subtags:

| Setting         | Description                                                                                                                                                                                                                                                                                                                                 |
| --------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `base_dn`       | modelo usado para construir o base DN da busca LDAP. O DN resultante será construído substituindo todas as substrings `\{user_name\}` e &#39;&#123;bind&#95;dn&#125;&#39; do modelo pelo nome de usuário real e pelo bind DN durante a busca LDAP.                                                                                          |
| `scope`         | escopo da busca LDAP. Os valores aceitos são: `base`, `one_level`, `children`, `subtree` (o padrão).                                                                                                                                                                                                                                        |
| `search_filter` | modelo usado para construir o search filter da busca LDAP. O filtro resultante será construído substituindo todas as substrings `\{user_name\}`, `\{bind_dn\}` e `\{base_dn\}` do modelo pelo nome de usuário real, bind DN e base DN durante a busca LDAP. Observe que os caracteres especiais precisam ser escapados corretamente em XML. |

Exemplo:

```xml
<my_ldap_server>
    <host>localhost</host>
    <port>636</port>
    <bind_dn>uid={user_name},ou=users,dc=example,dc=com</bind_dn>
    <verification_cooldown>300</verification_cooldown>
    <enable_tls>yes</enable_tls>
    <tls_minimum_protocol_version>tls1.2</tls_minimum_protocol_version>
    <tls_require_cert>demand</tls_require_cert>
    <tls_cert_file>/path/to/tls_cert_file</tls_cert_file>
    <tls_key_file>/path/to/tls_key_file</tls_key_file>
    <tls_ca_cert_file>/path/to/tls_ca_cert_file</tls_ca_cert_file>
    <tls_ca_cert_dir>/path/to/tls_ca_cert_dir</tls_ca_cert_dir>
    <tls_cipher_suite>ECDHE-ECDSA-AES256-GCM-SHA384:ECDHE-RSA-AES256-GCM-SHA384:AES256-GCM-SHA384</tls_cipher_suite>
</my_ldap_server>
```

Exemplo (configuração típica do Active Directory com detecção do DN do usuário para posterior mapeamento de função):

```xml
<my_ad_server>
    <host>localhost</host>
    <port>389</port>
    <bind_dn>EXAMPLE\{user_name}</bind_dn>
    <user_dn_detection>
        <base_dn>CN=Users,DC=example,DC=com</base_dn>
        <search_filter>(&amp;(objectClass=user)(sAMAccountName={user_name}))</search_filter>
    </user_dn_detection>
    <enable_tls>no</enable_tls>
</my_ad_server>
```

| `%V`          | Número da semana ISO 8601 (intervalo [01,53])                                                                                                                                              | `27`                       |
| `%j`          | Dia do ano como número decimal (intervalo [001,366])                                                                                                                                       | `187`                      |
| `%d`          | Dia do mês como número decimal preenchido com zero à esquerda (intervalo [01,31]). Um único dígito é precedido por zero.                                                                   | `06`                       |
| `%e`          | Dia do mês como número decimal preenchido com espaço à esquerda (intervalo [1,31]). Um único dígito é precedido por um espaço.                                                             | `&nbsp; 6`                 |
| `%a`          | Nome abreviado do dia da semana, por exemplo, Fri (dependente da localidade)                                                                                                               | `Thu`                      |
| `%A`          | Nome completo do dia da semana, por exemplo, Friday (dependente da localidade)                                                                                                             | `Thursday`                 |
| `%w`          | Dia da semana como número inteiro, em que domingo é 0 (intervalo [0-6])                                                                                                                    | `4`                        |
| `%u`          | Dia da semana como número decimal, em que segunda-feira é 1 (formato ISO 8601) (intervalo [1-7])                                                                                           | `4`                        |
| `%H`          | Hora como número decimal, no formato de 24 horas (intervalo [00-23])                                                                                                                       | `18`                       |
| `%I`          | Hora como número decimal, no formato de 12 horas (intervalo [01,12])                                                                                                                       | `06`                       |
| `%M`          | Minuto como número decimal (intervalo [00,59])                                                                                                                                             | `32`                       |
| `%S`          | Segundo como número decimal (intervalo [00,60])                                                                                                                                            | `07`                       |
| `%c`          | Representação padrão de data e hora, por exemplo, Sun Oct 17 04:41:13 2010 (dependente da localidade)                                                                                      | `Thu Jul  6 18:32:07 2023` |
| `%x`          | Representação localizada da data (dependente da localidade)                                                                                                                                | `07/06/23`                 |

```xml

    <size>1000M</size>
    <count>10</count>
    <stream_compress>true</stream_compress>
</logger>
```

Para imprimir mensagens de log somente no console:

```xml
<logger>
    <level>information</level>
    <console>true</console>
</logger>
```

**Sobrescritas por nível**

É possível sobrescrever o nível de log de loggers individuais. Por exemplo, para silenciar todas as mensagens dos loggers &quot;Backup&quot; e &quot;RBAC&quot;.

```xml
<logger>
    <levels>
        <logger>
            <name>Backup</name>
            <level>none</level>
        </logger>
        <logger>
            <name>RBAC</name>
            <level>none</level>
        </logger>
    </levels>
</logger>
```

**syslog**

Para também gravar mensagens de log no syslog:

```xml
<logger>
    <use_syslog>1</use_syslog>
    <syslog>
        <address>syslog.remote:10514</address>
        <hostname>myhost.local</hostname>
        <facility>LOG_LOCAL6</facility>
        <format>syslog</format>
    </syslog>
</logger>
```

Chaves para `<syslog>`:

| Key        | Description                                                                                                                                                                                                                                                                                               |
| ---------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `address`  | O endereço do syslog no formato `host\[:port\]`. Se omitido, o daemon local será usado.                                                                                                                                                                                                                   |
| `hostname` | O nome do host de onde os logs são enviados (opcional).                                                                                                                                                                                                                                                   |
| `facility` | A [palavra-chave de facility](https://en.wikipedia.org/wiki/Syslog#Facility) do syslog. Deve ser especificada em maiúsculas, com o prefixo &quot;LOG&#95;&quot;, por exemplo, `LOG_USER`, `LOG_DAEMON`, `LOG_LOCAL3` etc. Padrão: `LOG_USER` se `address` for especificado; caso contrário, `LOG_DAEMON`. |
| `format`   | Formato da mensagem de log. Valores possíveis: `bsd` e `syslog.`                                                                                                                                                                                                                                          |

**Formatos de log**

Você pode especificar o formato de log que será exibido no console. Atualmente, apenas JSON é compatível.

**Exemplo**

Aqui está um exemplo de log JSON de saída:

```json
{
  "date_time_utc": "2024-11-06T09:06:09Z",
  "date_time": "1650918987.180175",
  "thread_name": "#1",
  "thread_id": "254545",
  "level": "Trace",
  "query_id": "",
  "logger_name": "BaseDaemon",
  "message": "Received signal 2",
  "source_file": "../base/daemon/BaseDaemon.cpp; virtual void SignalListener::run()",
  "source_line": "192"
}
```

Para habilitar o suporte a logs em JSON, use o trecho a seguir:

```xml
<logger>
    <formatting>
        <type>json</type>
        <!-- Pode ser configurado por canal (log, errorlog, console, syslog) ou globalmente para todos os canais (nesse caso, basta omiti-lo). -->
        <!-- <channel></channel> -->
        <names>
            <date_time>date_time</date_time>
            <thread_name>thread_name</thread_name>
            <thread_id>thread_id</thread_id>
            <level>level</level>
            <query_id>query_id</query_id>
            <logger_name>logger_name</logger_name>
            <message>message</message>
            <source_file>source_file</source_file>
            <source_line>source_line</source_line>
        </names>
    </formatting>
</logger>
```

**Renomeando chaves em logs JSON**

Os nomes das chaves podem ser modificados alterando os valores das tags dentro da tag `<names>`. Por exemplo, para alterar `DATE_TIME` para `MY_DATE_TIME`, você pode usar `<date_time>MY_DATE_TIME</date_time>`.

**Omitindo chaves em logs JSON**

As propriedades do log podem ser omitidas comentando a propriedade correspondente. Por exemplo, se você não quiser que o log imprima `query_id`, pode comentar a tag `<query_id>`.

<SettingsInfoBlock type="UInt64" default_value="0" changeable_without_restart="Sim" />

Como caso especial, o valor `0` (padrão) significa que o servidor pode consumir toda a memória disponível (exceto por restrições adicionais impostas por `max_server_memory_usage_to_ram_ratio`).

<div id="max_server_memory_usage_to_ram_ratio">
  ## max_server_memory_usage_to_ram_ratio
</div>

<SettingsInfoBlock type="Double" default_value="0.9" changeable_without_restart="No" />

A quantidade máxima de memória que o servidor pode usar, expressa como uma proporção da memória total disponível.

Por exemplo, um valor de `0.9` (padrão) significa que o servidor pode consumir 90% da memória disponível.

Permite reduzir o uso de memória em sistemas com pouca memória.
Em hosts com pouca RAM e swap, pode ser necessário definir [`max_server_memory_usage_to_ram_ratio`](#max_server_memory_usage_to_ram_ratio) com um valor maior que 1.

Essa proporção é aplicada de duas formas:

* Na inicialização (e ao recarregar a configuração), ela limita o limite rígido de memória do servidor a esta fração da RAM física total, como descrito acima.
* Em tempo de execução, o worker de memória em segundo plano recalcula periodicamente o limite rígido de memória como `(resident memory + system available memory) * max_server_memory_usage_to_ram_ratio`, para que o servidor mantenha uma folga de recursos para outros processos em execução no mesmo host. Ao executar dentro de um cgroup com um limite de memória finito, a memória disponível do cgroup é usada em vez do valor do host como um todo. À medida que outros processos crescem e a memória disponível diminui, o limite rígido do servidor acompanha essa redução, limitando o crescimento antes que o host (ou o cgroup) fique sem memória.

Definir a proporção como `0` desabilita tanto o limite na inicialização quanto o ajuste em tempo de execução. O ajuste em tempo de execução também é um no-op em sistemas não Linux e em `clickhouse-keeper`, que não expõe essa configuração. Para manter o limite estático de inicialização/recarregamento, mas desabilitar o ajuste em tempo de execução (o comportamento das versões anteriores), defina `memory_worker_dynamic_hard_limit` como `0`.

<Note>
  O consumo máximo de memória do servidor é limitado adicionalmente pela configuração `max_server_memory_usage`.
</Note>

<div id="min_allocation_size_to_throw_on_memory_limit">
  ## min_allocation_size_to_throw_on_memory_limit
</div>

<SettingsInfoBlock type="UInt64" default_value="0" changeable_without_restart="Sim" />

Tamanho mínimo, em bytes, de uma alocação genérica em C++ (como as feitas por contêineres padrão, strings, crescimento de `std::vector`, smart pointers etc.) que pode gerar `MEMORY_LIMIT_EXCEEDED` quando `max_server_memory_usage` é atingido. Alocações genéricas menores ainda são contabilizadas pelo rastreador de memória, mas podem ser concluídas com sucesso mesmo acima do limite, o que reduz falhas espúrias durante rotinas de limpeza e tratamento de exceções perto de OOM.

As alocações feitas pelos próprios buffers grandes do ClickHouse — dados de coluna, tabelas hash, arenas, pipelines de consulta, buffers de E/S — sempre respeitam o limite de memória e podem lançar uma exceção independentemente desse valor. Essa configuração afeta apenas alocações implícitas que passam por `operator new`.

Um valor de `0` (padrão) preserva o comportamento legado: alocações implícitas de `operator new` nunca são recusadas pelo rastreador de memória, e apenas alocações explícitas do ClickHouse podem gerar `MEMORY_LIMIT_EXCEEDED`.

Observação: para evitar efeitos colaterais, recomenda-se definir um valor maior que `max_untracked_memory`.

<div id="mmap_cache_size">
  ## mmap_cache_size
</div>

<SettingsInfoBlock type="UInt64" default_value="1024" changeable_without_restart="Yes" />

Essa configuração ajuda a evitar chamadas frequentes de open/close (que são muito custosas devido às falhas de página subsequentes) e permite reutilizar mapeamentos entre várias threads e consultas. O valor da configuração é o número de regiões mapeadas (geralmente igual ao número de arquivos mapeados).

A quantidade de dados em arquivos mapeados pode ser monitorada nas seguintes tabelas de sistema, com as seguintes métricas:

* `MMappedFiles`/`MMappedFileBytes`/`MMapCacheCells` em [`system.metrics`](/pt-BR/reference/system-tables/metrics), [`system.metric_log`](/pt-BR/reference/system-tables/metric_log)
* `CreatedReadBufferMMap`/`CreatedReadBufferMMapFailed`/`MMappedFileCacheHits`/`MMappedFileCacheMisses` em [`system.events`](/pt-BR/reference/system-tables/events), [`system.processes`](/pt-BR/reference/system-tables/processes), [`system.query_log`](/pt-BR/reference/system-tables/query_log), [`system.query_thread_log`](/pt-BR/reference/system-tables/query_thread_log), [`system.query_views_log`](/pt-BR/reference/system-tables/query_views_log)

<Note>
  A quantidade de dados em arquivos mapeados não consome memória diretamente e não é contabilizada no uso de memória da consulta nem do servidor, porque essa memória pode ser descartada de forma semelhante ao cache de páginas do SO. O cache é descartado (os arquivos são fechados) automaticamente quando partes antigas são removidas em tabelas da família MergeTree, e também pode ser descartado manualmente pela consulta `SYSTEM DROP MMAP CACHE`.

  Essa configuração pode ser modificada em tempo de execução e passa a valer imediatamente.
</Note>

<div id="oom_canary_enable">
  ## oom_canary_enable
</div>

<SettingsInfoBlock type="Bool" default_value="0" changeable_without_restart="No" />

Experimental. Habilita o canário de OOM: um processo filho sacrificial que atrai o OOM killer do Linux
antes do processo principal do servidor ClickHouse, dando ao servidor a chance de aliviar a carga.
Requer Linux &gt;= 5.3 (para `pidfd_open`); o canário é desabilitado na inicialização em kernels mais antigos.
A resposta a OOM requer evidência de OOM-kill em `memory.events.local` do cgroup v2 e pode executar cancelamento global de
consultas, cancelamento de merge e gravações em `system.crash_log`.
O canário não consegue proteger o servidor quando `memory.oom.group` do cgroup v2 está habilitado para o
cgroup do servidor: nesse caso, o kernel mata todo o cgroup como uma única unidade, incluindo o servidor,
portanto a resposta a OOM nunca é executada. Um aviso é registrado na inicialização nesse modo.
O comportamento pode mudar entre versões do ClickHouse até que a validação em produção seja concluída.

<div id="oom_canary_initial_backoff_seconds">
  ## oom_canary_initial_backoff_seconds
</div>

<SettingsInfoBlock type="UInt64" default_value="1" changeable_without_restart="No" />

Atraso inicial de backoff, em segundos, entre reinicializações consecutivas do canário de OOM.
O atraso dobra a cada reinicialização até `oom_canary_max_backoff_seconds`.

<div id="oom_canary_max_backoff_seconds">
  ## oom_canary_max_backoff_seconds
</div>

<SettingsInfoBlock type="UInt64" default_value="60" changeable_without_restart="No" />

Atraso máximo de backoff, em segundos, entre reinicializações consecutivas do canário de OOM.

<div id="oom_canary_max_rapid_relaunches">
  ## oom_canary_max_rapid_relaunches
</div>

<SettingsInfoBlock type="UInt64" default_value="10" changeable_without_restart="No" />

Número máximo de relançamentos rápidos consecutivos do canário de OOM antes que o relançamento automático seja desativado
para evitar instabilidade sob pressão de memória persistente. O contador e o backoff de relançamento são redefinidos
assim que um canário sobrevive por mais tempo que `oom_canary_max_backoff_seconds`, de modo que um canário que morre apenas
esporadicamente ao longo de um longo tempo de atividade não acabe sendo desativado.
Aplica-se somente quando `oom_canary_relaunch` é true.

<div id="oom_canary_relaunch">
  ## oom_canary_relaunch
</div>

<SettingsInfoBlock type="Bool" default_value="1" changeable_without_restart="Não" />

Quando `true`, o canário de OOM é reiniciado automaticamente depois que o processo do canário é encerrado por qualquer
motivo que não seja uma falha permanente de Setup ou o desligamento do servidor — incluindo encerramentos por OOM, outros
sinais e saídas transitórias — de acordo com `oom_canary_max_rapid_relaunches` e as configurações de backoff.
A própria sequência de resposta a OOM é executada apenas quando `memory.events.local` do cgroup v2
fornece evidências de encerramento por OOM.

<div id="oom_canary_size">
  ## oom_canary_size
</div>

<SettingsInfoBlock type="UInt64" default_value="104857600" changeable_without_restart="No" />

Tamanho em bytes da região de memória que o processo filho do canário de OOM aloca e acessa.
O bloqueio da região com `mlock` é feito em regime de melhor esforço: requer `CAP_IPC_LOCK` ou um
`RLIMIT_MEMLOCK` suficiente, e, quando o bloqueio falha, o canário emite um aviso e a memória permanece
alocada, mas pode ser movida para swap.
O padrão é 100 MB (104857600). Valores maiores tornam o canário um alvo de OOM mais atraente.

<div id="proxy">
  ## proxy
</div>

Defina servidores proxy para solicitações HTTP e HTTPS, atualmente compatível com armazenamento S3, funções de tabela do S3 e funções de URL.

Há três maneiras de definir servidores proxy:

* variáveis de ambiente
* listas de proxy
* resolvedores remotos de proxy.

Também há suporte para ignorar servidores proxy em hosts específicos com o uso de `no_proxy`.

**Variáveis de ambiente**

As variáveis de ambiente `http_proxy` e `https_proxy` permitem especificar um
servidor proxy para um determinado protocolo. Se elas já estiverem definidas no seu sistema, isso deverá funcionar sem problemas.

Essa é a abordagem mais simples se um determinado protocolo tiver
apenas um servidor proxy e esse servidor proxy não mudar.

**Listas de proxy**

Essa abordagem permite especificar um ou mais
servidores proxy para um protocolo. Se mais de um servidor proxy for definido,
o ClickHouse usa os diferentes proxies em esquema round-robin, distribuindo a
carga entre os servidores. Essa é a abordagem mais simples se houver mais de
um servidor proxy para um protocolo e a lista de servidores proxy não mudar.

**Modelo de configuração**

```xml
<proxy>
    <http>
        <uri>http://proxy1</uri>
        <uri>http://proxy2:3128</uri>
    </http>
    <https>
        <uri>http://proxy1:3128</uri>
    </https>
</proxy>
```

Selecione um campo pai nas abas abaixo para ver seus campos filhos:

<Tabs>
  <Tab title="<proxy>">
    | Campo     | Descrição                             |
    | --------- | ------------------------------------- |
    | `<http>`  | Uma lista de um ou mais proxies HTTP  |
    | `<https>` | Uma lista de um ou mais proxies HTTPS |
  </Tab>

  <Tab title="<http> e <https>">
    | Campo   | Descrição      |
    | ------- | -------------- |
    | `<uri>` | O URI do proxy |
  </Tab>
</Tabs>

**Resolvedores remotos de proxy**

É possível que os servidores proxy mudem dinamicamente. Nesse
caso, você pode definir o endpoint de um resolvedor. O ClickHouse envia
uma solicitação GET vazia para esse endpoint, e o resolvedor remoto deve retornar o host do proxy.
O ClickHouse o usará para formar o URI do proxy usando o seguinte modelo: `\{proxy_scheme\}://\{proxy_host\}:{proxy_port}`

**Modelo de configuração**

```xml
<proxy>
    <http>
        <resolver>
            <endpoint>http://resolver:8080/hostname</endpoint>
            <proxy_scheme>http</proxy_scheme>
            <proxy_port>80</proxy_port>
            <proxy_cache_time>10</proxy_cache_time>
        </resolver>
    </http>

    <https>
        <resolver>
            <endpoint>http://resolver:8080/hostname</endpoint>
            <proxy_scheme>http</proxy_scheme>
            <proxy_port>3128</proxy_port>
            <proxy_cache_time>10</proxy_cache_time>
        </resolver>
    </https>

</proxy>
```

Selecione um campo pai nas abas abaixo para ver os respectivos campos filhos:

<Tabs>
  <Tab title="<proxy>">
    | Campo     | Descrição                           |
    | --------- | ----------------------------------- |
    | `<http>`  | Uma lista com um ou mais resolvedores* |
    | `<https>` | Uma lista com um ou mais resolvedores* |
  </Tab>

  <Tab title="<http> e <https>">
    | Campo        | Descrição                                   |
    | ------------ | ------------------------------------------- |
    | `<resolver>` | O endpoint e outros detalhes de um resolvedor |

    <Note>
      Você pode ter vários elementos `<resolver>`, mas apenas o primeiro
      `<resolver>` de um determinado protocolo é usado. Quaisquer outros elementos `<resolver>`
      desse protocolo são ignorados. Isso significa que o balanceamento de carga
      (se necessário) deve ser implementado pelo resolvedor remoto.
    </Note>
  </Tab>

  <Tab title="<resolver>">
    | Campo                | Descrição                                                                                                                                                                                                                             |
    | -------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
    | `<endpoint>`         | O URI do resolvedor de proxy                                                                                                                                                                                                            |
    | `<proxy_scheme>`     | O protocolo do URI final do proxy. Pode ser `http` ou `https`.                                                                                                                                                                        |
    | `<proxy_port>`       | O número da porta do resolvedor de proxy                                                                                                                                                                                                |
    | `<proxy_cache_time>` | O tempo, em segundos, durante o qual os valores do resolvedor devem ser armazenados em cache pelo ClickHouse. Definir esse valor como `0` faz com que o ClickHouse entre em contato com o resolvedor para cada solicitação HTTP ou HTTPS. |
  </Tab>
</Tabs>

**Precedência**

As configurações de proxy são determinadas na seguinte ordem:

| Ordem | Configuração               |
| ----- | -------------------------- |
| 1.    | Resolvedores remotos de proxy |
| 2.    | Listas de proxy            |
| 3.    | Variáveis de ambiente      |

O ClickHouse verificará o tipo de resolvedor de maior prioridade para o protocolo da solicitação. Se ele não estiver definido,
verificará o próximo tipo de resolvedor de maior prioridade, até chegar ao resolvedor de ambiente.
Isso também permite usar uma combinação de tipos de resolvedor.

<div id="query_masking_rules">
  ## query_masking_rules
</div>

Regras baseadas em expressões regulares, aplicadas às consultas e também a todas as mensagens de log antes de serem armazenadas nos logs do servidor, nas tabelas [`system.query_log`](/pt-BR/reference/system-tables/query_log), [`system.text_log`](/pt-BR/reference/system-tables/text_log) e [`system.processes`](/pt-BR/reference/system-tables/processes), e nos logs enviados ao cliente. Isso ajuda a evitar
o vazamento de dados sensíveis de consultas SQL, como nomes, e-mails, identificadores pessoais ou números de cartão de crédito, para os logs.

**Exemplo**

```xml
<query_masking_rules>
    <rule>
        <name>hide SSN</name>
        <regexp>(^|\D)\d{3}-\d{2}-\d{4}($|\D)</regexp>
        <replace>000-00-0000</replace>
    </rule>
</query_masking_rules>
```

**Campos de configuração**:

| Configuração | Descrição                                                                          |
| ------------ | ---------------------------------------------------------------------------------- |
| `name`       | nome da regra (opcional)                                                           |
| `regexp`     | expressão regular compatível com RE2 (obrigatório)                                 |
| `replace`    | texto de substituição para dados sensíveis (opcional; por padrão, seis asteriscos) |

As regras de mascaramento são aplicadas à consulta inteira (para evitar vazamentos de dados sensíveis em consultas malformadas / que não podem ser interpretadas).

A tabela [`system.events`](/pt-BR/reference/system-tables/events) tem o contador `QueryMaskingRulesMatch`, que indica o número total de correspondências com regras de mascaramento de consultas.

Para consultas distribuídas, cada servidor precisa ser configurado separadamente; caso contrário, as subconsultas enviadas para outros
nós serão armazenadas sem mascaramento.

<div id="remote_url_allow_hosts">
  ## remote_url_allow_hosts
</div>

Lista de hosts permitidos para uso em motores de armazenamento relacionados a URL e em table functions.

Ao adicionar um host com a tag XML `\<host\>`:

* ele deve ser especificado exatamente como na URL, pois o nome é verificado antes da resolução de DNS. Por exemplo: `<host>clickhouse.com</host>`
* se a porta for especificada explicitamente na URL, então host:porta será verificado como um todo. Por exemplo: `<host>clickhouse.com:80</host>`
* se o host for especificado sem porta, então qualquer porta desse host será permitida. Por exemplo: se `<host>clickhouse.com</host>` for especificado, então `clickhouse.com:20` (FTP), `clickhouse.com:80` (HTTP), `clickhouse.com:443` (HTTPS) etc. serão permitidos.
* se o host for especificado como um endereço IP, então ele será verificado conforme especificado na URL. Por exemplo: `[2a02:6b8:a::a]`.
* se houver redirecionamentos e o suporte a redirecionamentos estiver habilitado, então cada redirecionamento (o campo location) será verificado.

Por exemplo:

```sql
<remote_url_allow_hosts>
    <host>clickhouse.com</host>
</remote_url_allow_hosts>
```

Permite a configuração de armazenamento com vários discos.


<div id="configuration-of-disks">
  ### Configuração de disks
</div>

A configuração de `disks` segue a estrutura abaixo:

```xml
<storage_configuration>
    <disks>
        <disk_name_1>
            <path>/mnt/fast_ssd/clickhouse/</path>
        </disk_name_1>
        <disk_name_2>
            <path>/mnt/hdd1/clickhouse/</path>
            <keep_free_space_bytes>10485760</keep_free_space_bytes>
        </disk_name_2>
        <disk_name_3>
            <path>/mnt/hdd2/clickhouse/</path>
            <keep_free_space_bytes>10485760</keep_free_space_bytes>
        </disk_name_3>
        ...
    </disks>
</storage_configuration>
```

As subtags acima definem as seguintes configurações para `disks`:

| Configuração            | Descrição                                                                                                    |
| ----------------------- | ------------------------------------------------------------------------------------------------------------ |
| `<disk_name_N>`         | O nome do disco, que deve ser único.                                                                         |
| `path`                  | O caminho em que os dados do servidor serão armazenados (catálogos `data` e `shadow`). Deve terminar com `/` |
| `keep_free_space_bytes` | Tamanho do espaço livre reservado no disco.                                                                  |

<Note>
  A ordem dos `disks` não importa.
</Note>

<div id="configuration-of-policies">
  ### Configuração de políticas
</div>

As subtags acima definem as seguintes configurações para `policies`:

| Configuração                 | Descrição                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                     |
| ---------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `policy_name_N`              | Nome da política. Os nomes das políticas devem ser únicos.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                    |
| `volume_name_N`              | Nome do volume. Os nomes dos volumes devem ser únicos.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                        |
| `disk`                       | O disk localizado dentro do volume.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                           |
| `max_data_part_size_bytes`   | O tamanho máximo de um fragmento de dados que pode residir em qualquer disk deste volume. Se a operação de merge resultar em um fragmento com tamanho previsto maior que `max_data_part_size_bytes`, o fragmento será gravado no próximo volume. Basicamente, esse recurso permite armazenar fragmentos novos/pequenos em um volume quente (SSD) e movê-los para um volume frio (HDD) quando atingirem um tamanho maior. Não use essa opção se a política tiver apenas um volume.                                                                                                             |
| `move_factor`                | A proporção de espaço livre disponível no volume. Se o espaço ficar abaixo disso, os dados começarão a ser transferidos para o próximo volume, se houver. Para a transferência, os fragmentos são ordenados por tamanho, do maior para o menor (ordem decrescente), e são selecionados os fragmentos cujo tamanho total seja suficiente para atender à condição de `move_factor`; se o tamanho total de todos os fragmentos for insuficiente, todos os fragmentos serão movidos.                                                                                                              |
| `perform_ttl_move_on_insert` | Desabilita a movimentação, na inserção, de dados com TTL expirado. Por padrão (quando habilitado), se inserirmos uma parte de dados que já expirou de acordo com a regra de movimentação por tempo de vida, ela será movida imediatamente para o volume / disk especificado na regra de movimentação. Isso pode tornar a inserção significativamente mais lenta caso o volume / disk de destino seja lento (por exemplo, S3). Se desabilitado, a parte expirada dos dados é gravada no volume padrão e, em seguida, movida imediatamente para o volume especificado na regra do TTL expirado. |
| `load_balancing`             | Política de balanceamento de disk, `round_robin` ou `least_used`.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                             |
| `least_used_ttl_ms`          | Define o timeout (em milissegundos) para atualizar o espaço disponível em todos os disks (`0` - sempre atualizar, `-1` - nunca atualizar, o valor padrão é `60000`). Observe que, se o disk for usado apenas pelo ClickHouse e não estiver sujeito a redimensionamento do sistema de arquivos em tempo real, você pode usar o valor `-1`. Em todos os outros casos, isso não é recomendado, pois acabará levando a uma alocação incorreta de espaço.                                                                                                                                          |
| `prefer_not_to_merge`        | Desabilita a mesclagem de partes dos dados neste volume. Observação: isso é potencialmente prejudicial e pode causar lentidão. Quando essa configuração está habilitada (não faça isso), a mesclagem de dados neste volume é proibida (o que é ruim). Isso permite controlar como o ClickHouse interage com disks lentos. Recomendamos não usar isso de forma alguma.                                                                                                                                                                                                                         |
| `volume_priority`            | Define a prioridade (ordem) em que os volumes são preenchidos. Quanto menor o valor, maior a prioridade. Os valores do parâmetro devem ser números naturais e cobrir o intervalo de 1 a N (N é o maior valor de parâmetro especificado), sem lacunas.                                                                                                                                                                                                                                                                                                                                         |

Para `volume_priority`:

* Se todos os volumes tiverem esse parâmetro, eles serão priorizados na ordem especificada.
* Se apenas *alguns* volumes o tiverem, os volumes que não o tiverem terão a menor prioridade. Aqueles que o tiverem serão priorizados de acordo com o valor da tag, e a prioridade dos demais será determinada pela ordem em que aparecem no arquivo de configuração.
* Se *nenhum* volume receber esse parâmetro, a ordem deles será determinada pela ordem em que aparecem no arquivo de configuração.
* A prioridade dos volumes não pode ser igual.

<div id="use_minimalistic_part_header_in_zookeeper">
  ## use_minimalistic_part_header_in_zookeeper
</div>

Método de armazenamento para cabeçalhos de partes de dados no ZooKeeper. Esta configuração se aplica apenas à família [`MergeTree`](/pt-BR/reference/engines/table-engines/mergetree-family/index). Ela pode ser especificada:

**Globalmente na seção [merge&#95;tree](#merge_tree) do arquivo `config.xml`**

O ClickHouse usa essa configuração para todas as tabelas no servidor. Você pode alterá-la a qualquer momento. As tabelas existentes mudam de comportamento quando a configuração é alterada.

**Para cada tabela**

Ao criar uma tabela, especifique a [configuração do mecanismo](/pt-BR/reference/engines/table-engines/mergetree-family/mergetree#table_engine-mergetree-creating-a-table) correspondente. O comportamento de uma tabela existente com essa configuração não muda, mesmo que a configuração global seja alterada.

**Valores possíveis**

* `0` — A funcionalidade está desativada.
* `1` — A funcionalidade está ativada.

Se [`use_minimalistic_part_header_in_zookeeper = 1`](#use_minimalistic_part_header_in_zookeeper), então as tabelas [replicadas](/pt-BR/reference/engines/table-engines/mergetree-family/replication) armazenam os cabeçalhos das partes de dados de forma compacta usando um único `znode`. Se a tabela contiver muitas colunas, esse método de armazenamento reduz significativamente o volume de dados armazenados no ZooKeeper.

<Note>
  Após aplicar `use_minimalistic_part_header_in_zookeeper = 1`, você não poderá fazer downgrade do servidor ClickHouse para uma versão que não ofereça suporte a essa configuração. Tenha cuidado ao atualizar o ClickHouse em servidores de um cluster. Não atualize todos os servidores de uma só vez. É mais seguro testar novas versões do ClickHouse em um ambiente de teste ou em apenas alguns servidores do cluster.

  Cabeçalhos de partes de dados já armazenados com essa configuração não podem ser restaurados para sua representação anterior (não compacta).
</Note>

<div id="use_shared_merge_tree_log_pipeline">
  ## use_shared_merge_tree_log_pipeline
</div>

<SettingsInfoBlock type="Bool" default_value="0" changeable_without_restart="Não" />

Disponível apenas no ClickHouse Cloud. Quando habilitado, as tabelas `system.*_log` usam `SharedMergeTree` por meio de um pipeline baseado em S3.
Para cada log `<log>`, os seguintes objetos são criados automaticamente na inicialização:

* `system.<log>_s3` — uma tabela com suporte a `S3` que é o destino direto do flush; cada
  chamada de `SYSTEM FLUSH LOGS` grava aqui um novo arquivo particionado.
* `system.<log>_s3queue` — uma tabela `S3Queue` (modo ordenado) que coleta arquivos de
  `<log>_s3` e encaminha as linhas adiante. Cada nó processa apenas seus próprios arquivos por meio de
  particionamento baseado em `partition_regex`.
* `system.<log>_mv` — uma `MATERIALIZED VIEW` que encaminha linhas de `<log>_s3queue` para
  a tabela final `SharedMergeTree`.
* `system.<log>` — a tabela final `SharedMergeTree`, onde as linhas se acumulam e são consultadas.

Em caso de alteração no esquema ou nas configurações, a tabela afetada é renomeada para `<log>_0`, `<log>_1` etc.
e recriada, em linha com o comportamento existente de rotação de `SystemLog`.

Exige que `<shared_log_pipeline><endpoint>` esteja definido na configuração do servidor.
Veja também: `shared_log_pipeline.enable_sync_flush`, `shared_log_pipeline.flush_timeout_seconds`.

| Configuração | Descrição                                                                                                                                                                                                                                                                                                                                                                                    |
| ------------ | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `roles`      | seção com uma lista de roles definidas localmente que serão atribuídas a cada usuário obtido do servidor LDAP. Se nenhuma role for especificada, o usuário não poderá executar nenhuma ação após a autenticação. Se alguma das roles listadas não estiver definida localmente no momento da autenticação, a tentativa de autenticação falhará como se a senha fornecida estivesse incorreta. |
| `server`     | um dos nomes de servidor LDAP definidos na seção de configuração `ldap_servers`. Este parâmetro é obrigatório e não pode estar vazio.                                                                                                                                                                                                                                                        |

**Exemplo**

```xml
<ldap>
    <server>my_ldap_server</server>
        <roles>
            <my_local_role1 />
            <my_local_role2 />
        </roles>
```

<div id="wait_dictionaries_load_at_startup">

</div>

<SettingsInfoBlock type="Bool" default_value="1" changeable_without_restart="Não" />

(porque terão de aguardar o carregamento de alguns dicionários).

Se `wait_dictionaries_load_at_startup` for `true`, o servidor aguardará, na inicialização,
até que todos os dicionários concluam o carregamento (com sucesso ou não) antes de aceitar qualquer conexão.

<div id="zookeeper">
  ## zookeeper
</div>

Contém configurações que permitem ao ClickHouse interagir com um cluster do [ZooKeeper](http://zookeeper.apache.org/). O ClickHouse usa o ZooKeeper para armazenar os metadados das réplicas ao usar tabelas replicadas. Se tabelas replicadas não forem usadas, esta seção de parâmetros pode ser omitida.

As configurações a seguir podem ser definidas por meio de subtags:

| Setting                                         | Description                                                                                                                                                                                                                                                                                                                                                                                                                    |
| ----------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| `node`                                          | Endpoint do ZooKeeper. Você pode definir vários endpoints. Ex.: `<node index="1"><host>example_host</host><port>2181</port></node>`. O atributo `index` especifica a ordem dos nós ao tentar se conectar ao cluster do ZooKeeper.                                                                                                                                                                                              |
| `operation_timeout_ms`                          | Timeout máximo de uma operação, em milissegundos.                                                                                                                                                                                                                                                                                                                                                                              |
| `session_timeout_ms`                            | Timeout máximo da sessão do cliente, em milissegundos.                                                                                                                                                                                                                                                                                                                                                                         |
| `root` (optional)                               | O znode usado como raiz para os znodes usados pelo servidor ClickHouse.                                                                                                                                                                                                                                                                                                                                                        |
| `fallback_session_lifetime.min` (optional)      | Limite mínimo para a duração de uma sessão do ZooKeeper no nó de fallback quando o primário não está disponível (balanceamento de carga). Definido em segundos. Padrão: 3 horas.                                                                                                                                                                                                                                               |
| `fallback_session_lifetime.max` (optional)      | Limite máximo para a duração de uma sessão do ZooKeeper no nó de fallback quando o primário não está disponível (balanceamento de carga). Definido em segundos. Padrão: 6 horas.                                                                                                                                                                                                                                               |
| `identity` (optional)                           | Usuário e senha exigidos pelo ZooKeeper para acessar os znodes solicitados.                                                                                                                                                                                                                                                                                                                                                    |
| `use_compression` (optional)                    | Habilita a compressão no protocolo Keeper se definido como true.                                                                                                                                                                                                                                                                                                                                                               |
| `use_xid_64` (optional)                         | Habilita IDs de transação de 64 bits. Defina como `true` para ativar o formato estendido de ID de transação. Padrão: `false`.                                                                                                                                                                                                                                                                                                  |
| `pass_opentelemetry_tracing_context` (optional) | Habilita a propagação do contexto de rastreamento do OpenTelemetry para solicitações ao Keeper. Quando habilitado, spans de rastreamento serão criados para operações do Keeper, permitindo rastreamento distribuído entre ClickHouse e Keeper. Veja [Tracing ClickHouse Keeper Requests](/pt-BR/guides/oss/deployment-and-scaling/monitoring/opentelemetry#tracing-clickhouse-keeper-requests) para mais detalhes. Padrão: `false`. |

Também há a configuração `zookeeper_load_balancing` (opcional), que permite selecionar o algoritmo de seleção de nós do ZooKeeper:

| Algorithm Name                   | Description                                                                                                                           |
| -------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------- |
| `random`                         | seleciona aleatoriamente um dos nós do ZooKeeper.                                                                                     |
| `in_order`                       | seleciona o primeiro nó do ZooKeeper; se ele não estiver disponível, seleciona o segundo, e assim por diante.                         |
| `nearest_hostname`               | seleciona um nó do ZooKeeper com um hostname mais semelhante ao hostname do servidor; o hostname é comparado com o prefixo do nome.   |
| `hostname_levenshtein_distance`  | assim como nearest&#95;hostname, mas compara o hostname usando a distância de Levenshtein.                                            |
| `hostname_longest_common_prefix` | assim como nearest&#95;hostname, mas prefere o nó cujo hostname compartilha o prefixo em comum mais longo com o hostname do servidor. |
| `hostname_longest_common_suffix` | assim como nearest&#95;hostname, mas prefere o nó cujo hostname compartilha o sufixo em comum mais longo com o hostname do servidor.  |
| `first_or_random`                | seleciona o primeiro nó do ZooKeeper; se ele não estiver disponível, seleciona aleatoriamente um dos nós restantes do ZooKeeper.      |
| `round_robin`                    | seleciona o primeiro nó do ZooKeeper; se houver reconexão, seleciona o próximo.                                                       |

**Configuração de exemplo**

```xml
<zookeeper>
    <node>
        <host>example1</host>
        <port>2181</port>
    </node>
    <node>
        <host>example2</host>
        <port>2181</port>
    </node>
    <session_timeout_ms>30000</session_timeout_ms>
    <operation_timeout_ms>10000</operation_timeout_ms>
    <!-- Optional. Chroot suffix. Should exist. -->
    <root>/path/to/zookeeper/node</root>
    <!-- Optional. Zookeeper digest ACL string. -->
    <identity>user:password</identity>
    <!--<zookeeper_load_balancing>random / in_order / nearest_hostname / hostname_levenshtein_distance / hostname_longest_common_prefix / hostname_longest_common_suffix / first_or_random / round_robin</zookeeper_load_balancing>-->
    <zookeeper_load_balancing>random</zookeeper_load_balancing>
    <!-- Optional. Enable 64-bit transaction IDs. -->
    <use_xid_64>false</use_xid_64>
    <!-- Optional. Enable OpenTelemetry tracing context propagation. -->
    <pass_opentelemetry_tracing_context>false</pass_opentelemetry_tracing_context>
</zookeeper>
```
