<div id="description">
  ## Descrição
</div>

[Apache Parquet](https://parquet.apache.org/) é um formato de armazenamento colunar amplamente utilizado no ecossistema Hadoop. O ClickHouse oferece suporte a operações de leitura e gravação nesse formato.

<div id="data-types-matching-parquet">

Ao gravar um arquivo Parquet, os tipos de dados que não têm um tipo Parquet correspondente são convertidos para o tipo disponível mais próximo:

| tipo de dado do ClickHouse                                      | tipo do Parquet                                     |
| --------------------------------------------------------------- | --------------------------------------------------- |
| [IPv4](/pt-BR/reference/data-types/ipv4)                              | `UINT_32`                                           |
| [IPv6](/pt-BR/reference/data-types/ipv6)                              | `FIXED_LEN_BYTE_ARRAY` (16 bytes)                   |
| [Date](/pt-BR/reference/data-types/date) (16 bits)                    | `DATE` (32 bits)                                    |
| [DateTime](/pt-BR/reference/data-types/datetime) (32 bits, segundos)  | `TIMESTAMP` (64 bits, milissegundos)                |
| [Int128/UInt128/Int256/UInt256](/pt-BR/reference/data-types/int-uint) | `FIXED_LEN_BYTE_ARRAY` (16/32 bytes, little-endian) |
| [Point](/pt-BR/reference/data-types/geo#point)                        | `BYTE_ARRAY` (WKB) + metadados do GeoParquet        |
| [LineString](/pt-BR/reference/data-types/geo#linestring)              | `BYTE_ARRAY` (WKB) + metadados do GeoParquet        |
| [Polygon](/pt-BR/reference/data-types/geo#polygon)                    | `BYTE_ARRAY` (WKB) + metadados do GeoParquet        |
| [MultiLineString](/pt-BR/reference/data-types/geo#multilinestring)    | `BYTE_ARRAY` (WKB) + metadados do GeoParquet        |
| [MultiPolygon](/pt-BR/reference/data-types/geo#multipolygon)          | `BYTE_ARRAY` (WKB) + metadados do GeoParquet        |

Arrays podem ser aninhados e podem ter um valor do tipo `Nullable` como argumento. Os tipos `Tuple` e `Map` também podem ser aninhados.

Os tipos de dados das colunas de tabelas do ClickHouse podem diferir dos campos correspondentes dos dados Parquet inseridos. Ao inserir dados, o ClickHouse interpreta os tipos de dados de acordo com a tabela acima e depois [converte](/pt-BR/reference/functions/regular-functions/type-conversion-functions#CAST) os dados para o tipo de dado definido para a coluna da tabela do ClickHouse. Ex.: uma coluna Parquet `UINT_32` pode ser lida em uma coluna ClickHouse [IPv4](/pt-BR/reference/data-types/ipv4).

Para alguns tipos Parquet, não há um tipo ClickHouse com correspondência próxima. Nós os lemos da seguinte forma:

* `TIME` (hora do dia) é lido como um timestamp. Ex.: `10:23:13.000` torna-se `1970-01-01 10:23:13.000`.
* `TIMESTAMP`/`TIME` com `isAdjustedToUTC=false` é um horário de relógio local (campos de ano, mês, dia, hora, minuto, segundo e fração de segundo em um timezone local, independentemente de qual fuso horário específico seja considerado local), assim como em SQL `TIMESTAMP WITHOUT TIME ZONE`. O ClickHouse o lê como se fosse um timestamp em UTC. Ex.: `2025-09-29 18:42:13.000` (representando a leitura de um relógio local) torna-se `2025-09-29 18:42:13.000` (`DateTime64(3, 'UTC')` representando um ponto no tempo). Se convertido para String, ele mostra o ano, mês, dia, hora, minuto, segundo e fração de segundo corretos, que então podem ser interpretados como pertencentes a algum timezone local em vez de UTC. Contraintuitivamente, mudar o tipo de `DateTime64(3, 'UTC')` para `DateTime64(3)` não ajudaria, pois ambos os tipos representam um ponto no tempo, e não uma leitura de relógio, mas `DateTime64(3)` seria formatado incorretamente usando o timezone local.
* `INTERVAL` no momento é lido como `FixedString(12)`, com a representação binária bruta do intervalo de tempo, conforme codificada no arquivo Parquet.

<div id="geo-types">
  ## Tipos geo (GeoParquet)
</div>

O ClickHouse oferece suporte à leitura e à gravação de colunas de geometria de acordo com a especificação [GeoParquet](https://geoparquet.org/). As colunas de geometria são armazenadas como payloads `BYTE_ARRAY` codificados em [WKB](https://libgeos.org/specifications/wkb/) (ou WKT na leitura), com uma chave JSON `geo` nos metadados Parquet em nível de arquivo que descreve a codificação, o tipo de geometria e o CRS de cada coluna de geometria.

<div id="read">
  ### Comportamento de leitura
</div>

Na leitura, as colunas de geometria são mapeadas para os [tipos de dados geo](/pt-BR/reference/data-types/geo) correspondentes no ClickHouse:

* Uma coluna declarada como `Point`, `LineString`, `Polygon`, `MultiLineString` ou `MultiPolygon` é lida como o tipo geo correspondente no ClickHouse.
* Uma coluna com tipos de geometria múltiplos ou desconhecidos é lida como o tipo [`Geometry`](/pt-BR/reference/data-types/geo#geometry), que é um `Variant` de todos os tipos geo compatíveis.
* Se o tipo de coluna solicitado for `String`, os metadados do GeoParquet serão ignorados e o payload bruto da geometria codificada será retornado como está — bytes WKB ou WKT, de acordo com a codificação declarada pela coluna do GeoParquet. Isso também se aplica se a configuração [`input_format_parquet_allow_geoparquet_parser`](/pt-BR/reference/settings/formats#input_format_parquet_allow_geoparquet_parser) estiver definida como `0`.

<div id="write">
  ### Comportamento de escrita
</div>

Na escrita, colunas de nível superior dos tipos `Point`, `LineString`, `Polygon`, `MultiLineString` ou `MultiPolygon` são codificadas como `BYTE_ARRAY` (WKB), e os metadados JSON `geo` correspondentes são acrescentados ao footer do arquivo Parquet. Um `Variant` [`Geometry`](/pt-BR/reference/data-types/geo#geometry) de nível superior também é codificado como uma carga útil `BYTE_ARRAY` WKB (seus subvalores são convertidos em WKB e armazenados como uma coluna `Nullable(String)`), mas nenhum metadado `geo` é emitido para ele, portanto o resultado não é reconhecido como uma coluna de geometria GeoParquet na leitura. Outros tipos relacionados a geo, como [`Ring`](/pt-BR/reference/data-types/geo#ring), são gravados usando sua representação nativa subjacente, sem metadados GeoParquet. Esse comportamento pode ser desativado por completo definindo [`output_format_parquet_geometadata`](/pt-BR/reference/settings/formats#output_format_parquet_geometadata) como `0`; nesse caso, até mesmo os tipos geo compatíveis são gravados usando sua representação nativa subjacente (`Point` como `Tuple(Float64, Float64)`, `LineString` como `Array(Point)`, `Polygon` como `Array(Array(Point))`, etc.), e nenhum metadado GeoParquet é emitido.

Colunas de geometria devem aparecer na raiz do schema ou aninhadas em `Tuple` (`struct`); não há suporte para aninhá-las em `Array` ou `Map`. `Nullable` também não tem suporte para colunas geo.

```sql

```

<div id="reading-data">
  ### Leitura de dados
</div>

Leia os dados no formato `Parquet`:

```sql
SELECT *
FROM football
INTO OUTFILE 'football.parquet'
FORMAT Parquet
```

<Tip>
  Parquet é um formato binário que não é exibido no terminal de forma legível para humanos. Use `INTO OUTFILE` para gerar arquivos Parquet.
</Tip>

Para trocar dados com o Hadoop, você pode usar o [`motor de tabela HDFS`](/pt-BR/reference/engines/table-engines/integrations/hdfs).
