<div id="description">
  ## Descrição
</div>

No formato TabSeparated, os dados são gravados por linha. Cada linha contém valores separados por tabulações. Cada valor é seguido por uma tabulação, exceto o último valor da linha, que é seguido por uma quebra de linha. Assume-se estritamente o uso de quebras de linha Unix em todos os lugares. A última linha também deve terminar com uma quebra de linha. Os valores são gravados em formato de texto, sem aspas e com caracteres especiais escapados.

Este formato também está disponível sob o nome `TSV`.

O formato `TabSeparated` é conveniente para processar dados com programas e scripts personalizados. Ele é usado por padrão na interface HTTP e no modo batch do cliente de linha de comando. Este formato também permite transferir dados entre diferentes SGBDs. Por exemplo, você pode obter um dump do MySQL e carregá-lo no ClickHouse, ou vice-versa.

O formato `TabSeparated` oferece suporte à saída de valores totais (ao usar WITH TOTALS) e valores extremos (quando &#39;extremes&#39; está definido como 1). Nesses casos, os valores totais e os extremos são exibidos após os dados principais. O resultado principal, os valores totais e os extremos são separados entre si por uma linha vazia. Exemplo:

```sql
SELECT EventDate, count() AS c FROM test.hits GROUP BY EventDate WITH TOTALS ORDER BY EventDate FORMAT TabSeparated

2014-03-17      1406958
2014-03-18      1383658
2014-03-19      1405797
2014-03-20      1353623
2014-03-21      1245779
2014-03-22      1031592
2014-03-23      1046491

1970-01-01      8873898

2014-03-17      1031592
2014-03-23      1406958
```

<div id="tabseparated-data-formatting">
  ## Formatação de dados
</div>

Números inteiros são escritos em formato decimal. Os números podem conter um caractere &quot;+&quot; adicional no início (ignorado durante o parsing e não registrado durante a formatação). Números não negativos não podem conter o sinal de menos. Na leitura, é permitido interpretar uma string vazia como zero ou (para tipos com sinal) uma string composta apenas por um sinal de menos como zero. Números que não cabem no tipo de dado correspondente podem ser interpretados como um número diferente, sem mensagem de erro.

Números de ponto flutuante são escritos em formato decimal. O ponto é usado como separador decimal. Há suporte para notação exponencial, bem como para &#39;inf&#39;, &#39;+inf&#39;, &#39;-inf&#39; e &#39;nan&#39;. Um valor de número de ponto flutuante pode começar ou terminar com um ponto decimal.
Durante a formatação, pode haver perda de precisão em números de ponto flutuante.
Durante o parsing, não é estritamente necessário ler o número representável pela máquina mais próximo.

Datas são escritas no formato YYYY-MM-DD e interpretadas no mesmo formato, mas com quaisquer caracteres como separadores.
Datas com horário são escritas no formato `YYYY-MM-DD hh:mm:ss` e interpretadas no mesmo formato, mas com quaisquer caracteres como separadores.
Tudo isso ocorre no fuso horário do sistema no momento em que o cliente ou o servidor é iniciado (dependendo de qual deles formata os dados). Para datas com horário, o horário de verão não é especificado. Portanto, se um dump tiver horários durante o horário de verão, o dump não corresponderá de forma inequívoca aos dados, e o parsing selecionará um dos dois horários.
Durante uma operação de leitura, datas inválidas e datas com horário podem ser interpretadas com overflow natural ou como datas e horários nulos, sem mensagem de erro.

Como exceção, também há suporte para interpretar datas com horário no formato Unix timestamp, se ele consistir em exatamente 10 dígitos decimais. O resultado não depende do fuso horário. Os formatos `YYYY-MM-DD hh:mm:ss` e `NNNNNNNNNN` são diferenciados automaticamente.

Strings são geradas com caracteres especiais escapados com barra invertida. As seguintes sequências de escape são usadas na saída: `\b`, `\f`, `\r`, `\n`, `\t`, `\0`, `\'`, `\\`. O parsing também oferece suporte às sequências `\a`, `\v` e `\xHH` (sequências de escape hexadecimais) e a quaisquer sequências `\c`, em que `c` é qualquer caractere (essas sequências são convertidas em `c`). Assim, a leitura de dados oferece suporte a formatos em que uma quebra de linha pode ser escrita como `\n`, como `\`, ou como uma quebra de linha real. Por exemplo, a string `Hello world` com uma quebra de linha entre as palavras em vez de um espaço pode ser interpretada em qualquer uma das seguintes variações:

```text
Hello\nworld

Hello\
world
```

A segunda variante é compatível porque o MySQL a usa ao gravar dumps separados por tabulação.

O conjunto mínimo de caracteres que você precisa escapar ao passar dados no formato TabSeparated é: tabulação, quebra de linha (LF) e barra invertida.

Apenas um pequeno conjunto de símbolos é escapado. Você pode facilmente se deparar com um valor String que seu terminal acabará corrompendo na saída.

Arrays são gravados como uma lista de valores separados por vírgula em `[]`. Os itens Number no array são formatados normalmente. Os tipos `Date` e `DateTime` são gravados entre aspas simples. Strings são gravadas entre aspas simples, com as mesmas regras de escape acima.

[NULL](/pt-BR/reference/syntax) é formatado de acordo com a configuração [format&#95;tsv&#95;null&#95;representation](/pt-BR/reference/settings/formats#format_tsv_null_representation) (o valor padrão é `\N`).

Nos dados de entrada, os valores ENUM podem ser representados como nomes ou como IDs. Primeiro, tentamos corresponder o valor de entrada ao nome do ENUM. Se não conseguirmos e o valor de entrada for um número, tentamos corresponder esse número ao ID do ENUM.
Se os dados de entrada contiverem apenas IDs de ENUM, recomenda-se ativar a configuração [input&#95;format&#95;tsv&#95;enum&#95;as&#95;number](/pt-BR/reference/settings/formats#input_format_tsv_enum_as_number) para otimizar o parsing de ENUM.

Cada elemento das estruturas [Nested](/pt-BR/reference/data-types/nested-data-structures/index) é representado como um array.

Por exemplo:

```sql
CREATE TABLE nestedt
(
    `id` UInt8,
    `aux` Nested(
        a UInt8,
        b String
    )
)
ENGINE = TinyLog
```

```sql
INSERT INTO nestedt VALUES ( 1, [1], ['a'])
```

```sql
SELECT * FROM nestedt FORMAT TSV
```

```response
1  [1]    ['a']
```

| ------------------------------------------------------------------------------------------------------------------------------------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ | ------- |
| [`format_tsv_null_representation`](/pt-BR/reference/settings/formats#format_tsv_null_representation)                                             | Representação personalizada de NULL no formato TSV.                                                                                                                                                                                                          | `\N`    |
| [`input_format_tsv_empty_as_default`](/pt-BR/reference/settings/formats#input_format_tsv_empty_as_default)                                       | trata campos vazios na entrada TSV como valores padrão. Para expressões padrão complexas, [input&#95;format&#95;defaults&#95;for&#95;omitted&#95;fields](/pt-BR/reference/settings/formats#input_format_defaults_for_omitted_fields) também deve estar habilitada. | `false` |
| [`input_format_tsv_enum_as_number`](/pt-BR/reference/settings/formats#input_format_tsv_enum_as_number)                                           | trata valores de enum inseridos em formatos TSV como índices de enum.                                                                                                                                                                                        | `false` |
| [`input_format_tsv_use_best_effort_in_schema_inference`](/pt-BR/reference/settings/formats#input_format_tsv_use_best_effort_in_schema_inference) | usa alguns ajustes e heurísticas para inferir o schema no formato TSV. Se estiver desabilitada, todos os campos serão inferidos como Strings.                                                                                                                | `true`  |

| [`input_format_tsv_skip_first_lines`](/pt-BR/reference/settings/formats#input_format_tsv_skip_first_lines)                                       | ignora o número especificado de linhas no início dos dados.                                                                                                                                                                                                  | `0`     |
| [`input_format_tsv_detect_header`](/pt-BR/reference/settings/formats#input_format_tsv_detect_header)                                             | detecta automaticamente o cabeçalho com nomes e tipos no formato TSV.                                                                                                                                                                                        | `true`  |
| [`input_format_tsv_skip_trailing_empty_lines`](/pt-BR/reference/settings/formats#input_format_tsv_skip_trailing_empty_lines)                     | ignora linhas vazias no final dos dados.                                                                                                                                                                                                                     | `false` |
| [`input_format_tsv_allow_variable_number_of_columns`](/pt-BR/reference/settings/formats#input_format_tsv_allow_variable_number_of_columns)       | permite um número variável de colunas no formato TSV, ignora colunas extras e usa valores padrão para colunas ausentes.                                                                                                                                      | `false` |
