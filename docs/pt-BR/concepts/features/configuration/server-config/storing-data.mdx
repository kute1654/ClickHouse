Os dados processados no ClickHouse geralmente sГЈo armazenados no sistema de arquivos local da

<Note>
  O ClickHouse tambГ©m oferece suporte a motores de tabela externos, que sГЈo diferentes da

  a configuraГ§ГЈo de armazenamento para tabelas da famГӯlia `MergeTree` ou da famГӯlia `Log` do ClickHouse.

  1. para trabalhar com dados armazenados em discos `Amazon S3`, use o motor de tabela [S3](/pt-BR/reference/engines/table-engines/integrations/s3).
  2. para trabalhar com dados armazenados no Azure Blob Storage, use o motor de tabela [AzureBlobStorage](/pt-BR/reference/engines/table-engines/integrations/azureBlobStorage).
  3. para trabalhar com dados no Hadoop Distributed File System (sem suporte), use o motor de tabela [HDFS](/pt-BR/reference/engines/table-engines/integrations/hdfs).

<div id="refresh-parts-interval-and-table-disk">
  ## refresh_parts_interval and table_disk
</div>

Esta configuraГ§ГЈo se destina a tabelas MergeTree nГЈo replicadas, nas quais as partes podem ser gravadas externamente e a descoberta de metadados precisa ser atualizada a partir do armazenamento.

A configuraГ§ГЈo do MergeTree `refresh_parts_interval` permite atualizar periodicamente a lista de partes de dados a partir do armazenamento subjacente (por exemplo, para reconhecer partes gravadas externamente). Usar apenas armazenamento de objetos nГЈo implica metadados compartilhados.

* **Armazenamento de objetos (por exemplo, `disk = 's3'`) nГЈo implica metadados compartilhados.** Quando os metadados sГЈo armazenados localmente por rГ©plica (o padrГЈo), cada rГ©plica gerencia de forma independente seus ponteiros para blobs no armazenamento de objetos. AlteraГ§Гөes feitas em uma rГ©plica nГЈo ficam visГӯveis para as outras. Nesse caso, `refresh_parts_interval` nГЈo torna novas partes visГӯveis entre rГ©plicas, porque os metadados que cada rГ©plica lГӘ sГЈo locais Г  prГіpria rГ©plica.

* **A atualizaГ§ГЈo automГЎtica de partes exige que os metadados do sistema de arquivos sejam compartilhados** (ou que a tabela use metadados somente leitura pertencentes Г  prГіpria tabela, para que a atualizaГ§ГЈo se aplique). * **Com um disco definido globalmente** (por exemplo, `disk = 's3'` em `storage_configuration`) e metadados locais padrГЈo, cada rГ©plica tem seu prГіprio estado de metadados. Embora os blobs possam estar no S3, o armazenamento nГЈo Г© considerado compartilhado para fins de `refresh_parts_interval`, e novas partes criadas fora do ClickHouse ou em outra rГ©plica nГЈo serГЈo detectadas.

Para a atualizaГ§ГЈo automГЎtica de partes, garanta que os metadados sejam compartilhados ou use um disco no nГӯvel da tabela com `table_disk = true`, como acima. Confiar apenas em `refresh_parts_interval` com metadados locais Г  rГ©plica nГЈo atualizarГЎ as partes como esperado.

<Note>
  `refresh_parts_interval` nГЈo Г© usado para tabelas ReplicatedMergeTree.
  As tabelas replicadas jГЎ sincronizam partes por meio do mecanismo de replicaГ§ГЈo.
  Esta configuraГ§ГЈo se aplica apenas a tabelas MergeTree nГЈo replicadas nas quais as partes sГЈo gravadas externamente e a atualizaГ§ГЈo de metadados Г© necessГЎria.
