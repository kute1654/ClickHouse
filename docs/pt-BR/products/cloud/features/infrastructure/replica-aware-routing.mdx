O roteamento com reconhecimento de réplicas (também conhecido como sticky sessions, sticky routing ou afinidade de sessão) aumenta a chance de reutilização de cache ao rotear de forma consistente as solicitações relacionadas para a mesma réplica do ClickHouse. Isso não garante isolamento.

<Warning>
  **Suporte de protocolo — somente HTTP/HTTPS**

  O roteamento com reconhecimento de réplicas é aplicado na camada de proxy sobre a **interface HTTP/HTTPS**, usando o parâmetro de consulta `session_id` (veja abaixo). Ele **não está disponível no protocolo nativo** (porta nativa, por exemplo, o driver `clickhouse-go` em seu modo nativo padrão). Se você precisar de sticky routing a partir de um cliente com protocolo nativo, conecte essa carga de trabalho pela interface HTTP — para `clickhouse-go` (v2), defina `Protocol: clickhouse.HTTP` na conexão.
</Warning>

<div id="http-based-routing">
  ## Roteamento baseado em HTTP (`session_id`)
</div>

Para fixar uma carga de trabalho em uma réplica, defina o parâmetro de consulta `session_id` na interface HTTPS. O proxy analisa a solicitação e usa hash consistente no `session_id` para selecionar uma réplica, de modo que todas as solicitações com o mesmo `session_id` sejam roteadas para o mesmo servidor — até que a topologia do cluster mude.

```bash
echo 'SELECT hostName()' | curl \
  -H 'X-ClickHouse-User: default' -H 'X-ClickHouse-Key: <password>' \
  'https://<host>:8443/?session_id=my-workload-1' -d @-
```

Toda solicitação com `session_id=my-workload-1` vai para a mesma réplica. Um valor de `session_id` diferente passa por hash de forma independente e pode ir para a mesma réplica ou para outra réplica — o mapeamento é consistente, mas você não escolhe *para qual* réplica um determinado valor é mapeado.

<div id="subdomain-based-routing-deprecated">
  ## Roteamento baseado em subdomínio (descontinuado)
</div>

<Danger>
  **Descontinuado**

  O mecanismo baseado em subdomínio abaixo está **descontinuado** e não é mais habilitado em novos serviços. Ele não escala (cada endpoint sticky requer seu próprio certificado TLS). Use o [método `session_id` baseado em HTTP](#http-based-routing) no lugar.
</Danger>

Anteriormente, habilitar o roteamento com reconhecimento de réplicas permitia usar um subdomínio curinga sobre o hostname do serviço. Para um serviço com o host name `abcxyz123.us-west-2.aws.clickhouse.cloud`, qualquer hostname correspondente a `*.sticky.abcxyz123.us-west-2.aws.clickhouse.cloud` (por exemplo, `aaa.sticky.abcxyz123.us-west-2.aws.clickhouse.cloud`) era mapeado pelo Envoy, via hash, para uma réplica consistente. O hostname original continuava usando balanceamento de carga `LEAST_CONNECTION`, o algoritmo de roteamento padrão.

<div id="limitations-of-replica-aware-routing">
  ## Limitações do roteamento com reconhecimento de réplicas
</div>

<div id="replica-aware-routing-does-not-guarantee-isolation">
  ### Replica-aware routing não garante isolamento
</div>

Qualquer interrupção no serviço — reinicializações de pods do Kubernetes do servidor (upgrade de versão, falha, escala vertical) ou aumento/redução de escala — altera o anel de hash usado no roteamento. Isso faz com que requisições que compartilham o mesmo `session_id` sejam direcionadas para um pod do Kubernetes do servidor diferente.

<div id="replica-aware-routing-does-not-work-out-of-the-box-with-private-link">
  ### O roteamento com reconhecimento de réplicas não funciona por padrão com Private Link
</div>

Ao usar o método de subdomínio descontinuado, os clientes precisam adicionar manualmente uma entrada DNS para que a resolução de nomes funcione com o novo padrão de hostname. Se usado incorretamente, isso pode causar desequilíbrio na carga entre os servidores.

<div id="replica-aware-routing-requires-http">
  ### O roteamento com reconhecimento de réplica exige o protocolo HTTP
</div>

O roteamento sticky usa como chave o parâmetro de consulta `session_id`, que existe apenas na interface HTTP/HTTPS. O protocolo binário nativo não transporta esse parâmetro para que o proxy possa aplicar hash e, por isso, o roteamento com reconhecimento de réplica não está disponível no protocolo nativo. Clientes do protocolo nativo precisam mover a carga de trabalho relevante para a interface HTTP para usar esse recurso.
