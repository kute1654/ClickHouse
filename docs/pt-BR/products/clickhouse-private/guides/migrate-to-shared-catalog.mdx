O Shared Catalog é a nova abordagem para gerenciar metadados em instâncias ClickHouse Private. Em vez de usar bancos de dados com o engine [Replicated](https://clickhouse.com/docs/engines/database-engines/replicated), que utiliza uma DDL queue no ClickHouse Keeper para garantir que as instruções DDL sejam aplicadas em todas as réplicas, o engine de banco de dados [Shared](https://clickhouse.com/docs/engines/database-engines/shared) armazena todos os metadados no próprio Keeper, eliminando a necessidade de sincronizar instruções DDL entre as réplicas.

O Shared Catalog oferece vários benefícios:

* Estado consistente em todas as réplicas
* Nós de compute sem estado (permite nós do servidor sem estado)
* `CREATE TABLE ... AS SELECT` atômico
* Suporte a `UNDROP`
* `RENAME`/mover `TABLE` entre bancos de dados
* Inicialização rápida e confiável de réplicas (reduz os tempos de ativação, inicialização e provisionamento)

No entanto, isso exige uma migração se a sua instância não tiver sido configurada com o Shared Catalog na implantação inicial. Recomendamos testar primeiro a migração em um ambiente de teste antes de migrar instâncias de produção.

<div id="migration-requirements">
  ## Requisitos da migração
</div>

Antes de iniciar a migração, verifique os seguintes pré-requisitos:

<Steps>
  <Step>
    ### 1. Pelo menos 2 réplicas do servidor

    O Operator executa uma atualização gradual para migrar os bancos de dados para o novo Shared database engine. Defina o valor `server.replicas` do helm como 2 ou mais.
  </Step>

  <Step>
    ### 2. Todos os bancos de dados usam o engine Replicated

    Todos os bancos de dados (exceto `system`) devem usar o engine de banco de dados Replicated. Verifique com:

    ```sql
    SELECT DISTINCT
        name,
        engine
    FROM clusterAllReplicas('default', system.databases)
    WHERE (name NOT IN ('INFORMATION_SCHEMA', 'system', 'information_schema'))
      AND (engine != 'Replicated')
    ```

    A consulta não deve retornar nenhuma linha.
  </Step>

  <Step>
    ### 3. Todas as tabelas usam motores de tabela não replicados

    Nenhuma tabela (exceto em `system`) deve usar motores de tabela replicados ou legados. Verifique com:

    ```sql
    SELECT
        name,
        engine,
        database
    FROM system.tables
    WHERE ((engine LIKE '%Replicated%') OR (engine IN ('Ordinary', 'Atomic')))
      AND (database != 'system')
    ```

    A consulta não deve retornar nenhuma linha.
  </Step>

  <Step>
    ### 4. Não há tabelas desanexadas

    ```sql
    SELECT * FROM system.detached_tables
    ```

    A consulta não deve retornar nenhuma linha.

    Quando todos os requisitos forem atendidos, a migração poderá ser iniciada. Se algum requisito não for atendido, entre em contato com o suporte do ClickHouse para obter ajuda.
  </Step>
</Steps>

<div id="performing-the-migration">
  ## Realizando a migração
</div>

<Warning>
  Para instâncias Hydra, migre primeiro a instância principal antes de migrar quaisquer instâncias filhas. Repita as etapas abaixo para cada instância filha depois que a principal for migrada.
</Warning>

<Steps>
  <Step>
    ### 1. Habilite a configuração do Shared Catalog

    Defina `featureFlags.enableSharedCatalog` como `true` nos valores do Helm. Isso habilita a configuração, mas não aciona a migração dos bancos de dados. Implantar um ClickHouse Cluster com essa flag definida como `true`, sem migrar os bancos de dados `Replicated` existentes, fará com que o ClickHouse Cluster não consiga iniciar.
  </Step>

  <Step>
    ### 2. Adicione os marcadores de migração

    Defina `featureFlags.migrateToSharedCatalog` como `true` nos valores do Helm. Isso:

    * Adiciona a configuração `shared_database_catalog.migration_from_database_replicated` ao ClickHouse, informando que ele deve esperar bancos de dados `Replicated` existentes enquanto opera no modo Shared Catalog
    * Adiciona a anotação `clickhouse.com/start-shared-catalog-migration: "true"` ao recurso ClickHouseCluster do Kubernetes, notificando o Operator para migrar este cluster

    Depois da implantação, o Operator acionará uma atualização gradual dos nós do servidor. Isso também alterará os tipos de todos os bancos de dados para `Shared`.
  </Step>

  <Step>
    ### 3. Aguarde até que o status seja Ready

    ```bash
    kubectl get ClickHouseCluster <instance-name>
    ```
  </Step>

  <Step>
    ### 4. Remova os marcadores de migração

    Quando o status estiver Ready, defina `featureFlags.migrateToSharedCatalog` como `false` nos valores do Helm. Como todos os bancos de dados já foram migrados para o tipo `Shared`, em vez de `Replicated`, as configurações de migração não são mais relevantes.
  </Step>

  <Step>
    ### 5. Verifique se todos os bancos de dados foram migrados

    Confirme que todos os bancos de dados que não são do sistema usam o engine `Shared`:

    ```sql
    SELECT DISTINCT
        name,
        engine
    FROM clusterAllReplicas('default', system.databases)
    WHERE (name NOT IN ('INFORMATION_SCHEMA', 'system', 'information_schema'))
      AND (engine != 'Shared')
    ```

    A consulta não deve retornar nenhuma linha. Neste ponto, sua instância foi migrada para o Shared Catalog.
  </Step>
</Steps>

<div id="rollback">
  ## Reversão
</div>

Se houver qualquer problema com o Shared Catalog, você pode reverter para o estado anterior:

<Steps>
  <Step>
    ### 1. Remova os marcadores de migração

    Defina `featureFlags.migrateToSharedCatalog` como `false` nos valores do Helm.
  </Step>

  <Step>
    ### 2. Remova as configurações do Shared Catalog

    Defina `featureFlags.enableSharedCatalog` como `false` nos valores do Helm. Isso fará com que o Operator reverta as alterações.
  </Step>

  <Step>
    ### 3. Remova o catálogo no Keeper

    Crie um redirecionamento de porta para o Keeper e inicie uma sessão do keeper-client (usando `clickhouse keeper-client`).

    Depois que a sessão for iniciada, execute os seguintes comandos:

    ```text
    ls '/clickhouse/catalog'
    # Saída: migration_from_database_replicated_completed names references replicas uuids

    rmr '/clickhouse/catalog'
    # Você está prestes a excluir recursivamente o caminho /clickhouse/catalog. Continuar?
    # [y/n] y
    ```

    Isso limpará todo o estado do Shared Catalog no Keeper.
  </Step>

  <Step>
    ### 4. Verifique a reversão

    Confirme que todos os bancos de dados que não são do sistema estão usando o engine Replicated novamente:

    ```sql
    SELECT DISTINCT
        name,
        engine
    FROM clusterAllReplicas('default', system.databases)
    WHERE (name NOT IN ('INFORMATION_SCHEMA', 'system', 'information_schema'))
      AND (engine != 'Replicated')
    ```

    A consulta não deve retornar nenhuma linha.
  </Step>
</Steps>
