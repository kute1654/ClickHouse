<div id="operator-node-group-x86">
  #### Grupo de nós do operador (x86)
</div>

| Propriedade                 | Valor                                                                                             |
| --------------------------- | ------------------------------------------------------------------------------------------------- |
| AMI                         | `AL2023_x86_64`                                                                                   |
| Tamanho do disco            | 20 GB                                                                                             |
| Tamanho mínimo de instância | xlarge                                                                                            |
| Tipo de instância           | Qualquer tipo compatível com x86                                                                  |

**Todos os nós da AWS exigem [IMDS](https://docs.aws.amazon.com/AWSEC2/latest/UserGuide/instancedata-data-retrieval.html) para autenticação.**

Para os grupos de nós do Keeper e do servidor, **crie um grupo de nós por AZ** se quiser oferecer suporte ao cluster autoscaler entre AZs.

Tags do autoscaler (se estiver usando cluster-autoscaler): - `k8s.io/cluster-autoscaler/enabled: true` - `k8s.io/cluster-autoscaler/$CLUSTER_NAME: owned`

<div id="aws">
  ### AWS
</div>

| Recurso                  | Detalhes                                                                                                                               |
| ------------------------ | -------------------------------------------------------------------------------------------------------------------------------------- |

| VolumeSnapshot CRDs      | Obrigatórios para o operador                                                                                                           |

<div id="iam-service-account-requirements">
  ## Requisitos de IAM / conta de serviço
</div>

<div id="aws">
  ### AWS
</div>

<div id="ecr-pull-role">
  #### Função de pull do ECR
</div>

* **Nome da função:** `ClickHouseAirgapECRPullRole`
* **Finalidade:** Baixar imagens do ECR privado da ClickHouse
* **Permissões:** `ecr:GetDownloadUrlForLayer`, `ecr:BatchGetImage`, `ecr:BatchCheckLayerAvailability`, `ecr:DescribeImages`, `ecr:ListImages`, `ecr:GetAuthorizationToken`
* **Ação:** Forneça o ARN da função à equipe da ClickHouse

<div id="clickhouse-serverkeeper-s3-role">
  #### Função de S3 do ClickHouse Server/Keeper
</div>

* **Uma função por cluster do ClickHouse provisionado**
* **Convenção de nomenclatura:** `CH-S3-$NAME-$REGION-$ORDINAL-Role`
* `$NAME`: nome do cluster (por exemplo, `default-xx-01`)
* `$REGION`: região abreviada (por exemplo, `uw2` para `us-west-2`)
* `$ORDINAL`: reservado, definido como `00`
* Exemplo: `CH-S3-default-xx-01-uw2-00-Role`
* **Permissões:** `s3:*`, `s3:ListBucket` no recurso do bucket
* **Política de confiança:** IRSA via `sts:AssumeRoleWithWebIdentity` com o provedor OIDC do cluster EKS

<div id="ebs-csi-driver-role">
  #### Função do driver CSI do Amazon EBS
</div>

* **Nome de exemplo:** `ClickHouse_EksEbsCsiDriverRole`
* **Política gerenciada:** `arn:aws:iam::aws:policy/service-role/AmazonEBSCSIDriverPolicy`
* **Política de confiança:** IRSA com o provedor OIDC do cluster EKS

<div id="clickhouse-workload-identity-service-account">
  #### Conta de serviço do Workload Identity do ClickHouse
</div>

* **SA do GCP:** `clickhouse@$PROJECT.iam.gserviceaccount.com`
* **SA do K8s (manual):** `clickhouse` no espaço de nomes `ns-$CLUSTER_NAME`
* **SA do K8s (criada pelo helm):** `ch-$CLUSTER_NAME-sa` no espaço de nomes `ns-$CLUSTER_NAME`
* **Anotação da SA do K8s:** `iam.gke.io/gcp-service-account=<GSA_EMAIL>`
* **Membro da vinculação do IAM:** `serviceAccount:$PROJECT.svc.id.goog[<namespace>/<ksa>]`
* **Permissões do bucket:** `roles/storage.objectAdmin`, `roles/storage.legacyBucketReader`
* **Permissões do Artifact Registry:** `roles/artifactregistry.reader`

  ### Bare Metal
</div>

* **Credenciais da AWS:** chave de acesso e chave secreta para pull de imagens do ECR

<div id="cluster-naming-conventions">
  ## Convenções de nomenclatura de clusters
</div>

A convenção de nomenclatura dos clusters do ClickHouse é a mesma em todas as plataformas:

```text
$DESCRIPTOR-$LETTERS-$ORDINAL
```

| Componente    | Descrição                           | Exemplo   |
| ------------- | ----------------------------------- | --------- |
| `$DESCRIPTOR` | Nome descritivo (somente letras)    | `default` |
| `$LETTERS`    | Reservado; duas letras quaisquer    | `xx`      |
| `$ORDINAL`    | Ordinal sequencial a partir de `01` | `01`      |

**Nome de exemplo:** `default-xx-01`

O nome do cluster deve ser **único no cluster do Kubernetes de destino** e é usado em: - Espaço de nomes do Kubernetes: `ns-$CLUSTER_NAME` - Recursos do Kubernetes: com o prefixo `c-$CLUSTER_NAME` - Nomes de funções do IAM (AWS): `CH-S3-$CLUSTER_NAME-$REGION-$ORDINAL-Role` - Prefixos de chave no S3/GCS: `ch-s3-<uuid>` (único por cluster)
