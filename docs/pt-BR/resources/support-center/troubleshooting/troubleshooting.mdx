<div id="installation">
  ## Instalação
</div>

<div id="cant-import-gpg-keys-from-keyserverubuntucom-with-apt-key">
  ### Não é possível importar chaves GPG de keyserver.ubuntu.com com apt-key
</div>

A funcionalidade `apt-key` do [Advanced package tool (APT) foi descontinuada](https://manpages.debian.org/bookworm/apt/apt-key.8.en.html). Em vez dela, use o comando `gpg`. Consulte o artigo [guia de instalação](/pt-BR/get-started/setup/install).

<div id="cant-import-gpg-keys-from-keyserverubuntucom-with-gpg">
  ### Não é possível importar as chaves GPG de keyserver.ubuntu.com com o gpg
</div>

1. Verifique se o `gpg` está instalado:

```shell
sudo apt-get install gnupg
```

<div id="cant-get-deb-packages-from-clickhouse-repository-with-apt-get">
  ### Não é possível obter pacotes .deb do repositório do ClickHouse com apt-get
</div>

1. Verifique as configurações do firewall.
2. Se você não conseguir acessar o repositório por qualquer motivo, baixe os pacotes conforme descrito no artigo [guia de instalação](/pt-BR/get-started/setup/install) e instale-os manualmente com o comando `sudo dpkg -i <packages>`. Você também precisará do pacote `tzdata`.

<div id="cant-update-deb-packages-from-clickhouse-repository-with-apt-get">
  ### Não é possível atualizar pacotes deb do repositório do ClickHouse com apt-get
</div>

Esse problema pode ocorrer quando a chave GPG é alterada.

Use as instruções na página de [configuração](/pt-BR/get-started/setup/self-managed/debian-ubuntu) para atualizar a configuração do repositório.

<div id="you-get-different-warnings-with-apt-get-update">
  ### Você recebe avisos diferentes com `apt-get update`
</div>

As mensagens de aviso completas podem ser uma das seguintes:

```shell
N: Skipping acquire of configured file 'main/binary-i386/Packages' as repository 'https://packages.clickhouse.com/deb stable InRelease' doesn't support architecture 'i386'
```

```shell
E: Failed to fetch https://packages.clickhouse.com/deb/dists/stable/main/binary-amd64/Packages.gz  File has unexpected size (30451 != 28154). Mirror sync in progress?
```

```shell
E: Repository 'https://packages.clickhouse.com/deb stable InRelease' changed its 'Origin' value from 'Artifactory' to 'ClickHouse'
E: Repository 'https://packages.clickhouse.com/deb stable InRelease' changed its 'Label' value from 'Artifactory' to 'ClickHouse'
N: Repository 'https://packages.clickhouse.com/deb stable InRelease' changed its 'Suite' value from 'stable' to ''
N: This must be accepted explicitly before updates for this repository can be applied. See apt-secure(8) manpage for details.
```

```shell
Err:11 https://packages.clickhouse.com/deb stable InRelease
400  Bad Request [IP: 172.66.40.249 443]
```

Para resolver esse problema, use o script a seguir:

```shell
sudo rm /var/lib/apt/lists/packages.clickhouse.com_* /var/lib/dpkg/arch /var/lib/apt/lists/partial/packages.clickhouse.com_*
sudo apt-get clean
sudo apt-get autoclean
```

<div id="cant-get-packages-with-yum-because-of-wrong-signature">
  ### Não é possível obter pacotes com o Yum devido a uma assinatura incorreta
</div>

Possível problema: o cache está incorreto; talvez tenha sido corrompido após a atualização da chave GPG em 2022-09.

A solução é limpar o cache e o diretório lib do Yum:

```shell
sudo find /var/lib/yum/repos/ /var/cache/yum/ -name 'clickhouse-*' -type d -exec rm -rf {} +
sudo rm -f /etc/yum.repos.d/clickhouse.repo
```

Depois, siga o [guia de instalação](/pt-BR/get-started/setup/self-managed/redhat)

<div id="connecting-to-the-server">
  ## Conexão com o servidor
</div>

Possíveis problemas:

* O servidor não está em execução.
* Parâmetros de configuração inesperados ou incorretos.

<div id="server-is-not-running">
  ### O servidor não está em execução
</div>

<div id="check-if-server-is-running">
  #### Verifique se o servidor está em execução
</div>

```shell
sudo service clickhouse-server status
```

Se o servidor não estiver em execução, inicie-o com o comando:

```shell
sudo service clickhouse-server start
```

<div id="check-the-logs">
  #### Verifique os logs
</div>

O log principal do `clickhouse-server` fica em `/var/log/clickhouse-server/clickhouse-server.log` por padrão.

Se o servidor tiver sido iniciado com sucesso, você deverá ver as seguintes strings:

* `<Information> Application: starting up.` — Servidor iniciado.
* `<Information> Application: Ready for connections.` — O servidor está em execução e pronto para receber conexões.

Se a inicialização do `clickhouse-server` tiver falhado devido a um erro de configuração, você deverá ver a string `<Error>` com a descrição do erro. Por exemplo:

```plaintext
2019.01.11 15:23:25.549505 [ 45 ] {} <Error> ExternalDictionaries: Failed reloading 'event2id' external dictionary: Poco::Exception. Code: 1000, e.code() = 111, e.displayText() = Connection refused, e.what() = Connection refused
```

Se você não encontrar um erro no final do arquivo, examine o arquivo inteiro a partir da string:

```plaintext
<Information> Application: starting up.
```

Se você tentar iniciar uma segunda instância do `clickhouse-server` no servidor, verá o seguinte log:

```plaintext
2019.01.11 15:25:11.151730 [ 1 ] {} <Information> : Starting ClickHouse 19.1.0 with revision 54413
2019.01.11 15:25:11.154578 [ 1 ] {} <Information> Application: starting up
2019.01.11 15:25:11.156361 [ 1 ] {} <Information> StatusFile: Status file ./status already exists - unclean restart. Contents:
PID: 8510
Started at: 2019-01-11 15:24:23
Revision: 54413

2019.01.11 15:25:11.156673 [ 1 ] {} <Error> Application: DB::Exception: Cannot lock file ./status. Another server instance in same directory is already running.
2019.01.11 15:25:11.156682 [ 1 ] {} <Information> Application: shutting down
2019.01.11 15:25:11.156686 [ 1 ] {} <Debug> Application: Uninitializing subsystem: Logging Subsystem
2019.01.11 15:25:11.156716 [ 2 ] {} <Information> BaseDaemon: Stop SignalListener thread
```

<div id="see-systemd-logs">
  #### Ver os logs do system.d
</div>

Se você não encontrar nenhuma informação útil nos logs do `clickhouse-server` ou se não houver logs, poderá ver os logs do `system.d` com o comando:

```shell
sudo journalctl -u clickhouse-server
```

<div id="start-clickhouse-server-in-interactive-mode">
  #### Inicie o clickhouse-server em modo interativo
</div>

```shell
sudo -u clickhouse /usr/bin/clickhouse-server --config-file /etc/clickhouse-server/config.xml
```

Este comando inicia o servidor como uma aplicação interativa com os parâmetros padrão do script de inicialização automática. Nesse modo, `clickhouse-server` imprime todas as mensagens de eventos no console.

<div id="configuration-parameters">
  ### Parâmetros de configuração
</div>

Verifique:

1. Configurações do Docker:

   * Se você executar o ClickHouse no Docker em uma rede IPv6, certifique-se de que `network=host` esteja definido.

2. Configurações do endpoint.
   * Verifique as configurações [listen&#95;host](/pt-BR/reference/settings/server-settings/settings#listen_host) e [tcp&#95;port](/pt-BR/reference/settings/server-settings/settings#tcp_port).
   * Por padrão, o servidor ClickHouse aceita conexões apenas via localhost.

3. Configurações do protocolo HTTP:

   * Verifique as configurações de protocolo da API HTTP.

4. Configurações de conexão segura.

   * Verifique:
     * A configuração [tcp&#95;port&#95;secure](/pt-BR/reference/settings/server-settings/settings#tcp_port_secure).
     * As configurações dos [certificados SSL](/pt-BR/reference/settings/server-settings/settings#openssl).
   * Use os parâmetros corretos ao se conectar. Por exemplo, use o parâmetro `port_secure` com `clickhouse_client`.

5. Configurações do usuário:

   * Talvez você esteja usando o nome de usuário ou a senha incorretos.

<div id="query-processing">
  ## Processamento de consultas
</div>

Se o ClickHouse não conseguir processar a consulta, ele enviará ao cliente uma descrição do erro. No `clickhouse-client`, você recebe essa descrição no console. Se estiver usando a interface HTTP, o ClickHouse envia a descrição do erro no corpo da resposta. Por exemplo:

```shell
$ curl 'http://localhost:8123/' --data-binary "SELECT a"
Code: 47, e.displayText() = DB::Exception: Unknown identifier: a. Note that there are no tables (FROM clause) in your query, context: required_names: 'a' source_tables: table_aliases: private_aliases: column_aliases: public_columns: 'a' masked_columns: array_join_columns: source_columns: , e.what() = DB::Exception
```

Se você iniciar o `clickhouse-client` com o parâmetro `stack-trace`, o ClickHouse retornará o stack trace do servidor com a descrição do erro.

Você pode ver uma mensagem sobre uma conexão interrompida. Nesse caso, pode repetir a consulta. Se a conexão for interrompida toda vez que você executar a consulta, verifique os logs do servidor em busca de erros.

<div id="efficiency-of-query-processing">
  ## Eficiência do processamento de consultas
</div>

Se você perceber que o ClickHouse está funcionando muito lentamente, precisará avaliar a carga nos recursos do servidor e na rede gerada pelas suas consultas.

Você pode usar o utilitário `clickhouse-benchmark` para analisar o perfil das consultas. Ele mostra o número de consultas processadas por segundo, o número de linhas processadas por segundo e os percentis dos tempos de processamento das consultas.
