<div id="query-api-endpoints-ga">
  ### Query API endpoints (GA)
</div>

Temos o prazer de anunciar o lançamento GA de Query API Endpoints no ClickHouse Cloud. Query API Endpoints permitem criar endpoints de API REST para consultas salvas com apenas alguns cliques e começar a consumir dados no seu aplicativo sem precisar lidar com clientes específicos de linguagem nem com a complexidade da autenticação. Desde o lançamento inicial, lançamos uma série de melhorias, incluindo:

* Redução da latência dos endpoints, especialmente em cold starts
* Mais controles de RBAC para endpoints
* Domínios permitidos por CORS configuráveis
* Streaming de resultados
* Suporte a todos os formatos de saída compatíveis com ClickHouse

Além dessas melhorias, temos o prazer de anunciar os endpoints genéricos de API de consulta que, aproveitando nossa estrutura existente, permitem executar consultas SQL arbitrárias nos seus serviços do ClickHouse Cloud. Endpoints genéricos podem ser habilitados e configurados na página de configurações do serviço.

Para começar, siga a [documentação de Query API Endpoints](/pt-BR/products/cloud/guides/sql-console/query-endpoints).

<div id="vector-search-using-vector-similarity-indexes-early-access">
  ### Busca vetorial usando índices de similaridade vetorial (acesso antecipado)
</div>

Estamos anunciando índices de similaridade vetorial para busca vetorial aproximada em acesso antecipado.

O ClickHouse já oferece suporte robusto a casos de uso baseados em vetores, com uma ampla variedade de [funções de distância]https://clickhouse.com/blog/reinvent-2024-product-announcements#vector-search-using-vector-similarity-indexes-early-access) e a capacidade de realizar varreduras lineares. Além disso, mais recentemente, adicionamos uma abordagem experimental de [busca vetorial aproximada](/pt-BR/reference/engines/table-engines/mergetree-family/annindexes) com tecnologia da biblioteca [usearch](https://github.com/unum-cloud/usearch) e do algoritmo Hierarchical Navigable Small Worlds (HNSW) de busca aproximada por vizinho mais próximo.

Para começar, [inscreva-se na lista de espera para acesso antecipado](https://clickhouse.com/cloud/vector-search-index-waitlist).

<div id="clickpipes-update-custom-certificates-latency-insights-and-more">
  ### Atualização do ClickPipes: certificados personalizados, insights de latência e muito mais.
</div>

Temos o prazer de compartilhar as atualizações mais recentes do ClickPipes, a maneira mais fácil de fazer ingestão de dados no seu serviço ClickHouse. Esses novos recursos foram desenvolvidos para ampliar seu controle sobre a ingestão de dados e oferecer mais visibilidade das métricas de desempenho.

*Certificados de autenticação personalizados para Kafka*

O ClickPipes for Kafka agora oferece suporte a certificados de autenticação personalizados para brokers do Kafka usando SASL e SSL/TLS público. Você pode enviar facilmente seu próprio certificado na seção SSL Certificate durante o Setup do ClickPipe, garantindo uma conexão mais segura com o Kafka.

*Apresentando métricas de latência para Kafka e Kinesis*

A visibilidade do desempenho é crucial. O ClickPipes agora inclui um gráfico de latência, oferecendo insights sobre o tempo entre a produção da mensagem (seja em um tópico do Kafka ou em um Kinesis Stream) e a ingestão no ClickHouse Cloud. Com essa nova métrica, você pode acompanhar mais de perto o desempenho dos seus pipelines de dados e otimizá-los conforme necessário.

<Image img="/images/cloud/reference/oct-4-latency-insights.webp" size="lg" alt="Interface do ClickPipes mostrando gráfico de métricas de latência para o desempenho da ingestão de dados" border />

<br />

*Controles de scaling para Kafka e Kinesis (Beta privada)*

Alto throughput pode exigir recursos extras para atender às suas necessidades de volume de dados e latência. Estamos introduzindo horizontal scaling para o ClickPipes, disponível diretamente no nosso console em nuvem. Esse recurso está atualmente em beta privada, permitindo que você dimensione os recursos com mais eficiência com base nos seus requisitos. Entre em contato com o [suporte](https://clickhouse.com/support/program) para participar da beta.

*Ingestão de mensagens brutas para Kafka e Kinesis*

Agora é possível fazer a ingestão de uma mensagem inteira do Kafka ou do Kinesis sem parsing. O ClickPipes agora oferece suporte à [coluna virtual](/pt-BR/integrations/clickpipes/kafka/reference#kafka-virtual-columns) `_raw_message`, permitindo que os usuários mapeiem a mensagem completa para uma única coluna String. Isso oferece a flexibilidade de trabalhar com dados brutos conforme necessário.

<div id="clickpipes-for-s3-and-gcs-now-in-ga-continuous-mode-support">
  ### ClickPipes para S3 e GCS agora em GA, com suporte ao modo contínuo
</div>

ClickPipes é a maneira mais fácil de fazer a ingestão de dados no ClickHouse Cloud. Temos o prazer de anunciar que o [ClickPipes](https://clickhouse.com/cloud/clickpipes) para S3 e GCS agora está em **disponibilidade geral**. O ClickPipes oferece suporte tanto à ingestão única em lote quanto ao &quot;modo contínuo&quot;. Uma tarefa de ingestão carregará todos os arquivos que corresponderem a um padrão de um bucket remoto específico para a tabela de destino no ClickHouse. No &quot;modo contínuo&quot;, o job do ClickPipes será executado continuamente, fazendo a ingestão dos arquivos correspondentes que forem adicionados ao bucket remoto de armazenamento de objetos conforme chegarem. Isso permitirá que você transforme qualquer bucket de armazenamento de objetos em uma área de staging completa para ingestão de dados no ClickHouse Cloud. Leia mais sobre o ClickPipes em [nossa documentação](/pt-BR/integrations/clickpipes/home).

<div id="query-log-insights">
  ### Insights do log de consultas
</div>

Nossa nova UI do Query Insights no console do Cloud torna o log de consultas nativo do ClickHouse muito mais fácil de usar. A tabela `system.query_log` do ClickHouse é uma fonte essencial de informações para otimização de consultas, depuração e monitoramento da integridade e do desempenho geral do cluster. Há apenas uma ressalva: com mais de 70 campos e vários registros por consulta, interpretar o log de consultas exige uma curva de aprendizado íngreme. Esta versão inicial do Query Insights oferece uma base para trabalhos futuros voltados a simplificar a depuração de consultas e os padrões de otimização. <div id="configurable-offsets-for-kafka-clickpipes-connector-beta">
  ### Offsets configuráveis para o conector Kafka do ClickPipes (Beta)
</div>

Até recentemente, sempre que você configurava um novo [Conector Kafka do ClickPipes](/pt-BR/integrations/clickpipes/kafka/index), ele consumia dados desde o início do tópico do Kafka. Nessa situação, isso pode não ser flexível o suficiente para atender a casos de uso específicos em que você precisa reprocessar dados históricos, monitorar novos dados recebidos ou retomar a partir de um ponto exato.

ClickPipes for Kafka adicionou um novo recurso que aumenta a flexibilidade e o controle sobre o consumo de dados de tópicos do Kafka. Agora, você pode configurar o offset a partir do qual os dados serão consumidos.

As seguintes opções estão disponíveis:

* Desde o início: comece a consumir dados desde o começo do tópico do Kafka. Essa opção é ideal se você precisa reprocessar todos os dados históricos.
* Do mais recente: comece a consumir dados a partir do offset mais recente. Isso é útil se você estiver interessado apenas em novas mensagens.
* De um timestamp: comece a consumir dados de mensagens produzidas em um timestamp específico ou depois dele. Esse recurso permite um controle mais preciso, possibilitando retomar o processamento a partir de um ponto exato no tempo.

<div id="reliability-changes">
  ### Alterações de confiabilidade
</div>

* Alteração incompatível com versões anteriores que afeta o usuário: anteriormente, dois recursos ([is&#95;deleted](/pt-BR/reference/engines/table-engines/mergetree-family/replacingmergetree#is_deleted) e `OPTIMIZE CLEANUP`), em determinadas condições, podiam levar à corrupção de dados no ClickHouse. Para proteger a integridade dos dados dos nossos usuários, preservando a funcionalidade principal, ajustamos a forma como esse recurso funciona. Especificamente, a configuração do MergeTree `clean_deleted_rows` foi descontinuada e não tem mais efeito. A palavra-chave `CLEANUP` não é permitida por padrão (para usá-la, você precisará habilitar `allow_experimental_replacing_merge_with_cleanup`). Se você decidir usar `CLEANUP`, precisará garantir que ela seja sempre usada junto com `FINAL` e que nenhuma linha de versões mais antigas seja inserida depois de executar `OPTIMIZE FINAL CLEANUP`.
