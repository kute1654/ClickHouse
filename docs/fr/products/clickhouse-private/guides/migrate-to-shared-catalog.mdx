title: "Migrer vers Shared Catalog"
description: "Shared Catalog est la nouvelle approche de gestion des métadonnées dans les instances ClickHouse Private."
---

Shared Catalog est la nouvelle approche pour gérer les métadonnées dans les instances ClickHouse Private. Au lieu d&#39;utiliser des bases de données avec le moteur [Replicated](https://clickhouse.com/docs/engines/database-engines/replicated), qui s&#39;appuie sur une DDL queue dans ClickHouse Keeper pour garantir que les instructions DDL sont appliquées à tous les réplicas, le moteur de base de données [Shared](https://clickhouse.com/docs/engines/database-engines/shared) stocke toutes les métadonnées directement dans Keeper, ce qui supprime la nécessité de synchroniser les instructions DDL entre les réplicas.

Shared Catalog offre plusieurs avantages :

* État cohérent sur tous les réplicas
* Caractère sans état des nœuds de calcul (permet des nœuds serveur sans état)
* `CREATE TABLE ... AS SELECT` atomique
* Prise en charge de `UNDROP`
* `RENAME`/déplacement de `TABLE` entre bases de données
* Initialisation rapide et fiable des réplicas (réduit les temps de réveil, de démarrage et de provisionnement)

Cependant, une migration est nécessaire si votre instance n&#39;a pas été configurée avec Shared Catalog lors du premier déploiement. Nous vous recommandons de tester d&#39;abord la migration dans un environnement de test avant de migrer des instances de production.

<div id="migration-requirements">
  ## Prérequis pour la migration
</div>

Avant de démarrer la migration, vérifiez les prérequis suivants :

<Steps>
  <Step>
    ### 1. Au moins 2 réplicas de serveur

    L’Operator effectue une mise à niveau progressive pour migrer les bases de données vers le nouveau moteur de base de données Shared. Définissez la valeur Helm `server.replicas` sur 2 ou plus.
  </Step>

  <Step>
    ### 2. Toutes les bases de données utilisent le moteur Replicated

    Toutes les bases de données (sauf `system`) doivent utiliser le moteur de base de données Replicated. Vérifiez avec :

    ```sql
    SELECT DISTINCT
        name,
        engine
    FROM clusterAllReplicas('default', system.databases)
    WHERE (name NOT IN ('INFORMATION_SCHEMA', 'system', 'information_schema'))
      AND (engine != 'Replicated')
    ```

    La requête ne doit retourner aucune ligne.
  </Step>

  <Step>
    ### 3. Toutes les tables utilisent des moteurs de table non répliqués

    Aucune table (sauf dans `system`) ne doit utiliser de moteurs de table répliqués ou anciens. Vérifiez avec :

    ```sql
    SELECT
        name,
        engine,
        database
    FROM system.tables
    WHERE ((engine LIKE '%Replicated%') OR (engine IN ('Ordinary', 'Atomic')))
      AND (database != 'system')
    ```

    La requête ne doit retourner aucune ligne.
  </Step>

  <Step>
    ### 4. Aucune table détachée n’existe

    ```sql
    SELECT * FROM system.detached_tables
    ```

    La requête ne doit retourner aucune ligne.

    Une fois tous les prérequis remplis, la migration peut être lancée. Si certains prérequis ne sont pas remplis, contactez le support ClickHouse pour obtenir de l’aide.
  </Step>
</Steps>

<div id="performing-the-migration">
  ## Effectuer la migration
</div>

<Warning>
  Pour les instances Hydra, migrez d’abord l’instance parente avant de migrer les instances enfants. Répétez les étapes ci-dessous pour chaque instance enfant une fois l’instance parente migrée.
</Warning>

<Steps>
  <Step>
    ### 1. Activer la configuration Shared Catalog

    Définissez `featureFlags.enableSharedCatalog` sur `true` dans vos valeurs Helm. Cela active la configuration, mais ne déclenche pas la migration des bases de données. Le déploiement d’un cluster ClickHouse avec ce flag défini sur `true` sans migrer les bases de données répliquées existantes empêchera le cluster ClickHouse de démarrer.
  </Step>

  <Step>
    ### 2. Ajouter les marqueurs de migration

    Définissez `featureFlags.migrateToSharedCatalog` sur `true` dans vos valeurs Helm. Cela :

    * Ajoute le paramètre `shared_database_catalog.migration_from_database_replicated` à ClickHouse, pour lui indiquer de s’attendre à des bases de données `Replicated` existantes lorsqu’il s’exécute en mode Shared Catalog
    * Ajoute l’annotation `clickhouse.com/start-shared-catalog-migration: "true"` à la ressource Kubernetes ClickHouseCluster, afin d’indiquer à l’Operator de migrer ce cluster

    Une fois le déploiement effectué, l’Operator déclenchera une mise à niveau progressive des nœuds serveur. Cela changera également le type de toutes les bases de données en `Shared`.
  </Step>

  <Step>
    ### 3. Attendre que le statut soit Ready

    ```bash
    kubectl get ClickHouseCluster <instance-name>
    ```
  </Step>

  <Step>
    ### 4. Supprimer les marqueurs de migration

    Une fois le statut Ready, définissez `featureFlags.migrateToSharedCatalog` sur `false` dans vos valeurs Helm. Comme toutes les bases de données ont été migrées vers le type `Shared` au lieu de `Replicated`, les paramètres de migration ne sont plus nécessaires.
  </Step>

  <Step>
    ### 5. Vérifier que toutes les bases de données ont été migrées

    Vérifiez que toutes les bases de données hors système utilisent le moteur Shared :

    ```sql
    SELECT DISTINCT
        name,
        engine
    FROM clusterAllReplicas('default', system.databases)
    WHERE (name NOT IN ('INFORMATION_SCHEMA', 'system', 'information_schema'))
      AND (engine != 'Shared')
    ```

    La requête ne doit renvoyer aucune ligne. À ce stade, votre instance a été migrée vers le Shared Catalog.
  </Step>
</Steps>

<div id="rollback">
  ## Retour arrière
</div>

Si vous rencontrez des problèmes avec le Shared Catalog, vous pouvez revenir à l’état précédent :

<Steps>
  <Step>
    ### 1. Supprimer les marqueurs de migration

    Définissez `featureFlags.migrateToSharedCatalog` sur `false` dans vos valeurs Helm.
  </Step>

  <Step>
    ### 2. Supprimer les paramètres du Shared Catalog

    Définissez `featureFlags.enableSharedCatalog` sur `false` dans vos valeurs Helm. Cela amènera l’Operator à annuler ces modifications.
  </Step>

  <Step>
    ### 3. Supprimer le catalog dans Keeper

    Redirigez un port vers Keeper et démarrez une session keeper-client (à l’aide de `clickhouse keeper-client`).

    Une fois la session démarrée, exécutez les commandes suivantes :

    ```text
    ls '/clickhouse/catalog'
    # Sortie : migration_from_database_replicated_completed names references replicas uuids

    rmr '/clickhouse/catalog'
    # Vous êtes sur le point de supprimer récursivement le chemin /clickhouse/catalog. Continuer ?
    # [y/n] y
    ```

    Cela supprimera tout l’état du Shared Catalog dans Keeper.
  </Step>

  <Step>
    ### 4. Vérifier le retour arrière

    Confirmez que toutes les bases de données autres que système utilisent de nouveau le moteur Replicated :

    ```sql
    SELECT DISTINCT
        name,
        engine
    FROM clusterAllReplicas('default', system.databases)
    WHERE (name NOT IN ('INFORMATION_SCHEMA', 'system', 'information_schema'))
      AND (engine != 'Replicated')
    ```

    La requête ne doit renvoyer aucune ligne.
  </Step>
</Steps>
