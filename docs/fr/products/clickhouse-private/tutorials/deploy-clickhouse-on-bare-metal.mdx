Ce tutoriel vous guide pour déployer ClickHouse sur des nœuds bare metal à l’aide de Terraform et d’Ansible sur des instances AWS EC2, avec RKE2 comme distribution Kubernetes. À la fin, vous disposerez d’un cluster RKE2 opérationnel avec ClickHouse déployé via l’opérateur.

Pour des spécifications détaillées de l’infrastructure, consultez [reference/infrastructure-requirements.md](/fr/products/clickhouse-private/reference/infrastructure-requirements-reference).

***

<Steps>
  <Step>
    ## Phase 1 : installer Terraform

    ```text
    dnf install -y unzip git
    ```

    ```bash
    curl -O https://releases.hashicorp.com/terraform/1.13.2/terraform_1.13.2_linux_amd64.zip
    ```

    ```text
    unzip terraform_1.13.2_linux_amd64.zip
    ```

    ```bash
    sudo cp terraform /usr/local/bin
    ```

    Vérifiez l’installation :

    ```text
    terraform version
    ```

    Résultat attendu :

    ```text
    Terraform v1.13.2
    on linux_amd64
    ```

    ***
  </Step>

  <Step>
    ## Phase 2 : Télécharger les scripts d’exemple

    Téléchargez et extrayez les scripts d’exemple fournis par ClickHouse :

    ```text
    unzip example-scripts.zip
    cd ./bare-metal
    ```

    L’archive contient des modules Terraform, des playbooks Ansible, des modèles d’inventaire et des fichiers de valeurs pour Helm.

    ***
  </Step>

  <Step>
    ## Phase 3 : Configurer les identifiants AWS

    Créez un fichier d’identifiants contenant votre clé d’accès AWS IAM et votre clé secrète :

    ```bash
    cat > aws-creds.sh <<EOF
    export AWS_ACCESS_KEY_ID=<Your AWS ACCESS KEY ID>
    export AWS_SECRET_ACCESS_KEY=<Your AWS SECRET ACCESS KEY>
    export AWS_REGION=us-east-2
    EOF
    ```

    Remplacez `<Your AWS ACCESS KEY ID>` et `<Your AWS SECRET ACCESS KEY>` par les valeurs figurant dans la console AWS IAM.

    Chargez les informations d’identification :

    ```bash
    source ./aws-creds.sh
    ```

    ***
  </Step>

  <Step>
    ## Phase 4 : Initialiser et appliquer Terraform

    Initialisez le module Terraform RKE2. Ajoutez `-migrate-state` si vous migrez l’état du stockage local vers S3.

    ```text
    terraform -chdir=modules/rke2 init \
    -backend-config="encrypt=true" \
    -backend-config="region=us-east-2" \
    -backend-config="bucket=chb-tfstate-359774249275" \
    -backend-config="use_lockfile=true" \
    -backend-config="key=chb/rke2/terraform.tfstate"
    ```

    Appliquez la configuration Terraform pour provisionner l’infrastructure :

    ```text
    terraform -chdir=modules/rke2 apply \
    -var-file=../../vars/common.tfvars \
    -var-file=../../vars/instances.tfvars \
    -var-file=../../vars/rke2.tfvars
    ```

    Cela crée les instances EC2, le réseau et les groupes de sécurité nécessaires au cluster RKE2.

    ***
  </Step>

  <Step>
    ## Phase 5 : Installer les dépendances Ansible

    Accédez au répertoire Ansible et installez les rôles et les collections Galaxy requis :

    ```bash
    cd ~/bare-metal/ansible
    ```

    ```text
    ansible-galaxy role install lablabs.rke2
    ```

    ```text
    ansible-galaxy collection install kubernetes.core
    ```

    ***
  </Step>

  <Step>
    ## Phase 6 : Déployer RKE2

    ### Vérifier que les nœuds sont prêts

    ```text
    ansible all -i inventory.yml -a 'cloud-init status'
    ```

    ### Configurer les disques NVMe

    ```text
    ansible-playbook -i inventory.yml disks_nvme.yml
    ```

    ### Exécuter le playbook RKE2

    ```text
    ansible-playbook -i inventory.yml rke2_playbook.yml
    ```

    Ce playbook installe et configure RKE2 sur tous les nœuds. Il provisionne le premier nœud serveur, attend qu’il soit prêt, puis ajoute au cluster les nœuds serveur et agent restants.

    La sortie du playbook affichera les tâches en cours d’exécution sur tous les nœuds (par exemple, de `chb-rke2s101` à `chb-rke2s302`). Une fois l’opération terminée, vous devriez voir un résumé comme celui-ci :

    ```text

    chb-rke2s101               : ok=40   changed=12   unreachable=0    failed=0    skipped=45   rescued=0    ignored=0
    chb-rke2s102               : ok=23   changed=7    unreachable=0    failed=0    skipped=38   rescued=0    ignored=0
    chb-rke2s201               : ok=23   changed=7    unreachable=0    failed=0    skipped=38   rescued=0    ignored=0
    chb-rke2s202               : ok=23   changed=7    unreachable=0    failed=0    skipped=38   rescued=0    ignored=0
    ```

    ***
  </Step>

  <Step>
    ## Phase 7 : Configurer cert-manager et les certificats

    Installez cert-manager, trust-manager et générez des certificats pour ClickHouse à l’aide du playbook Ansible :

    ```bash
    cd ~/bare-metal/ansible
    ansible-playbook -i 127.0.0.1, --connection=local manage_helm_applications.yml
    ```

    Cette procédure effectue les opérations suivantes :

    1. Installe le chart Helm cert-manager
    2. Installe le chart Helm trust-manager
    3. Crée un issuer auto-signé et un certificat de CA
    4. Crée un issuer de certificat de CA
    5. Configure Rancher (facultatif)
    6. Crée l’espace de noms du cluster ClickHouse
    7. Génère les certificats du serveur ClickHouse et de Keeper
    8. Copie les certificats dans les secrets Kubernetes correspondants

    Résultat attendu :

    ```text
    PLAY RECAP *****************************************************************
    127.0.0.1                  : ok=18   changed=13   unreachable=0    failed=0    skipped=0    rescued=0    ignored=0
    ```

    ***
  </Step>

  <Step>
    ## Phase 8 : Installer le stockage local

    ### Installer local-path-provisioner

    ```bash
    kubectl apply -f https://raw.githubusercontent.com/rancher/local-path-provisioner/v0.0.32/deploy/local-path-storage.yaml
    ```

    ### Créer une StorageClass

    ```bash
    cd ~/bare-metal/ch-cluster-deployment
    kubectl apply -f local-nvme-sc.yaml
    ```

    Le YAML StorageClass doit ressembler à ceci (mettez à jour les valeurs de topologie pour qu’elles correspondent aux noms de vos nœuds) :

    ```yaml
    apiVersion: storage.k8s.io/v1
    kind: StorageClass
    metadata:
      name: local-nvme-sc
    allowVolumeExpansion: true
    parameters:
      path: /nvme/disk
    provisioner: rancher.io/local-path
    reclaimPolicy: Delete
    volumeBindingMode: WaitForFirstConsumer
    allowedTopologies:
    - matchLabelExpressions:
        - key: topology.kubernetes.io/zone
          values:
            - <your_cluster_name>-keeper
            - <your_host_1>
            - <your_host_2>
            - <your_host_3>
            - <your_host_4>
            - <your_host_5>
            - <your_host_6>
        - key: directpv.min.io/region
          values:
            - default
    ```

    ***
  </Step>

  <Step>
    ## Phase 9 : Ajouter des labels aux nœuds

    Ajoutez aux nœuds les labels de groupe ClickHouse et de zone de topologie requis par l’operator :

    ```bash
    kubectl get nodes |awk '{ print $1}'|grep '01'|xargs -I {} -n 1 kubectl label nodes '{}' clickhouseGroup=keeper-amd64  topology.kubernetes.io/zone=c-baremetal-ch-01-keeper --overwrite
    kubectl get nodes |awk '{ print $1}'|grep '02'|xargs -I {} -n 1 kubectl label nodes '{}' clickhouseGroup=server-amd64  --overwrite
    kubectl label nodes chb-rke2s102 topology.kubernetes.io/zone=chb-rke2s102 --overwrite
    kubectl label nodes chb-rke2s202 topology.kubernetes.io/zone=chb-rke2s202 --overwrite
    kubectl label nodes chb-rke2s302 topology.kubernetes.io/zone=chb-rke2s302 --overwrite
    ```

    Adaptez les noms de nœud et les labels à votre environnement.

    ***
  </Step>

  <Step>
    ## Phase 10 : Installer le ClickHouse Operator et le cluster

    ### Définir les variables d&#39;environnement

    Créez et sourcez le script de variables Helm. Mettez à jour toutes les valeurs des espaces réservés :

    ```bash
    #!/bin/bash
    # AWS/S3 creds
    export AWS_ACCESS_KEY_ID=<your_aws_key>
    export AWS_SECRET_ACCESS_KEY=<your_aws_secret>
    export AWS_REGION=<your_aws_region>
    export S3_ACCESS_KEY_ID=<s3_or_minio_key>
    export S3_SECRET_ACCESS_KEY=<s3_or_minio_secret>
    export S3_REGION=us-east-1
    export S3_BUCKET_NAME=<your_s3_bucket>
    export S3_KEY_PREFIX=<ch-s3-unique-uuid>
    export S3_ENDPOINT=https://s3.us-east-1.amazonaws.com
    export ECR_HOST=<ecr_host>
    # Set AZs as determined by VPC subnets. If not relevant then use your k8s host names.
    export AZ_LIST='["<your_host_1>", "<your_host_2>", "<your_host_3>","<your_host_4>", "<your_host_5>", "<your_host_6>"]'

    PASSWORD='My super secret p@$$w0rd'
    export HASHED_PASSWORD=$(echo -n "$PASSWORD" | sha256sum | awk '{printf $1}' | base64 -w 0)
    export CLUSTER_NAME="baremetal-ch-01"
    export CHART_VERSION="1.1.311"

    export SERVER_NG_GROUP="server-amd64"
    export KEEPER_NG_GROUP="keeper-amd64"
    export KEEPER_CPU="4"
    export KEEPER_MEMORY="16Gi"
    export SERVER_CPU="64"
    export SERVER_MEMORY="128Gi"
    export CACHE_DISK_SIZE_GI=379Gi
    ```

    Exécutez le script avec `source` :

    ```bash
    source ./set_helm_vars.sh
    ```

    ### Installer l&#39;Operator

    Connectez-vous au registre Helm et installez l&#39;opérateur :

    ```text
    # Log into the Helm registry
    aws ecr get-login-password | helm registry login --username AWS --password-stdin $ECR_HOST

    # Create image pull secret
    kubectl create secret docker-registry all-icr-io \
      --docker-server=$ECR_HOST \
      --docker-username=AWS \
      --docker-password=$(aws ecr get-login-password) \
      -n clickhouse-operator-system

    OPERATOR_VERSION=1.18587.1

    helm upgrade clickhouse-operator \
      oci://$ECR_HOST/helm/clickhouse-operator-helm \
      --install --reuse-values \
      --version="$OPERATOR_VERSION" \
      --create-namespace \
      -n clickhouse-operator-system \
      --set-json="image.repository=\"$ECR_HOST/clickhouse-operator\"" \
      --set-json='cilium.enabled=false' \
      --set-json='idleScalerEnabled=false' \
      --set-json='webhooks.enabled=false' \
      --set-json='operator.debug=true' \
      --set-json='operator.metricsScraper.enabled=false' \
      --set-json='operator.featureFlags.backupOnASeparatePod=true' \
      --set-json='operator.featureFlags.serverCMEKEnabled=true' \
      --set-json='imagePullSecrets=[{"name":"all-icr-io"}]' \
      --set-json="operator.availabilityZones=$AZ_LIST"
    ```

    ### Déployer le cluster ClickHouse

    Créez le fichier de valeurs Helm (`clickhouse-values.yaml`) avec la configuration de votre cluster :

    ```yaml expandable
    imagePullSecrets:
    - name: all-icr-io

    account:
      hashedPassword: "$HASHED_PASSWORD"
    server:
      ssdCacheConfiguration:
        cacheDiskSize: "$CACHE_DISK_SIZE_GI"
      openSSL:
        enabled: true
        required: true
      replicaCount: 3
      arm64: false
      podPolicy:
        nodeSelector:
          clickhouseGroup: "$SERVER_NG_GROUP"
        resources:
          limits:
            cpu: "$SERVER_CPU"
            memory: "$SERVER_MEMORY"
          requests:
            cpu: "$SERVER_CPU"
            memory: "$SERVER_MEMORY"
      tolerations:
        - effect: NoSchedule
          key: "clickhouse.com/do-not-schedule"
          operator: Exists
      storage:
        storageClassName: local-nvme-sc
        resources:
          requests: "300Gi"
        s3:
          bucketName: "$S3_BUCKET_NAME"
          keyPrefix: "$S3_KEY_PREFIX"
          endpoint: "$S3_ENDPOINT"
          region: "$S3_REGION"
      image:
        repository: "$ECR_HOST/clickhouse-server"
        tag: "25.12.1.1497"
      additionalEnvVars:
        - name: AWS_ACCESS_KEY_ID
          value: "${S3_ACCESS_KEY_ID}"
        - name: AWS_SECRET_ACCESS_KEY
          value: "${S3_SECRET_ACCESS_KEY}"

    keeper:
      openSSL:
        enabled: true
        required: true
      image:
        repository: "$ECR_HOST/clickhouse-keeper"
        tag: "26.2.1.258"
      arm64: false
      podPolicy:
        nodeSelector:
          clickhouseGroup: "$KEEPER_NG_GROUP"
        resources:
          limits:
            cpu: "$KEEPER_CPU"
            memory: "$KEEPER_MEMORY"
          requests:
            cpu: "$KEEPER_CPU"
            memory: "$KEEPER_MEMORY"
      tolerations:
        - effect: NoSchedule
          key: "clickhouse.com/do-not-schedule"
          operator: Exists
      storage:
        storageClassName: local-nvme-sc
        resources:
          requests: 30Gi

    serviceAccount:
      create: false
      name: default

    storageClass:
      create: false
      name: local-nvme-sc
      provisioner: rancher.io/local-path
    ```

    Installez le cluster à l&#39;aide du fichier de valeurs :

    ```text
    # Create namespace
    kubectl create namespace ns-$CLUSTER_NAME

    # Create image pull secret in the cluster namespace
    kubectl create secret docker-registry all-icr-io \
      --docker-server=$ECR_HOST \
      --docker-username=AWS \
      --docker-password=$(aws ecr get-login-password) \
      -n ns-$CLUSTER_NAME

    # Expand variables and install
    TEMP_VALUES_FILE=$(mktemp)
    envsubst < clickhouse-values.yaml > "$TEMP_VALUES_FILE"

    helm upgrade "$CLUSTER_NAME" \
      oci://$ECR_HOST/helm/onprem-clickhouse-cluster \
      --install --reuse-values \
      --version="$CHART_VERSION" \
      -n ns-$CLUSTER_NAME \
      --create-namespace \
      --values "$TEMP_VALUES_FILE" \
      --timeout 10m0s

    rm "$TEMP_VALUES_FILE"
    ```

    ### Correctifs post-installation

    Une fois le cluster déployé, il peut être nécessaire de patcher le StatefulSet Keeper pour y ajouter des image pull secrets et des credentials S3 :

    ```text
    # Copy pull secret to cluster namespace
    kubectl get secret all-icr-io -n default -o yaml | \
      sed "s/namespace: default/namespace: ns-$CLUSTER_NAME/" | \
      kubectl apply -n ns-$CLUSTER_NAME --force -f -

    # Patch keeper StatefulSet (use sts_keeper_patch_template.yaml from example scripts)
    kubectl patch sts c-$CLUSTER_NAME-keeper -n ns-$CLUSTER_NAME \
      --patch "$(envsubst < sts_keeper_patch_template.yaml)"

    # Restart keeper pods
    kubectl delete pod c-$CLUSTER_NAME-keeper-{0,1,2} -n ns-$CLUSTER_NAME
    ```

    Attendez que les pods Keeper soient redéployés, puis corrigez les PV du serveur si nécessaire :

    ```text
    ./sts_server_patch.sh "ns-$CLUSTER_NAME" all-icr-io
    kubectl delete pod -l kind=clickhouse-server
    ```

  </Step>
</Steps>

<div id="verify-installation">
  ## Vérifier l’installation
</div>

Une fois que tous les pods sont en cours d’exécution, effectuez un port-forward, puis vérifiez. Comme ce déploiement utilise TLS (openSSL activé), utilisez le port 9440 :

```bash
kubectl port-forward svc/c-$CLUSTER_NAME-server-any 9440:9440 -n ns-$CLUSTER_NAME
```

Connectez-vous à l’aide du client ClickHouse :

```bash
clickhouse client --host localhost --port 9440 --password "$PASSWORD" --secure --accept-invalid-certificate
```

Exécutez une requête de test :

```sql
SELECT 1;
```

Résultat attendu :

```text
   ┌─1─┐
1. │ 1 │
   └───┘
1 row in set. Elapsed: 0.001 sec.
```
