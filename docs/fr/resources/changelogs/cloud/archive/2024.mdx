<div id="marketplace-subscription-organization-attachment">
  ### Rattachement d’un abonnement marketplace à une organisation
</div>

Vous pouvez désormais rattacher votre nouvel abonnement marketplace à une organisation ClickHouse Cloud existante. Une fois l’abonnement au marketplace finalisé et la redirection vers ClickHouse Cloud effectuée, vous pouvez associer à ce nouvel abonnement marketplace une organisation existante créée précédemment. À partir de ce moment, les ressources de l’organisation seront facturées via le marketplace.

<Image img="/images/cloud/reference/add_marketplace.webp" size="md" alt="Interface ClickHouse Cloud montrant comment ajouter un abonnement marketplace à une organisation existante" border />

<div id="force-openapi-key-expiration">
  ### Forcer l’expiration des clés OpenAPI
</div>

Il est désormais possible de restreindre les durées de validité des clés API afin d’éviter de créer des clés OpenAPI sans expiration. Veuillez contacter l’équipe ClickHouse Cloud Support pour activer ces restrictions pour votre organisation.

<div id="custom-emails-for-notifications">
  ### Adresses e-mail personnalisées pour les notifications
</div>

Les Admins de l&#39;org peuvent désormais ajouter d&#39;autres adresses e-mail à une notification donnée en tant que destinataires supplémentaires. Cela est utile si vous souhaitez envoyer des notifications à un alias ou à d&#39;autres utilisateurs de votre organisation qui ne sont pas forcément des utilisateurs de ClickHouse Cloud. Pour ce faire, accédez aux paramètres de notification depuis la console Cloud et modifiez les adresses e-mail auxquelles vous souhaitez envoyer les notifications.

<div id="byoc-beta">
  ### BYOC (bêta)
</div>

Bring Your Own Cloud pour AWS est désormais disponible en bêta. Ce mode de déploiement vous permet de déployer et d’exploiter ClickHouse Cloud dans votre propre compte AWS. Nous prenons en charge les déploiements dans plus de 11 régions AWS, et d’autres arriveront bientôt. Veuillez [contacter le support](https://clickhouse.com/support/program) pour obtenir l’accès. Notez que ce mode de déploiement est réservé aux déploiements à grande échelle.

<div id="postgres-change-data-capture-cdc-connector-in-clickpipes">
  ### Connecteur Postgres de capture des changements de données (CDC) dans ClickPipes
</div>

Cette intégration clé en main permet aux clients de répliquer leurs bases de données Postgres vers ClickHouse Cloud en seulement quelques clics et d’exploiter ClickHouse pour des analyses ultra-rapides. Vous pouvez utiliser ce connecteur aussi bien pour la réplication continue que pour des migrations ponctuelles depuis Postgres.

<div id="query-api-endpoints-ga">
  ### Query API endpoints (GA)
</div>

Nous avons le plaisir d’annoncer la disponibilité générale (GA) de Query API endpoints dans ClickHouse Cloud. Query API endpoints vous permet de créer des endpoints d’API REST pour des requêtes enregistrées en seulement quelques clics, puis de commencer à exploiter ces données dans votre application sans avoir à gérer de clients pour différents langages ni la complexité de l’authentification. Depuis le lancement initial, nous avons apporté un certain nombre d’améliorations, notamment :

* Réduction de la latence des endpoints, en particulier lors des démarrages à froid
* Renforcement des contrôles RBAC des endpoints
* Domaines autorisés par CORS configurables
* Streaming des résultats
* Prise en charge de tous les formats de sortie compatibles avec ClickHouse

En plus de ces améliorations, nous avons le plaisir d’annoncer des endpoints génériques de Query API qui, en s’appuyant sur notre infrastructure existante, vous permettent d’exécuter des requêtes SQL arbitraires sur vos services ClickHouse Cloud. Les endpoints génériques peuvent être activés et configurés depuis la page des paramètres du service.

Pour commencer, consultez la [documentation Query API Endpoints](/fr/products/cloud/guides/sql-console/query-endpoints).

<Image img="/images/cloud/reference/api_endpoints.webp" size="lg" alt="Interface ClickHouse Cloud affichant la configuration des API Endpoints avec différents paramètres" border />

<div id="vector-search-using-vector-similarity-indexes-early-access">
  ### Recherche vectorielle à l’aide d’index de similarité vectorielle (accès anticipé)
</div>

Nous annonçons la disponibilité en accès anticipé des index de similarité vectorielle pour la recherche vectorielle approximative.

ClickHouse offre déjà une prise en charge solide des cas d’usage vectoriels, avec un large éventail de [fonctions de distance]https://clickhouse.com/blog/reinvent-2024-product-announcements#vector-search-using-vector-similarity-indexes-early-access) et la possibilité d’effectuer des parcours linéaires. Plus récemment, nous avons également ajouté une approche expérimentale de [recherche vectorielle approximative](/fr/reference/engines/table-engines/mergetree-family/annindexes), basée sur la bibliothèque [usearch](https://github.com/unum-cloud/usearch) et sur l’algorithme de recherche approximative des plus proches voisins Hierarchical Navigable Small Worlds (HNSW).

Pour commencer, [veuillez vous inscrire sur la liste d’attente pour l’accès anticipé](https://clickhouse.com/cloud/vector-search-index-waitlist).

Pour commencer et configurer un partage de ressources, vous pouvez consulter le [billet d’annonce](https://clickhouse.com/blog/clickpipes-crossvpc-resource-endpoints?utm_medium=web\&utm_source=changelog).

<div id="clickpipes-update-custom-certificates-latency-insights-and-more">
  ### Mise à jour de ClickPipes : certificats personnalisés, indicateurs de latence et plus encore.
</div>

Nous sommes ravis de partager les dernières mises à jour de ClickPipes, le moyen le plus simple d’ingérer des données dans votre service ClickHouse. Ces nouvelles fonctionnalités sont conçues pour vous offrir un meilleur contrôle sur l’ingestion des données et une visibilité accrue sur les métriques de performance.

*Certificats d’authentification personnalisés pour Kafka*

ClickPipes for Kafka prend désormais en charge les certificats d’authentification personnalisés pour les brokers Kafka utilisant SASL et SSL/TLS publics. Vous pouvez facilement téléverser votre propre certificat dans la section SSL Certificate lors de la configuration du ClickPipe, afin de garantir une connexion plus sécurisée à Kafka.

*Présentation des métriques de latence pour Kafka et Kinesis*

La visibilité sur les performances est essentielle. ClickPipes inclut désormais un graphique de latence, qui vous donne un aperçu du délai entre la production des messages (qu’ils proviennent d’un topic Kafka ou d’un Kinesis Stream) et leur ingestion dans ClickHouse Cloud. Grâce à cette nouvelle métrique, vous pouvez surveiller de plus près les performances de vos pipelines de données et les optimiser en conséquence.

<Image img="/images/cloud/reference/oct-4-latency-insights.webp" size="lg" alt="Interface ClickPipes affichant un graphique des métriques de latence pour les performances d’ingestion des données" border />

<br />

*Contrôles de mise à l’échelle pour Kafka et Kinesis (bêta privée)*

Un débit élevé peut nécessiter des ressources supplémentaires pour répondre à vos besoins en volume de données et en latence. Nous introduisons la mise à l’échelle horizontale pour ClickPipes, disponible directement depuis notre console Cloud. Cette fonctionnalité est actuellement en bêta privée, ce qui vous permet d’ajuster les ressources plus efficacement en fonction de vos besoins. Veuillez contacter le [support](https://clickhouse.com/support/program) pour rejoindre la bêta.

*Ingestion des messages bruts pour Kafka et Kinesis*

Il est désormais possible d’ingérer un message Kafka ou Kinesis dans son intégralité sans l’analyser. ClickPipes prend maintenant en charge la [colonne virtuelle](/fr/integrations/clickpipes/kafka/reference#kafka-virtual-columns) `_raw_message`, ce qui permet aux utilisateurs d’associer le message complet à une seule colonne String. Cela vous offre la flexibilité nécessaire pour travailler avec des données brutes selon vos besoins.

<div id="compute-compute-separation-is-now-in-private-preview-for-aws">
  ### Compute-compute separation est désormais disponible en aperçu privé sur AWS
</div>

Pour les services ClickHouse Cloud existants, les répliques gèrent à la fois les lectures et les écritures, et il n’existe aucun moyen de configurer une réplique donnée pour qu’elle ne gère qu’un seul type d’opération. Une nouvelle fonctionnalité à venir, appelée Compute-compute separation, vous permettra de désigner des services spécifiques comme services en lecture-écriture ou en lecture seule, afin de concevoir la configuration de compute la mieux adaptée à votre application et d’optimiser les coûts et les performances.

Notre nouvelle fonctionnalité Compute-compute separation vous permet de créer plusieurs groupes de nœuds de calcul, chacun avec son propre endpoint, qui utilisent le même dossier de stockage objet et donc les mêmes tables, vues, etc. Pour en savoir plus, consultez [Compute-compute separation](/fr/products/cloud/features/infrastructure/warehouses). Veuillez [contacter l’assistance](https://clickhouse.com/support/program) si vous souhaitez accéder à cette fonctionnalité en aperçu privé.

<div id="clickpipes-for-s3-and-gcs-now-in-ga-continuous-mode-support">
  ### ClickPipes pour S3 et GCS désormais en GA, avec prise en charge du mode continu
</div>

ClickPipes est le moyen le plus simple d’ingérer des données dans ClickHouse Cloud. Nous sommes heureux d’annoncer que [ClickPipes](https://clickhouse.com/cloud/clickpipes) pour S3 et GCS est désormais **en disponibilité générale**. ClickPipes prend en charge à la fois l’ingestion ponctuelle par lot et le « mode continu ». Une tâche d’ingestion charge tous les fichiers correspondant à un motif depuis un bucket distant donné vers la table de destination ClickHouse. En « mode continu », le job ClickPipes s’exécute en permanence et ingère les fichiers correspondants à mesure qu’ils sont ajoutés au bucket de stockage objet distant. Vous pouvez ainsi transformer n’importe quel bucket de stockage objet en véritable zone de staging pour l’ingestion de données dans ClickHouse Cloud. Pour en savoir plus sur ClickPipes, consultez [notre documentation](/fr/integrations/clickpipes/home).

<div id="table-inspector-in-cloud-console">
  ### Table Inspector dans la console Cloud
</div>

ClickHouse propose des commandes comme [`DESCRIBE`](/fr/reference/statements/describe-table) qui vous permettent d’inspecter votre table afin d’en examiner le schéma. Ces commandes affichent leurs résultats dans la console, mais elles sont souvent peu pratiques, car il faut combiner plusieurs requêtes pour obtenir toutes les informations pertinentes sur vos tables et vos colonnes.

Nous avons récemment lancé **Table Inspector** dans la console Cloud, qui vous permet de consulter dans l’UI des informations importantes sur les tables et les colonnes, sans avoir à écrire de SQL. Vous pouvez essayer Table Inspector pour vos services dans la console Cloud. Il regroupe dans une interface unifiée des informations sur votre schéma, le stockage, la compression, et bien plus encore.

<div id="configurable-offsets-for-kafka-clickpipes-connector-beta">
  ### Offsets configurables pour le connecteur Kafka de ClickPipes (bêta)
</div>

Jusqu&#39;à récemment, lorsque vous configuriez un nouveau [Kafka Connector for ClickPipes](/fr/integrations/clickpipes/kafka/index), il consommait toujours les données depuis le début du topic Kafka. Dans ce contexte, ce comportement pouvait manquer de souplesse pour certains cas d&#39;usage, notamment lorsque vous deviez retraiter des données historiques, surveiller l&#39;arrivée de nouvelles données ou reprendre à partir d&#39;un point précis.

ClickPipes for Kafka ajoute une nouvelle fonctionnalité qui améliore la flexibilité et le contrôle de la consommation des données depuis les topics Kafka. Vous pouvez désormais configurer l&#39;offset à partir duquel les données sont consommées.

Les options suivantes sont disponibles :

* Depuis le début : commencez à consommer les données depuis le tout début du topic Kafka. Cette option est idéale si vous devez retraiter l&#39;ensemble des données historiques.
* Depuis le plus récent : commencez à consommer les données à partir de l&#39;offset le plus récent. Cette option est utile si vous vous intéressez uniquement aux nouveaux messages.
* Depuis un timestamp : commencez à consommer les données à partir des messages produits à un timestamp donné ou après celui-ci. Cette fonctionnalité offre un contrôle plus précis et vous permet de reprendre le traitement à un moment exact.

<div id="reliability-changes">
  ### Changements liés à la fiabilité
</div>

* Changement non rétrocompatible pour les utilisateurs : auparavant, deux fonctionnalités ([is&#95;deleted](/fr/reference/engines/table-engines/mergetree-family/replacingmergetree#is_deleted) et `OPTIMIZE CLEANUP`) pouvaient, dans certaines conditions, entraîner une corruption des données dans ClickHouse. Afin de protéger l’intégrité des données de nos utilisateurs tout en préservant le cœur de cette fonctionnalité, nous en avons ajusté le fonctionnement. Plus précisément, le paramètre MergeTree `clean_deleted_rows` est désormais déprécié et n’a plus aucun effet. Le mot-clé `CLEANUP` n’est pas autorisé par défaut (pour l’utiliser, vous devrez activer `allow_experimental_replacing_merge_with_cleanup`). Si vous décidez d’utiliser `CLEANUP`, vous devez vous assurer qu’il est toujours utilisé avec `FINAL`, et vous devez garantir qu’aucune ligne avec une version antérieure ne soit insérée après l’exécution de `OPTIMIZE FINAL CLEANUP`.
