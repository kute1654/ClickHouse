  ### Tag de version ClickHouse : 24.2.2.15987

<div id="backward-incompatible-change">
  #### Changement incompatible avec les versions antérieures
</div>

* Validation des types suspects/expérimentaux dans les types imbriqués. Auparavant, nous ne validions pas ces types (à l’exception de JSON) dans des types imbriqués tels que Array/Tuple/Map. [#59385](https://github.com/ClickHouse/ClickHouse/pull/59385) ([Kruglov Pavel](https://github.com/Avogar)).
* La clause de tri `ORDER BY ALL` (introduite dans la v23.12) est remplacée par `ORDER BY *`. L’ancienne syntaxe était trop sujette aux erreurs pour les tables comportant une colonne `all`. [#59450](https://github.com/ClickHouse/ClickHouse/pull/59450) ([Robert Schulze](https://github.com/rschu1ze)).
* Ajout d’une vérification de cohérence du nombre de threads et de la taille des blocs. [#60138](https://github.com/ClickHouse/ClickHouse/pull/60138) ([Raúl Marín](https://github.com/Algunenano)).
* Rejet des requêtes INSERT entrantes lorsque les paramètres au niveau de la requête `async_insert` et `deduplicate_blocks_in_dependent_materialized_views` sont activés simultanément. Ce comportement est contrôlé par le paramètre `throw_if_deduplication_in_dependent_materialized_views_enabled_with_async_insert`, activé par défaut. Il s’agit de la suite de https://github.com/ClickHouse/ClickHouse/pull/59699, nécessaire pour débloquer https://github.com/ClickHouse/ClickHouse/pull/59915. [#60888](https://github.com/ClickHouse/ClickHouse/pull/60888) ([Nikita Mikhaylov](https://github.com/nikitamikhaylov)).
* L’utilitaire `clickhouse-copier` a été déplacé vers un dépôt GitHub distinct : https://github.com/ClickHouse/copier. Il n’est plus inclus dans le bundle, mais reste disponible en téléchargement séparé. Ceci clôt : [#60734](https://github.com/ClickHouse/ClickHouse/issues/60734) Ceci clôt : [#60540](https://github.com/ClickHouse/ClickHouse/issues/60540) Ceci clôt : [#60250](https://github.com/ClickHouse/ClickHouse/issues/60250) Ceci clôt : [#52917](https://github.com/ClickHouse/ClickHouse/issues/52917) Ceci clôt : [#51140](https://github.com/ClickHouse/ClickHouse/issues/51140) Ceci clôt : [#47517](https://github.com/ClickHouse/ClickHouse/issues/47517) Ceci clôt : [#47189](https://github.com/ClickHouse/ClickHouse/issues/47189) Ceci clôt : [#46598](https://github.com/ClickHouse/ClickHouse/issues/46598) Ceci clôt : [#40257](https://github.com/ClickHouse/ClickHouse/issues/40257) Ceci clôt : [#36504](https://github.com/ClickHouse/ClickHouse/issues/36504) Ceci clôt : [#35485](https://github.com/ClickHouse/ClickHouse/issues/35485) Ceci clôt : [#33702](https://github.com/ClickHouse/ClickHouse/issues/33702) Ceci clôt : [#26702](https://github.com/ClickHouse/ClickHouse/issues/26702) ### Entrée de documentation pour les changements visibles par les utilisateurs. [#61058](https://github.com/ClickHouse/ClickHouse/pull/61058) ([Nikita Mikhaylov](https://github.com/nikitamikhaylov)).
* Pour améliorer la compatibilité avec MySQL, la fonction `locate` accepte désormais par défaut les arguments `(needle, haystack[, start_pos])`. Le comportement précédent `(haystack, needle, [, start_pos])` peut être rétabli en définissant `function_locate_has_mysql_compatible_argument_order = 0`. [#61092](https://github.com/ClickHouse/ClickHouse/pull/61092) ([Robert Schulze](https://github.com/rschu1ze)).
* Les parties de données en mémoire obsolètes sont dépréciées depuis la version 23.5 et ne sont plus prises en charge depuis la version 23.10. Le code restant est désormais supprimé. Suite de [#55186](https://github.com/ClickHouse/ClickHouse/issues/55186) et [#45409](https://github.com/ClickHouse/ClickHouse/issues/45409). Il est peu probable que vous ayez utilisé des parties de données en mémoire, car elles n&#39;étaient disponibles qu&#39;avant la version 23.5, et seulement si vous les activiez manuellement en spécifiant les SETTINGS correspondants pour une table MergeTree. Pour vérifier si vous avez des parties de données en mémoire, exécutez la requête suivante : `SELECT part_type, count() FROM system.parts GROUP BY part_type ORDER BY part_type`. Pour désactiver l&#39;utilisation des parties de données en mémoire, exécutez `ALTER TABLE ... MODIFY SETTING min_bytes_for_compact_part = DEFAULT, min_rows_for_compact_part = DEFAULT`. Avant de mettre à niveau depuis d&#39;anciennes versions de ClickHouse, vérifiez d&#39;abord que vous n&#39;avez pas de parties de données en mémoire. S&#39;il y a des parties de données en mémoire, désactivez-les d&#39;abord, puis attendez qu&#39;il n&#39;y en ait plus avant de poursuivre la mise à niveau. [#61127](https://github.com/ClickHouse/ClickHouse/pull/61127) ([Alexey Milovidov](https://github.com/alexey-milovidov)).
* Interdire `SimpleAggregateFunction` dans `ORDER BY` des tables `MergeTree` par défaut (comme `AggregateFunction`, car ils ne sont pas comparables ; utilisez `allow_suspicious_primary_key` pour les autoriser). [#61399](https://github.com/ClickHouse/ClickHouse/pull/61399) ([Azat Khuzhin](https://github.com/azat)).
* ClickHouse autorise des données binaires arbitraires dans le type de données String, qui est généralement en UTF-8. Les Strings de Parquet/ORC/Arrow ne prennent en charge que l&#39;UTF-8. C&#39;est pourquoi vous pouvez choisir quel type de données Arrow utiliser pour le type de données String de ClickHouse : String ou Binary. Cela est contrôlé par les paramètres `output_format_parquet_string_as_string`, `output_format_orc_string_as_string`, `output_format_arrow_string_as_string`. Bien que Binary soit plus correct et plus compatible, utiliser String par défaut correspondra aux attentes des utilisateurs dans la plupart des cas. Parquet/ORC/Arrow prend en charge de nombreuses méthodes de compression, notamment lz4 et zstd. ClickHouse les prend toutes en charge. Certains outils de moindre qualité ne prennent pas en charge la méthode de compression `lz4`, plus rapide, c&#39;est pourquoi nous définissons `zstd` par défaut. Cela est contrôlé par les paramètres `output_format_parquet_compression_method`, `output_format_orc_compression_method` et `output_format_arrow_compression_method`. Nous avons remplacé la valeur par défaut par `zstd` pour Parquet et ORC, mais pas pour Arrow (qui reste privilégié pour des usages de bas niveau). [#61817](https://github.com/ClickHouse/ClickHouse/pull/61817) ([Alexey Milovidov](https://github.com/alexey-milovidov)).
* Correction d&#39;un problème de sécurité des vues matérialisées, qui permettait à un utilisateur d&#39;insérer dans une table sans disposer des privilèges requis. Le correctif vérifie que l&#39;utilisateur a le droit d&#39;insérer non seulement dans une vue matérialisée, mais aussi dans toutes les tables sous-jacentes. Cela signifie que certaines requêtes qui fonctionnaient auparavant peuvent désormais échouer avec Not enough privileges. Pour résoudre ce problème, cette release introduit une nouvelle fonctionnalité de SQL security pour les vues [https://clickhouse.com/docs/sql-reference/statements/create/view#sql&#95;security](/fr/reference/statements/create/view#sql_security). [#54901](https://github.com/ClickHouse/ClickHouse/pull/54901) ([pufit](https://github.com/pufit))

<div id="performance-improvement">
  #### Amélioration des performances
</div>

* Élimine les agrégateurs min/max/any/anyLast des clés de GROUP BY dans la section SELECT. [#52230](https://github.com/ClickHouse/ClickHouse/pull/52230) ([JackyWoo](https://github.com/JackyWoo)).
* Améliore les performances de la méthode d’agrégation sérialisée lorsqu’elle implique plusieurs colonnes [nullable]. Il s’agit d’une version générale de [#51399](https://github.com/ClickHouse/ClickHouse/issues/51399) qui ne compromet pas l’intégrité de l’abstraction. [#55809](https://github.com/ClickHouse/ClickHouse/pull/55809) ([Amos Bird](https://github.com/amosbird)).
* Construit paresseusement la sortie du join pour améliorer les performances de ALL join. [#58278](https://github.com/ClickHouse/ClickHouse/pull/58278) ([LiuNeng](https://github.com/liuneng1994)).
* Améliorations des fonctions d’agrégation ArgMin / ArgMax / any / anyLast / anyHeavy, ainsi que des requêtes `ORDER BY {u8/u16/u32/u64/i8/i16/u32/i64) LIMIT 1`. [#58640](https://github.com/ClickHouse/ClickHouse/pull/58640) ([Raúl Marín](https://github.com/Algunenano)).
* Optimise conditionnellement les performances de sum/avg pour les types bigint et big decimal en réduisant les ratés de prédiction de branchement. [#59504](https://github.com/ClickHouse/ClickHouse/pull/59504) ([李扬](https://github.com/taiyang-li)).
* Améliore les performances des requêtes SELECT avec des mutations actives. [#59531](https://github.com/ClickHouse/ClickHouse/pull/59531) ([Azat Khuzhin](https://github.com/azat)).
* Optimisation mineure du filtre de colonnes. Évite de filtrer avec `result_size_hint = -1` les colonnes dont le type de données sous-jacent n’est pas numérique. Le pic de mémoire peut être réduit à 44 % de la valeur d’origine dans certains cas. [#59698](https://github.com/ClickHouse/ClickHouse/pull/59698) ([李扬](https://github.com/taiyang-li)).
* La clé primaire utilisera moins de mémoire. [#60049](https://github.com/ClickHouse/ClickHouse/pull/60049) ([Alexey Milovidov](https://github.com/alexey-milovidov)).
* Améliore l’utilisation de la mémoire pour la clé primaire et certaines autres opérations. [#60050](https://github.com/ClickHouse/ClickHouse/pull/60050) ([Alexey Milovidov](https://github.com/alexey-milovidov)).
* Les clés primaires des tables seront chargées en mémoire de manière paresseuse lors du premier accès. Cela est contrôlé par le nouveau paramètre MergeTree `primary_key_lazy_load`, activé par défaut. Cela offre plusieurs avantages : - elles ne seront pas chargées pour les tables qui ne sont pas utilisées ; - s’il n’y a pas assez de mémoire, une exception sera levée lors de la première utilisation plutôt qu’au démarrage du serveur. Cela présente aussi plusieurs inconvénients : - la latence de chargement de la clé primaire sera supportée lors de la première requête plutôt qu’avant l’acceptation des connexions ; cela peut théoriquement introduire un effet de troupeau. Cela résout [#11188](https://github.com/ClickHouse/ClickHouse/issues/11188). [#60093](https://github.com/ClickHouse/ClickHouse/pull/60093) ([Alexey Milovidov](https://github.com/alexey-milovidov)).
* Fonction vectorisée `dotProduct`, utile pour la recherche vectorielle. [#60202](https://github.com/ClickHouse/ClickHouse/pull/60202) ([Robert Schulze](https://github.com/rschu1ze)).
* Si la clé primaire de la table contient principalement des colonnes inutiles, ne les gardez pas en mémoire. Cela est contrôlé par un nouveau paramètre `primary_key_ratio_of_unique_prefix_values_to_skip_suffix_columns`, avec la valeur `0.9` par défaut, ce qui signifie : pour une clé primaire composite, si une colonne change de valeur au moins 0.9 fois sur l’ensemble des cas, les colonnes suivantes ne seront pas chargées. [#60255](https://github.com/ClickHouse/ClickHouse/pull/60255) ([Alexey Milovidov](https://github.com/alexey-milovidov)).
* Exécute la fonction multiIf de manière colonnaire lorsque le type sous-jacent de result&#95;type est numérique. [#60384](https://github.com/ClickHouse/ClickHouse/pull/60384) ([李扬](https://github.com/taiyang-li)).
* Comme le montre la fig. 1, remplacer &quot;&amp;&amp;&quot; par &quot;&amp;&quot; peut générer du code SIMD. ![image](https://github.com/ClickHouse/ClickHouse/assets/26588299/a5a72ac4-6dc6-4d52-835a-4f512e55f0b9) Fig. 1. Code compilé à partir de &#39;&amp;&amp;&#39; (à gauche) et de &#39;&amp;&#39; (à droite). [#60498](https://github.com/ClickHouse/ClickHouse/pull/60498) ([Zhiguo Zhou](https://github.com/ZhiguoZh)).
* Mutex presque 2x plus rapides (ils étaient plus lents à cause de ThreadFuzzer). [#60823](https://github.com/ClickHouse/ClickHouse/pull/60823) ([Azat Khuzhin](https://github.com/azat)).
* Déplacement du drainage des connections de prepare vers work, avec drainage de plusieurs connections en parallèle. [#60845](https://github.com/ClickHouse/ClickHouse/pull/60845) ([lizhuoyu5](https://github.com/lzydmxy)).
* Optimisation de insertManyFrom pour les nombres Nullable ou les chaînes Nullable. [#60846](https://github.com/ClickHouse/ClickHouse/pull/60846) ([李扬](https://github.com/taiyang-li)).
* Optimisation de la fonction `dotProduct` pour éviter des copies mémoire inutiles et coûteuses. [#60928](https://github.com/ClickHouse/ClickHouse/pull/60928) ([Robert Schulze](https://github.com/rschu1ze)).
* Les opérations sur le filesystem cache souffriront moins de la contention sur les verrous. [#61066](https://github.com/ClickHouse/ClickHouse/pull/61066) ([Alexey Milovidov](https://github.com/alexey-milovidov)).
* Optimisation de ColumnString::replicate et prévention de l&#39;optimisation de memcpySmallAllowReadWriteOverflow15Impl en memcpy intégré. Close [#61074](https://github.com/ClickHouse/ClickHouse/issues/61074). ColumnString::replicate est 2,46x plus rapide sur x86-64. [#61075](https://github.com/ClickHouse/ClickHouse/pull/61075) ([李扬](https://github.com/taiyang-li)).
* Affichage 30x plus rapide pour les entiers 256 bits. [#61100](https://github.com/ClickHouse/ClickHouse/pull/61100) ([Raúl Marín](https://github.com/Algunenano)).
* Si une query contenant une erreur de syntaxe incluait un sélecteur COLUMNS avec une expression régulière, celle-ci était compilée à chaque retour arrière du parser, au lieu de l&#39;être une seule fois. C&#39;était une erreur fondamentale. La regexp compilée était placée dans l&#39;AST. Mais le A de AST signifie &quot;abstract&quot;, ce qui implique qu&#39;il ne devrait pas contenir d&#39;objets lourds. Certaines parties de l&#39;AST peuvent être créées puis supprimées pendant le parsing, notamment lors de nombreux retours arrière. Cela ralentit le parsing et permet donc à un utilisateur readonly de provoquer un DoS. Mais le principal problème est que cela freine les progrès des fuzzers. [#61543](https://github.com/ClickHouse/ClickHouse/pull/61543) ([Alexey Milovidov](https://github.com/alexey-milovidov)).

* Autorise la définition de `volume_priority` dans `storage_configuration`. [#58533](https://github.com/ClickHouse/ClickHouse/pull/58533) ([Andrey Zvonov](https://github.com/zvonand)).
