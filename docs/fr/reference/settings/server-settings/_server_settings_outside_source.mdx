<div id="asynchronous_metric_log">
  ## asynchronous_metric_log
</div>

Activé par défaut dans les déploiements ClickHouse Cloud.

Si ce paramètre n’est pas activé par défaut dans votre environnement, vous pouvez, selon la façon dont ClickHouse a été installé, suivre les instructions ci-dessous pour l’activer ou le désactiver.

**Activation**

Pour activer manuellement la collecte de l’historique du journal des métriques asynchrones [`system.asynchronous_metric_log`](/fr/reference/system-tables/asynchronous_metric_log), créez `/etc/clickhouse-server/config.d/asynchronous_metric_log.xml` avec le contenu suivant :

```xml
<clickhouse>
     <asynchronous_metric_log>
        <database>system</database>
        <table>asynchronous_metric_log</table>
        <flush_interval_milliseconds>7500</flush_interval_milliseconds>
        <collect_interval_milliseconds>1000</collect_interval_milliseconds>
        <max_size_rows>1048576</max_size_rows>
        <reserved_size_rows>8192</reserved_size_rows>
        <buffer_size_rows_flush_threshold>524288</buffer_size_rows_flush_threshold>
        <flush_on_crash>false</flush_on_crash>
    </asynchronous_metric_log>
</clickhouse>
```

**Désactivation**

Pour désactiver le paramètre `asynchronous_metric_log`, vous devez créer le fichier `/etc/clickhouse-server/config.d/disable_asynchronous_metric_log.xml` suivant avec le contenu suivant :

```xml
<clickhouse><asynchronous_metric_log remove="1" /></clickhouse>
```

<div id="auth_use_forwarded_address">
  ## auth_use_forwarded_address
</div>

Utilise l’adresse d’origine pour authentifier les clients connectés via un proxy.

<Note>
  Ce paramètre doit être utilisé avec une extrême prudence, car les adresses transmises peuvent être facilement usurpées ; les serveurs qui acceptent ce mode d’authentification ne doivent pas être accessibles directement, mais uniquement via un proxy de confiance.
</Note>

<div id="bcrypt_workfactor">
  ## bcrypt_workfactor
</div>

Facteur de coût du type d&#39;authentification `bcrypt_password`, qui utilise l&#39;[algorithme Bcrypt](https://wildlyinaccurate.com/bcrypt-choosing-a-work-factor/).
Le facteur de coût définit la quantité de calculs et le temps nécessaires pour calculer le hachage et vérifier le mot de passe.

```xml
<bcrypt_workfactor>12</bcrypt_workfactor>
```

<Warning>
  Pour les applications nécessitant une authentification fréquente,
  envisagez d’autres méthodes d’authentification en raison du
  surcoût de calcul de bcrypt avec des facteurs de travail élevés.
</Warning>

<div id="table_engines_require_grant">
  ## table_engines_require_grant
</div>

Si ce paramètre est défini sur true, les utilisateurs doivent disposer d&#39;un privilège pour créer une table avec un moteur spécifique, par ex. `GRANT TABLE ENGINE ON TinyLog to user`.

<Note>
  Par défaut, pour assurer la rétrocompatibilité, la création d&#39;une table avec un moteur de table spécifique ne tient pas compte du privilège, mais vous pouvez modifier ce comportement en définissant ce paramètre sur true.
</Note>

<div id="builtin_dictionaries_reload_interval">
  ## builtin_dictionaries_reload_interval
</div>

Intervalle, en secondes, avant le rechargement des dictionnaires intégrés.

ClickHouse recharge les dictionnaires intégrés toutes les x secondes. Cela permet de modifier les dictionnaires « à la volée » sans redémarrer le serveur.

**Exemple**


<div id="compression">
  ## compression
</div>

Paramètres de compression des données pour les tables utilisant le moteur [MergeTree](/fr/reference/engines/table-engines/mergetree-family/mergetree).

<Note>
  Nous vous recommandons de ne pas modifier ces paramètres si vous débutez avec ClickHouse.
</Note>

**Modèle de configuration**:

```xml
<compression>
    <case>
      <min_part_size>...</min_part_size>
      <min_part_size_ratio>...</min_part_size_ratio>
      <method>...</method>
      <level>...</level>
    </case>
    ...
</compression>
```

**Champs `<case>`** :

* `min_part_size` – Taille minimale d’une partie de données.
* `min_part_size_ratio` – Ratio entre la taille de la partie de données et celle de la table.
* `method` – Méthode de compression. Valeurs acceptées : `lz4`, `lz4hc`, `zstd`,`deflate_qpl`.
* `level` – Niveau de compression. Voir [Codecs](/fr/reference/statements/create/table#general-purpose-codecs).

<Note>
  Vous pouvez configurer plusieurs sections `<case>`.
</Note>

**Actions lorsque les conditions sont remplies** :

* Si une partie de données correspond à un jeu de conditions, ClickHouse utilise la méthode de compression spécifiée.
* Si une partie de données correspond à plusieurs jeux de conditions, ClickHouse utilise le premier jeu de conditions correspondant.

<Note>
  Si aucune condition n’est remplie pour une partie de données, ClickHouse utilise la compression `lz4`.
</Note>

**Exemple**

```xml
<compression incl="clickhouse_compression">
    <case>
        <min_part_size>10000000000</min_part_size>
        <min_part_size_ratio>0.01</min_part_size_ratio>
        <method>zstd</method>
        <level>1</level>
    </case>
</compression>
```

<div id="encryption">
  ## chiffrement
</div>

Configure une commande permettant d&#39;obtenir une clé à utiliser avec les [codecs de chiffrement](/fr/reference/statements/create/table#encryption-codecs). La ou les clés doivent être écrites dans des variables d&#39;environnement ou définies dans le fichier de configuration.

Les clés peuvent être hexadécimales ou prendre la forme de chaînes de 16 octets.

**Exemple**

Chargement depuis la configuration :

```xml
<encryption_codecs>
    <aes_128_gcm_siv>
        <key>1234567812345678</key>
    </aes_128_gcm_siv>
</encryption_codecs>
```

<Note>
  Il est déconseillé de stocker les clés dans le fichier de configuration. Ce n’est pas sûr. Vous pouvez déplacer les clés vers un fichier de configuration distinct sur un disque sécurisé, puis placer un lien symbolique vers ce fichier dans le dossier `config.d/`.
</Note>

Chargement depuis le fichier de configuration, lorsque la clé est au format hexadécimal :

```xml
<encryption_codecs>
    <aes_128_gcm_siv>
        <key_hex>00112233445566778899aabbccddeeff</key_hex>
    </aes_128_gcm_siv>
</encryption_codecs>
```

Chargement de la clé à partir de la variable d’environnement :

```xml
<encryption_codecs>
    <aes_128_gcm_siv>
        <key_hex from_env="ENVVAR"></key_hex>
    </aes_128_gcm_siv>
</encryption_codecs>
```

Ici, `current_key_id` définit la clé actuellement utilisée pour le chiffrement, et toutes les clés indiquées peuvent servir au déchiffrement.

Chacune de ces méthodes peut être appliquée à plusieurs clés :

```xml
<encryption_codecs>
    <aes_128_gcm_siv>
        <key_hex id="0">00112233445566778899aabbccddeeff</key_hex>
        <key_hex id="1" from_env="ENVVAR"></key_hex>
        <current_key_id>1</current_key_id>
    </aes_128_gcm_siv>
</encryption_codecs>
```

Ici, `current_key_id` indique la clé actuellement utilisée pour le chiffrement.

Les utilisateurs peuvent également ajouter un nonce, qui doit faire 12 octets de long (par défaut, les processus de chiffrement et de déchiffrement utilisent un nonce composé d’octets nuls) :

```xml
<encryption_codecs>
    <aes_128_gcm_siv>
        <nonce>012345678910</nonce>
    </aes_128_gcm_siv>
</encryption_codecs>
```

Ou il peut être indiqué en hexadécimal :

```xml
<encryption_codecs>
    <aes_128_gcm_siv>
        <nonce_hex>abcdefabcdef</nonce_hex>
    </aes_128_gcm_siv>
</encryption_codecs>
```

<Note>
  Tout ce qui est mentionné ci-dessus s’applique à `aes_256_gcm_siv` (mais la clé doit comporter 32 octets).
</Note>

<div id="error_log">
  ## error_log
</div>

Il est désactivé par défaut.

**Activation**

Pour activer manuellement la collecte de l’historique des erreurs [`system.error_log`](/fr/reference/system-tables/error_log), créez `/etc/clickhouse-server/config.d/error_log.xml` avec le contenu suivant :

```xml
<clickhouse>
    <error_log>
        <database>system</database>
        <table>error_log</table>
        <flush_interval_milliseconds>7500</flush_interval_milliseconds>
        <collect_interval_milliseconds>1000</collect_interval_milliseconds>
        <max_size_rows>1048576</max_size_rows>
        <reserved_size_rows>8192</reserved_size_rows>
        <buffer_size_rows_flush_threshold>524288</buffer_size_rows_flush_threshold>
        <flush_on_crash>false</flush_on_crash>
    </error_log>
</clickhouse>
```

**Désactivation**

Pour désactiver le paramètre `error_log`, créez le fichier suivant `/etc/clickhouse-server/config.d/disable_error_log.xml` avec le contenu ci-dessous :

```xml
<clickhouse>
    <error_log remove="1" />
</clickhouse>
```

<div id="graphite">
  ## graphite
</div>

Envoi de données vers [Graphite](https://github.com/graphite-project).

Paramètres :

* `host` – Le serveur Graphite.
* `port` – Le port du serveur Graphite.
* `interval` – L’intervalle d’envoi, en secondes.
* `timeout` – Le délai d’expiration pour l’envoi des données, en secondes.
* `root_path` – Préfixe pour les clés.
* `metrics` – Envoi de données depuis la table [system.metrics](/fr/reference/system-tables/metrics).
* `events` – Envoi des données delta accumulées sur la période depuis la table [system.events](/fr/reference/system-tables/events).
* `events_cumulative` – Envoi de données cumulées depuis la table [system.events](/fr/reference/system-tables/events).
* `asynchronous_metrics` – Envoi de données depuis la table [system.asynchronous&#95;metrics](/fr/reference/system-tables/asynchronous_metrics).

Vous pouvez configurer plusieurs sections `<graphite>`. Par exemple, vous pouvez les utiliser pour envoyer des données différentes à des intervalles différents.

**Exemple**

```xml
<graphite>
    <host>localhost</host>
    <port>42000</port>
    <timeout>0.1</timeout>
    <interval>60</interval>
    <root_path>one_min</root_path>
    <metrics>true</metrics>
    <events>true</events>
    <events_cumulative>false</events_cumulative>
    <asynchronous_metrics>true</asynchronous_metrics>
</graphite>
```

<div id="http_handlers">
  ## http_handlers
</div>

Permet d’utiliser des gestionnaires HTTP personnalisés.
Pour ajouter un nouveau gestionnaire HTTP, ajoutez simplement une nouvelle balise `<rule>`.
Les règles sont vérifiées de haut en bas dans l’ordre défini,
et la première correspondance exécute le gestionnaire.

Les paramètres suivants peuvent être configurés via des sous-balises :

| Sous-balises         | Définition                                                                                                                                                                                                                                                                    |
| -------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `url`                | Pour faire correspondre l’URL de la requête, vous pouvez utiliser le préfixe &#39;regex:&#39; pour effectuer une correspondance par expression régulière (facultatif)                                                                                                         |
| `methods`            | Pour faire correspondre les méthodes de requête, vous pouvez utiliser des virgules pour séparer plusieurs méthodes (facultatif)                                                                                                                                               |
| `headers`            | Pour faire correspondre les en-têtes de la requête, faites correspondre chaque élément enfant (le nom de l’élément enfant est le nom de l’en-tête) ; vous pouvez utiliser le préfixe &#39;regex:&#39; pour effectuer une correspondance par expression régulière (facultatif) |
| `handler`            | Le gestionnaire de requête                                                                                                                                                                                                                                                    |
| `empty_query_string` | Vérifie qu’il n’y a pas de query string dans l’URL                                                                                                                                                                                                                            |

`handler` contient les paramètres suivants, qui peuvent être configurés via des sous-balises :

| Sous-balises       | Définition                                                                                                                                                                                                                           |
| ------------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| `url`              | Une URL de redirection                                                                                                                                                                                                               |
| `type`             | Types pris en charge : static, dynamic&#95;query&#95;handler, predefined&#95;query&#95;handler, redirect                                                                                                                             |
| `status`           | À utiliser avec le type static : code d’état de la réponse                                                                                                                                                                           |
| `query_param_name` | À utiliser avec le type dynamic&#95;query&#95;handler : extrait et exécute la valeur correspondant à `<query_param_name>` dans les params de la requête HTTP                                                                         |
| `query`            | À utiliser avec le type predefined&#95;query&#95;handler : exécute la query lorsque le gestionnaire est appelé                                                                                                                       |
| `content_type`     | À utiliser avec le type static : type de contenu de la réponse                                                                                                                                                                       |
| `response_content` | À utiliser avec le type static : contenu de la réponse envoyé au client ; lors de l’utilisation du préfixe &#39;file://&#39; ou &#39;config://&#39;, récupère le contenu depuis le fichier ou la configuration et l’envoie au client |

En plus d’une liste de règles, vous pouvez spécifier `<defaults/>`, ce qui permet d’activer tous les gestionnaires par défaut.

Exemple :

```xml
<http_handlers>
    <rule>
        <url>/</url>
        <methods>POST,GET</methods>
        <headers><pragma>no-cache</pragma></headers>
        <handler>
            <type>dynamic_query_handler</type>
            <query_param_name>query</query_param_name>
        </handler>
    </rule>

    <rule>
        <url>/predefined_query</url>
        <methods>POST,GET</methods>
        <handler>
            <type>predefined_query_handler</type>
```

<div id="interserver_http_credentials">
  ## interserver_http_credentials
</div>

Un nom d’utilisateur et un mot de passe utilisés pour se connecter à d’autres serveurs lors de la [réplication](/fr/reference/engines/table-engines/mergetree-family/replication). De plus, le serveur authentifie les autres répliques à l’aide de ces identifiants.
`interserver_http_credentials` doit donc être identique pour toutes les répliques d’un cluster.

<Note>
  * Par défaut, si la section `interserver_http_credentials` est omise, l’authentification n’est pas utilisée lors de la réplication.
  * Les paramètres `interserver_http_credentials` ne sont pas liés à la [configuration](/fr/concepts/features/interfaces/client#configuration_files) des identifiants du client ClickHouse.
  * Ces identifiants sont communs à la réplication via `HTTP` et `HTTPS`.
</Note>

Les paramètres suivants peuvent être configurés à l’aide de sous-balises :

* `user` — Nom d’utilisateur.
* `password` — Mot de passe.
* `allow_empty` — Si `true`, les autres répliques sont autorisées à se connecter sans authentification, même si des identifiants sont définis. Si `false`, les connexions sans authentification sont refusées. Valeur par défaut : `false`.
* `old` — Contient les anciens `user` et `password` utilisés lors de la rotation des identifiants. Plusieurs sections `old` peuvent être spécifiées.

**Rotation des identifiants**

ClickHouse prend en charge la rotation dynamique des identifiants interserveurs sans devoir arrêter toutes les répliques en même temps pour mettre à jour leur configuration. Les identifiants peuvent être modifiés en plusieurs étapes.

Pour activer l’authentification, définissez `interserver_http_credentials.allow_empty` sur `true` et ajoutez des identifiants. Cela autorise les connexions avec ou sans authentification.

```xml
<interserver_http_credentials>
    <user>admin</user>
    <password>111</password>
    <allow_empty>true</allow_empty>
</interserver_http_credentials>
```

Après avoir configuré toutes les répliques, définissez `allow_empty` sur `false` ou supprimez ce paramètre. Cela rend obligatoire l’authentification avec les nouveaux identifiants.

Pour modifier les identifiants existants, déplacez le nom d’utilisateur et le mot de passe dans la section `interserver_http_credentials.old`, puis mettez à jour `user` et `password` avec les nouvelles valeurs. À ce stade, le serveur utilise les nouveaux identifiants pour se connecter aux autres répliques et accepte les connexions avec les anciens comme avec les nouveaux identifiants.

```xml
<interserver_http_credentials>
    <user>admin</user>
    <password>222</password>
    <old>
        <user>admin</user>
        <password>111</password>
    </old>
    <old>
        <user>temp</user>
        <password>000</password>
    </old>
</interserver_http_credentials>
```

Lorsque les nouveaux identifiants ont été appliqués à toutes les répliques, les anciens peuvent être supprimés.

<div id="ldap_servers">
  ## ldap_servers
</div>

Indiquez ici les serveurs LDAP avec leurs paramètres de connexion afin de :

* les utiliser comme mécanismes d&#39;authentification pour des utilisateurs locaux dédiés, pour lesquels le mécanisme d&#39;authentification `ldap` est spécifié au lieu de `password`
* les utiliser comme répertoires d&#39;utilisateurs distants.

Les paramètres suivants peuvent être configurés à l&#39;aide de sous-balises :

| Setting                        | Description                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                         |
| ------------------------------ | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `bind_dn`                      | Modèle utilisé pour construire le DN à utiliser pour l&#39;opération de bind. Le DN résultant est construit en remplaçant toutes les sous-chaînes `\{user_name\}` du modèle par le nom d&#39;utilisateur réel à chaque tentative d&#39;authentification.                                                                                                                                                                                                                                                                                                                            |
| `enable_tls`                   | Indicateur qui active l&#39;utilisation d&#39;une connexion sécurisée au serveur LDAP. Spécifiez `no` pour le protocole en texte brut (`ldap://`) (non recommandé). Spécifiez `yes` pour le protocole LDAP sur SSL/TLS (`ldaps://`) (recommandé, valeur par défaut). Spécifiez `starttls` pour le protocole StartTLS legacy (protocole en texte brut (`ldap://`), mis à niveau vers TLS).                                                                                                                                                                                           |
| `host`                         | Nom d&#39;hôte ou adresse IP du serveur LDAP ; ce paramètre est obligatoire et ne peut pas être vide.                                                                                                                                                                                                                                                                                                                                                                                                                                                                               |
| `port`                         | Port du serveur LDAP ; la valeur par défaut est 636 si `enable_tls` est défini sur true, sinon `389`.                                                                                                                                                                                                                                                                                                                                                                                                                                                                               |
| `tls_ca_cert_dir`              | chemin vers le répertoire contenant les certificats de CA.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                          |
| `tls_ca_cert_file`             | chemin vers le fichier de certificat de CA.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                         |
| `tls_cert_file`                | chemin vers le fichier de certificat.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                               |
| `tls_cipher_suite`             | suite de chiffrement autorisée (notation OpenSSL).                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                  |
| `tls_key_file`                 | chemin vers le fichier de clé du certificat.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                        |
| `tls_minimum_protocol_version` | Version minimale du protocole SSL/TLS. Les valeurs acceptées sont : `ssl2`, `ssl3`, `tls1.0`, `tls1.1`, `tls1.2` (valeur par défaut).                                                                                                                                                                                                                                                                                                                                                                                                                                               |
| `tls_require_cert`             | Comportement de la vérification du certificat du pair SSL/TLS. Les valeurs acceptées sont : `never`, `allow`, `try`, `demand` (valeur par défaut).                                                                                                                                                                                                                                                                                                                                                                                                                                  |
| `user_dn_detection`            | Section contenant les paramètres de recherche LDAP permettant de détecter le DN utilisateur réel de l&#39;utilisateur lié. Cela est principalement utilisé dans les filtres de recherche pour un mappage de rôles ultérieur lorsque le serveur est Active Directory. Le DN utilisateur résultant sera utilisé lors du remplacement des sous-chaînes `\{user_dn\}` partout où elles sont autorisées. Par défaut, le DN utilisateur est défini comme étant égal au bind DN, mais une fois la recherche effectuée, il sera mis à jour avec la valeur réelle du DN utilisateur détecté. |
| `verification_cooldown`        | Période, en secondes, après une tentative de bind réussie, pendant laquelle un utilisateur est considéré comme authentifié avec succès pour toutes les requêtes consécutives sans contacter le serveur LDAP. Spécifiez `0` (valeur par défaut) pour désactiver la mise en cache et forcer le contact avec le serveur LDAP pour chaque requête d&#39;authentification.                                                                                                                                                                                                               |

Le paramètre `user_dn_detection` peut être configuré avec des sous-balises :

| Setting         | Description                                                                                                                                                                                                                                                                                                                                                                          |
| --------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| `base_dn`       | modèle utilisé pour construire le base DN pour la recherche LDAP. Le DN résultant sera construit en remplaçant toutes les sous-chaînes `\{user_name\}` et `\{bind_dn\}` du modèle par le nom d&#39;utilisateur réel et le bind DN pendant la recherche LDAP.                                                                                                                         |
| `scope`         | portée de la recherche LDAP. Les valeurs acceptées sont : `base`, `one_level`, `children`, `subtree` (valeur par défaut).                                                                                                                                                                                                                                                            |
| `search_filter` | modèle utilisé pour construire le filtre de recherche pour la recherche LDAP. Le filtre résultant sera construit en remplaçant toutes les sous-chaînes `\{user_name\}`, `\{bind_dn\}` et `\{base_dn\}` du modèle par le nom d&#39;utilisateur réel, le bind DN et le base DN pendant la recherche LDAP. Notez que les caractères spéciaux doivent être correctement échappés en XML. |

Exemple :

```xml
<my_ldap_server>
    <host>localhost</host>
    <port>636</port>
    <bind_dn>uid={user_name},ou=users,dc=example,dc=com</bind_dn>
    <verification_cooldown>300</verification_cooldown>
    <enable_tls>yes</enable_tls>
    <tls_minimum_protocol_version>tls1.2</tls_minimum_protocol_version>
    <tls_require_cert>demand</tls_require_cert>
    <tls_cert_file>/path/to/tls_cert_file</tls_cert_file>
    <tls_key_file>/path/to/tls_key_file</tls_key_file>
    <tls_ca_cert_file>/path/to/tls_ca_cert_file</tls_ca_cert_file>
    <tls_ca_cert_dir>/path/to/tls_ca_cert_dir</tls_ca_cert_dir>
    <tls_cipher_suite>ECDHE-ECDSA-AES256-GCM-SHA384:ECDHE-RSA-AES256-GCM-SHA384:AES256-GCM-SHA384</tls_cipher_suite>
</my_ldap_server>
```

Exemple (cas typique d’Active Directory avec détection configurée du DN utilisateur en vue d’un mappage ultérieur des rôles) :

```xml
<my_ad_server>
    <host>localhost</host>
    <port>389</port>
    <bind_dn>EXAMPLE\{user_name}</bind_dn>
    <user_dn_detection>
        <base_dn>CN=Users,DC=example,DC=com</base_dn>
        <search_filter>(&amp;(objectClass=user)(sAMAccountName={user_name}))</search_filter>
    </user_dn_detection>
    <enable_tls>no</enable_tls>
</my_ad_server>
```

| `facility` | Le [mot-clé facility](https://en.wikipedia.org/wiki/Syslog#Facility) de syslog. | `format`   | Format du message de log. Valeurs possibles : `bsd` et `syslog.`                                                                                                                                                                                                                                |

**Formats de logs**

Vous pouvez spécifier le format de log qui sera affiché dans les logs de la console. Actuellement, seul JSON est pris en charge.

**Exemple**

Voici un exemple de log JSON en sortie :

```json
{
  "date_time_utc": "2024-11-06T09:06:09Z",
  "date_time": "1650918987.180175",
  "thread_name": "#1",
  "thread_id": "254545",
  "level": "Trace",
  "query_id": "",
  "logger_name": "BaseDaemon",
  "message": "Received signal 2",
  "source_file": "../base/daemon/BaseDaemon.cpp; virtual void SignalListener::run()",
  "source_line": "192"
}
```

Pour activer la prise en charge de la journalisation au format JSON, utilisez l&#39;extrait suivant :

```xml
<logger>
    <formatting>
        <type>json</type>
        <!-- Can be configured on a per-channel basis (log, errorlog, console, syslog), or globally for all channels (then just omit it). -->
        <!-- <channel></channel> -->
        <names>
            <date_time>date_time</date_time>
            <thread_name>thread_name</thread_name>
            <thread_id>thread_id</thread_id>
            <level>level</level>
            <query_id>query_id</query_id>
            <logger_name>logger_name</logger_name>
            <message>message</message>
            <source_file>source_file</source_file>
            <source_line>source_line</source_line>
        </names>
    </formatting>
</logger>
```

**Renommer les clés des logs JSON**

Les noms de clés peuvent être modifiés en changeant les valeurs des balises à l’intérieur de la balise `<names>`. Par exemple, pour remplacer `DATE_TIME` par `MY_DATE_TIME`, vous pouvez utiliser `<date_time>MY_DATE_TIME</date_time>`.

**Omettre des clés dans les logs JSON**

Les propriétés des logs peuvent être omises en les mettant en commentaire. Par exemple, si vous ne voulez pas que votre log affiche `query_id`, vous pouvez mettre la balise `<query_id>` en commentaire.

```xml
<storage_configuration>
    <disks>
        <disk_name_1>
```

Les sous-balises ci-dessus définissent les paramètres suivants pour `disks` :

| Setting                 | Description                                                                                                       |
| ----------------------- | ----------------------------------------------------------------------------------------------------------------- |
| `<disk_name_N>`         | Le nom du disque, qui doit être unique.                                                                           |
| `path`                  | Le chemin où seront stockées les données du serveur (répertoires `data` et `shadow`). Il doit se terminer par `/` |
| `keep_free_space_bytes` | Taille de l’espace libre réservé sur le disque.                                                                   |

<Note>
  L’ordre des disques n’a pas d’importance.
</Note>

<div id="configuration-of-policies">
  ### Configuration des politiques
</div>

Les sous-balises ci-dessus définissent les paramètres suivants pour `policies` :

| Setting                      | Description                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                            |
| ---------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `policy_name_N`              | Nom de la politique. Les noms de politique doivent être uniques.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                       |
| `volume_name_N`              | Nom du volume. Les noms de volume doivent être uniques.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                |
| `disk`                       | Le disque situé dans le volume.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                        |
| `max_data_part_size_bytes`   | Taille maximale d&#39;un fragment de données pouvant résider sur l&#39;un des disques de ce volume. Si une fusion produit un fragment dont la taille prévue dépasse `max_data_part_size_bytes`, ce fragment sera écrit sur le volume suivant. En pratique, cette fonctionnalité vous permet de stocker les nouveaux petits fragments sur un volume chaud (SSD), puis de les déplacer vers un volume froid (HDD) lorsqu&#39;ils deviennent volumineux. N&#39;utilisez pas cette option si la politique ne comporte qu&#39;un seul volume.                                                                                               |
| `move_factor`                | La part d&#39;espace libre disponible sur le volume. Si l&#39;espace disponible passe en dessous de cette valeur, les données commenceront à être transférées vers le volume suivant, s&#39;il y en a un. Pour ce transfert, les fragments sont triés par taille, du plus grand au plus petit (ordre décroissant), puis les fragments dont la taille cumulée suffit à satisfaire la condition `move_factor` sont sélectionnés. Si la taille totale de tous les fragments est insuffisante, tous les fragments seront déplacés.                                                                                                         |
| `perform_ttl_move_on_insert` | Désactive le déplacement, lors de l&#39;insertion, des données dont le TTL a expiré. Par défaut (si cette option est activée), si des données déjà expirées selon la règle de déplacement basée sur la durée de vie sont insérées, elles sont immédiatement déplacées vers le volume / disque spécifié par cette règle. Cela peut considérablement ralentir l&#39;insertion si le volume / disque cible est lent (par exemple S3). Si cette option est désactivée, la partie expirée des données est d&#39;abord écrite sur le volume par défaut, puis immédiatement déplacée vers le volume spécifié par la règle pour le TTL expiré. |
| `load_balancing`             | Politique d&#39;équilibrage des disques : `round_robin` ou `least_used`.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                               |
| `least_used_ttl_ms`          | Définit le délai d&#39;expiration (en millisecondes) pour la mise à jour de l&#39;espace disponible sur tous les disques (`0` - toujours mettre à jour, `-1` - ne jamais mettre à jour, la valeur par défaut est `60000`). Notez que si le disque est utilisé uniquement par ClickHouse et ne fait pas l&#39;objet d&#39;un redimensionnement du système de fichiers à chaud, vous pouvez utiliser la valeur `-1`. Dans tous les autres cas, cela n&#39;est pas recommandé, car cela finira par entraîner une allocation incorrecte de l&#39;espace.                                                                                   |
| `prefer_not_to_merge`        | Désactive la fusion des parties de données sur ce volume. Remarque : cela peut être préjudiciable et entraîner des ralentissements. Lorsque ce paramètre est activé (ne faites pas cela), la fusion des données sur ce volume est interdite (ce qui est déconseillé). Cela permet de contrôler la manière dont ClickHouse interagit avec les disques lents. Nous recommandons de ne pas utiliser ce paramètre.                                                                                                                                                                                                                         |
| `volume_priority`            | Définit la priorité (l&#39;ordre) de remplissage des volumes. Plus la valeur est petite, plus la priorité est élevée. Les valeurs du paramètre doivent être des nombres naturels et couvrir la plage de 1 à N (N étant la plus grande valeur de paramètre spécifiée) sans lacunes.                                                                                                                                                                                                                                                                                                                                                     |

Pour `volume_priority` :

* Si tous les volumes ont ce paramètre, ils sont priorisés dans l&#39;ordre spécifié.
* Si seuls *certains* volumes l&#39;ont, les volumes qui ne l&#39;ont pas ont la priorité la plus basse. Ceux qui l&#39;ont sont priorisés selon la valeur de la balise, tandis que la priorité des autres est déterminée par leur ordre de description dans le fichier de configuration.
* Si *aucun* volume ne reçoit ce paramètre, leur ordre est déterminé par l&#39;ordre de leur description dans le fichier de configuration.
* La priorité des volumes ne peut pas être identique.

* `endpoint` – endpoint HTTP de collecte des métriques par le serveur Prometheus. Doit commencer par &#39;/&#39;.
* `port` – Port de `endpoint`.
* `metrics` – Expose les métriques de la table [system.metrics](/fr/reference/system-tables/metrics).
* `events` – Expose les métriques de la table [system.events](/fr/reference/system-tables/events).
* `asynchronous_metrics` – Expose les valeurs actuelles des métriques de la table [system.asynchronous&#95;metrics](/fr/reference/system-tables/asynchronous_metrics).
* `errors` - Expose le nombre d’erreurs par code d’erreur survenues depuis le dernier redémarrage du serveur. Ces informations peuvent également être obtenues à partir de [system.errors](/fr/reference/system-tables/errors).

**Exemple**

```xml highlight={5-12}
<clickhouse>
    <listen_host>0.0.0.0</listen_host>
    <http_port>8123</http_port>
    <tcp_port>9000</tcp_port>
    <prometheus>
        <endpoint>/metrics</endpoint>
        <port>9363</port>
        <metrics>true</metrics>
        <events>true</events>
        <asynchronous_metrics>true</asynchronous_metrics>
        <errors>true</errors>
    </prometheus>
</clickhouse>
```

Vérifiez (remplacez `127.0.0.1` par l’adresse IP ou le nom d’hôte de votre serveur ClickHouse) :

```bash
curl 127.0.0.1:9363/metrics
```

<div id="query_masking_rules">
  ## query_masking_rules
</div>

Règles fondées sur des expressions régulières, appliquées aux requêtes ainsi qu’à tous les messages de journalisation avant leur enregistrement dans les journaux du serveur,
les tables [`system.query_log`](/fr/reference/system-tables/query_log), [`system.text_log`](/fr/reference/system-tables/text_log), [`system.processes`](/fr/reference/system-tables/processes), ainsi que dans les journaux envoyés au client. Cela permet d’éviter
la divulgation de données sensibles issues de requêtes SQL, telles que des noms, des adresses e-mail, des identifiants personnels ou des numéros de carte de crédit, dans les journaux.

**Exemple**

```xml
<query_masking_rules>
    <rule>
        <name>hide SSN</name>
        <regexp>(^|\D)\d{3}-\d{2}-\d{4}($|\D)</regexp>
        <replace>000-00-0000</replace>
    </rule>
</query_masking_rules>
```

**Champs de configuration**:

| Paramètre | Description                                                                                  |
| --------- | -------------------------------------------------------------------------------------------- |
| `name`    | nom de la règle (facultatif)                                                                 |
| `regexp`  | expression régulière compatible avec RE2 (obligatoire)                                       |
| `replace` | chaîne de substitution pour les données sensibles (facultatif, par défaut : six astérisques) |

Les règles de masquage s’appliquent à l’ensemble de la requête (afin d’éviter toute fuite de données sensibles provenant de requêtes malformées / impossibles à analyser).

La table [`system.events`](/fr/reference/system-tables/events) comporte le compteur `QueryMaskingRulesMatch`, qui indique le nombre total de correspondances avec les règles de masquage des requêtes.

Pour les requêtes distribuées, chaque serveur doit être configuré séparément, sinon les sous-requêtes transmises à d’autres
nœuds seront stockées sans masquage.

<div id="remote_url_allow_hosts">
  ## remote_url_allow_hosts
</div>

Liste des hôtes autorisés à être utilisés dans les moteurs de stockage et les fonctions de table liés aux URL.

Lors de l’ajout d’un hôte avec la balise XML `\<host\>` :

* il doit être indiqué exactement comme dans l’URL, car le nom est vérifié avant la résolution DNS. Par exemple : `<host>clickhouse.com</host>`
* si le port est explicitement indiqué dans l’URL, alors `host:port` est vérifié dans son ensemble. Par exemple : `<host>clickhouse.com:80</host>`
* si l’hôte est indiqué sans port, alors tous les ports de cet hôte sont autorisés. Par exemple : si `<host>clickhouse.com</host>` est indiqué, alors `clickhouse.com:20` (FTP), `clickhouse.com:80` (HTTP), `clickhouse.com:443` (HTTPS), etc. sont autorisés.
* si l’hôte est indiqué sous la forme d’une adresse IP, alors il est vérifié tel qu’il apparaît dans l’URL. Par exemple : `[2a02:6b8:a::a]`.
* s’il y a des redirections et que leur prise en charge est activée, alors chaque redirection (le champ `location`) est vérifiée.

Par exemple :

```sql
<remote_url_allow_hosts>
    <host>clickhouse.com</host>
</remote_url_allow_hosts>
```

<div id="zookeeper">
  ## zookeeper
</div>

Contient des paramètres permettant à ClickHouse d&#39;interagir avec un cluster [ZooKeeper](http://zookeeper.apache.org/). ClickHouse utilise ZooKeeper pour stocker les métadonnées des répliques lors de l&#39;utilisation de tables répliquées. Si vous n&#39;utilisez pas de tables répliquées, cette section de paramètres peut être omise.

Les paramètres suivants peuvent être configurés via des sous-balises :

| Paramètre                                         | Description                                                                                                                                                                                                                                                                                                                                                                                                                        |
| ------------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `node`                                            | Point de terminaison ZooKeeper. Vous pouvez définir plusieurs endpoints. Ex. `<node index="1"><host>example_host</host><port>2181</port></node>`. L&#39;attribut `index` indique l&#39;ordre des nœuds lors des tentatives de connexion au cluster ZooKeeper.                                                                                                                                                                      |
| `operation_timeout_ms`                            | Délai d&#39;expiration maximal pour une opération, en millisecondes.                                                                                                                                                                                                                                                                                                                                                               |
| `session_timeout_ms`                              | Délai d&#39;expiration maximal pour la session du client, en millisecondes.                                                                                                                                                                                                                                                                                                                                                        |
| `root` (facultatif)                               | Le znode utilisé comme racine pour les znodes employés par le serveur ClickHouse.                                                                                                                                                                                                                                                                                                                                                  |
| `fallback_session_lifetime.min` (facultatif)      | Limite minimale de la durée de vie d&#39;une session ZooKeeper vers le nœud de secours lorsque le primaire n&#39;est pas disponible (équilibrage de charge). Exprimée en secondes. Par défaut : 3 heures.                                                                                                                                                                                                                          |
| `fallback_session_lifetime.max` (facultatif)      | Limite maximale de la durée de vie d&#39;une session ZooKeeper vers le nœud de secours lorsque le primaire n&#39;est pas disponible (équilibrage de charge). Exprimée en secondes. Par défaut : 6 heures.                                                                                                                                                                                                                          |
| `identity` (facultatif)                           | Utilisateur et mot de passe requis par ZooKeeper pour accéder aux znodes demandés.                                                                                                                                                                                                                                                                                                                                                 |
| `use_compression` (facultatif)                    | Active la compression dans le protocole Keeper si défini sur `true`.                                                                                                                                                                                                                                                                                                                                                               |
| `use_xid_64` (facultatif)                         | Active les identifiants de transaction 64 bits. Définissez `true` pour activer le format étendu d&#39;identifiant de transaction. Par défaut : `false`.                                                                                                                                                                                                                                                                            |
| `pass_opentelemetry_tracing_context` (facultatif) | Active la propagation du contexte de tracing OpenTelemetry vers les requêtes Keeper. Lorsqu&#39;elle est activée, des spans de tracing sont créés pour les opérations Keeper, ce qui permet un traçage distribué entre ClickHouse et Keeper. Voir [Tracing ClickHouse Keeper Requests](/fr/guides/oss/deployment-and-scaling/monitoring/opentelemetry#tracing-clickhouse-keeper-requests) pour plus de détails. Par défaut : `false`. |

Il existe également le paramètre `zookeeper_load_balancing` (facultatif), qui permet de sélectionner l&#39;algorithme de sélection des nœuds ZooKeeper :

| Nom de l&#39;algorithme         | Description                                                                                                                                        |
| ------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------- |
| `random`                        | sélectionne aléatoirement l&#39;un des nœuds ZooKeeper.                                                                                            |
| `in_order`                      | sélectionne le premier nœud ZooKeeper ; s&#39;il n&#39;est pas disponible, le deuxième, et ainsi de suite.                                         |
| `nearest_hostname`              | sélectionne un nœud ZooKeeper dont le nom d&#39;hôte est le plus proche de celui du serveur, en comparant le préfixe du nom d&#39;hôte.            |
| `hostname_levenshtein_distance` | identique à nearest&#95;hostname, mais compare les noms d&#39;hôte selon la distance de Levenshtein.                                               |
| `first_or_random`               | sélectionne le premier nœud ZooKeeper ; s&#39;il n&#39;est pas disponible, en sélectionne ensuite un autre aléatoirement parmi les nœuds restants. |
| `round_robin`                   | sélectionne le premier nœud ZooKeeper, puis, en cas de reconnexion, sélectionne le suivant.                                                        |

**Exemple de configuration**

```xml
<zookeeper>
    <node>
        <host>example1</host>
        <port>2181</port>
    </node>
    <node>
        <host>example2</host>
        <port>2181</port>
    </node>
    <session_timeout_ms>30000</session_timeout_ms>
    <operation_timeout_ms>10000</operation_timeout_ms>
    <!-- Optional. Chroot suffix. Should exist. -->
    <root>/path/to/zookeeper/node</root>
    <!-- Optional. Zookeeper digest ACL string. -->
    <identity>user:password</identity>
    <!--<zookeeper_load_balancing>random / in_order / nearest_hostname / hostname_levenshtein_distance / first_or_random / round_robin</zookeeper_load_balancing>-->
    <zookeeper_load_balancing>random</zookeeper_load_balancing>
    <!-- Optional. Enable 64-bit transaction IDs. -->
    <use_xid_64>false</use_xid_64>
    <!-- Optional. Enable OpenTelemetry tracing context propagation. -->
    <pass_opentelemetry_tracing_context>false</pass_opentelemetry_tracing_context>
</zookeeper>
```

<div id="use_minimalistic_part_header_in_zookeeper">
  ## use_minimalistic_part_header_in_zookeeper
</div>

Méthode de stockage des en-têtes des fragments de données dans ZooKeeper. Ce paramètre s&#39;applique uniquement à la famille [`MergeTree`](/fr/reference/engines/table-engines/mergetree-family/index). Il peut être défini :

**Globalement dans la section [merge&#95;tree](#merge_tree) du fichier `config.xml`**

ClickHouse utilise ce paramètre pour toutes les tables du serveur. Vous pouvez le modifier à tout moment. Les tables existantes changent de comportement lorsque ce paramètre est modifié.

**Pour chaque table**

Lors de la création d&#39;une table, spécifiez le [paramètre de moteur](/fr/reference/engines/table-engines/mergetree-family/mergetree#table_engine-mergetree-creating-a-table) correspondant. Le comportement d&#39;une table existante avec ce paramètre ne change pas, même si le paramètre global est modifié.

**Valeurs possibles**

* `0` — La fonctionnalité est désactivée.
* `1` — La fonctionnalité est activée.

Si [`use_minimalistic_part_header_in_zookeeper = 1`](#use_minimalistic_part_header_in_zookeeper), les tables [répliquées](/fr/reference/engines/table-engines/mergetree-family/replication) stockent les en-têtes des fragments de données de façon compacte à l&#39;aide d&#39;un seul `znode`. Si la table contient de nombreuses colonnes, cette méthode de stockage réduit considérablement le volume de données stockées dans ZooKeeper.

<Note>
  Après avoir appliqué `use_minimalistic_part_header_in_zookeeper = 1`, vous ne pouvez pas rétrograder le serveur ClickHouse vers une version qui ne prend pas en charge ce paramètre. Soyez prudent lors de la mise à niveau de ClickHouse sur les serveurs d&#39;un cluster. Ne mettez pas à niveau tous les serveurs en même temps. Il est plus sûr de tester les nouvelles versions de ClickHouse dans un environnement de test, ou sur seulement quelques serveurs du cluster.

  Les en-têtes de fragments de données déjà stockés avec ce paramètre ne peuvent pas être restaurés dans leur représentation précédente (non compacte).
</Note>

<div id="distributed_ddl">
  ## distributed_ddl
</div>

Gère l’exécution des [requêtes DDL distribuées](/fr/reference/statements/distributed-ddl) (`CREATE`, `DROP`, `ALTER`, `RENAME`) sur le cluster.
Fonctionne uniquement si [ZooKeeper](/fr/reference/settings/server-settings/settings#zookeeper) est activé.

Les paramètres configurables dans `<distributed_ddl>` comprennent :

| Paramètre              | Description                                                                                                                                                          | Valeur par défaut                            |
| ---------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------- | -------------------------------------------- |
| `cleanup_delay_period` | le nettoyage démarre après la réception d’un événement de nouveau nœud si le dernier nettoyage n’a pas été effectué il y a moins de `cleanup_delay_period` secondes. | `60` secondes                                |
| `max_tasks_in_queue`   | le nombre maximal de tâches pouvant se trouver dans la file d’attente.                                                                                               | `1,000`                                      |
| `path`                 | le chemin dans Keeper pour la `task_queue` des requêtes DDL                                                                                                          |                                              |
| `pool_size`            | le nombre de requêtes `ON CLUSTER` pouvant être exécutées simultanément                                                                                              |                                              |
| `profile`              | le profil utilisé pour exécuter les requêtes DDL                                                                                                                     |                                              |
| `task_max_lifetime`    | supprime le nœud si son ancienneté dépasse cette valeur.                                                                                                             | `7 * 24 * 60 * 60` (une semaine en secondes) |

**Exemple**

```xml
<distributed_ddl>
    <!-- Path in ZooKeeper to queue with DDL queries -->
    <path>/clickhouse/task_queue/ddl</path>

    <!-- Settings from this profile will be used to execute DDL queries -->
    <profile>default</profile>

    <!-- Controls how much ON CLUSTER queries can be run simultaneously. -->
    <pool_size>1</pool_size>

    <!--
         Cleanup settings (active tasks will not be removed)
    -->

    <!-- Controls task TTL (default 1 week) -->
    <task_max_lifetime>604800</task_max_lifetime>

    <!-- Controls how often cleanup should be performed (in seconds) -->
    <cleanup_delay_period>60</cleanup_delay_period>

    <!-- Controls how many tasks could be in the queue -->
    <max_tasks_in_queue>1000</max_tasks_in_queue>
</distributed_ddl>
```

<div id="proxy">
  ## proxy
</div>

Définissez des serveurs proxy pour les requêtes HTTP et HTTPS, actuellement pris en charge par le stockage S3, les fonctions de table S3 et les fonctions URL.

Il existe trois façons de définir des serveurs proxy :

* variables d’environnement
* listes de proxy
* résolveurs de proxy distants.

Le contournement des serveurs proxy pour des hôtes spécifiques est également pris en charge à l’aide de `no_proxy`.

**Variables d’environnement**

Les variables d’environnement `http_proxy` et `https_proxy` vous permettent de spécifier un
serveur proxy pour un protocole donné. Si elles sont définies sur votre système, cela devrait fonctionner sans problème.

Il s’agit de l’approche la plus simple si un protocole donné n’a
qu’un seul serveur proxy et que ce serveur proxy ne change pas.

**Listes de proxy**

Cette approche vous permet de spécifier un ou plusieurs
serveurs proxy pour un protocole. Si plusieurs serveurs proxy sont définis,
ClickHouse utilise les différents proxy selon une méthode round-robin, en répartissant la
charge entre les serveurs. Il s’agit de l’approche la plus simple s’il existe plusieurs
serveurs proxy pour un protocole et que la liste des serveurs proxy ne change pas.

**Modèle de configuration**

```xml
<proxy>
    <http>
        <uri>http://proxy1</uri>
        <uri>http://proxy2:3128</uri>
    </http>
    <https>
        <uri>http://proxy1:3128</uri>
    </https>
</proxy>
```

Sélectionnez un champ parent dans les onglets ci-dessous pour afficher les champs enfants correspondants :

<Tabs>
  <Tab title="<proxy>">
    | Champ     | Description                                 |
    | --------- | ------------------------------------------- |
    | `<http>`  | Une liste d&#39;un ou plusieurs proxy HTTP  |
    | `<https>` | Une liste d&#39;un ou plusieurs proxy HTTPS |
  </Tab>

  <Tab title="<http> et <https>">
    | Champ   | Description        |
    | ------- | ------------------ |
    | `<uri>` | L&#39;URI du proxy |
  </Tab>
</Tabs>

**Résolveurs de proxy distants**

Il est possible que les serveurs proxy changent dynamiquement. Dans ce
cas, vous pouvez définir l&#39;endpoint d&#39;un résolveur. ClickHouse envoie
une requête GET vide à cet endpoint, et le résolveur distant doit renvoyer l&#39;hôte du proxy.
ClickHouse l&#39;utilisera pour construire l&#39;URI du proxy à l&#39;aide du modèle suivant : `\{proxy_scheme\}://\{proxy_host\}:{proxy_port}`

**Modèle de configuration**

```xml
<proxy>
    <http>
        <resolver>
            <endpoint>http://resolver:8080/hostname</endpoint>
            <proxy_scheme>http</proxy_scheme>
            <proxy_port>80</proxy_port>
            <proxy_cache_time>10</proxy_cache_time>
        </resolver>
    </http>

    <https>
        <resolver>
            <endpoint>http://resolver:8080/hostname</endpoint>
            <proxy_scheme>http</proxy_scheme>
            <proxy_port>3128</proxy_port>
            <proxy_cache_time>10</proxy_cache_time>
        </resolver>
    </https>

</proxy>
```

<Tabs>
  <Tab title="<proxy>">
    | Champ     | Description                                     |
    | --------- | ----------------------------------------------- |
    | `<http>`  | Une liste d&#39;un ou plusieurs résolveurs* |
    | `<https>` | Une liste d&#39;un ou plusieurs résolveurs* |
  </Tab>

  <Tab title="<http> et <https>">
    | Champ        | Description                                             |
    | ------------ | ------------------------------------------------------- |
    | `<resolver>` | L&#39;endpoint et d&#39;autres détails sur un résolveur |

    <Note>
      Vous pouvez avoir plusieurs éléments `<resolver>`, mais seul le premier
      `<resolver>` pour un protocole donné est utilisé. Tous les autres éléments `<resolver>`
      de ce protocole sont ignorés. Cela signifie que l&#39;équilibrage de charge
      (si nécessaire) doit être implémenté par le résolveur distant.
    </Note>
  </Tab>

  <Tab title="<resolver>">
    | Champ                | Description                                                                                                                                                                                                            |
    | -------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
    | `<endpoint>`         | L&#39;URI du résolveur de proxy                                                                                                                                                                                        |
    | `<proxy_scheme>`     | Le protocole de l&#39;URI finale du proxy. Il peut s&#39;agir de `http` ou de `https`.                                                                                                                                 |
    | `<proxy_port>`       | Le numéro de port du résolveur de proxy                                                                                                                                                                                |
    | `<proxy_cache_time>` | La durée, en secondes, pendant laquelle les valeurs du résolveur doivent être mises en cache par ClickHouse. Définir cette valeur sur `0` amène ClickHouse à contacter le résolveur pour chaque requête HTTP ou HTTPS. |
  </Tab>
</Tabs>

**Priorité**

Les paramètres de proxy sont déterminés dans l&#39;ordre suivant :

| Ordre | Paramètre                     |
| ----- | ----------------------------- |
| 1.    | Résolveurs de proxy distants  |
| 2.    | Listes de proxy               |
| 3.    | Variables d&#39;environnement |

ClickHouse vérifiera le type de résolveur ayant la plus haute priorité pour le protocole de la requête. S&#39;il n&#39;est pas défini,
il vérifiera alors le type de résolveur ayant la priorité immédiatement inférieure, jusqu&#39;à atteindre le résolveur d&#39;environnement.
Cela permet également d&#39;utiliser une combinaison de types de résolveur.

<div id="disable_tunneling_for_https_requests_over_http_proxy">
  ## disable_tunneling_for_https_requests_over_http_proxy
</div>

Par défaut, le tunneling (c.-à-d. `HTTP CONNECT`) est utilisé pour effectuer des requêtes `HTTPS` via un proxy `HTTP`. Ce paramètre permet de le désactiver.

**no&#95;proxy**

Par défaut, toutes les requêtes passent par le proxy. Pour le désactiver pour des hôtes spécifiques, la variable `no_proxy` doit être définie.
Elle peut être définie dans la clause `<proxy>` pour les résolveurs de liste et distants, et comme variable d’environnement pour le résolveur d’environnement.
Elle prend en charge les adresses IP, les domaines, les sous-domaines et le caractère générique `'*'` pour un contournement complet. Les points initiaux sont supprimés, comme avec curl.

**Example**

La configuration ci-dessous contourne le proxy pour les requêtes vers `clickhouse.cloud` et tous ses sous-domaines (par ex., `auth.clickhouse.cloud`).
Il en va de même pour GitLab, même s’il comporte un point initial. `gitlab.com` et `about.gitlab.com` contourneraient tous deux le proxy.

```xml
<proxy>
    <no_proxy>clickhouse.cloud,.gitlab.com</no_proxy>
    <http>
        <uri>http://proxy1</uri>
        <uri>http://proxy2:3128</uri>
    </http>
    <https>
        <uri>http://proxy1:3128</uri>
    </https>
</proxy>
```
