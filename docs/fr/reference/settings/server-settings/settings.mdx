Cette section contient la description des paramètres du serveur. Il s’agit de paramètres qui
ne peuvent pas être modifiés au niveau de la session ou de la requête.

Pour plus d’informations sur les fichiers de configuration dans ClickHouse, consultez [&quot;&quot;Fichiers de configuration&quot;&quot;](/fr/concepts/features/configuration/server-config/configuration-files).

Les autres paramètres sont décrits dans la section &quot;&quot;[Paramètres](/fr/concepts/features/configuration/settings/overview)&quot;&quot;.
Avant d’étudier les paramètres, nous vous recommandons de lire la section [Fichiers de configuration](/fr/concepts/features/configuration/server-config/configuration-files)
et de noter l’utilisation des substitutions (les attributs `incl` et `optional`).

<div id="access_control_improvements">
  ## access_control_improvements
</div>

Paramètres des améliorations optionnelles du système de contrôle d&#39;accès.

| Paramètre                                       | Description                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                     | Par défaut |
| ----------------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ---------- |
| `on_cluster_queries_require_cluster_grant`      | Définit si les requêtes `ON CLUSTER` nécessitent le privilège `CLUSTER`.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                        | `true`     |
| `role_cache_expiration_time_seconds`            | Définit le nombre de secondes pendant lesquelles un rôle est conservé dans le cache des rôles après son dernier accès.                                                                                                                                                                                                                                                                                                                                                                                                                                          | `600`      |
| `select_from_information_schema_requires_grant` | Définit si `SELECT * FROM information_schema.<table>` nécessite des privilèges ou peut être exécuté par n&#39;importe quel utilisateur. Si la valeur est `true`, cette requête nécessite `GRANT SELECT ON information_schema.<table>`, comme pour les tables ordinaires.                                                                                                                                                                                                                                                                                        | `true`     |
| `select_from_system_db_requires_grant`          | Définit si `SELECT * FROM system.<table>` nécessite des privilèges ou peut être exécuté par n&#39;importe quel utilisateur. Si la valeur est `true`, cette requête nécessite `GRANT SELECT ON system.<table>`, comme pour les tables non système. Exceptions : quelques tables système (`tables`, `columns`, `databases` et certaines tables constantes comme `one`, `contributors`) restent accessibles à tous ; et si un privilège `SHOW` (par exemple `SHOW USERS`) a été accordé, la table système correspondante (c.-à-d. `system.users`) sera accessible. | `true`     |
| `settings_constraints_replace_previous`         | Définit si une contrainte d&#39;un profil de paramètres pour un paramètre donné annule les effets de la contrainte précédente (définie dans d&#39;autres profils) pour ce paramètre, y compris pour les champs qui ne sont pas définis par la nouvelle contrainte. Active également le type de contrainte `changeable_in_readonly`.                                                                                                                                                                                                                             | `true`     |
| `table_engines_require_grant`                   | Définit si la création d&#39;une table avec un moteur de table spécifique nécessite un privilège.                                                                                                                                                                                                                                                                                                                                                                                                                                                               | `false`    |
| `throw_on_unmatched_row_policies`               | Définit si la lecture d&#39;une table doit lever une exception lorsque la table possède des politiques de ligne, mais qu&#39;aucune d&#39;elles ne s&#39;applique à l&#39;utilisateur courant                                                                                                                                                                                                                                                                                                                                                                   | `false`    |
| `users_without_row_policies_can_read_rows`      | Définit si les utilisateurs sans politiques de ligne permissives peuvent quand même lire des lignes à l&#39;aide d&#39;une requête `SELECT`. Par exemple, s&#39;il y a deux utilisateurs A et B et qu&#39;une politique de ligne est définie uniquement pour A, alors si ce paramètre vaut `true`, l&#39;utilisateur B verra toutes les lignes. Si ce paramètre vaut `false`, l&#39;utilisateur B ne verra aucune ligne.                                                                                                                                        | `true`     |

Exemple :

```xml
<access_control_improvements>
    <throw_on_unmatched_row_policies>true</throw_on_unmatched_row_policies>
    <users_without_row_policies_can_read_rows>true</users_without_row_policies_can_read_rows>
    <on_cluster_queries_require_cluster_grant>true</on_cluster_queries_require_cluster_grant>
    <select_from_system_db_requires_grant>true</select_from_system_db_requires_grant>
    <select_from_information_schema_requires_grant>true</select_from_information_schema_requires_grant>
    <settings_constraints_replace_previous>true</settings_constraints_replace_previous>
    <table_engines_require_grant>false</table_engines_require_grant>
    <role_cache_expiration_time_seconds>600</role_cache_expiration_time_seconds>
</access_control_improvements>
```

<div id="allow_feature_tier">
  ## allow_feature_tier
</div>

<SettingsInfoBlock type="UInt32" default_value="0" changeable_without_restart="Oui" />

Détermine si l&#39;utilisateur peut modifier les paramètres associés aux différents niveaux de maturité des fonctionnalités.

* `0` - Les modifications de tous les paramètres sont autorisées (expérimental, bêta, production).
* `1` - Seules les modifications des paramètres des fonctionnalités bêta et de production sont autorisées. Les modifications des paramètres expérimentaux sont rejetées.
* `2` - Seules les modifications des paramètres des fonctionnalités de production sont autorisées. Les modifications des paramètres expérimentaux ou bêta sont rejetées.

Cela revient à définir une contrainte readonly sur toutes les fonctionnalités `EXPERIMENTAL` / `BETA`.

<Note>
  Une valeur de `0` signifie que tous les paramètres peuvent être modifiés.
</Note>

<div id="async_load_databases">
  ## async_load_databases
</div>

<SettingsInfoBlock type="Bool" default_value="1" changeable_without_restart="No" />

Chargement asynchrone des bases de données et des tables.

* Si `true`, toutes les bases de données non système utilisant le moteur `Ordinary`, `Atomic` ou `Replicated` seront chargées de manière asynchrone après le démarrage du serveur ClickHouse. Voir la table `system.asynchronous_loader` ainsi que les paramètres du serveur `tables_loader_background_pool_size` et `tables_loader_foreground_pool_size`. Toute requête qui tente d’accéder à une table qui n’est pas encore chargée attendra que cette table précise soit démarrée. Si la tâche de chargement échoue, la requête relancera une erreur (au lieu d’arrêter l’ensemble du serveur dans le cas de `async_load_databases = false`). La table attendue par au moins une requête sera chargée avec une priorité plus élevée. Les requêtes DDL sur une base de données attendront que cette base de données précise soit démarrée. Pensez également à définir une limite `max_waiting_queries` pour le nombre total de requêtes en attente.
* Si `false`, toutes les bases de données sont chargées au démarrage du serveur.

**Exemple**

```xml
<async_load_databases>true</async_load_databases>
```

<div id="async_load_system_database">
  ## async_load_system_database
</div>

<SettingsInfoBlock type="Bool" default_value="0" changeable_without_restart="Non" />

Chargement asynchrone des tables système. Utile lorsqu’il y a un grand nombre de tables de logs et de parts dans la base de données `system`. Indépendant du paramètre `async_load_databases`.

* Si cette valeur est définie sur `true`, toutes les bases de données système utilisant les engines `Ordinary`, `Atomic` et `Replicated` seront chargées de manière asynchrone après le démarrage du serveur ClickHouse. Voir la table `system.asynchronous_loader`, ainsi que les paramètres du serveur `tables_loader_background_pool_size` et `tables_loader_foreground_pool_size`. Toute query qui tente d’accéder à une table système qui n’est pas encore chargée attendra que cette table soit initialisée. Une table attendue par au moins une query sera chargée avec une priorité plus élevée. Pensez également à définir le paramètre `max_waiting_queries` pour limiter le nombre total de queries en attente.
* Si cette valeur est définie sur `false`, les bases de données système sont chargées avant le démarrage du serveur.

**Exemple**

```xml
<async_load_system_database>true</async_load_system_database>
```

<div id="asynchronous_metric_log">
  ## asynchronous_metric_log
</div>

Activé par défaut dans les déploiements ClickHouse Cloud.

Si ce paramètre n’est pas activé par défaut dans votre environnement, selon la manière dont ClickHouse a été installé, vous pouvez suivre les instructions ci-dessous pour l’activer ou le désactiver.

**Activation**

Pour activer manuellement la collecte de l’historique des logs de métriques asynchrones [`system.asynchronous_metric_log`](/fr/reference/system-tables/asynchronous_metric_log), créez `/etc/clickhouse-server/config.d/asynchronous_metric_log.xml` avec le contenu suivant :

```xml
     <asynchronous_metric_log>
        <database>system</database>
        <table>asynchronous_metric_log</table>
        <flush_interval_milliseconds>7500</flush_interval_milliseconds>
        <collect_interval_milliseconds>1000</collect_interval_milliseconds>
        <max_size_rows>1048576</max_size_rows>
        <reserved_size_rows>8192</reserved_size_rows>
        <buffer_size_rows_flush_threshold>524288</buffer_size_rows_flush_threshold>
        <flush_on_crash>false</flush_on_crash>
    </asynchronous_metric_log>
</clickhouse>
```

**Désactivation**

Pour désactiver le paramètre `asynchronous_metric_log`, créez le fichier suivant `/etc/clickhouse-server/config.d/disable_asynchronous_metric_log.xml` avec le contenu ci-dessous :

```xml
<clickhouse><asynchronous_metric_log remove="1" /></clickhouse>
```

<div id="background_merges_mutations_concurrency_ratio">
  ## background_merges_mutations_concurrency_ratio
</div>

<SettingsInfoBlock type="Float" default_value="2" changeable_without_restart="Non" />

Définit un ratio entre le nombre de threads et le nombre de merges et de mutations en arrière-plan pouvant être exécutées de manière concurrente.

Par exemple, si le ratio est égal à 2 et que [`background_pool_size`](/fr/reference/settings/server-settings/settings#background_pool_size) est défini sur 16, alors ClickHouse peut exécuter 32 merges en arrière-plan simultanément. Cela est possible, car les opérations en arrière-plan peuvent être suspendues et reportées. Cela est nécessaire pour accorder une priorité d’exécution plus élevée aux petites fusions.

<Note>
  Vous ne pouvez augmenter ce ratio qu’à chaud. Pour le diminuer, vous devez redémarrer le serveur.

  Comme pour le paramètre [`background_pool_size`](/fr/reference/settings/server-settings/settings#background_pool_size), [`background_merges_mutations_concurrency_ratio`](/fr/reference/settings/server-settings/settings#background_merges_mutations_concurrency_ratio) peut être appliqué depuis le profil `default` pour assurer la compatibilité descendante.
</Note>

<div id="background_merges_mutations_scheduling_policy">
  ## background_merges_mutations_scheduling_policy
</div>

<SettingsInfoBlock type="String" default_value="round_robin" changeable_without_restart="No" />

Politique définissant la manière dont l’ordonnancement des merges et mutations en arrière-plan est effectué. Les valeurs possibles sont : `round_robin` et `shortest_task_first`.

Algorithme utilisé pour sélectionner le prochain merge ou la prochaine mutation à exécuter par le pool de threads en arrière-plan. La politique peut être modifiée à l’exécution sans redémarrer le serveur.
Peut être appliquée à partir du profil `default` pour assurer la compatibilité descendante.

Valeurs possibles :

* `round_robin` — Chaque merge et mutation concurrent est exécuté selon un ordre round-robin afin de garantir un fonctionnement sans risque de famine. Les merges de plus petite taille se terminent plus rapidement que les plus gros, simplement parce qu’ils ont moins de blocs à fusionner.
* `shortest_task_first` — Exécute toujours le merge ou la mutation de plus petite taille. Les merges et mutations se voient attribuer des priorités en fonction de leur taille résultante. Les merges de plus petite taille sont strictement privilégiés par rapport aux plus gros. Cette politique garantit la fusion la plus rapide possible des petites parts, mais peut entraîner une famine indéfinie des gros merges dans des partitions fortement surchargées par des `INSERT`s.

<div id="background_pool_size">
  ## background_pool_size
</div>

<SettingsInfoBlock type="UInt64" default_value="16" changeable_without_restart="Non" />

Définit le nombre de threads qui effectuent les fusions en arrière-plan et les mutations pour les tables utilisant des moteurs MergeTree.

<Note>
  * Ce paramètre peut également être appliqué au démarrage du serveur à partir de la configuration du profil `default`, pour préserver la rétrocompatibilité lors du démarrage du serveur ClickHouse.
  * Vous pouvez uniquement augmenter le nombre de threads à l&#39;exécution.
  * Pour réduire le nombre de threads, vous devez redémarrer le serveur.
  * En ajustant ce paramètre, vous gérez la charge du CPU et du disque.
</Note>

<Danger>
  Une taille de pool plus petite utilise moins de ressources CPU et disque, mais les processus en arrière-plan progressent plus lentement, ce qui peut finir par affecter les performances des requêtes.
</Danger>

Avant de le modifier, veuillez également consulter les paramètres MergeTree associés, tels que :

* [`number_of_free_entries_in_pool_to_lower_max_size_of_merge`](/fr/reference/settings/merge-tree-settings#number_of_free_entries_in_pool_to_lower_max_size_of_merge).
* [`number_of_free_entries_in_pool_to_execute_mutation`](/fr/reference/settings/merge-tree-settings#number_of_free_entries_in_pool_to_execute_mutation).
* [`number_of_free_entries_in_pool_to_execute_optimize_entire_partition`](/fr/reference/settings/merge-tree-settings#number_of_free_entries_in_pool_to_execute_optimize_entire_partition)

**Exemple**

```xml
<background_pool_size>16</background_pool_size>
```

<div id="concurrent_threads_scheduler">
  ## concurrent_threads_scheduler
</div>

<SettingsInfoBlock type="String" default_value="max_min_fair" changeable_without_restart="Yes" />

Politique définissant l’ordonnancement des slots CPU spécifiés par `concurrent_threads_soft_limit_num` et `concurrent_threads_soft_limit_ratio_to_cores`. Algorithme utilisé pour déterminer comment un nombre limité de slots CPU est réparti entre des requêtes concurrentes. Le planificateur peut être modifié à l’exécution sans redémarrer le serveur.

Valeurs possibles :

* `round_robin` — Chaque requête avec le paramètre `use_concurrency_control` = 1 alloue jusqu’à `max_threads` slots CPU. Un slot par thread. En cas de contention, les slots CPU sont accordés aux requêtes selon un ordonnancement round-robin. Notez que le premier slot est accordé de manière inconditionnelle, ce qui peut entraîner un manque d’équité et une hausse de la latence pour les requêtes ayant une valeur `max_threads` élevée lorsqu’un grand nombre de requêtes ont `max_threads` = 1.
* `fair_round_robin` — Chaque requête avec le paramètre `use_concurrency_control` = 1 alloue jusqu’à `max_threads - 1` slots CPU. Variante de `round_robin` qui ne nécessite pas de slot CPU pour le premier thread de chaque requête. Ainsi, les requêtes ayant `max_threads` = 1 ne nécessitent aucun slot et ne peuvent pas monopoliser injustement tous les slots. Aucun slot n’est accordé de manière inconditionnelle.
* `max_min_fair` — Chaque requête avec le paramètre `use_concurrency_control` = 1 alloue jusqu’à `max_threads - 1` slots CPU. Semblable à `fair_round_robin`, mais les slots libérés sont toujours accordés à la requête qui possède actuellement le plus petit nombre de slots alloués. Cela offre une meilleure équité en cas de forte surallocation, lorsque de nombreuses requêtes se disputent un nombre limité de slots CPU. Les requêtes de courte durée ne sont pas pénalisées par les requêtes de longue durée qui ont accumulé davantage de slots au fil du temps.

<div id="concurrent_threads_soft_limit_num">
  ## concurrent_threads_soft_limit_num
</div>

<SettingsInfoBlock type="UInt64" default_value="0" changeable_without_restart="Oui" />

<div id="cpu_slot_preemption">
  ## cpu_slot_preemption
</div>

<SettingsInfoBlock type="Bool" default_value="1" changeable_without_restart="Oui" />

Définit comment l’ordonnancement des workloads pour les ressources CPU (MASTER THREAD et WORKER THREAD) est réalisé.

* Si `true` (par défaut), la comptabilisation se fait en fonction du temps CPU réellement consommé. Une part équitable de temps CPU est allouée aux workloads en concurrence. Les slots sont alloués pour une durée limitée, puis redemandés à expiration. La demande de slots peut bloquer l’exécution des threads en cas de surcharge des ressources CPU, c’est-à-dire qu’une préemption peut se produire. Cela garantit une répartition équitable du temps CPU.
* Si `false`, la comptabilisation se base sur le nombre de slots CPU alloués. Un nombre équitable de slots CPU est alloué aux workloads en concurrence. Un slot est alloué lorsqu’un thread démarre, conservé pendant toute son exécution, puis libéré lorsque le thread se termine. Le nombre de threads alloués à l’exécution des requêtes peut uniquement passer de 1 à `max_threads` et ne jamais diminuer. Cela favorise davantage les requêtes de longue durée et peut entraîner une famine CPU pour les requêtes courtes.

**Exemple**

```xml
<cpu_slot_preemption>true</cpu_slot_preemption>
```

**Voir aussi**

* [Ordonnancement des charges de travail](/fr/concepts/features/configuration/server-config/workload-scheduling)

<div id="cpu_slot_preemption_timeout_ms">
  ## cpu_slot_preemption_timeout_ms
</div>

<SettingsInfoBlock type="UInt64" default_value="1000" changeable_without_restart="Oui" />

Définit le nombre de millisecondes pendant lesquelles un worker thread peut attendre lors de la préemption, c’est-à-dire en attendant qu’un autre CPU slot lui soit attribué. Passé ce délai, si le thread n’a pas pu obtenir un nouveau CPU slot, il se termine et la query est dynamiquement ramenée à un nombre plus faible de threads s’exécutant de manière concurrente. Notez que le master thread n’est jamais réduit, mais peut être préempté indéfiniment. Ce paramètre n’a de sens que lorsque `cpu_slot_preemption` est activé et qu’une ressource CPU est définie pour WORKER THREAD.

<div id="disable_tunneling_for_https_requests_over_http_proxy">
  ## disable_tunneling_for_https_requests_over_http_proxy
</div>

Par défaut, le tunneling (c.-à-d. `HTTP CONNECT`) est utilisé pour effectuer des requêtes `HTTPS` via un proxy `HTTP`. Ce paramètre permet de le désactiver.

**no&#95;proxy**

Par défaut, toutes les requêtes passent par le proxy. Pour le désactiver pour des hôtes spécifiques, la variable `no_proxy` doit être définie.
Elle peut être définie dans la clause `<proxy>` pour les resolveurs de liste et distants, et comme variable d’environnement pour le resolveur d’environnement.
Elle prend en charge les adresses IP, les domaines, les sous-domaines et le caractère générique `'*'` pour un contournement complet. Les points en tête sont supprimés, comme avec curl.

**Example**

La configuration ci-dessous contourne le proxy pour `clickhouse.cloud` et tous ses sous-domaines (par ex. `auth.clickhouse.cloud`).
Il en va de même pour GitLab, même s’il comporte un point en tête. `gitlab.com` et `about.gitlab.com` contourneraient tous deux le proxy.

```xml
<proxy>
    <no_proxy>clickhouse.cloud,.gitlab.com</no_proxy>
    <http>
        <uri>http://proxy1</uri>
        <uri>http://proxy2:3128</uri>
    </http>
    <https>
        <uri>http://proxy1:3128</uri>
    </https>
</proxy>
```

<div id="encryption">
  ## chiffrement
</div>

Configure une commande permettant d’obtenir une clé à utiliser avec les [codecs de chiffrement](/fr/reference/statements/create/table#encryption-codecs). La clé (ou les clés) doit être stockée dans des variables d’environnement ou définie dans le fichier de configuration.

Les clés peuvent être hexadécimales ou des chaînes d’une longueur de 16 octets.

**Exemple**

Chargement depuis la config :

```xml
<encryption_codecs>
    <aes_128_gcm_siv>
        <key>1234567812345678</key>
    </aes_128_gcm_siv>
</encryption_codecs>
```

<Note>
  Il n’est pas recommandé de stocker des clés dans le fichier de configuration. Ce n’est pas sûr. Vous pouvez déplacer les clés dans un fichier de configuration distinct sur un disque sécurisé et placer un lien symbolique vers ce fichier de configuration dans le dossier `config.d/`.
</Note>

Chargement à partir de la configuration, lorsque la clé est en hexadécimal :

```xml
<encryption_codecs>
    <aes_128_gcm_siv>
        <key_hex>00112233445566778899aabbccddeeff</key_hex>
    </aes_128_gcm_siv>
</encryption_codecs>
```

Chargement de la clé à partir de la variable d’environnement :

```xml
<encryption_codecs>
    <aes_128_gcm_siv>
        <key_hex from_env="ENVVAR"></key_hex>
    </aes_128_gcm_siv>
</encryption_codecs>
```

Ici, `current_key_id` définit la clé active pour le chiffrement, et toutes les clés spécifiées peuvent être utilisées pour le déchiffrement.

Chacune de ces méthodes peut s’appliquer à plusieurs clés :

```xml
<encryption_codecs>
    <aes_128_gcm_siv>
        <key_hex id="0">00112233445566778899aabbccddeeff</key_hex>
        <key_hex id="1" from_env="ENVVAR"></key_hex>
        <current_key_id>1</current_key_id>
    </aes_128_gcm_siv>
</encryption_codecs>
```

Ici, `current_key_id` indique la clé actuellement utilisée pour le chiffrement.

Les utilisateurs peuvent également ajouter un nonce, qui doit comporter 12 octets (par défaut, les processus de chiffrement et de déchiffrement utilisent un nonce composé d’octets nuls) :

```xml
<encryption_codecs>
    <aes_128_gcm_siv>
        <nonce>012345678910</nonce>
    </aes_128_gcm_siv>
</encryption_codecs>
```

Ou être défini en hexadécimal :

```xml
<encryption_codecs>
    <aes_128_gcm_siv>
        <nonce_hex>abcdefabcdef</nonce_hex>
    </aes_128_gcm_siv>
</encryption_codecs>
```

<Note>
  Tout ce qui précède peut s’appliquer à `aes_256_gcm_siv` (mais la clé doit faire 32 octets).
</Note>

<div id="http_handlers">
  ## http_handlers
</div>

Permet d’utiliser des gestionnaires HTTP personnalisés.
Pour ajouter un nouveau gestionnaire HTTP, ajoutez simplement une nouvelle `<rule>`.
Les règles sont vérifiées de haut en bas dans l’ordre défini,
et la première correspondance exécute le gestionnaire.
Une règle sans condition de correspondance (uniquement `handler`) correspond à toutes les requêtes ; comme les règles sont vérifiées dans l’ordre,
une telle règle n’est utile qu’en solution de repli, placée en dernier.

Les paramètres suivants peuvent être configurés via des sous-balises (toutes ces sous-balises sont facultatives, sauf `handler`) :

| Sous-balises         | Définition                                                                                                                                                                                                                                                                                                                                 |
| -------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| `url`                | Permet de faire correspondre le chemin URL de la requête. La chaîne de requête est ignorée lors de la correspondance                                                                                                                                                                                                                       |
| `url_prefix`         | Permet de faire correspondre le chemin URL de la requête à un chemin de base : le chemin lui-même ou tout ce qui se trouve en dessous, sur une limite de segment de chemin (par ex. &#39;/api/v1&#39; correspond à /api/v1, /api/v1/ et /api/v1/write, mais pas à /api/v1beta). La chaîne de requête est ignorée lors de la correspondance |
| `url_regexp`         | Permet de faire correspondre le chemin URL de la requête à une expression régulière. La chaîne de requête est ignorée lors de la correspondance                                                                                                                                                                                            |
| `full_url`           | Permet de faire correspondre l’URL complète de la requête `scheme://host:port/path`. La chaîne de requête est ignorée lors de la correspondance, et l’hôte est l’adresse IP de la connexion (et non l’en-tête `Host`)                                                                                                                      |
| `full_url_prefix`    | Permet de faire correspondre l’URL complète de la requête `scheme://host:port/path` à l’URL de base `scheme://host:port/base_path`, sur une limite de segment de chemin (voir `url_prefix`). La chaîne de requête est ignorée lors de la correspondance                                                                                    |
| `full_url_regexp`    | Permet de faire correspondre l’URL complète de la requête `scheme://host:port/path` à une expression régulière. La chaîne de requête est ignorée lors de la correspondance                                                                                                                                                                 |
| `methods`            | Permet de faire correspondre les méthodes de requête ; vous pouvez utiliser des virgules pour séparer plusieurs méthodes                                                                                                                                                                                                                   |
| `headers`            | Permet de faire correspondre les en-têtes de requête en faisant correspondre chaque élément enfant (le nom de l’élément enfant est le nom de l’en-tête)                                                                                                                                                                                    |
| `headers_regexp`     | Comme `headers`, mais la valeur de chaque élément enfant est comparée à une expression régulière                                                                                                                                                                                                                                           |
| `empty_query_string` | Vérifie qu’il n’y a pas de chaîne de requête dans l’URL                                                                                                                                                                                                                                                                                    |
| `handler`            | Le gestionnaire de requête (obligatoire)                                                                                                                                                                                                                                                                                                   |

<Note>
  Au lieu de `url_regexp`, `full_url_regexp` et `headers_regexp`, vous pouvez aussi écrire une expression régulière dans `url`, `full_url` ou `headers` en utilisant le préfixe `regex:` (par ex. `<url>regex:/api/.*</url>`). Cette syntaxe reste prise en charge pour la rétrocompatibilité, mais elle est obsolète : préférez les sous-balises dédiées `url_regexp`, `full_url_regexp` et `headers_regexp`.
</Note>

`handler` contient les paramètres suivants, qui peuvent être configurés via des sous-balises :

| Sous-balises       | Définition                                                                                                                                                                                                                          |
| ------------------ | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `url`              | Une destination de redirection                                                                                                                                                                                                      |
| `type`             | Types pris en charge : static, dynamic&#95;query&#95;handler, predefined&#95;query&#95;handler, redirect                                                                                                                            |
| `status`           | À utiliser avec le type static, code d’état de la réponse                                                                                                                                                                           |
| `query_param_name` | À utiliser avec le type dynamic&#95;query&#95;handler, extrait et exécute la valeur correspondant à la valeur de `<query_param_name>` dans les paramètres de la requête HTTP                                                        |
| `query`            | À utiliser avec le type predefined&#95;query&#95;handler, exécute la requête lorsque le gestionnaire est appelé                                                                                                                     |
| `content_type`     | À utiliser avec le type static, type de contenu de la réponse                                                                                                                                                                       |
| `response_content` | À utiliser avec le type static, contenu de la réponse envoyé au client ; lors de l’utilisation du préfixe &#39;file://&#39; ou &#39;config://&#39;, récupère le contenu depuis le fichier ou la configuration et l’envoie au client |

En plus d’une liste de règles, vous pouvez spécifier `<defaults/>`, ce qui active tous les gestionnaires par défaut.

Exemple :

```xml
<http_handlers>
    <rule>
        <url>/</url>
        <methods>POST,GET</methods>
        <headers><pragma>no-cache</pragma></headers>
        <handler>
            <type>dynamic_query_handler</type>
            <query_param_name>query</query_param_name>
        </handler>
    </rule>

    <rule>
        <url>/predefined_query</url>
        <methods>POST,GET</methods>
        <handler>
            <type>predefined_query_handler</type>
            <query>SELECT * FROM system.settings</query>
        </handler>
    </rule>

    <rule>
        <handler>
            <type>static</type>
            <status>200</status>
            <content_type>text/plain; charset=UTF-8</content_type>
            <response_content>config://http_server_default_response</response_content>
        </handler>
    </rule>
</http_handlers>
```

</Note>

<div id="insert_deduplication_version">
  ## insert_deduplication_version
</div>

<SettingsInfoBlock type="InsertDeduplicationVersions" default_value="compatible_double_hashes" changeable_without_restart="Non" />

Ce paramètre permet de migrer d’une version du code où la déduplication des inserts synchrones et asynchrones est gérée de façon totalement distincte et non transparente vers une version du code où les données insérées sont dédupliquées de manière uniforme entre les inserts synchrones et asynchrones.
La valeur `old_separate_hashes` signifie que ClickHouse utilisera des hachages de déduplication différents pour les inserts synchrones et asynchrones (comme auparavant).
Cette valeur doit être utilisée comme valeur par défaut s’il n’est pas prévu de lancer la migration.
La valeur `compatible_double_hashes` signifie que ClickHouse utilisera deux hachages de déduplication : l’ancien pour les inserts synchrones ou asynchrones, et le nouveau pour tous les inserts. Cette valeur doit être utilisée pour migrer les instances existantes vers le nouveau comportement en toute sécurité.
Cette valeur doit être activée pendant un certain temps (voir les paramètres replicated&#95;deduplication&#95;window et non&#95;replicated&#95;deduplication&#95;window) afin de garantir qu’aucun insert synchrone ou asynchrone n’est perdu pendant la migration.
Enfin, la valeur `new_unified_hash` signifie que ClickHouse utilisera le nouveau hachage de déduplication pour les inserts synchrones et asynchrones. Cette valeur peut être activée sur de nouvelles instances de ClickHouse ou sur des instances ayant déjà utilisé la valeur `compatible_double_hashes` pendant un certain temps.
La valeur par défaut sera d’abord définie sur `compatible_double_hashes`, puis ultérieurement sur `new_unified_hash` dans de futures versions de ClickHouse, afin de mener cette migration à bien en toute sécurité et en deux étapes.

<div id="interserver_http_credentials">
  ## interserver_http_credentials
</div>

Nom d’utilisateur et mot de passe utilisés pour se connecter à d’autres serveurs pendant la [réplication](/fr/reference/engines/table-engines/mergetree-family/replication). De plus, le serveur authentifie les autres répliques à l’aide de ces identifiants.
`interserver_http_credentials` doit donc être identique pour toutes les répliques d’un cluster.

<Note>
  * Par défaut, si la section `interserver_http_credentials` est omise, l’authentification n’est pas utilisée pendant la réplication.
  * Les paramètres `interserver_http_credentials` ne sont pas liés à la [configuration](/fr/concepts/features/interfaces/client#configuration_files) des identifiants du client ClickHouse.
  * Ces identifiants sont communs à la réplication via `HTTP` et `HTTPS`.
</Note>

Les paramètres suivants peuvent être configurés à l’aide de sous-balises :

* `user` — Nom d’utilisateur.
* `password` — Mot de passe.
* `allow_empty` — Si `true`, les autres répliques sont autorisées à se connecter sans authentification, même si des identifiants sont définis. Si `false`, les connexions sans authentification sont refusées. Par défaut : `false`.
* `old` — Contient les anciens `user` et `password` utilisés lors de la rotation des identifiants. Plusieurs sections `old` peuvent être spécifiées.

**Rotation des identifiants**

ClickHouse prend en charge la rotation dynamique des identifiants interserveur sans devoir arrêter toutes les répliques en même temps pour mettre à jour leur configuration. Les identifiants peuvent être modifiés en plusieurs étapes.

Pour activer l’authentification, définissez `interserver_http_credentials.allow_empty` sur `true` et ajoutez des identifiants. Cela autorise les connexions avec ou sans authentification.

```xml
<interserver_http_credentials>
    <user>admin</user>
    <password>111</password>
    <allow_empty>true</allow_empty>
</interserver_http_credentials>
```

Après avoir configuré toutes les répliques, définissez `allow_empty` sur `false` ou supprimez ce paramètre. Cela rend obligatoire l&#39;authentification avec les nouveaux identifiants.

Pour modifier les identifiants existants, déplacez le nom d&#39;utilisateur et le mot de passe dans la section `interserver_http_credentials.old`, puis mettez à jour `user` et `password` avec les nouvelles valeurs. À ce stade, le serveur utilise les nouveaux identifiants pour se connecter aux autres répliques et accepte les connexions avec les nouveaux comme avec les anciens identifiants.

```xml
<interserver_http_credentials>
    <user>admin</user>
    <password>222</password>
    <old>
        <user>admin</user>
        <password>111</password>
    </old>
    <old>
        <user>temp</user>
        <password>000</password>
    </old>
</interserver_http_credentials>
```

Lorsque de nouveaux identifiants sont configurés sur toutes les répliques, les anciens identifiants peuvent être supprimés.

<div id="ldap_servers">
  ## ldap_servers
</div>

Listez ici les serveurs LDAP avec leurs paramètres de connexion afin de :

* les utiliser comme mécanismes d&#39;authentification pour des utilisateurs locaux dédiés, pour lesquels un mécanisme d&#39;authentification `ldap` est spécifié à la place de `password`
* les utiliser comme répertoires d&#39;utilisateurs distants.

Les paramètres suivants peuvent être configurés à l&#39;aide de sous-balises :

| Setting                        | Description                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                              |
| ------------------------------ | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `bind_dn`                      | Modèle utilisé pour construire le bind DN. Le DN résultant sera construit en remplaçant toutes les sous-chaînes `\{user_name\}` du modèle par le nom d&#39;utilisateur réel lors de chaque tentative d&#39;authentification.                                                                                                                                                                                                                                                                                                                             |
| `enable_tls`                   | Indicateur qui active l&#39;utilisation d&#39;une connexion sécurisée au serveur LDAP. Spécifiez `no` pour le protocole en texte brut (`ldap://`) (non recommandé). Spécifiez `yes` pour le protocole LDAP sur SSL/TLS (`ldaps://`) (recommandé, valeur par défaut). Spécifiez `starttls` pour le protocole StartTLS legacy (protocole en texte brut (`ldap://`), mis à niveau vers TLS).                                                                                                                                                                |
| `host`                         | Nom d&#39;hôte ou adresse IP du serveur LDAP ; ce paramètre est obligatoire et ne peut pas être vide.                                                                                                                                                                                                                                                                                                                                                                                                                                                    |
| `port`                         | Port du serveur LDAP. La valeur par défaut est 636 si `enable_tls` est défini sur true, `389` sinon.                                                                                                                                                                                                                                                                                                                                                                                                                                                     |
| `tls_ca_cert_dir`              | chemin vers le répertoire contenant les certificats CA.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                  |
| `tls_ca_cert_file`             | chemin vers le fichier du certificat CA.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                 |
| `tls_cert_file`                | chemin vers le fichier de certificat.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                    |
| `tls_cipher_suite`             | suite de chiffrement autorisée (notation OpenSSL).                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                       |
| `tls_key_file`                 | chemin vers le fichier de clé du certificat.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                             |
| `tls_minimum_protocol_version` | Version minimale du protocole SSL/TLS. Les valeurs acceptées sont : `ssl2`, `ssl3`, `tls1.0`, `tls1.1`, `tls1.2` (par défaut).                                                                                                                                                                                                                                                                                                                                                                                                                           |
| `tls_require_cert`             | Comportement de vérification du certificat du pair SSL/TLS. Les valeurs acceptées sont : `never`, `allow`, `try`, `demand` (par défaut).                                                                                                                                                                                                                                                                                                                                                                                                                 |
| `user_dn_detection`            | Section contenant les paramètres de recherche LDAP permettant de détecter le user DN réel de l&#39;utilisateur lié. Cela est principalement utilisé dans les filtres de recherche pour un role mapping supplémentaire lorsque le serveur est Active Directory. Le user DN résultant sera utilisé lors du remplacement des sous-chaînes `\{user_dn\}` partout où elles sont autorisées. Par défaut, le user DN est défini comme étant égal au bind DN, mais une fois la recherche effectuée, il sera mis à jour avec la valeur réelle du user DN détecté. |
| `verification_cooldown`        | Période, en secondes, après une tentative de bind réussie, pendant laquelle un utilisateur sera considéré comme authentifié avec succès pour toutes les requêtes consécutives sans contacter le serveur LDAP. Spécifiez `0` (par défaut) pour désactiver la mise en cache et forcer le contact avec le serveur LDAP pour chaque requête d&#39;authentification.                                                                                                                                                                                          |

Le paramètre `user_dn_detection` peut être configuré à l&#39;aide de sous-balises :

| Setting         | Description                                                                                                                                                                                                                                                                                                                                                                          |
| --------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| `base_dn`       | modèle utilisé pour construire le base DN pour la recherche LDAP. Le DN résultant sera construit en remplaçant toutes les sous-chaînes `\{user_name\}` et &#39;&#123;bind&#95;dn&#125;&#39; du modèle par le nom d&#39;utilisateur réel et le bind DN pendant la recherche LDAP.                                                                                                     |
| `scope`         | portée de la recherche LDAP. Les valeurs acceptées sont : `base`, `one_level`, `children`, `subtree` (par défaut).                                                                                                                                                                                                                                                                   |
| `search_filter` | modèle utilisé pour construire le filtre de recherche pour la recherche LDAP. Le filtre résultant sera construit en remplaçant toutes les sous-chaînes `\{user_name\}`, `\{bind_dn\}` et `\{base_dn\}` du modèle par le nom d&#39;utilisateur réel, le bind DN et le base DN pendant la recherche LDAP. Notez que les caractères spéciaux doivent être correctement échappés en XML. |

Exemple :

```xml
<my_ldap_server>
    <host>localhost</host>
    <port>636</port>
    <bind_dn>uid={user_name},ou=users,dc=example,dc=com</bind_dn>
    <verification_cooldown>300</verification_cooldown>
    <enable_tls>yes</enable_tls>
    <tls_minimum_protocol_version>tls1.2</tls_minimum_protocol_version>
    <tls_require_cert>demand</tls_require_cert>
    <tls_cert_file>/path/to/tls_cert_file</tls_cert_file>
    <tls_key_file>/path/to/tls_key_file</tls_key_file>
    <tls_ca_cert_file>/path/to/tls_ca_cert_file</tls_ca_cert_file>
    <tls_ca_cert_dir>/path/to/tls_ca_cert_dir</tls_ca_cert_dir>
    <tls_cipher_suite>ECDHE-ECDSA-AES256-GCM-SHA384:ECDHE-RSA-AES256-GCM-SHA384:AES256-GCM-SHA384</tls_cipher_suite>
</my_ldap_server>
```

Exemple (Active Directory classique avec détection du user DN configurée en vue d’un mappage ultérieur des rôles) :

```xml
<my_ad_server>
    <host>localhost</host>
    <port>389</port>
    <bind_dn>EXAMPLE\{user_name}</bind_dn>
    <user_dn_detection>
        <base_dn>CN=Users,DC=example,DC=com</base_dn>
        <search_filter>(&amp;(objectClass=user)(sAMAccountName={user_name}))</search_filter>
    </user_dn_detection>
    <enable_tls>no</enable_tls>
</my_ad_server>
```

**Surcharges par niveau**

Le niveau de journalisation de noms de log spécifiques peut être surchargé. Par exemple, pour désactiver tous les messages des loggers &quot;Backup&quot; et &quot;RBAC&quot;.

```xml
<logger>
    <levels>
        <logger>
            <name>Backup</name>
            <level>none</level>
        </logger>
        <logger>
            <name>RBAC</name>
            <level>none</level>
        </logger>
    </levels>
</logger>
```

**syslog**

Pour envoyer également les messages de journalisation vers syslog :

```xml
<logger>
    <use_syslog>1</use_syslog>
    <syslog>
        <address>syslog.remote:10514</address>
        <hostname>myhost.local</hostname>
        <facility>LOG_LOCAL6</facility>
        <format>syslog</format>
    </syslog>
</logger>
```

Clés pour `<syslog>` :

| Key        | Description                                                                                                                                                                                                                                                                          |
| ---------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| `address`  | L&#39;adresse de syslog au format `host\[:port\]`. Si elle est omise, le démon local est utilisé.                                                                                                                                                                                    |
| `hostname` | Le nom d&#39;hôte à partir duquel les logs sont envoyés (facultatif).                                                                                                                                                                                                                |
| `facility` | Le [mot-clé facility](https://en.wikipedia.org/wiki/Syslog#Facility) de syslog. Il doit être indiqué en majuscules avec le préfixe &quot;LOG&#95;&quot;, par ex. `LOG_USER`, `LOG_DAEMON`, `LOG_LOCAL3`, etc. Par défaut : `LOG_USER` si `address` est spécifié, sinon `LOG_DAEMON`. |
| `format`   | Format du message de log. Valeurs possibles : `bsd` et `syslog.`                                                                                                                                                                                                                     |

**Formats de logs**

Vous pouvez spécifier le format de log qui sera affiché dans les logs de la console. Actuellement, seul JSON est pris en charge.

**Exemple**

Voici un exemple de log JSON en sortie :

```json
{
  "date_time_utc": "2024-11-06T09:06:09Z",
  "date_time": "1650918987.180175",
  "thread_name": "#1",
  "thread_id": "254545",
  "level": "Trace",
  "query_id": "",
  "logger_name": "BaseDaemon",
  "message": "Received signal 2",
  "source_file": "../base/daemon/BaseDaemon.cpp; virtual void SignalListener::run()",
  "source_line": "192"
}
```

Pour activer la prise en charge de la journalisation au format JSON, utilisez l&#39;extrait suivant :

```xml
<logger>
    <formatting>
        <type>json</type>
        <!-- Can be configured on a per-channel basis (log, errorlog, console, syslog), or globally for all channels (then just omit it). -->
        <!-- <channel></channel> -->
        <names>
            <date_time>date_time</date_time>
            <thread_name>thread_name</thread_name>
            <thread_id>thread_id</thread_id>
            <level>level</level>
            <query_id>query_id</query_id>
            <logger_name>logger_name</logger_name>
            <message>message</message>
            <source_file>source_file</source_file>
            <source_line>source_line</source_line>
        </names>
    </formatting>
</logger>
```

**Renommer les clés des logs JSON**

Les noms de clés peuvent être modifiés en changeant les valeurs des balises à l’intérieur de la balise `<names>`. Par exemple, pour remplacer `DATE_TIME` par `MY_DATE_TIME`, vous pouvez utiliser `<date_time>MY_DATE_TIME</date_time>`.

**Omettre des clés dans les logs JSON**

Les propriétés des logs peuvent être omises en commentant la propriété correspondante. Par exemple, si vous ne souhaitez pas que votre log affiche `query_id`, vous pouvez commenter la balise `<query_id>`.

<SettingsInfoBlock type="UInt64" default_value="0" changeable_without_restart="Oui" />

<div id="max_server_memory_usage_to_ram_ratio">
  ## max_server_memory_usage_to_ram_ratio
</div>

<SettingsInfoBlock type="Double" default_value="0.9" changeable_without_restart="No" />

La quantité maximale de mémoire que le serveur est autorisé à utiliser, exprimée sous forme de ratio par rapport à la mémoire totale disponible.

Par exemple, une valeur de `0.9` (par défaut) signifie que le serveur peut consommer 90 % de la mémoire disponible.

Permet de réduire l’utilisation de la mémoire sur les systèmes disposant de peu de mémoire.
Sur les hôtes disposant de peu de RAM et de swap, il peut être nécessaire de définir [`max_server_memory_usage_to_ram_ratio`](#max_server_memory_usage_to_ram_ratio) sur une valeur supérieure à 1.

Ce ratio est appliqué de deux façons :

* Au démarrage (et lors du rechargement de la configuration), il plafonne la limite stricte de mémoire du serveur à cette fraction de la RAM physique totale, comme décrit ci-dessus.
* À l’exécution, le background memory worker recalcule périodiquement la limite stricte de mémoire selon la formule `(resident memory + system available memory) * max_server_memory_usage_to_ram_ratio`, afin que le serveur conserve une marge pour les autres processus exécutés sur le même hôte. Lorsqu’il s’exécute dans un cgroup avec une limite mémoire finie, la mémoire disponible du cgroup est utilisée à la place de la valeur à l’échelle de l’hôte. À mesure que les autres processus consomment davantage de mémoire et que la mémoire disponible diminue, la limite stricte du serveur suit cette baisse, ce qui plafonne sa croissance avant que l’hôte (ou le cgroup) n’épuise sa mémoire.

Définir ce ratio sur `0` désactive à la fois le plafonnement au démarrage et l’ajustement à l’exécution. L’ajustement à l’exécution est également un no-op sur les systèmes non Linux et dans `clickhouse-keeper`, qui n’expose pas ce paramètre. Pour conserver le plafonnement statique au démarrage/rechargement tout en désactivant l’ajustement à l’exécution (le comportement des versions précédentes), définissez `memory_worker_dynamic_hard_limit` sur `0`.

<Note>
  La consommation maximale de mémoire du serveur est en outre limitée par le paramètre `max_server_memory_usage`.
</Note>

<div id="min_allocation_size_to_throw_on_memory_limit">
  ## min_allocation_size_to_throw_on_memory_limit
</div>

<SettingsInfoBlock type="UInt64" default_value="0" changeable_without_restart="Yes" />

Taille minimale, en octets, d&#39;une allocation C++ générique (du type de celles effectuées par les conteneurs standard, les chaînes, l&#39;agrandissement de `std::vector`, les pointeurs intelligents, etc.) pouvant déclencher `MEMORY_LIMIT_EXCEEDED` une fois `max_server_memory_usage` atteint. Les allocations génériques plus petites sont toujours comptabilisées par le memory tracker, mais restent autorisées même au-delà de la limite, ce qui réduit les échecs intempestifs lors du nettoyage et dans les chemins de gestion des exceptions à l&#39;approche d&#39;un OOM.

Les allocations effectuées par les grands tampons internes de ClickHouse — données de colonnes, tables de hachage, arenas, pipelines de requête, tampons d&#39;E/S — respectent toujours la memory limit et peuvent lever une exception quelle que soit cette valeur. Ce paramètre n&#39;affecte que les allocations implicites qui passent par `operator new`.

Une valeur de `0` (par défaut) préserve le legacy behaviour : les allocations implicites de `operator new` ne sont jamais refusées par le memory tracker, et seules les allocations explicites de ClickHouse peuvent déclencher `MEMORY_LIMIT_EXCEEDED`.

Remarque : pour éviter les effets de bord, il est recommandé de définir une valeur supérieure à `max_untracked_memory`.

<div id="mmap_cache_size">
  ## mmap_cache_size
</div>

<SettingsInfoBlock type="UInt64" default_value="1024" changeable_without_restart="Yes" />

Ce paramètre permet d’éviter des appels open/close fréquents (très coûteux en raison des défauts de page qu’ils entraînent) et de réutiliser des mappages entre plusieurs threads et requêtes. <div id="oom_canary_enable">
  ## oom_canary_enable
</div>

<SettingsInfoBlock type="Bool" default_value="0" changeable_without_restart="No" />

Expérimental. Active l’OOM canary : un processus enfant sacrificiel qui attire l’OOM killer de Linux
avant le processus principal du serveur ClickHouse, laissant ainsi au serveur une chance de réduire sa charge.
Nécessite Linux &gt;= 5.3 (pour `pidfd_open`) ; le canary est désactivé au démarrage sur les noyaux plus anciens.
La réponse à l’OOM nécessite la détection d’un OOM-kill via `memory.events.local` de cgroup v2 et peut déclencher
une annulation globale des requêtes, une annulation des fusions et des écritures dans `system.crash_log`.
Le canary ne peut pas protéger le serveur lorsque `memory.oom.group` de cgroup v2 est activé pour le
cgroup du serveur : le noyau tue alors l’ensemble du cgroup comme une seule unité, y compris le serveur,
de sorte que la réponse à l’OOM ne s’exécute jamais. Un avertissement est consigné au démarrage dans ce mode.
Le comportement peut changer d’une version de ClickHouse à l’autre jusqu’à ce que la validation en production soit terminée.

<div id="oom_canary_initial_backoff_seconds">
  ## oom_canary_initial_backoff_seconds
</div>

<SettingsInfoBlock type="UInt64" default_value="1" changeable_without_restart="Non" />

Délai de backoff initial, en secondes, entre deux relances consécutives de l&#39;OOM canary.
Le délai double à chaque relance jusqu&#39;à `oom_canary_max_backoff_seconds`.

<div id="oom_canary_max_backoff_seconds">
  ## oom_canary_max_backoff_seconds
</div>

<SettingsInfoBlock type="UInt64" default_value="60" changeable_without_restart="No" />

Délai maximal de backoff, en secondes, entre deux relances consécutives de l’OOM canary.

<div id="oom_canary_max_rapid_relaunches">
  ## oom_canary_max_rapid_relaunches
</div>

<SettingsInfoBlock type="UInt64" default_value="10" changeable_without_restart="No" />

Nombre maximal de relances rapides consécutives de l’OOM canary avant la désactivation de la relance automatique,
afin d’éviter les redémarrages en boucle sous une pression mémoire persistante. Le compteur et le backoff de relance sont réinitialisés
dès qu’un canary survit plus longtemps que `oom_canary_max_backoff_seconds`, de sorte qu’un canary qui ne meurt que
sporadiquement sur une longue période de fonctionnement ne soit pas désactivé à terme.
S’applique uniquement lorsque `oom_canary_relaunch` vaut true.

<div id="oom_canary_relaunch">
  ## oom_canary_relaunch
</div>

<SettingsInfoBlock type="Bool" default_value="1" changeable_without_restart="Non" />

Lorsque cette valeur est définie sur `true`, l’OOM canary est automatiquement relancé après l’arrêt du processus canari pour toute
raison autre qu’un échec permanent de l’initialisation ou un arrêt du serveur — y compris les OOM kills, d’autres
signaux et des arrêts temporaires — sous réserve de `oom_canary_max_rapid_relaunches` et des paramètres
de backoff. La séquence de réponse OOM elle-même ne s’exécute que lorsque cgroup v2 `memory.events.local`
fournit des indications d’OOM kill.

<div id="oom_canary_size">
  ## oom_canary_size
</div>

<SettingsInfoBlock type="UInt64" default_value="104857600" changeable_without_restart="No" />

Taille en octets de la région mémoire que le processus enfant OOM canary alloue et active.
Le verrouillage de la région avec `mlock` est effectué au mieux : il nécessite `CAP_IPC_LOCK` ou un
`RLIMIT_MEMLOCK` suffisant, et en cas d’échec du verrouillage, le canary consigne un avertissement dans les logs et la mémoire reste
allouée, mais peut être déplacée vers l’espace d’échange.
La valeur par défaut est de 100 Mo (104857600). Des valeurs plus élevées font du canary une cible OOM plus probable.

<div id="proxy">
  ## proxy
</div>

Définissez des serveurs proxy pour les requêtes HTTP et HTTPS, actuellement pris en charge par le stockage S3, les fonctions de table S3 et les fonctions URL.

Il existe trois façons de définir des serveurs proxy :

* variables d’environnement
* listes de proxys
* résolveurs proxy distants.

Il est également possible de contourner les serveurs proxy pour des hôtes spécifiques à l’aide de `no_proxy`.

**Variables d’environnement**

Les variables d’environnement `http_proxy` et `https_proxy` vous permettent de spécifier un
serveur proxy pour un protocole donné. Si elles sont définies sur votre système, cela devrait fonctionner de façon transparente.

Il s’agit de l’approche la plus simple si un protocole donné n’a
qu’un seul serveur proxy et que ce serveur proxy ne change pas.

**Listes de proxys**

Cette approche vous permet de spécifier un ou plusieurs
serveurs proxy pour un protocole. Si plusieurs serveurs proxy sont définis,
ClickHouse utilise les différents proxys selon une méthode round-robin, en répartissant la
charge entre les serveurs. C’est l’approche la plus simple s’il existe plusieurs
serveurs proxy pour un protocole et que la liste des serveurs proxy ne change pas.

**Modèle de configuration**

```xml
<proxy>
    <http>
        <uri>http://proxy1</uri>
        <uri>http://proxy2:3128</uri>
    </http>
    <https>
        <uri>http://proxy1:3128</uri>
    </https>
</proxy>
```

Sélectionnez un champ parent dans les onglets ci-dessous pour afficher ses champs enfants :

<Tabs>
  <Tab title="<proxy>">
    | Champ     | Description                              |
    | --------- | ---------------------------------------- |
    | `<http>`  | Une liste d’un ou plusieurs proxys HTTP  |
    | `<https>` | Une liste d’un ou plusieurs proxys HTTPS |
  </Tab>

  <Tab title="<http> et <https>">
    | Champ   | Description    |
    | ------- | -------------- |
    | `<uri>` | L’URI du proxy |
  </Tab>
</Tabs>

**Résolveurs de proxy distants**

Il est possible que les serveurs proxy changent dynamiquement. Dans ce
cas, vous pouvez définir l’endpoint d’un résolveur. ClickHouse envoie
une requête GET vide à cet endpoint, et le résolveur distant doit renvoyer l’hôte du proxy.
ClickHouse l’utilisera pour construire l’URI du proxy à l’aide du modèle suivant : `\{proxy_scheme\}://\{proxy_host\}:{proxy_port}`

**Modèle de configuration**

```xml
<proxy>
    <http>
        <resolver>
            <endpoint>http://resolver:8080/hostname</endpoint>
            <proxy_scheme>http</proxy_scheme>
            <proxy_port>80</proxy_port>
            <proxy_cache_time>10</proxy_cache_time>
        </resolver>
    </http>

    <https>
        <resolver>
            <endpoint>http://resolver:8080/hostname</endpoint>
            <proxy_scheme>http</proxy_scheme>
            <proxy_port>3128</proxy_port>
            <proxy_cache_time>10</proxy_cache_time>
        </resolver>
    </https>

</proxy>
```

Sélectionnez un champ parent dans les onglets ci-dessous pour afficher ses éléments enfants :

<Tabs>
  <Tab title="<proxy>">
    | Champ     | Description                                     |
    | --------- | ----------------------------------------------- |
    | `<http>`  | Une liste d&#39;un ou plusieurs résolveurs* |
    | `<https>` | Une liste d&#39;un ou plusieurs résolveurs* |
  </Tab>

  <Tab title="<http> et <https>">
    | Champ        | Description                                               |
    | ------------ | --------------------------------------------------------- |
    | `<resolver>` | L&#39;endpoint et d&#39;autres détails d&#39;un résolveur |

    <Note>
      Vous pouvez avoir plusieurs éléments `<resolver>`, mais seul le premier
      `<resolver>` pour un protocole donné est utilisé. Tous les autres éléments `<resolver>`
      pour ce protocole sont ignorés. Cela signifie que la répartition de charge
      (si nécessaire) doit être implémentée par le résolveur distant.
    </Note>
  </Tab>

  <Tab title="<resolver>">
    | Champ                | Description                                                                                                                                                                                                           |
    | -------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
    | `<endpoint>`         | L&#39;URI du résolveur proxy                                                                                                                                                                                          |
    | `<proxy_scheme>`     | Le protocole de l&#39;URI proxy finale. Il peut s&#39;agir de `http` ou de `https`.                                                                                                                                   |
    | `<proxy_port>`       | Le numéro de port du résolveur proxy                                                                                                                                                                                  |
    | `<proxy_cache_time>` | La durée, en secondes, pendant laquelle les valeurs du résolveur doivent être mises en cache par ClickHouse. Si cette valeur est définie sur `0`, ClickHouse contacte le résolveur pour chaque requête HTTP ou HTTPS. |
  </Tab>
</Tabs>

**Priorité**

Les paramètres proxy sont déterminés dans l&#39;ordre suivant :

| Ordre | Paramètre                     |
| ----- | ----------------------------- |
| 1.    | Résolveurs proxy distants     |
| 2.    | Listes de proxys              |
| 3.    | Variables d&#39;environnement |

ClickHouse vérifie le type de résolveur ayant la priorité la plus élevée pour le protocole de la requête. S’il n’est pas défini,
il vérifie alors le type de résolveur de priorité immédiatement inférieure, jusqu’à atteindre le résolveur d’environnement.
Cela permet également d’utiliser une combinaison de types de résolveurs.

<div id="query_masking_rules">
  ## query_masking_rules
</div>

Règles basées sur des expressions régulières, appliquées aux requêtes ainsi qu’à tous les messages de journal avant leur stockage dans les journaux du serveur,
les tables [`system.query_log`](/fr/reference/system-tables/query_log), [`system.text_log`](/fr/reference/system-tables/text_log), [`system.processes`](/fr/reference/system-tables/processes), ainsi que dans les journaux envoyés au client. Cela permet d’éviter
la fuite de données sensibles issues des requêtes SQL, telles que des noms, des e-mails, des identifiants personnels ou des numéros de carte de crédit, dans les journaux.

**Exemple**

```xml
<query_masking_rules>
    <rule>
        <name>hide SSN</name>
        <regexp>(^|\D)\d{3}-\d{2}-\d{4}($|\D)</regexp>
        <replace>000-00-0000</replace>
    </rule>
</query_masking_rules>
```

**Champs de configuration** :

| Paramètre | Description                                                                                  |
| --------- | -------------------------------------------------------------------------------------------- |
| `name`    | nom de la règle (facultatif)                                                                 |
| `regexp`  | expression régulière compatible avec RE2 (obligatoire)                                       |
| `replace` | chaîne de substitution pour les données sensibles (facultatif, par défaut : six astérisques) |

Les règles de masquage s’appliquent à l’ensemble de la requête (afin d’éviter les fuites de données sensibles provenant de requêtes mal formées / impossibles à analyser).

La table [`system.events`](/fr/reference/system-tables/events) contient le compteur `QueryMaskingRulesMatch`, qui comptabilise le nombre total de correspondances avec les règles de masquage des requêtes.

Pour les requêtes distribuées, chaque serveur doit être configuré séparément, sinon les sous-requêtes transmises à d’autres
nœuds seront stockées sans masquage.

<div id="remote_url_allow_hosts">
  ## remote_url_allow_hosts
</div>

Liste des hôtes autorisés à être utilisés dans les engines de stockage liés aux URL et les fonctions de table.

Lors de l’ajout d’un hôte avec la balise XML `\<host\>` :

* il doit être spécifié exactement comme dans l’URL, car le nom est vérifié avant la résolution DNS. Par exemple : `<host>clickhouse.com</host>`
* si le port est explicitement indiqué dans l’URL, `host:port` est alors vérifié dans son ensemble. Par exemple : `<host>clickhouse.com:80</host>`
* si l’hôte est spécifié sans port, alors n’importe quel port de cet hôte est autorisé. Par exemple : si `<host>clickhouse.com</host>` est spécifié, alors `clickhouse.com:20` (FTP), `clickhouse.com:80` (HTTP), `clickhouse.com:443` (HTTPS), etc. sont autorisés.
* si l’hôte est spécifié sous forme d’adresse IP, alors il est vérifié tel qu’il apparaît dans l’URL. Par exemple : `[2a02:6b8:a::a]`.
* s’il y a des redirections et que la prise en charge des redirections est activée, alors chaque redirection (le champ lieu) est vérifiée.

Par exemple :

```sql
<remote_url_allow_hosts>
    <host>clickhouse.com</host>
</remote_url_allow_hosts>
```

<div id="configuration-of-disks">
  ### Configuration de `disks`
</div>

La configuration de `disks` suit la structure ci-dessous :

```xml
<storage_configuration>
    <disks>
        <disk_name_1>
            <path>/mnt/fast_ssd/clickhouse/</path>
        </disk_name_1>
        <disk_name_2>
            <path>/mnt/hdd1/clickhouse/</path>
            <keep_free_space_bytes>10485760</keep_free_space_bytes>
        </disk_name_2>
        <disk_name_3>
            <path>/mnt/hdd2/clickhouse/</path>
            <keep_free_space_bytes>10485760</keep_free_space_bytes>
        </disk_name_3>
        ...
    </disks>
</storage_configuration>
```

Les sous-balises ci-dessus définissent les paramètres suivants pour `disks` :

| Paramètre               | Description                                                                                                      |
| ----------------------- | ---------------------------------------------------------------------------------------------------------------- |
| `<disk_name_N>`         | Le nom du disque, qui doit être unique.                                                                          |
| `path`                  | Le chemin où seront stockées les données du server (répertoires `data` et `shadow`). Il doit se terminer par `/` |
| `keep_free_space_bytes` | Taille de l’espace libre réservé sur le disque.                                                                  |

<Note>
  L’ordre des disques n’a pas d’importance.
</Note>

<div id="configuration-of-policies">
  ### Configuration des politiques
</div>

Les sous-balises ci-dessus définissent les paramètres suivants pour `policies` :

| Setting                      | Description                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                             |
| ---------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `policy_name_N`              | Nom de la politique. Les noms de politique doivent être uniques.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                        |
| `volume_name_N`              | Nom du volume. Les noms de volume doivent être uniques.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                 |
| `disk`                       | Le disque situé dans le volume.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                         |
| `max_data_part_size_bytes`   | Taille maximale d’une part de données pouvant résider sur l’un des disques de ce volume. Si une fusion produit une part dont la taille prévue dépasse `max_data_part_size_bytes`, la part sera écrite sur le volume suivant. Cette fonctionnalité permet essentiellement de stocker les nouveaux petits parts sur un volume rapide (SSD), puis de les déplacer vers un volume plus lent (HDD) lorsqu’ils deviennent volumineux. N’utilisez pas cette option si la politique ne comporte qu’un seul volume.                                                                                                 |
| `move_factor`                | Part de l’espace libre disponible sur le volume. Si l’espace disponible passe en dessous de ce seuil, les données commenceront à être transférées vers le volume suivant, s’il en existe un. Pour ce transfert, les parts sont triées par taille du plus grand au plus petit (ordre décroissant), puis les parts dont la taille cumulée suffit à satisfaire la condition `move_factor` sont sélectionnés. Si la taille totale de tous les parts est insuffisante, tous les parts seront déplacés.                                                                                                        |
| `perform_ttl_move_on_insert` | Désactive le déplacement, lors de l’insertion, des données dont le TTL a expiré. Par défaut (si cette option est activée), si des données déjà expirées selon la règle de déplacement basée sur la durée de vie sont insérées, elles sont immédiatement déplacées vers le volume / disque spécifié par cette règle. Cela peut considérablement ralentir l’insertion si le volume / disque cible est lent (par exemple S3). Si cette option est désactivée, la partie expirée des données est écrite sur le volume par défaut, puis immédiatement déplacée vers le volume indiqué par la règle applicable au TTL expiré. |
| `load_balancing`             | Politique d’équilibrage des disques : `round_robin` ou `least_used`.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                    |
| `least_used_ttl_ms`          | Définit le délai d’expiration (en millisecondes) pour la mise à jour de l’espace disponible sur tous les disques (`0` - toujours mettre à jour, `-1` - ne jamais mettre à jour, valeur par défaut : `60000`). Notez que si le disque est utilisé uniquement par ClickHouse et ne fera pas l’objet d’un redimensionnement du système de fichiers à la volée, vous pouvez utiliser la valeur `-1`. Dans tous les autres cas, cela n’est pas recommandé, car cela finira par entraîner une allocation incorrecte de l’espace.                                                                                              |
| `prefer_not_to_merge`        | Désactive la fusion des parties de données sur ce volume. Remarque : ce paramètre est potentiellement nuisible et peut provoquer des ralentissements. Lorsqu’il est activé (ce qui est déconseillé), la fusion des données sur ce volume est interdite. Cela permet de contrôler la manière dont ClickHouse interagit avec les disques lents. Nous recommandons de ne pas l’utiliser du tout.                                                                                                                                                                                                                           |
| `volume_priority`            | Définit la priorité (ordre) selon laquelle les volumes sont remplis. Plus la valeur est petite, plus la priorité est élevée. Les valeurs de ce paramètre doivent être des entiers naturels et couvrir l’intervalle de 1 à N (N étant la plus grande valeur spécifiée pour ce paramètre) sans lacunes.                                                                                                                                                                                                                                                                                                                   |

Pour `volume_priority` :

* Si tous les volumes ont ce paramètre, ils sont priorisés dans l’ordre spécifié.
* Si seuls *certains* volumes l’ont, les volumes qui ne l’ont pas ont la priorité la plus faible. Ceux qui l’ont sont priorisés selon la valeur de la balise, tandis que la priorité des autres est déterminée, entre eux, par leur ordre de description dans le fichier de configuration.
* Si *aucun* volume n’a ce paramètre, leur ordre est déterminé par l’ordre de description dans le fichier de configuration.
* La priorité des volumes ne peut pas être identique.

<div id="use_minimalistic_part_header_in_zookeeper">
  ## use_minimalistic_part_header_in_zookeeper
</div>

Méthode de stockage des en-têtes des parties de données dans ZooKeeper. Ce paramètre s’applique uniquement à la famille [`MergeTree`](/fr/reference/engines/table-engines/mergetree-family/index). Il peut être défini :

**Globalement dans la section [merge&#95;tree](#merge_tree) du fichier `config.xml`**

ClickHouse utilise ce paramètre pour toutes les tables du serveur. Vous pouvez le modifier à tout moment. Les tables existantes changent alors de comportement.

**Pour chaque table**

Lors de la création d’une table, spécifiez le [paramètre de moteur](/fr/reference/engines/table-engines/mergetree-family/mergetree#table_engine-mergetree-creating-a-table) correspondant. Le comportement d’une table existante dotée de ce paramètre ne change pas, même si le paramètre global change.

**Valeurs possibles**

* `0` — La fonctionnalité est désactivée.
* `1` — La fonctionnalité est activée.

Si [`use_minimalistic_part_header_in_zookeeper = 1`](#use_minimalistic_part_header_in_zookeeper), les tables [répliquées](/fr/reference/engines/table-engines/mergetree-family/replication) stockent les en-têtes des parties de données de manière compacte à l’aide d’un seul `znode`. Si la table contient de nombreuses colonnes, cette méthode de stockage réduit considérablement le volume de données stockées dans ZooKeeper.

<Note>
  Après avoir appliqué `use_minimalistic_part_header_in_zookeeper = 1`, vous ne pouvez pas revenir à une version antérieure du serveur ClickHouse qui ne prend pas en charge ce paramètre. Soyez prudent lors de la mise à niveau de ClickHouse sur les serveurs d’un cluster. Ne mettez pas à niveau tous les serveurs en une seule fois. Il est plus sûr de tester les nouvelles versions de ClickHouse dans un environnement de test, ou sur seulement quelques serveurs du cluster.

  Les en-têtes des parties de données déjà stockés avec ce paramètre ne peuvent pas être restaurés dans leur représentation précédente (non compacte).
</Note>

<div id="use_shared_merge_tree_log_pipeline">
  ## use_shared_merge_tree_log_pipeline
</div>

<SettingsInfoBlock type="Bool" default_value="0" changeable_without_restart="No" />

Disponible uniquement sur ClickHouse Cloud. Lorsqu’elle est activée, les tables `system.*_log` s’appuient sur `SharedMergeTree` via un pipeline basé sur S3.
Pour chaque log `<log>`, les objets suivants sont créés automatiquement au démarrage :

* `system.<log>_s3` — une table adossée à `S3` qui constitue la cible directe du flush ; chaque
  appel à `SYSTEM FLUSH LOGS` y écrit un nouveau fichier partitionné.
* `system.<log>_s3queue` — une table `S3Queue` (mode ordonné) qui récupère les fichiers depuis
  `<log>_s3` et transmet les lignes en aval. Chaque nœud ne traite que ses propres fichiers via
  un partitionnement basé sur `partition_regex`.
* `system.<log>_mv` — une `MATERIALIZED VIEW` qui achemine les lignes de `<log>_s3queue` vers
  la table `SharedMergeTree` finale.
* `system.<log>` — la table `SharedMergeTree` finale dans laquelle les lignes s’accumulent et sont interrogées.

Lors d’un changement de schéma ou de paramètres, la table concernée est renommée en `<log>_0`, `<log>_1`, etc.
puis recréée, conformément au comportement existant de rotation de `SystemLog`.

Nécessite que `<shared_log_pipeline><endpoint>` soit défini dans la configuration du serveur.
Voir aussi : `shared_log_pipeline.enable_sync_flush`, `shared_log_pipeline.flush_timeout_seconds`.

| Paramètre | Description                                                                                                                                                                                                                                                                                                                                                                                                                      |
| --------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |

| `server`  | l’un des noms de serveur LDAP définis dans la section de config `ldap_servers`. Ce paramètre est obligatoire et ne peut pas être vide.                                                                                                                                                                                                                                                                                           |

**Exemple**

<div id="wait_dictionaries_load_at_startup">
  ## wait_dictionaries_load_at_startup
</div>

<SettingsInfoBlock type="Bool" default_value="1" changeable_without_restart="Non" />

Ce paramètre permet de définir le comportement si `dictionaries_lazy_load` est `false`.
(Si `dictionaries_lazy_load` est `true`, ce paramètre n&#39;a aucun effet.)

Si `wait_dictionaries_load_at_startup` est `false`, le serveur
commencera à charger tous les dictionnaires au démarrage et acceptera des connexions en parallèle de ce chargement.
Lorsqu&#39;un dictionnaire est utilisé pour la première fois dans une requête, celle-ci attendra que le dictionnaire soit chargé s&#39;il ne l&#39;est pas encore.
Définir `wait_dictionaries_load_at_startup` sur `false` peut permettre à ClickHouse de démarrer plus rapidement ; en revanche, certaines requêtes peuvent s&#39;exécuter plus lentement
(car elles devront attendre le chargement de certains dictionnaires).

Si `wait_dictionaries_load_at_startup` est `true`, le serveur attendra au démarrage
que tous les dictionnaires aient terminé leur chargement (avec succès ou non) avant d&#39;accepter la moindre connexion.

**Exemple**

```xml
<wait_dictionaries_load_at_startup>true</wait_dictionaries_load_at_startup>
```

<div id="zookeeper">
  ## zookeeper
</div>

Contient des paramètres qui permettent à ClickHouse d’interagir avec un cluster [ZooKeeper](http://zookeeper.apache.org/). ClickHouse utilise ZooKeeper pour stocker les métadonnées des répliques lors de l’utilisation de tables répliquées. Si les tables répliquées ne sont pas utilisées, cette section de paramètres peut être omise.

Les paramètres suivants peuvent être configurés à l’aide de sous-balises :

| Paramètre                                       | Description                                                                                                                                                                                                                                                                                                                                                                                                                    |
| ----------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| `node`                                          | Point de terminaison ZooKeeper. Vous pouvez définir plusieurs endpoints. Par ex. `<node index="1"><host>example_host</host><port>2181</port></node>`. L’attribut `index` indique l’ordre des nœuds lors des tentatives de connexion au cluster ZooKeeper.                                                                                                                                                                      |
| `operation_timeout_ms`                          | Délai d’expiration maximal pour une opération, en millisecondes.                                                                                                                                                                                                                                                                                                                                                               |
| `session_timeout_ms`                            | Délai d’expiration maximal pour la session du client, en millisecondes.                                                                                                                                                                                                                                                                                                                                                        |
| `root` (optional)                               | Le znode utilisé comme racine pour les znodes utilisés par le serveur ClickHouse.                                                                                                                                                                                                                                                                                                                                              |
| `fallback_session_lifetime.min` (optional)      | Limite minimale de durée de vie d’une session ZooKeeper vers le nœud de secours lorsque le primaire n’est pas disponible (équilibrage de charge). Définie en secondes. Par défaut : 3 heures.                                                                                                                                                                                                                                  |
| `fallback_session_lifetime.max` (optional)      | Limite maximale de durée de vie d’une session ZooKeeper vers le nœud de secours lorsque le primaire n’est pas disponible (équilibrage de charge). Définie en secondes. Par défaut : 6 heures.                                                                                                                                                                                                                                  |
| `identity` (optional)                           | Nom d’utilisateur et mot de passe requis par ZooKeeper pour accéder aux znodes demandés.                                                                                                                                                                                                                                                                                                                                       |
| `use_compression` (optional)                    | Active la compression dans le protocole Keeper si ce paramètre est défini sur true.                                                                                                                                                                                                                                                                                                                                            |
| `use_xid_64` (optional)                         | Active les identifiants de transaction sur 64 bits. Définissez `true` pour activer le format étendu d’identifiant de transaction. Par défaut : `false`.                                                                                                                                                                                                                                                                        |
| `pass_opentelemetry_tracing_context` (optional) | Active la propagation du contexte de traçage OpenTelemetry vers les requêtes Keeper. Lorsqu’elle est activée, des spans de traçage sont créés pour les opérations Keeper, ce qui permet le traçage distribué entre ClickHouse et Keeper. Voir [Tracing ClickHouse Keeper Requests](/fr/guides/oss/deployment-and-scaling/monitoring/opentelemetry#tracing-clickhouse-keeper-requests) pour plus de détails. Par défaut : `false`. |

Il existe également le paramètre `zookeeper_load_balancing` (optional), qui permet de sélectionner l’algorithme de sélection des nœuds ZooKeeper :

| Nom de l’algorithme              | Description                                                                                                                     |
| -------------------------------- | ------------------------------------------------------------------------------------------------------------------------------- |
| `random`                         | sélectionne aléatoirement l’un des nœuds ZooKeeper.                                                                             |
| `in_order`                       | sélectionne le premier nœud ZooKeeper ; s’il n’est pas disponible, le deuxième, et ainsi de suite.                              |
| `nearest_hostname`               | sélectionne un nœud ZooKeeper dont le hostname est le plus proche de celui du serveur, en comparant le préfixe du nom.          |
| `hostname_levenshtein_distance`  | comme nearest&#95;hostname, mais compare le hostname selon la distance de Levenshtein.                                          |
| `hostname_longest_common_prefix` | comme nearest&#95;hostname, mais préfère le nœud dont le hostname partage le plus long préfixe commun avec celui du serveur.    |
| `hostname_longest_common_suffix` | comme nearest&#95;hostname, mais préfère le nœud dont le hostname partage le plus long suffixe commun avec celui du serveur.    |
| `first_or_random`                | sélectionne le premier nœud ZooKeeper ; s’il n’est pas disponible, sélectionne aléatoirement l’un des nœuds ZooKeeper restants. |
| `round_robin`                    | sélectionne le premier nœud ZooKeeper, puis sélectionne le suivant en cas de reconnexion.                                       |

**Exemple de configuration**

```xml
<zookeeper>
    <node>
        <host>example1</host>
        <port>2181</port>
    </node>
    <node>
        <host>example2</host>
        <port>2181</port>
    </node>
    <session_timeout_ms>30000</session_timeout_ms>
    <operation_timeout_ms>10000</operation_timeout_ms>
    <!-- Optional. Chroot suffix. Should exist. -->
    <root>/path/to/zookeeper/node</root>
    <!-- Optional. Zookeeper digest ACL string. -->
    <identity>user:password</identity>
    <!--<zookeeper_load_balancing>random / in_order / nearest_hostname / hostname_levenshtein_distance / hostname_longest_common_prefix / hostname_longest_common_suffix / first_or_random / round_robin</zookeeper_load_balancing>-->
    <zookeeper_load_balancing>random</zookeeper_load_balancing>
    <!-- Optional. Enable 64-bit transaction IDs. -->
    <use_xid_64>false</use_xid_64>
    <!-- Optional. Enable OpenTelemetry tracing context propagation. -->
    <pass_opentelemetry_tracing_context>false</pass_opentelemetry_tracing_context>
</zookeeper>
```
