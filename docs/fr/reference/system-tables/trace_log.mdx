<div id="description">
  ## Description
</div>

Contient les traces de pile collectées par le [profileur de requêtes par échantillonnage](/fr/concepts/features/performance/troubleshoot/sampling-query-profiler).

ClickHouse crée cette table lorsque la section de configuration du serveur [trace&#95;log](/fr/reference/settings/server-settings/settings#trace_log) est définie. Voir aussi les paramètres : [query&#95;profiler&#95;real&#95;time&#95;period&#95;ns](/fr/reference/settings/session-settings#query_profiler_real_time_period_ns), [query&#95;profiler&#95;cpu&#95;time&#95;period&#95;ns](/fr/reference/settings/session-settings#query_profiler_cpu_time_period_ns), [memory&#95;profiler&#95;step](/fr/reference/settings/session-settings#memory_profiler_step),
[memory&#95;profiler&#95;sample&#95;probability](/fr/reference/settings/session-settings#memory_profiler_sample_probability), [trace&#95;profile&#95;events](/fr/reference/settings/session-settings#trace_profile_events).

Pour analyser les journaux, utilisez les fonctions d’introspection `addressToLine`, `addressToLineWithInlines`, `addressToSymbol` et `demangle`.

<div id="columns">
  ## Colonnes
</div>

{/*AUTOGENERATED_START*/}

* `hostname` ([LowCardinality(String)](/fr/reference/data-types/lowcardinality)) — Nom d’hôte du serveur qui exécute la requête.
* `event_date` ([Date](/fr/reference/data-types/date)) — Date du moment d’échantillonnage.
* `event_time` ([DateTime](/fr/reference/data-types/datetime)) — Horodatage du moment d’échantillonnage.
* `event_time_microseconds` ([DateTime64(6)](/fr/reference/data-types/datetime64)) — Horodatage du moment d’échantillonnage avec une précision à la microseconde.
* `timestamp_ns` ([UInt64](/fr/reference/data-types/int-uint)) — Horodatage du moment d’échantillonnage en nanosecondes.
* `revision` ([UInt32](/fr/reference/data-types/int-uint)) — Révision de build du serveur ClickHouse. Lors de la connexion au serveur avec `clickhouse-client`, vous voyez une chaîne semblable à `Connected to ClickHouse server version 19.18.1.`. Ce champ contient la `revision`, mais pas la `version` d’un serveur.
* `trace_type` ([Enum8(&#39;Real&#39; = 0, &#39;CPU&#39; = 1, &#39;Memory&#39; = 2, &#39;MemorySample&#39; = 3, &#39;MemoryPeak&#39; = 4, &#39;ProfileEvent&#39; = 5, &#39;JemallocSample&#39; = 6, &#39;MemoryAllocatedWithoutCheck&#39; = 7, &#39;Instrumentation&#39; = 8)](/fr/reference/data-types/enum)) — Type de trace : `Real` représente la collecte de stack traces selon le temps réel. `CPU` représente la collecte de stack traces selon le temps CPU. `Memory` représente la collecte des allocations et désallocations lorsque l’allocation mémoire dépasse le watermark suivant. `MemorySample` représente la collecte aléatoire d’allocations et de désallocations. `MemoryPeak` représente la collecte des mises à jour du pic d’utilisation mémoire. `ProfileEvent` représente la collecte des incréments des profile events. `JemallocSample` représente la collecte d’échantillons jemalloc. `MemoryAllocatedWithoutCheck` représente la collecte d’allocations importantes (&gt;16MiB) effectuée en ignorant toute limite mémoire (réservé aux développeurs ClickHouse). `Instrumentation` représente les traces collectées par l’instrumentation effectuée via XRay.
* `cpu_id` ([UInt64](/fr/reference/data-types/int-uint)) — Identifiant CPU.
* `thread_id` ([UInt64](/fr/reference/data-types/int-uint)) — Identifiant du thread.
* `thread_name` ([LowCardinality(String)](/fr/reference/data-types/lowcardinality)) — Nom du thread.
* `query_id` ([String](/fr/reference/data-types/string)) — Identifiant de requête pouvant être utilisé pour obtenir les détails d’une requête en cours d’exécution à partir de la table système query&#95;log.
* `trace` ([Array(UInt64)](/fr/reference/data-types/array)) — Stack trace au moment de l’échantillonnage. Chaque élément est une adresse de mémoire virtuelle à l’intérieur du processus serveur ClickHouse.
* `size` ([Int64](/fr/reference/data-types/int-uint)) — Pour les types de trace Memory, MemorySample, MemoryAllocatedWithoutCheck ou MemoryPeak, il s’agit de la quantité de mémoire allouée ; pour les autres types de trace, la valeur est 0.
* `ptr` ([UInt64](/fr/reference/data-types/int-uint)) — Adresse du chunk alloué.
* `memory_context` ([Enum8(&#39;Unknown&#39; = -1, &#39;Global&#39; = 0, &#39;User&#39; = 1, &#39;Process&#39; = 2, &#39;Thread&#39; = 3, &#39;Max&#39; = 4)](/fr/reference/data-types/enum)) — Contexte du Memory Tracker (uniquement pour Memory/MemoryPeak) : le contexte `Unknown` n’est pas défini pour ce trace&#95;type. `Global` représente le contexte du serveur. `User` représente le contexte utilisateur/merge. `Process` représente le contexte du processus (c.-à-d. de la requête). `Thread` représente le contexte du thread (thread d’un processus particulier). `Max` est une valeur spéciale qui signifie que le memory tracker n’est pas bloqué (pour la colonne blocked&#95;context).
* `memory_blocked_context` ([Enum8(&#39;Unknown&#39; = -1, &#39;Global&#39; = 0, &#39;User&#39; = 1, &#39;Process&#39; = 2, &#39;Thread&#39; = 3, &#39;Max&#39; = 4)](/fr/reference/data-types/enum)) — Contexte pour lequel le traqueur de mémoire est bloqué (réservé aux développeurs ClickHouse) : le contexte `Unknown` n’est pas défini pour ce trace&#95;type. `Global` représente le contexte du serveur. `User` représente le contexte utilisateur/de fusion. `Process` représente le contexte du processus (c.-à-d. de la requête). `Thread` représente le contexte du thread (thread d’un processus particulier). `Max` est une valeur spéciale qui signifie que le traqueur de mémoire n’est pas bloqué (pour la colonne blocked&#95;context).
* `event` ([LowCardinality(String)](/fr/reference/data-types/lowcardinality)) — Pour le type de trace ProfileEvent, il s’agit du nom de l’événement de profil mis à jour ; pour les autres types de trace, c’est une chaîne vide.
* `increment` ([Int64](/fr/reference/data-types/int-uint)) — Pour le type de trace ProfileEvent, il s’agit de la valeur d’incrément de l’événement de profil ; pour les autres types de trace, c’est 0.
* `symbols` ([Array(LowCardinality(String))](/fr/reference/data-types/array)) — Si la symbolisation est activée, contient les noms de symboles démanglés correspondant à la `trace`. La symbolisation peut être activée ou désactivée dans le paramètre `symbolize` sous `trace_log` dans le fichier de configuration du serveur.
* `lines` ([Array(LowCardinality(String))](/fr/reference/data-types/array)) — Si la symbolisation est activée, contient des chaînes avec les noms de fichiers et les numéros de ligne correspondant à la `trace`.
* `function_id` ([Nullable(Int32)](/fr/reference/data-types/nullable)) — Pour le type de trace Instrumentation, ID attribué à la fonction dans la section xray&#95;instr&#95;map du binaire ELF.
* `function_name` ([Nullable(String)](/fr/reference/data-types/nullable)) — Pour le type de trace Instrumentation, nom de la fonction instrumentée.
* `handler` ([Nullable(String)](/fr/reference/data-types/nullable)) — Pour le type de trace Instrumentation, gestionnaire de la fonction instrumentée.
* `entry_type` ([Nullable(Enum8(&#39;Entry&#39; = 0, &#39;Exit&#39; = 1))](/fr/reference/data-types/nullable)) — Pour le type de trace Instrumentation, type d’entrée de la fonction instrumentée.
* `duration_nanoseconds` ([Nullable(UInt64)](/fr/reference/data-types/nullable)) — Pour le type de trace Instrumentation, durée d’exécution de la fonction en nanosecondes.

**Alias :**

* `build_id` — alias de `'E2EEB60ED9503BFFA825B628D480CCDC40B2D416'`.

{/*AUTOGENERATED_END*/}

La symbolisation peut être activée ou désactivée dans `symbolize`, sous `trace_log`, dans le fichier de configuration du serveur.

<div id="chrome-event-trace-format">
  ## Conversion vers le format Event Trace de Chrome
</div>

Les données de profilage peuvent être converties au format Event Trace de Chrome à l’aide de la requête suivante. Enregistrez cette requête dans un fichier `chrome_trace.sql` :

```sql
WITH traces AS (
    SELECT * FROM system.trace_log
    WHERE event_date >= today() AND trace_type = 'Instrumentation' AND handler = 'profile'
    ORDER BY event_time, entry_type
)
SELECT
    format(
        '{{"traceEvents": [{}\n]}}',
        arrayStringConcat(
            groupArray(
                format(
                    '\n{{"name": "{}", "cat": "clickhouse", "ph": "{}", "ts": {}, "pid": 1, "tid": {}, "args": {{"query_id": "{}", "cpu_id": {}, "stack": [{}]}}}},',
                    function_name,
                    if(entry_type = 0, 'B', 'E'),
                    timestamp_ns/1000,
                    toString(thread_id),
                    query_id,
                    cpu_id,
                    arrayStringConcat(arrayMap((x, y) -> concat('"', x, ': ', y, '", '), lines, symbols))
                )
            )
        )
    )
FROM traces;
```

Et l’exécuter avec ClickHouse Client afin de l’exporter dans un fichier `trace.json`, que nous pouvons importer soit dans [Perfetto](https://ui.perfetto.dev/), soit dans [speedscope](https://www.speedscope.app/).

```bash
echo $(clickhouse client --query "$(cat chrome_trace.sql)") > trace.json
```

Nous pouvons omettre la partie stack si nous voulons une trace plus compacte, mais moins riche en informations.
