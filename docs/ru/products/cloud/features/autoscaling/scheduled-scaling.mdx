Сервисы ClickHouse Cloud автоматически масштабируются в зависимости от загрузки CPU и памяти, но для многих рабочих нагрузок характерны предсказуемые паттерны — ежедневные всплески ингестии, батч-задачи, выполняемые по ночам, или резкое падение трафика по выходным. Для таких сценариев запланированное масштабирование позволяет точно определить, когда ваш сервис должен масштабироваться вверх или вниз, независимо от метрик в реальном времени.

С помощью запланированного масштабирования вы настраиваете набор правил по времени прямо в консоли ClickHouse Cloud. Каждое правило задаёт время, периодичность (ежедневно, еженедельно или по пользовательскому расписанию) и целевой размер — либо число реплик (горизонтальное масштабирование), либо уровень памяти (вертикальное масштабирование). В назначенное время ClickHouse Cloud автоматически применяет изменение, чтобы сервис был масштабирован должным образом ещё до роста нагрузки, а не реагировал на неё постфактум.

Это отличается от автомасштабирования на основе метрик, которое динамически реагирует на нагрузку CPU и памяти. Запланированное масштабирование детерминировано: вы точно знаете, когда произойдёт масштабирование и до какого размера. Эти два подхода дополняют друг друга — у сервиса может быть базовое расписание масштабирования, и при этом он всё равно может выигрывать от автомасштабирования в этих рамках, если рабочая нагрузка неожиданно меняется.

Сейчас запланированное масштабирование доступно в **закрытой предварительной версии**. Чтобы включить эту возможность для вашей организации, свяжитесь с командой поддержки ClickHouse.

<div id="setting-up-a-scaling-schedule">
  ## Настройка расписания масштабирования
</div>

Чтобы настроить расписание, перейдите к своему сервису в консоли ClickHouse Cloud и откройте настройки. Затем выберите **Schedule Override** и добавьте новое правило.

<Image img="/images/cloud/features/autoscaling/scheduled-scaling-1.webp" size="md" alt="Интерфейс Scaling Schedules в консоли ClickHouse Cloud с правилами масштабирования по времени" border />

<Image img="/images/cloud/features/autoscaling/scheduled-scaling-2.webp" size="md" alt="Настройка правила запланированного масштабирования в консоли ClickHouse Cloud" border />

Для каждого правила нужно указать:

* **Время:** когда должно выполняться действие масштабирования (в вашем местном часовом поясе)
* **Повторение:** как часто повторяется правило (например, каждый будний день или каждое воскресенье)
* **Целевой размер:** количество реплик или объём памяти, до которого нужно масштабировать

Несколько правил можно объединить в полное недельное расписание. Например, можно увеличивать число реплик до 5 каждый будний день в 6:00 утра, а в 20:00 уменьшать его обратно до 2.

<div id="use-cases">
  ## Сценарии использования
</div>

**Батч- и ETL-нагрузки:** Увеличивайте масштаб перед запуском ночной задачи по приёму данных и уменьшайте его после её завершения, чтобы избежать избыточного выделения ресурсов в дневные часы простоя.

**Предсказуемые паттерны трафика:** Сервисы со стабильными периодами пиковой нагрузки (например, с трафиком запросов в рабочие часы) можно заранее масштабировать под ожидаемую нагрузку, а не ждать, пока среагирует автомасштабирование.

**Уменьшение масштаба на выходных:** Снижайте число реплик или уровень памяти на выходных, когда спрос ниже, а затем восстанавливайте ресурсы перед утренним всплеском в понедельник.

**Контроль затрат:** Для команд, контролирующих затраты на ClickHouse Cloud, запланированное уменьшение масштаба в периоды заведомо низкой нагрузки может заметно сократить потребление ресурсов без какого-либо ручного вмешательства.

<Note>
  Действие запланированного масштабирования и одновременная рекомендация по автомасштабированию могут взаимодействовать — в момент срабатывания приоритет у расписания.
</Note>

<div id="handling-bursty-workloads">
  ## Обработка всплесков рабочей нагрузки
</div>

Если вы ожидаете всплеск рабочей нагрузки, можно использовать
[ClickHouse Cloud API](/ru/products/cloud/features/admin-features/api/api-overview), чтобы
заранее увеличить масштаб сервиса, а после снижения нагрузки — уменьшить его.

Чтобы понять, сколько ядер CPU и памяти сейчас используется для
каждой из ваших реплик, выполните запрос ниже:

```sql
SELECT *
FROM clusterAllReplicas('default', view(
    SELECT
        hostname() AS server,
        anyIf(value, metric = 'CGroupMaxCPU') AS cpu_cores,
        formatReadableSize(anyIf(value, metric = 'CGroupMemoryTotal')) AS memory
    FROM system.asynchronous_metrics
))
ORDER BY server ASC
SETTINGS skip_unavailable_shards = 1
```
