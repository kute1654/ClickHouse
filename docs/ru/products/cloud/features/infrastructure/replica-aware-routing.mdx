Маршрутизация с учетом реплик (также известная как sticky sessions, липкая маршрутизация или session affinity) повышает вероятность повторного использования кэша, последовательно направляя связанные запросы на одну и ту же реплику ClickHouse. Она работает по принципу best-effort и не гарантирует изоляцию.

<Warning>
  **Поддержка протоколов — только HTTP/HTTPS**

  Маршрутизация с учетом реплик применяется на уровне прокси через **интерфейс HTTP/HTTPS** с использованием параметра запроса `session_id` (см. ниже). Она **недоступна через собственный протокол** (native port, например для драйвера `clickhouse-go` в стандартном native-режиме). Если вам нужна липкая маршрутизация для клиента, использующего собственный протокол, подключайте эту рабочую нагрузку через интерфейс HTTP — для `clickhouse-go` (v2) установите `Protocol: clickhouse.HTTP` в параметрах подключения.
</Warning>

<div id="http-based-routing">
  ## Маршрутизация на основе HTTP (`session_id`)
</div>

Чтобы закрепить рабочую нагрузку за репликой, задайте параметр запроса `session_id` в интерфейсе HTTPS. Прокси разбирает запрос и использует согласованное хеширование по `session_id` для выбора реплики, поэтому все запросы с одинаковым `session_id` направляются на один и тот же сервер — пока не изменится топология кластера.

```bash
echo 'SELECT hostName()' | curl \
  -H 'X-ClickHouse-User: default' -H 'X-ClickHouse-Key: <password>' \
  'https://<host>:8443/?session_id=my-workload-1' -d @-
```

Каждый запрос с `session_id=my-workload-1` попадает на одну и ту же реплику. Другое значение `session_id` хэшируется независимо и может попасть на ту же или на другую реплику — соответствие остаётся постоянным, но выбрать, *на какую именно* реплику попадёт конкретное значение, нельзя.

<div id="subdomain-based-routing-deprecated">
  ## Маршрутизация на основе поддоменов (устарело)
</div>

<Danger>
  **Устарело**

  Описанный ниже механизм на основе поддоменов **устарел** и больше не включён для новых сервисов. Он не масштабируется (для каждой sticky-конечной точки требуется собственный TLS-сертификат). Вместо него используйте [HTTP-метод `session_id`](#http-based-routing).
</Danger>

Ранее включение маршрутизации с учётом реплик позволяло использовать подстановочный поддомен для имени хоста сервиса. Для сервиса с именем хоста `abcxyz123.us-west-2.aws.clickhouse.cloud` любое имя хоста, соответствующее шаблону `*.sticky.abcxyz123.us-west-2.aws.clickhouse.cloud` (например, `aaa.sticky.abcxyz123.us-west-2.aws.clickhouse.cloud`), Envoy по хешу направлял на одну и ту же реплику. Исходное имя хоста по-прежнему использовало балансировку нагрузки `LEAST_CONNECTION` — алгоритм маршрутизации по умолчанию.

<div id="limitations-of-replica-aware-routing">
  ## Ограничения маршрутизации с учетом реплик
</div>

<div id="replica-aware-routing-does-not-guarantee-isolation">
  ### Маршрутизация с учетом реплик не гарантирует изоляцию
</div>

Любое нарушение в работе сервиса — перезапуск серверных подов (обновление версии, сбой, вертикальное масштабирование) или горизонтальное масштабирование в сторону увеличения или уменьшения числа экземпляров — изменяет хеш-кольцо маршрутизации. Из-за этого запросы с одним и тем же `session_id` могут направляться в другой серверный под.

<div id="replica-aware-routing-does-not-work-out-of-the-box-with-private-link">
  ### Маршрутизация с учетом реплик не работает «из коробки» с Private Link
</div>

При использовании устаревшего метода с поддоменом клиентам нужно вручную добавить DNS-запись, чтобы для нового шаблона имен хостов работало разрешение имен. При неправильном использовании это может привести к неравномерной нагрузке на серверы.

<div id="replica-aware-routing-requires-http">
  ### Для маршрутизации с учетом реплик требуется HTTP-протокол
</div>

Липкая маршрутизация использует параметр запроса `session_id`, который существует только в HTTP/HTTPS-интерфейсе. Собственный бинарный протокол не передает такой параметр, по которому прокси мог бы вычислить хеш, поэтому маршрутизация с учетом реплик недоступна через собственный протокол. Чтобы использовать эту возможность, клиентам собственного протокола необходимо перенести соответствующую рабочую нагрузку на HTTP-интерфейс.
