  ## FAQ
</div>

<div id="compute">
  ### Вычислительные ресурсы
</div>

<Accordion title="Могу ли я создать несколько сервисов в одном кластере EKS?">
  Да. Инфраструктуру нужно развернуть только один раз для каждой комбинации аккаунта AWS и региона.
</Accordion>

<Accordion title="Какие регионы поддерживаются для BYOC?">
  Для развертываний BYOC доступны все **публичные регионы**, перечисленные в нашей документации о [поддерживаемых регионах](/ru/products/cloud/reference/supported-regions).
</Accordion>

<Accordion title="Будут ли дополнительные накладные расходы по ресурсам? Какие ресурсы нужны для запуска сервисов помимо инстансов ClickHouse?">
  Помимо самих инстансов ClickHouse (серверов ClickHouse и ClickHouse Keeper), мы также запускаем вспомогательные сервисы, такие как `clickhouse-operator`, `aws-cluster-autoscaler`, Istio и стек мониторинга.

  Потребление ресурсов этими общими компонентами относительно стабильно и не растет линейно с увеличением числа или размера ваших сервисов ClickHouse. В качестве ориентировочной оценки в AWS мы обычно используем выделенную группу узлов примерно из четырех экземпляров EC2 `4xlarge` для этих рабочих нагрузок.
</Accordion>

<div id="network-and-security">
  ### Сеть и безопасность
</div>

<Accordion title="Можно ли отозвать разрешения, настроенные во время установки, после завершения настройки?">
  В настоящее время это невозможно.
</Accordion>

<Accordion title="Рассматривали ли вы в будущем дополнительные меры безопасности, чтобы инженеры ClickHouse могли получать доступ к инфраструктуре клиента для устранения неполадок?">
  Да. Реализация контролируемого клиентом механизма, при котором клиенты смогут одобрять доступ инженеров к кластеру, есть в нашей дорожной карте. В настоящий момент инженеры должны пройти через наш внутренний процесс эскалации, чтобы получить оперативный доступ к кластеру. Это фиксируется в журналах и проверяется нашей командой безопасности.
</Accordion>

<Accordion title="Каков размер создаваемого диапазона IP-адресов VPC?">
  По умолчанию мы используем `10.0.0.0/16` для BYOC VPC. Мы рекомендуем зарезервировать как минимум /22 для возможного будущего масштабирования,
  но если вы предпочитаете ограничить размер, можно использовать /23, если предполагается, что вы будете ограничены
  30 серверными подами.
</Accordion>

<Accordion title="Могу ли я определить частоту технического обслуживания?">
  Обратитесь в службу поддержки, чтобы запланировать окна технического обслуживания. Имейте в виду, что минимальная частота обновлений — раз в неделю.
</Accordion>

<Accordion title="Как работает сетевое взаимодействие между BYOC VPC и S3?">
  Трафик между вашей клиентской BYOC VPC и S3 использует HTTPS (порт 443) через API AWS S3 для табличных данных, резервных копий и журналов. При использовании конечных точек S3 VPC этот трафик остается внутри сети AWS и не проходит через публичный интернет.
</Accordion>

<Accordion title="Какие порты используются для внутреннего взаимодействия в кластере ClickHouse?">
  Для внутреннего взаимодействия в кластере ClickHouse внутри клиентской BYOC VPC используются:

  * Нативный протокол ClickHouse на порту 9000
  * HTTP/HTTPS на портах 8123/8443
  * Межсерверное взаимодействие на порту 9009 для репликации и распределенных запросов
</Accordion>

<div id="uptime-sla">
  ### SLA по доступности
</div>

<Accordion title="Предоставляет ли ClickHouse SLA по доступности для BYOC?">
  Нет, поскольку плоскость данных размещается в облачной среде клиента, доступность сервиса зависит от ресурсов, которые находятся вне контроля ClickHouse. Поэтому ClickHouse не предоставляет формальный SLA по доступности для развертываний BYOC. Если у вас есть дополнительные вопросы, пожалуйста, обратитесь в службу поддержки: support@clickhouse.com.
</Accordion>
