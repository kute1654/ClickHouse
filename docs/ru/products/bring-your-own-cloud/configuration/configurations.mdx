На этой странице описаны различные параметры конфигурации инфраструктуры, доступные для вашего развертывания BYOC. Эти конфигурации позволяют настраивать сетевые параметры, безопасность и вычислительные ресурсы в соответствии с вашими требованиями.

<div id="load-balancers">
  ## Балансировщики нагрузки
</div>

В развертываниях BYOC используются **сетевые балансировщики нагрузки (NLB)** для управления и маршрутизации трафика к вашим сервисам ClickHouse. В зависимости от вашей сетевой модели вы можете использовать *публичные* или *частные* конечные точки балансировщика нагрузки.

| Тип балансировщика нагрузки | Выделенный VPC под управлением ClickHouse | VPC под управлением клиента |
| --------------------------- | :---------------------------------------: | :-------------------------: |
| **Публичный NLB**           |            Включен по умолчанию           |    Отключен по умолчанию    |
| **Частный NLB**             |           Отключен по умолчанию           |     Включен по умолчанию    |

**Публичный балансировщик нагрузки:**

* Предоставляет публичный доступ к вашим сервисам ClickHouse через интернет.
* Обычно включен по умолчанию при использовании выделенного VPC под управлением ClickHouse.
* По умолчанию отключен при использовании VPC под управлением клиента для повышения безопасности.

**Частный балансировщик нагрузки:**

* Предоставляет частный (внутренний) доступ, доступный только из подключенных к вам сетей.
* Обычно включен по умолчанию при использовании VPC под управлением клиента.
* По умолчанию отключен при использовании выделенного VPC под управлением ClickHouse.

Вы можете обратиться в **ClickHouse Cloud Support**, чтобы настроить, какие конечные точки будут включены в соответствии с вашими требованиями.

<div id="private-load-balancer-security-group">
  ### Группа безопасности для приватного балансировщика нагрузки в AWS
</div>

Если вы используете приватный балансировщик нагрузки для развертывания BYOC, убедитесь, что заданы соответствующие правила группы безопасности, разрешающие доступ из нужных вам частных сетей (например, из VPC, связанных через пиринг). По умолчанию группа безопасности разрешает трафик только внутри VPC.

Чтобы настроить группу безопасности для приватного балансировщика нагрузки:

**Обратитесь в ClickHouse Support**, чтобы запросить изменения правил входящего трафика в группе безопасности, разрешающие трафик из ваших сетей-источников:

* **Пиринг VPC**: Запросите правила, разрешающие трафик из CIDR-диапазонов ваших VPC, связанных через пиринг.
* **PrivateLink**: Изменения группы безопасности не требуются, так как трафик не регулируется группой безопасности балансировщика нагрузки.
* **Другие сетевые конфигурации**: Опишите свой сценарий, чтобы служба поддержки могла помочь соответствующим образом.

<Note>
  Любые изменения групп безопасности приватного балансировщика нагрузки должны выполняться ClickHouse Support. Это обеспечивает согласованность конфигурации и помогает избежать конфликтов в среде, управляемой ClickHouse Cloud.
</Note>

<div id="privatelink-or-private-service-connect">
  ## PrivateLink или Private Service Connect
</div>

Для максимальной сетевой изоляции и безопасности в развертываниях BYOC можно использовать **AWS PrivateLink** или **GCP Private Service Connect**. Эти варианты позволяют приложениям подключаться к сервисам ClickHouse Cloud по частной сети без необходимости настраивать пиринг VPC или открывать конечные точки в публичный интернет.

Пошаговые инструкции по настройке см. в [руководстве по настройке частной сети](/ru/products/bring-your-own-cloud/onboarding/network).

<div id="k8s-api-private-connection">
  ## Закрытое подключение к Kubernetes API
</div>

По умолчанию конечная точка API-сервера Kubernetes для вашего BYOC-кластера доступна из публичного интернета, но доступ к ней ограничен IP-фильтрацией: разрешены только IP-адреса NAT gateway ClickHouse. Для более высокого уровня безопасности можно ограничить доступ к API-серверу Kubernetes так, чтобы он был доступен только через частные сетевые подключения с использованием Tailscale.

<Note>
  Если для закрытого подключения вы используете только Tailscale, есть риск, что ClickHouse Support потеряет доступ к вашей среде, если агент Tailscale станет недоступен. Это может замедлить устранение неполадок или увеличить время ответа службы поддержки.
</Note>

**Обратитесь в ClickHouse Support**, чтобы запросить настройку закрытой конечной точки API.

<div id="node-groups">
  ## Группы узлов
</div>

Группы узлов Kubernetes — это наборы вычислительных экземпляров, которые предоставляют ресурсы, необходимые для работы сервисов ClickHouse в развертывании BYOC. ClickHouse Cloud управляет этими группами узлов, автоматически настраивая их и масштабируя.

<div id="default-configuration">
  ### Конфигурация по умолчанию
</div>

Кластеры BYOC разворачиваются с двумя основными типами групп узлов:

* **Системная группа узлов**
  На них размещаются критически важные системные рабочие нагрузки — такие как ClickHouse Operator, Istio (для service mesh), компоненты мониторинга (Prometheus, Grafana, AlertManager), cluster autoscaler и другие базовые сервисы. Эти узлы обычно используют стандартные типы x86-инстансов.

* **Группы узлов для рабочих нагрузок**
  Они предназначены для рабочих нагрузок ClickHouse, связанных с данными, включая серверы и сервисы Keeper. По умолчанию узлы рабочих нагрузок работают на ARM-инстансах, обеспечивая оптимальный баланс производительности и стоимости. Однако при необходимости их можно настроить с альтернативными профилями CPU/памяти или переключить на архитектуру x86.

<div id="customizing-node-groups">
  ### Настройка групп узлов
</div>

Нужны специализированные ресурсы или архитектуры? Доступны следующие варианты настройки — свяжитесь с ClickHouse Support, чтобы обсудить и реализовать их:

* **Выбор типа инстанса**
  Выберите конкретные типы инстансов в соответствии с такими требованиями, как производительность, соответствие нормативным требованиям, увеличенный объём памяти/CPU или использование зарезервированных ресурсов.
* **Соотношение CPU/памяти**
  При необходимости скорректируйте вычислительный профиль для групп узлов для рабочих нагрузок по мере необходимости.
* **Архитектура**
  При необходимости переключите группы узлов для рабочих нагрузок с ARM на x86.

> **Примечание:** Spot (вытесняемые) инстансы *не* поддерживаются; по умолчанию все группы узлов BYOC работают на инстансах on-demand.

<Note>
  Все изменения настроек и конфигурации групп узлов должны согласовываться через ClickHouse Support. Это обеспечивает совместимость, стабильность и оптимальную производительность.
</Note>

<div id="auto-scaling">
  ### Автоматическое масштабирование
</div>

Группы узлов кластера масштабируются автоматически с помощью Cluster Autoscaler в зависимости от следующих факторов:

* Запросы и лимиты ресурсов подов
* Общая емкость кластера и уровень его загрузки
* Потребности сервиса ClickHouse в масштабировании

Ручное вмешательство не требуется. ClickHouse Cloud берет на себя текущее управление ресурсами и масштабированием вашего развертывания.
