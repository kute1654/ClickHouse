Настройте ресурсы CPU и памяти для кластера ClickHouse с помощью Private API.

<div id="prerequisites">
  ## Предварительные требования
</div>

* ClickHouse Private API установлен и доступен (например, через проброс порта на `http://localhost:8080/`)
* Развернут как минимум один кластер ClickHouse (в этом руководстве используется `default-xx-01`)

<Steps>
  <Step>
    ## 1. Создайте запрос на масштабирование

    Создайте JSON‑файл с нужным распределением ресурсов:

    ```json
    {
      "vertical": {
        "resources": {
          "cpu": "4",
          "memory": "16Gi"
        }
      }
    }
    ```

    Сохраните его как `scale.json`.
  </Step>

  <Step>
    ## 2. Отправьте запрос

    ```bash
    curl -X POST "http://localhost:8080/api/v1/instances/default-xx-01/scale" \
        -H "Content-Type: application/json" \
        --data-binary "@scale.json"
    ```
  </Step>

  <Step>
    ## 3. Контроль соотношения CPU и памяти

    API поддерживает **соотношение CPU к памяти 1:4** (1 ядро CPU на 4 GiB памяти) с допуском 5%.

    Если указать только один тип ресурса, API автоматически вычислит второй. Например, если указать только `"memory": "16Gi"`, значение CPU будет установлено в 4 ядра.

    **Допустимые форматы объема ресурсов:**

    | Тип    | Примеры                                                           |
    | ------ | ----------------------------------------------------------------- |
    | CPU    | `"4"` (ядра), `"2.5"` (десятичное значение), `"500m"` (миллиядер) |
    | Память | `"16Gi"` (гибибайты), `"4096Mi"` (мебибайты)                      |

    Для памяти рекомендуется использовать двоичные единицы (Ki, Mi, Gi).
  </Step>

  <Step>
    ## 4. Отслеживание хода масштабирования

    Проверьте статус кластера, чтобы отслеживать выполнение масштабирования:

    ```bash
    curl "http://localhost:8080/api/v1/instances/default-xx-01/status"
    ```

    Ответ содержит: - `state`: текущее состояние кластера - `previousState`: состояние до перехода - `message`: подробности о смене состояния

    Масштабирование завершится, когда кластер вернется в состояние running.
  </Step>
</Steps>

<div id="rolling-restart-behavior">
  ## Поведение при поочередном перезапуске
</div>

API обновляет `ServerPodPolicy` пользовательского ресурса `ClickhouseCluster`. Затем оператор выполняет поочередный перезапуск StatefulSets с обновленным выделением ресурсов.

> **Важно:** В отличие от ClickHouse Cloud, ClickHouse Private не использует [масштабирование Make Before Break (MBB)](https://clickhouse.com/docs/manage/scaling#how-scaling-works-in-clickhouse-cloud). Вертикальное масштабирование может вызвать кратковременные перебои в работе сервиса во время поочередного перезапуска.
