В этом документе объясняется, что означает соответствие требованиям FIPS 140-2 в контексте ClickHouse Private и что меняется при развертывании в FIPS-совместимой конфигурации.

<div id="what-fips-140-2-means-for-clickhouse-private">
  ## Что FIPS 140-2 означает для ClickHouse Private
</div>

FIPS 140-2 (Federal Information Processing Standard) — это федеральный стандарт США для криптографических модулей. Организации, работающие в государственных средах (например, AWS GovCloud) или обрабатывающие чувствительные государственные данные, часто обязаны обеспечивать использование во всех криптографических операциях алгоритмов и реализаций, прошедших валидацию FIPS.

Когда ClickHouse Private развернут в режиме FIPS, все криптографические операции -- включая TLS‑соединения между компонентами, соединения с сервисами AWS и генерацию сертификатов -- должны использовать FIPS‑совместимые алгоритмы и библиотеки.

<div id="what-changes-in-a-fips-deployment">
  ## Что меняется при развертывании FIPS
</div>

Развертывание FIPS отличается от стандартного развертывания ClickHouse Private в нескольких ключевых аспектах:

<div id="fips-server-images">
  ### FIPS-образы сервера
</div>

Для образов ClickHouse server и Keeper необходимо использовать FIPS-варианты. Они обозначаются суффиксом `-fips` в теге образа (например, `25.12.1.1497-fips` для сервера и `26.2.1.258-fips` для Keeper). FIPS-образы собираются с использованием криптографических библиотек, сертифицированных по FIPS. Для образа оператора отдельный FIPS-вариант не предусмотрен.

<div id="fips-ecr-endpoints">
  ### Конечные точки ECR с поддержкой FIPS
</div>

При загрузке образов контейнеров из Amazon ECR необходимо использовать FIPS-совместимые конечные точки ECR. Эти конечные точки имеют вид `0000000000.dkr-ecr-fips.<region>.on.aws` вместо стандартного имени хоста ECR.

<div id="fips-s3-endpoints">
  ### Конечные точки S3 для FIPS
</div>

ClickHouse хранит данные в S3, и в режиме FIPS необходимо настроить конечные точки S3, совместимые с FIPS. AWS публикует список FIPS конечных точек S3 для каждого сервиса и региона по адресу [https://aws.amazon.com/compliance/fips/](https://aws.amazon.com/compliance/fips/).

<div id="s3-bucket-naming-restriction">
  ### Ограничение на имя S3 бакета
</div>

В режиме FIPS **имя S3 бакета не должно содержать точек** (`.`). Это ограничение связано с тем, что конечные точки S3 для FIPS требуют адресации в стиле virtual-host (например, `https://my-bucket.s3-fips.us-gov-west-1.amazonaws.com`). Если имя бакета содержит точки, это нарушает проверку TLS-сертификата для URL-адресов в стиле virtual-host, поскольку точки создают дополнительные уровни поддоменов, которые не соответствуют wildcard-сертификату.

Это важно учитывать при настройке инфраструктуры S3. Если у вас уже есть бакеты с точками в именах, их нельзя использовать с конечными точками S3 для FIPS.

<div id="certificate-requirements">
  ## Требования к сертификатам
</div>

Для FIPS-совместимого развертывания требуются TLS-сертификаты для зашифрованного взаимодействия между всеми компонентами ClickHouse. Необходимо сгенерировать три набора сертификатов:

| Набор сертификатов | Назначение                                                                                     | Файлы                      |
| ------------------ | ---------------------------------------------------------------------------------------------- | -------------------------- |
| **Server**         | Защищает входящие соединения к clickhouse-server                                               | `server.crt`, `server.key` |
| **Keeper**         | Защищает соединения с clickhouse-keeper                                                        | `keeper.crt`, `keeper.key` |
| **Client**         | Используется clickhouse-server для исходящих соединений (например, с keeper и HTTPS-словарями) | `client.crt`, `client.key` |

Все три набора подписываются единым центром сертификации (`ca.crt`).

<div id="fips-compliant-algorithm-requirements">
  ### Требования к FIPS-совместимым алгоритмам
</div>

Если требуется соответствие стандарту FIPS, все сертификаты должны быть:

* **Сгенерированы с использованием FIPS-совместимых алгоритмов** -- Например, RSA с длиной ключа не менее 3072 бит.
* **Созданы в среде с включенным FIPS** -- Криптографические операции при генерации ключей и подписывании должны выполняться с использованием модуля, валидированного по FIPS. Один из возможных подходов — использовать контейнер Red Hat UBI с включенным режимом FIPS.
* **Подписаны центром сертификации**, собственный ключ которого был сгенерирован FIPS-совместимым способом.

<div id="subject-alternative-names-sans">
  ### Альтернативные имена субъекта (SAN)
</div>

Каждый сертификат должен содержать корректные записи SAN, чтобы проверка TLS успешно проходила при межкомпонентном взаимодействии внутри кластера Kubernetes. Формат SAN соответствует следующему шаблону:

* Сервер: `*.c-<cluster-name>-server-headless.ns-<cluster-name>.svc.cluster.local` и `c-<cluster-name>-server-any.ns-<cluster-name>.svc.cluster.local`
* Keeper: `*.c-<cluster-name>-keeper-headless.ns-<cluster-name>.svc.cluster.local`

Для каждого развёртываемого кластера ClickHouse необходимо создать уникальный набор сертификатов (за исключением CA‑сертификата).

<div id="certificate-expiration">
  ### Срок действия сертификата
</div>

У сертификатов есть ограниченный срок действия. Следует наладить процесс контроля срока действия сертификатов и их ротации до истечения. Часто по умолчанию используется срок 365 дней, однако его следует скорректировать в соответствии с требованиями безопасности вашей организации.
