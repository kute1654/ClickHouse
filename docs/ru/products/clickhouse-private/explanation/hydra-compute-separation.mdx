Этот документ посвящён Hydra — функции разделения вычислительных ресурсов для кластеров ClickHouse Private.

<div id="what-is-hydra">
  ## Что такое Hydra?
</div>

Hydra позволяет создавать **несколько групп узлов в рамках одного логического кластера ClickHouse**, при этом каждая группа узлов может иметь:

* Разное количество узлов
* Разные ресурсы (CPU, память)
* Отдельную сетевую конечную точку

Это позволяет запускать разные рабочие нагрузки (например, ресурсоемкую аналитику и легковесные панели мониторинга) на одном и том же наборе данных, не создавая конкуренции за одни и те же вычислительные ресурсы.

<div id="how-it-works">
  ## Как это работает
</div>

Hydra обеспечивает разделение вычислительных ресурсов, сохраняя при этом единый слой данных:

* **Общий ClickHouse Keeper** -- Все группы узлов в кластере Hydra используют один и тот же ансамбль ClickHouse Keeper для распределённой координации. Отдельные экземпляры Keeper для каждой группы узлов не нужны.
* **Общие данные в S3** -- Все группы узлов читают данные из одного и того же префикса в S3 (и при необходимости записывают в него). Поскольку ClickHouse использует общее объектное хранилище для данных таблиц, несколько вычислительных групп могут работать с одними и теми же исходными данными без дублирования.
* **Раздельные вычислительные ресурсы** -- Каждая группа узлов запускает собственный набор подов сервера ClickHouse с независимыми конфигурациями ресурсов и собственной конечной точкой Kubernetes-сервиса.

<div id="parent-child-relationship">
  ## Связь «родитель — потомок»
</div>

Hydra использует **модель «родитель — потомок»**:

* **Родитель** — это существующий кластер ClickHouse, которому принадлежат данные и ансамбль Keeper.
* **Дочерние узлы** — это дополнительные группы узлов, которые разворачиваются с указанием имени и пространства имен родителя. При создании дочернего узла родитель указывается через `parentCluster.name` и `parentCluster.namespace`.

У одного родителя может быть несколько дочерних узлов, каждый со своим сайзингом и характеристиками доступа.

<div id="read-only-vs-read-write-children">
  ## Дочерние инстансы: только для чтения и для чтения/записи
</div>

По умолчанию дочерние инстансы работают в режиме **чтения и записи**, то есть могут и читать, и записывать данные в общее хранилище.

При необходимости дочерний инстанс можно настроить в режиме **только для чтения** (задав `isReadonly=true` при создании). Дочерние инстансы только для чтения могут выполнять запросы к данным, но не могут их изменять, поэтому они подходят для таких рабочих нагрузок, как отчётность, панели мониторинга или аналитика, где доступ на запись не требуется и желательна изоляция от операций записи.

<div id="limitations">
  ## Ограничения
</div>

При использовании Hydra нужно учитывать два важных ограничения:

1. **Нельзя создавать дочерние инстансы, если родительский инстанс остановлен, завершён или переведён в состояние простоя.** Для создания дочерних инстансов родительский инстанс должен находиться в рабочем состоянии, поскольку дочерние инстансы зависят от ансамбля Keeper родительского инстанса и конфигурации общих данных.

2. **Перед удалением родительского кластера необходимо удалить все дочерние инстансы.** Родительский кластер нельзя удалить, пока на него всё ещё ссылаются дочерние инстансы. Попытка удалить родительский кластер при наличии активных дочерних инстансов завершится ошибкой.
