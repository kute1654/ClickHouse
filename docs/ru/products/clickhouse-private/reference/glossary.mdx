<div id="clickhouse-components">
  ## Компоненты ClickHouse
</div>

**ClickHouse Server**
Основной процесс базы данных ClickHouse. В ClickHouse Private каждая реплика сервера работает как под, управляемый собственным StatefulSet. См. [официальную документацию](https://clickhouse.com/docs/operations/server-configuration-parameters/settings).

**ClickHouse Keeper**
Распределённый сервис координации для метаданных репликации. Для HA требуется минимум 3 экземпляра. См. [официальную документацию](https://clickhouse.com/docs/operations/clickhouse-keeper).

**ClickHouse Operator**
Оператор Kubernetes, который управляет жизненным циклом пользовательских ресурсов ClickHouseCluster, включая развёртывание, масштабирование, резервное копирование и реконсиляцию.

**ClickHouseCluster (CR)**
Пользовательский ресурс Kubernetes, который определяет желаемое состояние кластера ClickHouse — реплики, хранилище, сетевые настройки и конфигурацию. Создаётся через Helm-чарт `onprem-clickhouse-cluster`.

**Airgap Management API**
Необязательный компонент REST API для управления кластерами ClickHouse Private. Предоставляет конечные точки для резервного копирования, вертикального масштабирования, сброса паролей и мониторинга состояния.

<div id="architecture-concepts">
  ## Концепции архитектуры
</div>

**MultiStatefulSet (MultiSTS)**
Модель StatefulSet оператора, в которой каждая реплика ClickHouse server управляется собственным StatefulSet (1 STS = 1 под). Это обеспечивает более безопасный rolling update и независимое управление репликами. MultiSTS заменил устаревшую модель SingleSTS, которая помечена как Deprecated.

**ReplicaStateMap**
Внутренняя структура данных в status ресурса ClickHouseCluster CR, которая отслеживает состояние каждой реплики в режиме MultiSTS. Состояния включают **Pending**, **Ready**, **Stopped**, **Condemned**, **Dropped**, **VolumeRestoring** и **VolumeRestored**. Это предназначено только для внутреннего использования оператором и не должно использоваться внешними системами.

**Condemned Replica**
Реплика, помеченная на удаление при операциях уменьшения масштаба. Оператор переводит её из состояния **Ready** в **Condemned** в ReplicaStateMap, а затем безопасно удаляет после синхронизации данных.

**Hydra (Compute Separation)**
Возможность, которая позволяет использовать несколько профилей compute (групп узлов) в рамках одного кластера ClickHouse. Каждый профиль может иметь разные выделения ресурсов и отдельные конечные точки, совместно используя один инстанс Keeper и данные в S3. Child instances могут быть только для чтения или с возможностью чтения и записи.

**Backup Pod**
В режиме MultiSTS одна реплика назначается подом для резервного копирования (`isBackupPod: true` в ReplicaStateMap). Резервные копии всегда выполняются на этом поде, и он никогда не помечается на удаление при уменьшении масштаба.

<div id="infrastructure-terms">
  ## Термины инфраструктуры
</div>

**IRSA (роли IAM для сервисных аккаунтов)**
Механизм AWS, который позволяет сервисным аккаунтам Kubernetes принимать роли IAM. Используется для предоставления подам ClickHouse доступа к S3 без статических учетных данных.

**Workload Identity**
Эквивалент IRSA в GCP. Связывает сервисный аккаунт Kubernetes с сервисным аккаунтом GCP для доступа к GCS и другим сервисам GCP.

**FIPS 140-2**
Государственный стандарт США для криптографических модулей. ClickHouse Private поддерживает FIPS-совместимые развертывания с использованием специально собранных образов сервера и генерации TLS-сертификатов, валидированных по FIPS.

**Кэш NVMe**
Локальное хранилище NVMe SSD, подключенное к узлам сервера и используемое как дисковый кэш для данных, хранящихся в S3/GCS. Повышает производительность запросов за счет локального кэширования часто используемых данных. Настраивается через `bytesPerGiRAM` или `cacheDiskSize` в Helm-чарте.

**Префикс ключа S3**
Уникальный идентификатор (обычно UUID), который задает пространство имен для данных кластера в S3 бакете. Несколько кластеров могут использовать один и тот же бакет, применяя разные префиксы ключей.

<div id="operational-terms">
  ## Термины эксплуатации
</div>

**Реконсиляция**
Процесс, в ходе которого оператор сравнивает желаемое состояние (CR spec) с фактическим состоянием (запущенными подами, StatefulSet&#39;ами) и вносит необходимые исправления. Отслеживается по метрике `last_cluster_reconcile`.

**Аннотация Skip-Reconcile**
Аннотация `clickhouse.com/skip-reconcile` в CR ClickHouseCluster. Если она указана, оператор не будет выполнять реконсиляцию этого кластера. Используется при ручных операциях, например при удалении реплики.

**Проверки Preflight**
Инструмент проверки (на базе Troubleshoot.sh), который перед развертыванием ClickHouse проверяет предварительные требования кластера: метки узлов, классы хранилища, CRD и другое.

**PDB (PodDisruptionBudget)**
Ресурс Kubernetes, который ограничивает количество подов, одновременно недоступных в системе. Во время rolling update оператор учитывает PDB, чтобы сохранить доступность.
