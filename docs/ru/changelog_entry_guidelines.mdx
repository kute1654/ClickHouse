<div id="changelog-entry-guidelines">
  # Рекомендации по составлению записей для журнала изменений
</div>

Хорошие записи для журнала изменений помогают пользователям быстро понять, что нового появилось и как это на них повлияет. Мы просим участников проекта добавлять понятную пользователю запись, которая будет включена в журнал изменений каждого релиза.

Ниже приведено несколько простых рекомендаций о том, как составить хорошую запись для журнала изменений.

<div id="write-with-the-user-in-mind-not-the-developer">
  ## Пишите с учетом пользователя, а не разработчика
</div>

Запись в журнале изменений предназначена для того, чтобы донести суть изменения до *пользователя*, а не только до *разработчика*.
При составлении записи, где это уместно, старайтесь объяснить не только, ***что*** изменилось, но и ***почему*** это полезно пользователю, или ***как*** это на него влияет.

Например, вместо:

> Добавлена таблица `system.iceberg_history`

Напишите:

> Теперь пользователи могут просматривать исторические снимки таблиц Iceberg с помощью новой таблицы `system.iceberg_history`.

Вместо:

> Добавлены функции `stringBytesUniq` и `stringBytesEntropy` для поиска потенциально случайных или зашифрованных данных.&quot;

Напишите:

> Теперь вы можете выявлять потенциально зашифрованные или случайные данные в строках с помощью новых функций `stringBytesUniq` и `stringBytesEntropy`, что помогает находить проблемы с качеством данных или риски безопасности."

<div id="keep-it-simple">
  ## Не усложняйте
</div>

Избегайте технического жаргона, который пользователь может не понять без пояснений. Старайтесь укладываться в 1–5 предложений и
не бойтесь использовать LLM, чтобы находить опечатки, грамматические ошибки или переформулировать текст пункта в
понятной для пользователя форме (это не жульничество, обещаю!)

Вместо:

> Поддержка коррелированных подзапросов в качестве аргумента выражения `EXISTS`

Напишите:

> Теперь можно использовать подзапросы, которые ссылаются на столбцы внешнего запроса, в конструкциях `EXISTS`.

Пример ясной и простой записи:

> Позволяет настраивать параметры кэша страниц для каждого запроса. Это нужно для более быстрых экспериментов и более тонкой настройки запросов с высокой пропускной способностью и низкой задержкой.

<div id="follow-a-few-simple-formatting-guidelines">
  ## Соблюдайте несколько простых правил оформления
</div>

<div id="write-in-full-sentences-and-in-the-present-tense">
  ### Пишите полными предложениями и в настоящем времени
</div>

Вместо:

> Исправлен сбой: при попытке удалить временный файл генерируется исключение

Пишите:

> Устраняет сбой, при котором при попытке удалить временный файл генерируется исключение.

<div id="use-backticks-where-necessary">
  ### Используйте обратные кавычки там, где это необходимо
</div>

Заключайте в обратные кавычки такие элементы кода, как настройки, имена функций, SQL-операторы, названия форматов и типы данных. Как правило,
всё, что вы вводите в clickhouse-client, следует заключать в обратные кавычки. Это делает записи в журнале изменений более читаемыми.

Вместо:

> Settings use&#95;skip&#95;indexes&#95;if&#95;final and use&#95;skip&#95;indexes&#95;if&#95;final&#95;exact&#95;mode now default to True

Пишите:

> Settings `use_skip_indexes_if_final` and `use_skip_indexes_if_final_exact_mode` now default to `True`"

<div id="try-to-follow-a-consistent-format">
  ### Старайтесь придерживаться единого формата
</div>

Старайтесь использовать один и тот же формат: что делает → почему это важно для пользователя → как это использовать (если нужно). Так читателям будет проще быстро просматривать записи, а их структура станет более предсказуемой.

Например:

> Теперь вы можете фильтровать результаты векторного поиска либо до, либо после выполнения поиска, что дает больше контроля над компромиссом между производительностью и точностью. Используйте новую настройку `vector_search_filter_mode`, чтобы выбрать подходящий вариант.
