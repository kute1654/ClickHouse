<div id="answer">
  ## Ответ
</div>

Таблица `query_log` в базе данных `system` хранит информацию обо всех ваших запросах, включая:

* сколько памяти потребил запрос, и
* сколько процессорного времени потребовалось

Следующий запрос возвращает 10 запросов, использовавших больше всего памяти:

```sql
SELECT
    type,
    event_time,
    initial_query_id,
    query_id,
    formatReadableSize(memory_usage) AS memory,
    ProfileEvents.Values[indexOf(ProfileEvents.Names, 'UserTimeMicroseconds')] AS userCPU,
    ProfileEvents.Values[indexOf(ProfileEvents.Names, 'SystemTimeMicroseconds')] AS systemCPU,
    normalizedQueryHash(query) AS normalized_query_hash
FROM clusterAllReplicas(default, system.query_log)
ORDER BY memory_usage DESC
LIMIT 10;
```

<Note>
  `initial_query_id` — это идентификатор исходного запроса при выполнении распределённого запроса, запущенного с узла, принимающего запрос. `query_id` содержит идентификатор дочернего запроса, который выполняется на другом узле. Подробнее см. [в этой статье](/ru/resources/support-center/knowledge-base/performance-optimization/find-expensive-queries#initial_query_id-vs-query_id).

Идентификатор запроса можно использовать, чтобы получить о нём более подробные сведения. Давайте рассмотрим самый долгий запрос из приведённых выше (первый):

```sql
SELECT query
FROM clusterAllReplicas(default, system.query_log)
WHERE initial_query_id = '7fc488a5-838f-410d-88ee-2f492825a26b'
```

Как оказалось, это тот самый запрос, который мы использовали для вставки нескольких миллиардов строк в таблицу `youtube` (см. [набор данных о дизлайках YouTube](/ru/get-started/sample-datasets/youtube-dislikes)):

<div id="initial_query_id-vs-query_id">
  ## initial_query_id VS query_id
</div>

Обратите внимание: в кластерной среде ClickHouse (например, в ClickHouse Cloud) `initial_query_id` — это идентификатор исходного запроса для выполнения распределённого запроса, запущенного с узла, принимающего запрос;
а поле `query_id` содержит идентификатор дочернего запроса, выполняемого на другом узле.

<Info>
  **Распределённое выполнение — не единственная причина различий в значениях `query_id`**

  Один запрос также может порождать внутренние подзапросы, которые записываются со своими собственными `query_id` и `is_initial_query = 0` — чаще всего это внутренние запросы, которые ClickHouse выполняет для обработки [представлений](/ru/reference/system-tables/query_views_log). Эти внутренние идентификаторы назначаются ClickHouse и могут не совпадать по формату с идентификатором, который передаётся вашему клиенту; на этапах, связанных с представлениями, вы можете увидеть значения `query_id`, содержащие метку вроде `queryView...`. Если вам нужен запрос именно в том виде, в котором вы его отправили, фильтруйте по `is_initial_query = 1` (или `query_id = initial_query_id`). Это серверное поведение `query_log`, и оно одинаково для любого клиента, включая Cloud SQL Console.
</Info>

Если добавить `query_id` в приведённый выше запрос, поиск будет ограничен значениями `initial_query_id` = `a7262fa2-bd8b-4b51-a359-621ccf282417` и `hostname()`:

```sql
SELECT
    hostname(),
    type,
    event_time,
    initial_query_id,
    query_id,
    formatReadableSize(memory_usage) AS memory,
    ProfileEvents.Values[indexOf(ProfileEvents.Names, 'UserTimeMicroseconds')] AS userCPU,
    ProfileEvents.Values[indexOf(ProfileEvents.Names, 'SystemTimeMicroseconds')] AS systemCPU,
    normalizedQueryHash(query) AS normalized_query_hash
FROM clusterAllReplicas(default, system.query_log)
WHERE initial_query_id = 'a7262fa2-bd8b-4b51-a359-621ccf282417'
ORDER BY memory_usage DESC
LIMIT 10 FORMAT Pretty;
```

Чтобы сузить выборку и получить только дочерние запросы, мы также можем добавить условие `query_id != initial_query_id` в предложение WHERE:

```sql
SELECT
    hostname(),
    type,
    event_time,
    initial_query_id,
    query_id,
    formatReadableSize(memory_usage) AS memory,
    ProfileEvents.Values[indexOf(ProfileEvents.Names, 'UserTimeMicroseconds')] AS userCPU,
    ProfileEvents.Values[indexOf(ProfileEvents.Names, 'SystemTimeMicroseconds')] AS systemCPU,
    normalizedQueryHash(query) AS normalized_query_hash
FROM clusterAllReplicas(default, system.query_log)
WHERE (query_id != initial_query_id) AND (initial_query_id = 'a7262fa2-bd8b-4b51-a359-621ccf282417')
ORDER BY memory_usage DESC
LIMIT 10 FORMAT Pretty;
```

возвращает все дочерние запросы, выполненные на удалённых узлах (то есть узлах, удалённых относительно узла, на котором запрос был впервые запущен):

```sql
SELECT
    hostname(),
    type,
    event_time,
    initial_query_id,
    query_id,
    formatReadableSize(memory_usage) AS memory,
    ProfileEvents.Values[indexOf(ProfileEvents.Names, 'UserTimeMicroseconds')] AS userCPU,
    ProfileEvents.Values[indexOf(ProfileEvents.Names, 'SystemTimeMicroseconds')] AS systemCPU,
    normalizedQueryHash(query) AS normalized_query_hash
FROM clusterAllReplicas(default, system.query_log)
WHERE (query_id = initial_query_id) AND (initial_query_id = 'a7262fa2-bd8b-4b51-a359-621ccf282417')
ORDER BY memory_usage DESC
LIMIT 10 FORMAT Pretty;
```
