<div id="ttl-rules-and-control">
  ## Правила TTL и управление
</div>

TTL будет применён ***не сразу***. Что это значит? Настройка таблицы `MergeTree` [`merge_with_ttl_timeout`](/ru/reference/engines/table-engines/mergetree-family/mergetree#merge_with_ttl_timeout) задаёт минимальную задержку в секундах перед повторным слиянием с delete TTL. Значение по умолчанию — 14400 секунд (4 часа). Но это лишь минимальная задержка: фактический запуск слияния для delete TTL может произойти позже.

Вы можете просмотреть все текущие настройки TTL (например, `merge_with_ttl_timeout`) с помощью этого запроса:

```sql
SELECT *
FROM system.merge_tree_settings
WHERE name like '%ttl%'
```

Ответ выглядит так:

```response
┌─name───────────────────────────────────────────────────────────┬─value───┬─changed─┬─description────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────┬─min──┬─max──┬─readonly─┬─type───┐
│ max_replicated_merges_with_ttl_in_queue                        │ 1       │       0 │ Сколько задач слияния частей с TTL разрешено выполнять одновременно в очереди ReplicatedMergeTree.                                                                                        │ ᴺᵁᴸᴸ │ ᴺᵁᴸᴸ │        0 │ UInt64 │
```

Вы можете использовать `SHOW CREATE TABLE`, чтобы проверить, содержит ли ваша таблица правила TTL, а также изменяли ли какие-либо `SETTINGS` таблицы значения указанных выше настроек:

```sql
SHOW CREATE TABLE <TableName>
```

<div id="force-a-ttl-rule-to-be-applied">
  ## Принудительно применить правило TTL
</div>

Это не самое изящное решение, но можно явно вызвать `MATERIALIZE TTL`, чтобы принудительно материализовать все правила TTL таблицы:

```sql
ALTER TABLE my_table
    MATERIALIZE TTL
```

<div id="background-threads-affecting-ttl">
  ## Фоновые потоки, влияющие на TTL
</div>

Возможно, ваши правила TTL не применяются из-за недостаточного числа рабочих потоков во фоновом пуле. Например, при интенсивной вставке данных весь фоновый пул может быть занят обычными слияниями. Однако размер фонового пула можно увеличить.

Проверить текущий размер фонового пула можно с помощью этого запроса:

```sql
SELECT *
FROM system.settings
WHERE name = 'background_pool_size';
```

Ответ будет выглядеть так:

```response
┌─name─────────────────┬─value─┬─changed─┬─description─────────────────────┬─min──┬─max──┬─readonly─┬─type───┬─default─┬─alias_for─┐
│ background_pool_size │ 16    │       0 │ Устаревшая настройка, не выполняет никаких действий. │ ᴺᵁᴸᴸ │ ᴺᵁᴸᴸ │        0 │ UInt64 │ 16      │           │
└──────────────────────┴───────┴─────────┴─────────────────────────────────┴──────┴──────┴──────────┴────────┴─────────┴───────────┘
```

О том, как изменить [параметр `background_pool_size`](/ru/reference/settings/server-settings/settings#background_pool_size), см. в документации; он задается следующим образом:

```xml
<background_pool_size>16</background_pool_size>
```

Текущую активность фонового пула можно проверить с помощью следующего запроса:

```sql
SELECT *
FROM system.metrics
WHERE metric like 'Background%'
```
