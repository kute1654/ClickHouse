<div id="december-19-2025">
  ## 2025 年 12 月 19 日
</div>

* AWS ap-south-1 现已支持创建符合 PCI 标准的服务。
* **统一用户身份私有预览**
  有意在控制台中管理数据库用户的客户，现在可以为 SQL 控制台启用一种新的身份验证方法。
  这样，客户就可以在我们继续将数据库用户管理功能引入控制台的同时，试用这种新的身份验证方法。
* **S3 ClickPipes 现已支持无序模式**：
  客户现在可以按任意顺序将数据从 Amazon S3 摄取到 ClickHouse Cloud 中，以支持事件驱动分析。
  文件不再需要按词典序排列即可被消费。更多详情请参阅这篇[公告博文](https://clickhouse.com/blog/clickpipes-s3-unordered-mode)。
* Fivetran 连接器最近已进入 Beta 阶段。如果您正在使用 Fivetran，并希望将 ClickHouse 设置为目标端，请查看这些[文档](https://fivetran.com/docs/destinations/clickhouse/setup-guide)。

<div id="december-12-2025">
  ## 2025 年 12 月 12 日
</div>

* **SAML 单点登录自助设置**

  Enterprise 客户现在无需提交支持工单，即可在控制台中完成 SAML 设置。
  此外，SAML 客户还可以设置默认角色，将其分配给通过其身份提供商添加的新用户，并设置自定义会话超时。
  如需了解更多信息，请参阅我们的[文档](/zh/products/cloud/guides/security/cloud-access-management/saml-sso-setup)。
* **Azure 中的最大副本大小和扩缩容限制**

  现在，客户可在除 `eastus2` 外的所有 Azure 区域将最大副本大小设为 356 GiB；而在 `eastus2` 中，可用的最大副本大小为 120 GiB。

<div id="november-7-2025">
  ## 2025 年 11 月 7 日
</div>

* ClickHouse Cloud 控制台现已支持以 1 vCPU、4 GiB 为增量配置副本大小。
  无论是在创建新 service 时，还是在 settings 页面设置最小和最大副本大小时，这些选项都可用。
* 自定义硬件 profile (适用于 Enterprise 层级) 现已支持 idling。
* ClickHouse Cloud 现已通过 AWS Marketplace 提供更简化的购买体验，并分别提供 [按使用量付费](https://aws.amazon.com/marketplace/pp/prodview-p4gwofrqpkltu?sr=0-2\&ref_=beagle\&applicationId=AWSMPContessa) 和 [承诺消费合同](https://aws.amazon.com/marketplace/pp/prodview-4qyeihstyym2s?sr=0-3\&ref_=beagle\&applicationId=AWSMPContessa) 两种选项。
* ClickHouse Cloud 中的 ClickStack 用户现已可使用告警功能。
  现在，你可以直接在 HyperDX UI 中创建和管理日志、指标和链路追踪的告警，无需额外设置、基础设施或 service，也无需 config。告警可与 Slack、PagerDuty 等集成。
  更多信息，请参阅[告警文档](/zh/clickstack/features/alerts)。

<div id="october-17-2025">
  ## 2025 年 10 月 17 日
</div>

* **服务监控 - 资源利用率仪表板**
  CPU 利用率和内存利用率指标的显示方式将由显示平均值改为显示特定时间段内的最大利用率指标，以便更好地发现资源配置不足的实例。
  此外，CPU 利用率指标还将显示 Kubernetes 层级的 CPU 利用率指标，这与 ClickHouse Cloud 自动扩缩器使用的指标更接近。
* **外部存储桶**
  ClickHouse Cloud 现已支持将备份直接导出到您自己的云服务提供商账号。
  连接您的外部存储桶——AWS S3、Google Cloud Storage 或 Azure Blob 存储——即可自主管理备份。

* **适用于 AWS Glue 的 ClickHouse 连接器**
  现在，您可以使用在 [AWS Marketplace](https://aws.amazon.com/marketplace/pp/prodview-eqvmuopqzdg7s) 上提供的官方[适用于 AWS Glue 的 ClickHouse 连接器](/zh/integrations/connectors/data-ingestion/AWS/glue)。该连接器利用 AWS Glue 基于 Apache

<div id="clickhouse-byoc-for-aws-ga">
  ### 适用于 AWS 的 ClickHouse 自备 Cloud (BYOC) 现已正式发布
</div>

在这种部署模式下，数据平面组件 (计算资源、存储、备份、日志、指标)
运行在客户的 VPC 中，而控制平面 (Web 访问、API 和计费)
仍位于 ClickHouse 的 VPC 内。这种设置非常适合需要满足严格数据驻留要求的大型工作负载，
因为它能够确保所有数据都保留在安全的客户环境中。

* 如需了解更多详情，你可以参阅 BYOC 的[文档](/zh/products/cloud/guides/infrastructure/deployment-options/byoc/overview)
  或阅读我们的[公告博文](https://clickhouse.com/blog/announcing-general-availability-of-clickhouse-bring-your-own-cloud-on-aws)。
* [联系我们](https://clickhouse.com/cloud/bring-your-own-cloud)以申请开通权限。

<div id="single-replica-services">
  ### 单副本服务
</div>

我们正在引入“单副本服务”这一概念：它既可以作为独立形态提供，也可以在仓库中使用。作为独立形态时，单副本服务的规模受限，主要适用于较小的测试工作负载。在仓库中，单副本服务则可以部署为更大规模，并可用于无需大规模高可用性的工作负载，例如可重启的 ETL 作业。

<div id="vertical-auto-scaling-improvements">
  ### 垂直自动扩缩容改进
</div>

我们为计算副本引入了一种新的垂直扩缩容机制，称为“Make Before Break” (MBB) 。这种方法会先添加一个或多个新规格的副本，再移除旧副本，从而避免在扩缩容过程中出现容量损失。通过消除移除现有副本与添加新副本之间的时间空档，MBB 让扩缩容过程更加平滑，对业务的影响也更小。它在 scale-up 场景中尤其有用；在这类场景下，较高的资源利用率会触发对额外容量的需求，因为过早移除副本只会进一步加剧资源压力。

我们对 Prometheus 集成进行了多项增强：
