<div id="marketplace-subscription-organization-attachment">
  ### 将 Marketplace 订阅关联到组织
</div>

现在，您可以将新的 Marketplace 订阅关联到现有的 ClickHouse Cloud 组织。完成 Marketplace 订阅流程并跳转到 ClickHouse Cloud 后，您可以把之前创建的现有组织连接到新的 Marketplace 订阅。此后，该组织中的资源将通过 Marketplace 计费。

<Image img="/images/cloud/reference/add_marketplace.webp" size="md" alt="显示如何将 Marketplace 订阅添加到现有组织的 ClickHouse Cloud 界面" border />

<div id="force-openapi-key-expiration">
  ### 强制 OpenAPI 密钥到期
</div>

现在可以限制 API 密钥的到期选项，以避免创建永不过期的 OpenAPI 密钥。请联系 ClickHouse Cloud Support 团队，为您的组织启用这些限制。

<div id="custom-emails-for-notifications">
  ### 通知的自定义邮箱
</div>

组织 Admin 现在可以为特定通知添加更多邮箱地址作为额外收件人。如果您希望将通知发送到某个别名邮箱，或发送给组织内其他并非 ClickHouse Cloud 用户的人员，这项功能会很有用。要进行配置，请前往 Cloud 控制台中的通知设置，编辑您希望接收邮件通知的邮箱地址。

<div id="byoc-beta">
  ### BYOC (beta)
</div>

面向 AWS 的自备 Cloud 现已进入 Beta 阶段。该部署模式允许您在自己的 AWS 账户中部署并运行 ClickHouse Cloud。我们目前支持在 11 个以上的 AWS 区域部署，且很快会支持更多区域。如需开通权限，请[联系支持团队](https://clickhouse.com/support/program)。请注意，此部署仅适用于大规模场景。

<div id="postgres-change-data-capture-cdc-connector-in-clickpipes">
  ### ClickPipes 中的 Postgres CDC (变更数据捕获) 连接器
</div>

这一开箱即用的集成可让客户只需点击几下，即可将 Postgres 数据库复制到 ClickHouse Cloud，并利用 ClickHouse 实现极速分析。您既可以使用此连接器从 Postgres 进行持续复制，也可以进行一次性迁移。

<div id="query-api-endpoints-ga">
  ### 查询 API 端点 (GA)
</div>

我们很高兴地宣布，ClickHouse Cloud 中的查询 API 端点已正式进入 GA 阶段。借助查询 API 端点，您只需点击几下，即可为已保存查询创建 RESTful API 端点，并在应用程序中开始使用数据，无需再为语言客户端或复杂的身份验证而费心。自首次发布以来，我们已推出多项改进，包括：

* 降低端点延迟，尤其是冷启动时的延迟
* 增强端点的 RBAC 控制
* 可配置 CORS 允许的域名
* 结果流式传输
* 支持所有与 ClickHouse 兼容的输出格式

除了这些改进之外，我们还很高兴地宣布推出通用查询 API 端点。它基于我们现有的框架，可让您对 ClickHouse Cloud 服务执行任意 SQL 查询。通用端点可在服务设置页面启用和配置。

要开始使用，请参阅[查询 API 端点文档](/zh/products/cloud/guides/sql-console/query-endpoints)。

<Image img="/images/cloud/reference/api_endpoints.webp" size="lg" alt="显示 API 端点配置及各项设置的 ClickHouse Cloud 界面" border />

<div id="vector-search-using-vector-similarity-indexes-early-access">
  ### 使用向量相似度索引进行向量搜索 (抢先体验)
</div>

我们宣布推出适用于近似向量搜索的向量相似度索引，现已开放抢先体验。

ClickHouse 已为基于向量的用例提供了强大支持，包括丰富的[距离函数]https://clickhouse.com/blog/reinvent-2024-product-announcements#vector-search-using-vector-similarity-indexes-early-access)以及执行线性扫描的能力。此外，最近我们还引入了一种 Experimental 的[近似向量搜索](/zh/reference/engines/table-engines/mergetree-family/annindexes)方法，由 [usearch](https://github.com/unum-cloud/usearch) 库和 Hierarchical Navigable Small Worlds (HNSW) 近似最近邻搜索算法提供支持。

如需开始使用，[请加入抢先体验等候名单](https://clickhouse.com/cloud/vector-search-index-waitlist)。

</div>

对于希望为受保护健康信息 (PHI) 提供更高级别安全保障的客户，现在可以在 [Google Cloud Platform (GCP)](https://cloud.google.com/) 上启用 ClickHouse Cloud。ClickHouse 已落实 [HIPAA Security Rule](https://www.hhs.gov/hipaa/for-professionals/security/index.html) 规定的管理、物理和技术防护措施，并提供可配置的安全设置，可根据您的具体用例和工作负载进行配置。有关可用安全设置的更多信息，请参阅我们的[安全功能页面](/zh/products/cloud/features/security)。

<div id="compute-compute-separation-is-now-in-private-preview-for-gcp-and-azure">
  ### Compute-compute separation 现已在 GCP 和 Azure 上开放私有预览
</div>

我们最近宣布，AWS 上的 Compute-Compute Separation 已进入私有预览。现在，我们很高兴地宣布，GCP 和 Azure 也已支持该功能。

Compute-compute separation 允许您将特定服务指定为读写服务或只读服务，从而为您的应用设计最优的计算配置，优化成本与性能。更多详情请[阅读文档](/zh/products/cloud/features/infrastructure/warehouses)。

<div id="self-service-mfa-recovery-codes">
  ### 自助式 MFA 恢复代码
</div>

启用多重身份验证的客户现在可以获取恢复代码，以便在手机丢失或令牌被误删时使用。首次启用 MFA 的客户会在设置过程中收到该代码。已启用 MFA 的客户可以通过移除现有的 MFA 令牌并添加新的令牌来获取恢复代码。

<div id="clickpipes-update-custom-certificates-latency-insights-and-more">
  ### ClickPipes 更新：自定义证书、延迟洞察等。
</div>

我们很高兴分享 ClickPipes 的最新更新。作为将数据摄取到 ClickHouse 服务的最简便方式，ClickPipes 此次新增的功能可帮助您更好地控制数据摄取过程，并提升对性能指标的可观测性。

*适用于 Kafka 的自定义身份验证证书*

ClickPipes for Kafka 现已支持为使用 SASL 和公共 SSL/TLS 的 Kafka 消息代理配置自定义身份验证证书。您可以在 ClickPipe 设置过程中的 SSL Certificate 部分轻松上传自己的证书，从而确保与 Kafka 建立更安全的连接。

*为 Kafka 和 Kinesis 推出延迟指标*

性能可观测性至关重要。ClickPipes 现已提供延迟图表，让您了解消息从生成 (无论来自 Kafka topic 还是 Kinesis Stream) 到摄取至 ClickHouse Cloud 之间的耗时。借助这一新指标，您可以更密切地关注数据管道的性能，并据此进行优化。

<Image img="/images/cloud/reference/oct-4-latency-insights.webp" size="lg" alt="显示数据摄取性能延迟指标图表的 ClickPipes 界面" border />

<br />

*Kafka 和 Kinesis 的扩缩容控制 (私有 Beta)&#x20;*

高吞吐量场景可能需要更多资源，才能满足数据量和延迟方面的需求。我们正在为 ClickPipes 推出横向扩缩容功能，可直接通过 Cloud Console 使用。该功能目前处于私有 Beta 阶段，使您能够根据实际需求更高效地调整资源规模。如需加入 Beta，请联系 [support](https://clickhouse.com/support/program)。

*Kafka 和 Kinesis 的原始消息摄取*

现在，您可以在不解析的情况下直接摄取完整的 Kafka 或 Kinesis 消息。ClickPipes 现已支持 `_raw_message` [虚拟列](/zh/integrations/clickpipes/kafka/reference#kafka-virtual-columns)，允许用户将完整消息映射到单个 String 列。这让您能够根据需要灵活处理原始数据。

<div id="compute-compute-separation-is-now-in-private-preview-for-aws">
  ### Compute-compute separation 现已在 AWS 上进入私有预览
</div>

对于现有的 ClickHouse Cloud 服务，副本同时处理读写，且无法将某个特定副本配置为只处理其中一种操作。我们即将推出一项名为 Compute-compute separation 的新特性，它允许你将特定服务指定为读写或只读服务，从而为你的应用设计最佳的计算配置，以优化成本和性能。

全新的 Compute-compute separation 特性支持你创建多个计算节点组，每个节点组都有各自的端点，同时使用同一个对象存储文件夹，因此共享相同的表、视图等。请在[此处了解有关 Compute-compute separation 的更多信息](/zh/products/cloud/features/infrastructure/warehouses)。如果你希望获得这项私有预览特性的访问权限，请[联系支持团队](https://clickhouse.com/support/program)。

<div id="clickpipes-for-s3-and-gcs-now-in-ga-continuous-mode-support">
  ### 适用于 S3 和 GCS 的 ClickPipes 现已正式可用，并支持连续模式
</div>

ClickPipes 是将数据摄取到 ClickHouse Cloud 的最简单方式。我们很高兴地宣布，适用于 S3 和 GCS 的 [ClickPipes](https://clickhouse.com/cloud/clickpipes) 现已**正式可用**。ClickPipes 同时支持一次性批次摄取和“连续模式”。摄取任务会根据指定的匹配模式，将特定远程存储桶中所有匹配的文件加载到 ClickHouse 目标表中。在“连续模式”下，ClickPipes 作业会持续运行，不断摄取远程对象存储桶中新到达的匹配文件。这让你能够将任何对象存储桶用作功能完备的暂存区，以便将数据摄取到 ClickHouse Cloud。请在[我们的文档](/zh/integrations/clickpipes/home)中了解有关 ClickPipes 的更多信息。

  ### Prometheus 指标端点现已正式可用

在上一期 ClickHouse Cloud 更新日志中，我们宣布了从 ClickHouse Cloud 导出 [Prometheus](https://prometheus.io/) 指标功能进入私有预览。此功能允许你使用 [ClickHouse Cloud API](/zh/products/cloud/features/admin-features/api/api-overview)，将指标接入 [Grafana](https://grafana.com/) 和 [Datadog](https://www.datadoghq.com/) 等工具进行可视化。我们很高兴地宣布，该功能现已**正式可用**。请参阅[我们的文档](/zh/products/cloud/features/monitoring/prometheus)了解此功能的更多信息。

<div id="table-inspector-in-cloud-console">
  ### Cloud 控制台中的表检查器
</div>

ClickHouse 提供了诸如 [`DESCRIBE`](/zh/reference/statements/describe-table) 之类的命令，让你能够检查表并查看 schema。这些命令会将结果输出到控制台，但通常用起来并不方便，因为你需要组合多条查询，才能获取有关表和列的所有相关信息。

我们最近在 Cloud 控制台中推出了 **表检查器**，让你无需编写 SQL，就能在 UI 中获取重要的表和列信息。你可以前往 Cloud 控制台，为你的服务试用表检查器。它在一个统一界面中集中展示 schema、存储、压缩等信息。

<Image img="/images/cloud/reference/july-18-table-inspector.webp" size="lg" alt="显示详细 schema 和存储信息的 ClickHouse Cloud 表检查器界面" border />

<div id="new-java-client-api">
  ### 全新的 Java Client API
</div>

我们的 [Java Client](https://github.com/ClickHouse/clickhouse-java) 是用户连接 ClickHouse 时最常用的客户端之一。为了让它更易用、更直观，我们重新设计了 API，并进行了多项性能优化。这些改进将让你从 Java 应用连接到 ClickHouse 变得更加轻松。你可以在这篇[博客文章](https://clickhouse.com/blog/java-client-sequel)中详细了解如何使用更新后的 Java Client。

<div id="new-analyzer-is-enabled-by-default">
  ### 新版 analyzer 现已默认启用
</div>

过去几年里，我们一直在开发一个新的 analyzer，用于查询分析和优化。这个 analyzer 提升了查询性能，也让我们能够进一步进行优化，包括实现更快、更高效的 `JOIN`。此前，新用户需要通过设置 `allow_experimental_analyzer` 来启用这项功能。现在，新的 ClickHouse Cloud 服务已默认提供这一改进后的 analyzer。

敬请期待 analyzer 的更多改进，我们还规划了许多进一步的优化。

<div id="configurable-offsets-for-kafka-clickpipes-connector-beta">
  ### Kafka ClickPipes Connector 的可配置 offset (Beta)
</div>

直到最近，每当你设置新的 [Kafka Connector for ClickPipes](/zh/integrations/clickpipes/kafka/index) 时，它都会始终从 Kafka topic 的开头开始消费数据。在需要重新处理历史数据、监控新写入的数据，或从某个精确位置恢复处理时，这种方式可能不够灵活，无法满足特定用例。

ClickPipes for Kafka 新增了一项功能，进一步提升了从 Kafka topics 消费数据时的灵活性和控制力。你现在可以配置数据从哪个 offset 开始消费。

可用选项如下：

* 从开头开始：从 Kafka topic 的最开头开始消费数据。如果你需要重新处理全部历史数据，此选项非常适合。
* 从最新位置开始：从最新的 offset 开始消费数据。如果你只关心新消息，这个选项很有用。
* 从某个 timestamp 开始：从在某个指定 timestamp 或之后生成的消息开始消费数据。此功能可实现更精确的控制，让你能够从准确的时间点恢复处理。

<div id="reliability-changes">
  ### 可靠性变更
</div>

* 面向用户的向后不兼容变更：此前，在特定条件下，两个功能 ([is&#95;deleted](/zh/reference/engines/table-engines/mergetree-family/replacingmergetree#is_deleted) 和 `OPTIMIZE CLEANUP`) 可能会导致 ClickHouse 中的数据损坏。为了保护用户数据的完整性，同时保留该功能的核心能力，我们调整了这一功能的工作机制。具体来说，MergeTree 设置 `clean_deleted_rows` 已弃用，且不再生效。默认情况下不允许使用 `CLEANUP` 关键字 (如需使用，必须启用 `allow_experimental_replacing_merge_with_cleanup`) 。如果你决定使用 `CLEANUP`，则需要确保它始终与 `FINAL` 一同使用，并且必须保证在运行 &#96;&#96;OPTIMIZE FINAL CLEANUP&#96;&#96;&#96; 之后，不会再插入旧版本的行。
