本节包含服务器设置的说明。这些设置
无法在会话或查询级别更改。

有关 ClickHouse 配置文件的更多信息，请参阅[&quot;&quot;配置文件&quot;&quot;](/zh/concepts/features/configuration/server-config/configuration-files)。

其他设置见 &quot;&quot;[Settings](/zh/concepts/features/configuration/settings/overview)&quot;&quot; 部分。
在了解这些设置之前，建议先阅读[配置文件](/zh/concepts/features/configuration/server-config/configuration-files)
部分，并留意替换 (`incl` 和 `optional` 属性) 的用法。

<div id="access_control_improvements">
  ## access_control_improvements
</div>

访问控制系统可选增强功能的设置。

| 设置                                              | 描述                                                                                                                                                                                                                                                                      | 默认值     |
| ----------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ------- |
| `on_cluster_queries_require_cluster_grant`      | 设置 `ON CLUSTER` 查询是否需要 `CLUSTER` 授权。                                                                                                                                                                                                                                    | `true`  |
| `role_cache_expiration_time_seconds`            | 设置角色自上次访问后在 Role Cache 中保留的秒数。                                                                                                                                                                                                                                          | `600`   |
| `select_from_information_schema_requires_grant` | 设置 `SELECT * FROM information_schema.<table>` 是否需要授权，还是可由任意用户执行。如果设为 true，则此查询需要 `GRANT SELECT ON information_schema.<table>`，与普通表相同。                                                                                                                                   | `true`  |
| `select_from_system_db_requires_grant`          | 设置 `SELECT * FROM system.<table>` 是否需要授权，还是可由任意用户执行。如果设为 true，则此查询需要 `GRANT SELECT ON system.<table>`，与非系统表相同。例外情况：少数系统表 (`tables`、`columns`、`databases`，以及一些常量表，如 `one`、`contributors`) 仍对所有人开放；此外，如果已授予 `SHOW` 权限 (例如 `SHOW USERS`) ，则对应的系统表 (即 `system.users`) 也可访问。 | `true`  |
| `settings_constraints_replace_previous`         | 设置某个设置的 settings profile 中的约束，是否会取消该设置上一个约束 (定义在其他 profile 中) 的作用，包括新约束未设置的字段。它还会启用 `changeable_in_readonly` 约束类型。                                                                                                                                                      | `true`  |
| `table_engines_require_grant`                   | 设置使用特定表引擎创建表是否需要授权。                                                                                                                                                                                                                                                     | `false` |
| `throw_on_unmatched_row_policies`               | 设置从表中读取时，如果该表存在行策略，但当前用户没有任何匹配的行策略，是否应抛出异常。                                                                                                                                                                                                                             | `false` |
| `users_without_row_policies_can_read_rows`      | 设置没有允许型行策略的用户是否仍可通过 `SELECT` 查询读取行。例如，如果有两个用户 A 和 B，且仅为 A 定义了行策略，那么当此设置为 true 时，用户 B 将看到所有行；当此设置为 false 时，用户 B 将看不到任何行。                                                                                                                                                 | `true`  |

示例：

```xml
<access_control_improvements>
    <throw_on_unmatched_row_policies>true</throw_on_unmatched_row_policies>
    <users_without_row_policies_can_read_rows>true</users_without_row_policies_can_read_rows>
    <on_cluster_queries_require_cluster_grant>true</on_cluster_queries_require_cluster_grant>
    <select_from_system_db_requires_grant>true</select_from_system_db_requires_grant>
    <select_from_information_schema_requires_grant>true</select_from_information_schema_requires_grant>
    <settings_constraints_replace_previous>true</settings_constraints_replace_previous>
    <table_engines_require_grant>false</table_engines_require_grant>
    <role_cache_expiration_time_seconds>600</role_cache_expiration_time_seconds>
</access_control_improvements>
```

<div id="allow_feature_tier">
  ## allow_feature_tier
</div>

<SettingsInfoBlock type="UInt32" default_value="0" changeable_without_restart="Yes" />

控制用户是否可以修改与不同功能层级相关的设置。

* `0` - 允许修改任何设置 (Experimental、Beta、生产) 。
* `1` - 仅允许修改 Beta 和生产功能设置。对 Experimental 设置的修改会被拒绝。
* `2` - 仅允许修改生产设置。对 Experimental 或 Beta 设置的修改会被拒绝。

这相当于为所有 `EXPERIMENTAL` / `BETA` 功能设置只读约束。

<Note>
  值为 `0` 表示可以修改所有设置。
</Note>

<div id="async_load_databases">
  ## async_load_databases
</div>

<SettingsInfoBlock type="Bool" default_value="1" changeable_without_restart="No" />

异步加载数据库和表。

* 如果为 `true`，所有使用 `Ordinary`、`Atomic` 和 `Replicated` 引擎的非 system 数据库都会在 ClickHouse server 启动后异步加载。请参见 `system.asynchronous_loader` 表，以及 `tables_loader_background_pool_size` 和 `tables_loader_foreground_pool_size` 服务器设置。任何试图访问尚未加载表的查询，都会只等待该表完成启动。如果加载 job 失败，查询会重新抛出 error (而不是像 `async_load_databases = false` 时那样关闭整个 server) 。被至少一个查询等待的表会以更高优先级加载。针对某个数据库的 DDL 查询也会只等待该数据库完成启动。还应考虑为等待中的查询总数设置限制 `max_waiting_queries`。
* 如果为 `false`，所有数据库都会在 server 启动时加载。

**示例**

```xml
<async_load_databases>true</async_load_databases>
```

<div id="async_load_system_database">
  ## async_load_system_database
</div>

<SettingsInfoBlock type="Bool" default_value="0" changeable_without_restart="No" />

异步加载系统表。当 `system` 数据库中的日志表和 parts 数量较多时，此设置会很有帮助。该设置独立于 `async_load_databases`。

* 如果设置为 `true`，则所有使用 `Ordinary`、`Atomic` 和 `Replicated` 引擎的系统数据库都会在 ClickHouse server 启动后异步加载。请参见 `system.asynchronous_loader` 表，以及 `tables_loader_background_pool_size` 和 `tables_loader_foreground_pool_size` 服务器设置。任何尝试访问尚未加载的系统表的查询，都会一直等待该表完成启动。只要某个表被至少一个查询等待，它就会以更高优先级加载。你还可以考虑设置 `max_waiting_queries`，以限制等待中的查询总数。
* 如果设置为 `false`，系统数据库会在服务器启动前加载。

**示例**

```xml
<async_load_system_database>true</async_load_system_database>
```

<div id="asynchronous_metric_log">
  ## asynchronous_metric_log
</div>

在 ClickHouse Cloud 部署中，此项默认启用。

如果您的环境中该设置默认未启用，可根据 ClickHouse 的安装方式，按以下说明启用或禁用。

**启用**

如需手动启用异步指标日志历史记录 [`system.asynchronous_metric_log`](/zh/reference/system-tables/asynchronous_metric_log) 的收集功能，请创建 `/etc/clickhouse-server/config.d/asynchronous_metric_log.xml`，内容如下：

```xml
<clickhouse>
     <asynchronous_metric_log>
        <database>system</database>
        <table>asynchronous_metric_log</table>
        <flush_interval_milliseconds>7500</flush_interval_milliseconds>
        <collect_interval_milliseconds>1000</collect_interval_milliseconds>
        <max_size_rows>1048576</max_size_rows>
        <reserved_size_rows>8192</reserved_size_rows>
        <buffer_size_rows_flush_threshold>524288</buffer_size_rows_flush_threshold>
        <flush_on_crash>false</flush_on_crash>
    </asynchronous_metric_log>
</clickhouse>
```

**禁用**

要禁用 `asynchronous_metric_log` 设置，请创建以下文件 `/etc/clickhouse-server/config.d/disable_asynchronous_metric_log.xml`，内容如下：

```xml
<clickhouse><asynchronous_metric_log remove="1" /></clickhouse>
```

<div id="background_merges_mutations_concurrency_ratio">
  ## background_merges_mutations_concurrency_ratio
</div>

<SettingsInfoBlock type="Float" default_value="2" changeable_without_restart="No" />

设置线程数与可并发执行的后台合并和变更数量之间的比率。

例如，如果该比率为 2，且 [`background_pool_size`](/zh/reference/settings/server-settings/settings#background_pool_size) 设置为 16，那么 ClickHouse 最多可以并发执行 32 个后台合并。这之所以可行，是因为后台操作可以被挂起并延后执行。这样做是为了让较小的合并获得更高的执行优先级。

<Note>
  你只能在运行时提高此比率。若要降低，则必须重启服务器。

  与 [`background_pool_size`](/zh/reference/settings/server-settings/settings#background_pool_size) 设置一样，出于向后兼容性的考虑，[`background_merges_mutations_concurrency_ratio`](/zh/reference/settings/server-settings/settings#background_merges_mutations_concurrency_ratio) 也可以从 `default` profile 中应用。
</Note>

<div id="background_merges_mutations_scheduling_policy">
  ## background_merges_mutations_scheduling_policy
</div>

<SettingsInfoBlock type="String" default_value="round_robin" changeable_without_restart="No" />

用于后台 合并 和 变更 调度的策略。可能的值有：`round_robin` 和 `shortest_task_first`。

用于选择下一个由后台线程池执行的 合并 或 变更 的算法。该策略可在运行时更改，无需重启 server。
出于向后兼容考虑，可通过 `default` profile 应用。

可能的值：

* `round_robin` — 每个并发 合并 和 变更 都按轮询顺序执行，以确保不会发生饥饿。较小的 合并 比较大的 合并 完成得更快，仅仅是因为它们需要合并的块更少。
* `shortest_task_first` — 始终执行较小的 合并 或 变更。合并 和 变更 会根据结果大小分配优先级。较小的 合并 会被严格优先于较大的 合并。此策略可确保小 parts 尽快完成合并，但在因 `INSERT` 导致负载很高的分区中，可能会使大型 合并 长期得不到执行。

<div id="background_pool_size">
  ## background_pool_size
</div>

<SettingsInfoBlock type="UInt64" default_value="16" changeable_without_restart="No" />

设置用于为采用 MergeTree 引擎的表执行后台合并和变更的线程数。

<Note>
  * 出于向后兼容性的考虑，该设置也可以在 ClickHouse server 启动时通过 `default` profile 配置应用。
  * 您只能在运行时增加线程数。
  * 若要减少线程数，则必须重启 server。

</Note>

<Danger>
  较小的线程池会占用更少的 CPU 和磁盘资源，但后台进程推进得更慢，最终可能会影响查询性能。
</Danger>

* [`number_of_free_entries_in_pool_to_lower_max_size_of_merge`](/zh/reference/settings/merge-tree-settings#number_of_free_entries_in_pool_to_lower_max_size_of_merge)。
* [`number_of_free_entries_in_pool_to_execute_mutation`](/zh/reference/settings/merge-tree-settings#number_of_free_entries_in_pool_to_execute_mutation)。
* [`number_of_free_entries_in_pool_to_execute_optimize_entire_partition`](/zh/reference/settings/merge-tree-settings#number_of_free_entries_in_pool_to_execute_optimize_entire_partition)

**示例**

```xml

```

<div id="concurrent_threads_scheduler">
  ## concurrent_threads_scheduler
</div>

<SettingsInfoBlock type="String" default_value="max_min_fair" changeable_without_restart="Yes" />

由 `concurrent_threads_soft_limit_num` 和 `concurrent_threads_soft_limit_ratio_to_cores` 指定的 CPU 插槽的调度策略。该算法用于控制有限数量的 CPU 插槽如何在并发查询之间分配。调度器可在运行时更改，无需重启服务器。

可能的值：

* `round_robin` — 每个设置了 `use_concurrency_control` = 1 的查询最多可分配 `max_threads` 个 CPU 插槽。每个线程占用一个插槽。发生争用时，CPU 插槽会以轮询方式分配给各个查询。请注意，第一个插槽会无条件分配，这可能导致不公平；在存在大量 `max_threads` = 1 的查询时，还会增加 `max_threads` 较高的查询的延迟。
* `fair_round_robin` — 每个设置了 `use_concurrency_control` = 1 的查询最多可分配 `max_threads - 1` 个 CPU 插槽。这是 `round_robin` 的一种变体，不要求为每个查询的第一个线程分配 CPU 插槽。这样，`max_threads` = 1 的查询不需要任何插槽，因此也无法不公平地占用全部插槽。不会无条件分配任何插槽。
* `max_min_fair` — 每个设置了 `use_concurrency_control` = 1 的查询最多可分配 `max_threads - 1` 个 CPU 插槽。它与 `fair_round_robin` 类似，但释放出的插槽始终会分配给当前已分配插槽数最少的查询。在高超额订阅场景下 (即许多查询竞争有限 CPU 插槽时) ，这种方式能提供更好的公平性。运行时间短的查询不会因长期运行、且随着时间推移已累积更多插槽的查询而受到不利影响。

<div id="concurrent_threads_soft_limit_num">
  ## concurrent_threads_soft_limit_num
</div>

<SettingsInfoBlock type="UInt64" default_value="0" changeable_without_restart="Yes" />

允许运行所有查询的查询处理线程最大数量，不包括用于从远程服务器获取数据的线程。这不是硬性限制。即使达到该限制，每个查询仍至少会分配到一个线程运行。如果后续有更多线程可用，查询在执行过程中还可以扩展到所需的线程数。

<div id="cpu_slot_preemption">
  ## cpu_slot_preemption
</div>

<SettingsInfoBlock type="Bool" default_value="1" changeable_without_restart="Yes" />

定义如何对 CPU 资源 (MASTER THREAD 和 WORKER THREAD) 进行工作负载调度。

* 如果为 `true` (默认值) ，则按实际消耗的 CPU 时间进行核算。竞争中的各个工作负载会获得相对公平的 CPU 时间分配。CPU 插槽按有限时长分配，到期后需要重新申请。在 CPU 资源过载时，申请插槽可能会阻塞线程执行，也就是说，可能发生抢占。这样可确保 CPU 时间分配的公平性。
* 如果为 `false`，则按已分配的 CPU 插槽数量进行核算。竞争中的各个工作负载会获得相对公平的 CPU 插槽分配。线程启动时会分配一个插槽，并在整个执行期间持续占用，直到线程执行结束后释放。分配给查询执行的线程数只能从 1 增加到 `max_threads`，不会减少。这种方式更有利于长时间运行的查询，但也可能导致短查询出现 CPU 饥饿。

**示例**

```xml
<cpu_slot_preemption>true</cpu_slot_preemption>
```

**另请参阅**

* [工作负载调度](/zh/concepts/features/configuration/server-config/workload-scheduling)

<div id="disable_tunneling_for_https_requests_over_http_proxy">
  ## disable_tunneling_for_https_requests_over_http_proxy
</div>

默认情况下，通过 `HTTP` 代理发起 `HTTPS` 请求时会使用隧道 (即 `HTTP CONNECT`) 。此设置可用于禁用该行为。

**no&#95;proxy**

默认情况下，所有请求都会通过代理。要为特定主机禁用代理，必须设置 `no_proxy` 变量。
对于列表解析器和远程解析器，可在 `<proxy>` 子句中进行设置；对于环境解析器，则可将其设置为环境变量。
它支持 IP 地址、域名、子域名，以及用于完全绕过代理的 `'*'` 通配符。与 curl 一样，前导点会被去掉。

**示例**

以下配置会绕过发往 `clickhouse.cloud` 及其所有子域名 (例如 `auth.clickhouse.cloud`) 的请求代理。
同样的规则也适用于 GitLab，即使它带有前导点。`gitlab.com` 和 `about.gitlab.com` 都会绕过代理。

```xml
<proxy>
    <no_proxy>clickhouse.cloud,.gitlab.com</no_proxy>
    <http>
        <uri>http://proxy1</uri>
        <uri>http://proxy2:3128</uri>
    </http>
    <https>
        <uri>http://proxy1:3128</uri>
    </https>
</proxy>
```

```xml
    <!-- 控制任务的生存时间 (TTL)（默认 1 周） -->
```

```xml
<encryption_codecs>
```

```xml
        <key_hex>00112233445566778899aabbccddeeff</key_hex>
    </aes_128_gcm_siv>
</encryption_codecs>
```

从环境变量中加载密钥：

```xml
<encryption_codecs>
    <aes_128_gcm_siv>
        <key_hex from_env="ENVVAR"></key_hex>
    </aes_128_gcm_siv>
</encryption_codecs>
```

这里，`current_key_id` 用于设置当前的加密密钥，而所有指定的密钥均可用于解密。

以下每种方法都可应用于多个密钥：

```xml
<encryption_codecs>
    <aes_128_gcm_siv>
        <key_hex id="0">00112233445566778899aabbccddeeff</key_hex>
        <key_hex id="1" from_env="ENVVAR"></key_hex>
        <current_key_id>1</current_key_id>
    </aes_128_gcm_siv>
</encryption_codecs>
```

这里，`current_key_id` 表示当前用于加密的密钥。

此外，用户还可以添加长度必须为 12 字节的 nonce (默认情况下，加密和解密过程使用由零字节组成的 nonce) ：

```xml
<encryption_codecs>
    <aes_128_gcm_siv>
        <nonce>012345678910</nonce>
    </aes_128_gcm_siv>
</encryption_codecs>
```

也可以将其设置为十六进制：

```xml
<encryption_codecs>
    <aes_128_gcm_siv>
        <nonce_hex>abcdefabcdef</nonce_hex>
    </aes_128_gcm_siv>
</encryption_codecs>
```

<Note>
  上述提到的所有内容同样适用于 `aes_256_gcm_siv` (但密钥长度必须为 32 字节) 。
</Note>

<div id="http_handlers">
  ## http_handlers
</div>

允许使用自定义 HTTP 处理程序。
要添加新的 http 处理程序，只需新增一条 `<rule>`。
规则会按照定义顺序自上而下进行检查，
第一个匹配的规则将运行相应的处理程序。
没有匹配条件的规则 (仅包含 `handler`) 会匹配每个请求；由于规则会按顺序进行检查，
因此，这类规则只有作为放在最后的回退规则时才有用。

以下设置可通过子标签进行配置 (除 `handler` 外，所有这些子标签都是可选的) ：

| Sub-tags             | Definition                                                                                                                            |
| -------------------- | ------------------------------------------------------------------------------------------------------------------------------------- |
| `url`                | 用于匹配请求 URL 路径。匹配时会忽略查询字符串                                                                                                             |
| `url_prefix`         | 用于将请求 URL 路径与基础路径进行匹配：路径本身或其下方按路径段边界划分的任何内容 (例如，&#39;/api/v1&#39; 可匹配 /api/v1、/api/v1/ 和 /api/v1/write，但不匹配 /api/v1beta) 。匹配时会忽略查询字符串 |
| `url_regexp`         | 用于将请求 URL 路径与正则表达式进行匹配。匹配时会忽略查询字符串                                                                                                    |
| `full_url`           | 用于匹配完整的请求 URL `scheme://host:port/path`。匹配时会忽略查询字符串，且 host 是连接 IP 地址 (而不是 `Host` 请求头)                                                 |
| `full_url_prefix`    | 用于将完整的请求 URL `scheme://host:port/path` 与基础 URL `scheme://host:port/base_path` 在路径段边界上进行匹配 (见 `url_prefix`) 。匹配时会忽略查询字符串               |
| `full_url_regexp`    | 用于将完整的请求 URL `scheme://host:port/path` 与正则表达式进行匹配。匹配时会忽略查询字符串                                                                         |
| `methods`            | 用于匹配请求方法；可使用逗号分隔多个方法                                                                                                                  |
| `headers`            | 用于匹配请求头；会匹配每个子元素 (子元素名称即请求头名称)                                                                                                        |
| `headers_regexp`     | 与 `headers` 类似，但每个子元素的值都会与正则表达式进行匹配                                                                                                   |
| `empty_query_string` | 检查 URL 中是否不包含查询字符串                                                                                                                    |
| `handler`            | 请求处理程序 (必需)                                                                                                                           |

<Note>
  除了 `url_regexp`、`full_url_regexp` 和 `headers_regexp` 之外，也可以在 `url`、`full_url` 或 `headers` 中使用 `regex:` 前缀来编写正则表达式 (例如 `<url>regex:/api/.*</url>`) 。出于向后兼容性考虑，这种写法仍然受支持，但已过时：建议优先使用专用的 `url_regexp`、`full_url_regexp` 和 `headers_regexp` 子标签。
</Note>

`handler` 包含以下设置，这些设置可通过子标签进行配置：

| Sub-tags           | Definition                                                                                       |
| ------------------ | ------------------------------------------------------------------------------------------------ |
| `url`              | 重定向目标地址                                                                                          |
| `type`             | 支持的类型：static、dynamic&#95;query&#95;handler、predefined&#95;query&#95;handler、redirect             |
| `status`           | 与 static 类型配合使用，表示响应状态码                                                                          |
| `query_param_name` | 与 dynamic&#95;query&#95;handler 类型配合使用，提取并执行 HTTP 请求参数中名称与 `<query_param_name>` 值对应的参数值          |
| `query`            | 与 predefined&#95;query&#95;handler 类型配合使用，在调用该处理程序时执行查询                                          |
| `content_type`     | 与 static 类型配合使用，表示响应的 content-type                                                               |
| `response_content` | 与 static 类型配合使用，表示发送给客户端的响应内容；使用 &#39;file://&#39; 或 &#39;config://&#39; 前缀时，会从文件或配置中查找内容并发送给客户端 |

除规则列表外，还可以指定 `<defaults/>`，用于启用所有默认处理程序。

示例：

```xml
<http_handlers>
    <rule>
        <url>/</url>
        <methods>POST,GET</methods>
        <headers><pragma>no-cache</pragma></headers>
        <handler>
            <type>dynamic_query_handler</type>
            <query_param_name>query</query_param_name>
        </handler>
    </rule>

    <rule>
        <url>/predefined_query</url>
        <methods>POST,GET</methods>
        <handler>
            <type>predefined_query_handler</type>
            <query>SELECT * FROM system.settings</query>
        </handler>
    </rule>

    <rule>
        <handler>
            <type>static</type>
            <status>200</status>
            <content_type>text/plain; charset=UTF-8</content_type>
            <response_content>config://http_server_default_response</response_content>
        </handler>
    </rule>
</http_handlers>
```

<div id="insert_deduplication_version">
  ## insert_deduplication_version
</div>

<SettingsInfoBlock type="InsertDeduplicationVersions" default_value="compatible_double_hashes" changeable_without_restart="No" />

此设置可用于从这样一个代码版本迁移：该版本对同步和异步插入的插入去重采用完全不同且不透明的方式；迁移到另一个代码版本：在该版本中，已插入的数据会在同步和异步插入之间统一去重。
值 `old_separate_hashes` 表示 ClickHouse 会为同步和异步插入使用不同的去重哈希 (与此前相同) 。
如果不打算开始迁移，则应将此值作为默认值。
值 `compatible_double_hashes` 表示 ClickHouse 会使用两个去重哈希：一个旧哈希用于同步或异步插入，另一个新哈希用于所有插入。应使用此值，以安全的方式将现有实例迁移到新行为。
应在一段时间内启用此值 (参见 replicated&#95;deduplication&#95;window 和 non&#95;replicated&#95;deduplication&#95;window 设置) ，以确保迁移期间不会丢失任何同步或异步插入。
最后，值 `new_unified_hash` 表示 ClickHouse 会为同步和异步插入使用新的去重哈希。此值可在新的 ClickHouse 实例上启用，也可在已使用 `compatible_double_hashes` 一段时间的实例上启用。
默认值将先设置为 `compatible_double_hashes`，之后在未来的 ClickHouse 版本中再设置为 `new_unified_hash`，以便分两个阶段安全地完成迁移。

<div id="interserver_http_credentials">
  ## interserver_http_credentials
</div>

在[复制](/zh/reference/engines/table-engines/mergetree-family/replication)过程中，用于连接其他服务器的用户名和密码。此外，服务器还会使用这些凭据对其他副本进行身份验证。
因此，集群中所有副本的 `interserver_http_credentials` 必须保持一致。

<Note>
  * 默认情况下，如果省略 `interserver_http_credentials` 部分，则复制过程中不使用身份验证。
  * `interserver_http_credentials` 设置与 ClickHouse 客户端凭据[配置](/zh/concepts/features/interfaces/client#configuration_files)无关。
  * 这些凭据同时用于通过 `HTTP` 和 `HTTPS` 进行的复制。
</Note>

以下设置可通过子标签进行配置：

* `user` — 用户名。
* `password` — 密码。
* `allow_empty` — 如果为 `true`，则即使已设置凭据，也允许其他副本在不进行身份验证的情况下连接。如果为 `false`，则拒绝未经过身份验证的连接。默认值：`false`。
* `old` — 包含凭据轮换期间使用的旧 `user` 和 `password`。可以指定多个 `old` 部分。

**凭据轮换**

ClickHouse 支持动态轮换服务器间凭据，无需为了更新配置而同时停止所有副本。凭据可分多个步骤进行更改。

要启用身份验证，请将 `interserver_http_credentials.allow_empty` 设置为 `true` 并添加凭据。这样会同时允许已进行身份验证和未进行身份验证的连接。

```xml
<interserver_http_credentials>
    <user>admin</user>
    <password>111</password>
    <allow_empty>true</allow_empty>
</interserver_http_credentials>
```

在完成所有副本的配置后，将 `allow_empty` 设置为 `false`，或删除此设置。这样会强制要求使用新的凭据进行身份验证。

要更改现有凭据，请将用户名和密码移至 `interserver_http_credentials.old` 部分，并将 `user` 和 `password` 更新为新值。此时，server 会使用新凭据连接到其他副本，同时接受使用新旧任一凭据建立的连接。

```xml
<interserver_http_credentials>
    <user>admin</user>
    <password>222</password>
    <old>
        <user>admin</user>
        <password>111</password>
    </old>
    <old>
        <user>temp</user>
        <password>000</password>
    </old>
</interserver_http_credentials>
```

待所有副本都应用新凭据后，即可移除旧凭据。

<div id="ldap_servers">
  ## ldap_servers
</div>

在此处列出 LDAP 服务器及其连接参数，以便：

* 将其用作专用本地用户的身份验证器，这些用户指定的是 `ldap` 身份验证机制而非 `password`
* 将其用作远程用户目录。

以下设置可通过子标签进行配置：

| 设置                             | 说明                                                                                                                                                                                      |
| ------------------------------ | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `bind_dn`                      | 用于构造绑定 DN 的模板。在每次身份验证尝试期间，会将模板中的所有 `\{user_name\}` 子串替换为实际用户名，从而生成相应的 DN。                                                                                                               |
| `enable_tls`                   | 用于启用与 LDAP 服务器安全连接的标志。对于明文 (`ldap://`) 协议，指定 `no` (不推荐) 。对于 LDAP over SSL/TLS (`ldaps://`) 协议，指定 `yes` (推荐，也是默认值) 。对于旧版 StartTLS 协议，指定 `starttls` (即明文 (`ldap://`) 协议再升级为 TLS) 。        |
| `host`                         | LDAP 服务器主机名或 IP，此参数必填且不能为空。                                                                                                                                                             |
| `port`                         | LDAP 服务器端口；如果 `enable_tls` 设置为 true，则默认值为 636，否则为 `389`。                                                                                                                                |
| `tls_ca_cert_dir`              | 包含 CA 证书的目录路径。                                                                                                                                                                          |
| `tls_ca_cert_file`             | CA 证书文件的路径。                                                                                                                                                                             |
| `tls_cert_file`                | 证书文件的路径。                                                                                                                                                                                |
| `tls_cipher_suite`             | 允许使用的密码套件 (采用 OpenSSL 表示法) 。                                                                                                                                                            |
| `tls_key_file`                 | 证书密钥文件的路径。                                                                                                                                                                              |
| `tls_minimum_protocol_version` | SSL/TLS 的最低协议版本。可接受的值为：`ssl2`、`ssl3`、`tls1.0`、`tls1.1`、`tls1.2` (默认值) 。                                                                                                                 |
| `tls_require_cert`             | SSL/TLS 对端证书验证行为。可接受的值为：`never`、`allow`、`try`、`demand` (默认值) 。                                                                                                                          |
| `user_dn_detection`            | 包含 LDAP 搜索参数的部分，用于检测已绑定用户的实际 user DN。当 server 为 Active Directory 时，这主要用于后续角色映射中的搜索过滤器。生成的 user DN 将在允许使用的地方替换 `\{user_dn\}` 子串。默认情况下，user DN 与 bind DN 相同；但执行搜索后，它会更新为实际检测到的 user DN 值。 |
| `verification_cooldown`        | 成功完成一次绑定尝试后的一段时间 (以秒为单位) ；在此期间，对于后续所有请求，都会假定用户已成功通过身份验证，而无需联系 LDAP 服务器。指定 `0` (默认值) 可禁用缓存，并强制每次身份验证请求都联系 LDAP 服务器。                                                                      |

设置 `user_dn_detection` 可通过子标签进行配置：

| 设置              | 说明                                                                                                                                                               |
| --------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `base_dn`       | 用于构造 LDAP 搜索 base DN 的模板。在 LDAP 搜索期间，会将模板中的所有 `\{user_name\}` 和 `\{bind_dn\}` 子串替换为实际用户名和 bind DN，从而生成相应的 DN。                                                    |
| `scope`         | LDAP 搜索的范围。可接受的值为：`base`、`one_level`、`children`、`subtree` (默认值) 。                                                                                                |
| `search_filter` | 用于构造 LDAP 搜索 search filter 的模板。在 LDAP 搜索期间，会将模板中的所有 `\{user_name\}`、`\{bind_dn\}` 和 `\{base_dn\}` 子串替换为实际用户名、bind DN 和 base DN，从而生成相应的过滤器。请注意，特殊字符必须在 XML 中正确转义。 |

示例：

```xml
<my_ldap_server>
    <host>localhost</host>
    <port>636</port>
    <bind_dn>uid={user_name},ou=users,dc=example,dc=com</bind_dn>
    <verification_cooldown>300</verification_cooldown>
    <enable_tls>yes</enable_tls>
    <tls_minimum_protocol_version>tls1.2</tls_minimum_protocol_version>
    <tls_require_cert>demand</tls_require_cert>
    <tls_cert_file>/path/to/tls_cert_file</tls_cert_file>
    <tls_key_file>/path/to/tls_key_file</tls_key_file>
    <tls_ca_cert_file>/path/to/tls_ca_cert_file</tls_ca_cert_file>
    <tls_ca_cert_dir>/path/to/tls_ca_cert_dir</tls_ca_cert_dir>
    <tls_cipher_suite>ECDHE-ECDSA-AES256-GCM-SHA384:ECDHE-RSA-AES256-GCM-SHA384:AES256-GCM-SHA384</tls_cipher_suite>
</my_ldap_server>
```

示例 (典型的 Active Directory，已配置 user DN 检测以用于后续角色映射) ：

```xml
<my_ad_server>
    <host>localhost</host>
    <port>389</port>
    <bind_dn>EXAMPLE\{user_name}</bind_dn>
    <user_dn_detection>
        <base_dn>CN=Users,DC=example,DC=com</base_dn>

    </user_dn_detection>
    <enable_tls>no</enable_tls>
</my_ad_server>
```

**按级别覆盖**

可以单独覆盖各个日志名称的日志级别。例如，如需屏蔽记录器 &quot;Backup&quot; 和 &quot;RBAC&quot; 的所有消息。

```xml
<logger>
    <levels>
        <logger>
            <name>Backup</name>
            <level>none</level>
        </logger>
        <logger>
            <name>RBAC</name>
            <level>none</level>
        </logger>
    </levels>
</logger>
```

**syslog**

要将日志消息同时写入 syslog：

```xml
<logger>
    <use_syslog>1</use_syslog>
    <syslog>
        <address>syslog.remote:10514</address>
        <hostname>myhost.local</hostname>
        <facility>LOG_LOCAL6</facility>
        <format>syslog</format>
    </syslog>
</logger>
```

`<syslog>` 的键：

| Key        | Description                                                                                                                                                                                              |
| ---------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `address`  | syslog 的地址，格式为 `host\[:port\]`。如果省略，则使用本地守护进程。                                                                                                                                                           |
| `hostname` | 发送日志的主机名 (可选) 。                                                                                                                                                                                          |
| `facility` | syslog 的 [facility 关键字](https://en.wikipedia.org/wiki/Syslog#Facility)。必须使用带有 &quot;LOG&#95;&quot; 前缀的大写形式指定，例如 `LOG_USER`、`LOG_DAEMON`、`LOG_LOCAL3` 等。如果指定了 `address`，默认值为 `LOG_USER`；否则为 `LOG_DAEMON`。 |
| `format`   | 日志消息格式。可选值：`bsd` 和 `syslog.`                                                                                                                                                                             |

**日志格式**

您可以指定控制台日志的输出格式。目前仅支持 JSON。

**示例**

以下是一个输出的 JSON 日志示例：

```json
{
  "date_time_utc": "2024-11-06T09:06:09Z",
  "date_time": "1650918987.180175",
  "thread_name": "#1",
  "thread_id": "254545",
  "level": "Trace",
  "query_id": "",
  "logger_name": "BaseDaemon",
  "message": "Received signal 2",
  "source_file": "../base/daemon/BaseDaemon.cpp; virtual void SignalListener::run()",
  "source_line": "192"
}
```

要启用 JSON 日志支持，请使用以下代码片段：

```xml
<logger>
    <formatting>
        <type>json</type>
        <!-- 可按通道（log、errorlog、console、syslog）单独配置，也可全局应用于所有通道（全局配置时省略此标签即可）。-->
        <!-- <channel></channel> -->
        <names>
            <date_time>date_time</date_time>
            <thread_name>thread_name</thread_name>
            <thread_id>thread_id</thread_id>
            <level>level</level>
            <query_id>query_id</query_id>
            <logger_name>logger_name</logger_name>
            <message>message</message>
            <source_file>source_file</source_file>
            <source_line>source_line</source_line>
        </names>
    </formatting>
</logger>
```

**重命名 JSON 日志中的键**

可以通过修改 `<names>` 标签内各项的值来更改键名。例如，要将 `DATE_TIME` 改为 `MY_DATE_TIME`，可以使用 `<date_time>MY_DATE_TIME</date_time>`。

**省略 JSON 日志中的键**

可以通过将相应属性注释掉来省略日志属性。例如，如果不希望日志输出 `query_id`，可以将 `<query_id>` 标签注释掉。

特殊情况下，值为 `0` (默认值) 表示服务器可以消耗所有可用内存 (不包括 `max_server_memory_usage_to_ram_ratio` 施加的进一步限制) 。

<div id="max_server_memory_usage_to_ram_ratio">
  ## max_server_memory_usage_to_ram_ratio
</div>

<SettingsInfoBlock type="Double" default_value="0.9" changeable_without_restart="No" />

服务器允许使用的最大内存量，以占全部可用内存的比例表示。

例如，值为 `0.9` (默认值) 表示服务器最多可使用 90% 的可用内存。

可用于在内存较小的系统上降低内存使用量。
在 RAM 和交换空间都较小的主机上，你可能需要将设置 [`max_server_memory_usage_to_ram_ratio`](#max_server_memory_usage_to_ram_ratio) 设为大于 1。

此比例会通过两种方式生效：

* 在启动时 (以及重新加载配置时) ，它会将服务器的硬内存限制设为总物理 RAM 的这一比例，如上所述。
* 在运行时，后台内存工作线程会定期按 `(常驻内存 + system available memory) * max_server_memory_usage_to_ram_ratio` 重新计算硬内存限制，从而为同一主机上运行的其他进程预留余量。当在具有有限内存限制的 cgroup 中运行时，会使用 cgroup 的可用内存，而不是整个主机范围内的值。随着其他进程占用增加、可用内存减少，服务器的硬限制也会随之下调，从而在主机 (或 cgroup) 耗尽内存之前限制其继续增长。

将该比例设为 `0` 会同时禁用启动时上限和运行时调整。在非 Linux 系统中，以及在不提供此设置的 `clickhouse-keeper` 中，运行时调整同样是空操作。若要保留静态的启动/重新加载上限，但禁用运行时调整 (即先前版本的行为) ，请将 `memory_worker_dynamic_hard_limit` 设为 `0`。

<Note>
  服务器的最大内存消耗还会进一步受到设置 `max_server_memory_usage` 的限制。
</Note>

<div id="min_allocation_size_to_throw_on_memory_limit">
  ## min_allocation_size_to_throw_on_memory_limit
</div>

<SettingsInfoBlock type="UInt64" default_value="0" changeable_without_restart="Yes" />

达到 `max_server_memory_usage` 后，允许触发 `MEMORY_LIMIT_EXCEEDED` 的通用 C++ 内存分配的最小大小 (以字节为单位) 。这里的“通用分配”是指标准容器、字符串、`std::vector` 扩容、智能指针等所进行的那类分配。较小的通用分配仍会计入 memory tracker，但即使超出限制也会允许成功，从而减少接近 OOM 时在清理和异常处理路径中出现的误报失败。

由 ClickHouse 自身的大型缓冲区发起的分配——列数据、哈希表、Arena、查询管道、IO 缓冲区——始终遵守内存限制，因此无论此值如何都可能抛出异常。此设置仅影响通过 `operator new` 进行的隐式分配。

值为 `0` (默认值) 时，会保留旧版行为：memory tracker 永远不会拒绝隐式 `operator new` 分配，只有显式的 ClickHouse 分配才会触发 `MEMORY_LIMIT_EXCEEDED`。

请注意，为避免副作用，建议将该值设置为大于 `max_untracked_memory`。

<div id="oom_canary_enable">
  ## oom_canary_enable
</div>

<SettingsInfoBlock type="Bool" default_value="0" changeable_without_restart="No" />

Experimental。启用 OOM canary：这是一个充当“牺牲品”的子进程，会在主 ClickHouse 服务器进程之前吸引 Linux OOM killer，
从而让服务器有机会主动卸载负载。
需要 Linux &gt;= 5.3 (用于 `pidfd_open`) ；在较旧的内核上，canary 会在启动时被禁用。
OOM 响应要求 cgroup v2 `memory.events.local` 中存在 OOM-kill 证据，并且可能会执行全局
查询取消、merge 取消以及向 `system.crash_log` 写入记录。
当服务器所在的 cgroup 启用了 cgroup v2 `memory.oom.group` 时，canary 无法保护服务器：
此时内核会将整个 cgroup 作为一个整体一并杀死，其中也包括服务器，
因此 OOM 响应根本不会运行。在这种模式下，启动时会记录一条警告日志。
在完成生产环境验证之前，此行为在不同 ClickHouse 版本之间可能会发生变化。

<div id="oom_canary_initial_backoff_seconds">
  ## oom_canary_initial_backoff_seconds
</div>

<SettingsInfoBlock type="UInt64" default_value="1" changeable_without_restart="No" />

连续两次重新启动 OOM canary 之间的初始退避延迟，单位为秒。
该延迟会在每次重新启动时加倍，直到达到 `oom_canary_max_backoff_seconds`。

<div id="oom_canary_max_backoff_seconds">
  ## oom_canary_max_backoff_seconds
</div>

<SettingsInfoBlock type="UInt64" default_value="60" changeable_without_restart="No" />

连续两次 OOM canary 重新启动之间的最大退避延迟 (秒) 。

<div id="oom_canary_max_rapid_relaunches">
  ## oom_canary_max_rapid_relaunches
</div>

<SettingsInfoBlock type="UInt64" default_value="10" changeable_without_restart="No" />

在自动重新启动被禁用之前，OOM canary 可连续快速重新启动的最大次数，
以避免在持续的内存压力下发生抖动。当 canary 的存活时间超过 `oom_canary_max_backoff_seconds` 时，
计数器和重新启动退避会重置，因此，如果 canary 只是在较长运行时间内偶尔终止，
最终不会被禁用。
仅在 `oom_canary_relaunch` 为 true 时适用。

<div id="oom_canary_relaunch">
  ## oom_canary_relaunch
</div>

<SettingsInfoBlock type="Bool" default_value="1" changeable_without_restart="No" />

当值为 true 时，除永久性设置失败或服务器关闭外，只要 canary 进程因任何其他原因终止，OOM canary 都会自动重新启动——包括被 OOM kill、收到其他
信号以及临时退出——但会受到 `oom_canary_max_rapid_relaunches` 和 退避
设置的限制。只有在 cgroup v2 `memory.events.local`
提供 OOM-kill 证据时，OOM 响应序列本身才会运行。

<div id="oom_canary_size">
  ## oom_canary_size
</div>

<SettingsInfoBlock type="UInt64" default_value="104857600" changeable_without_restart="No" />

OOM canary 子进程分配并触碰的内存区域大小 (以字节为单位) 。
使用 `mlock` 锁定该区域属于尽力而为：这需要 `CAP_IPC_LOCK` 或足够的
`RLIMIT_MEMLOCK`。如果锁定失败，canary 会在日志中记录一条警告，并且该内存仍会保持
已分配状态，但可能会被换出。
默认值为 100 MB (104857600) 。值越大，canary 就越可能成为 OOM 的优先目标。

<div id="proxy">
  ## 代理
</div>

为 HTTP 和 HTTPS 请求定义代理服务器，目前 S3 存储、S3 表函数和 URL 函数支持此功能。

定义代理服务器有三种方式：

* 环境变量
* 代理列表
* 远程代理解析器。

也支持使用 `no_proxy` 为特定主机绕过代理服务器。

**环境变量**

`http_proxy` 和 `https_proxy` 环境变量允许你为指定协议设置代理服务器。如果你已在系统中设置了它们，通常即可直接生效。

如果某个协议只有一个代理服务器，且该代理服务器不会变更，这是最简单的方法。

**代理列表**

这种方法允许你为某个协议指定一个或多个代理服务器。如果定义了多个代理服务器，ClickHouse 会以轮询方式使用不同的代理，从而在各个服务器之间均衡负载。如果某个协议有多个代理服务器，且代理服务器列表不会变更，这是最简单的方法。

**配置模板**

```xml
<proxy>
    <http>
        <uri>http://proxy1</uri>
        <uri>http://proxy2:3128</uri>
    </http>
    <https>
        <uri>http://proxy1:3128</uri>
    </https>
</proxy>
```

在下方标签页中选择一个父字段，查看其子字段：

<Tabs>
  <Tab title="<proxy>">
    | Field     | Description       |
    | --------- | ----------------- |
    | `<http>`  | 一个或多个 HTTP 代理的列表  |
    | `<https>` | 一个或多个 HTTPS 代理的列表 |
  </Tab>

  <Tab title="<http> and <https>">
    | Field   | Description |
    | ------- | ----------- |
    | `<uri>` | 代理的 URI     |
  </Tab>
</Tabs>

**远程代理解析器**

代理服务器可能会动态变化。在这种情况下，
你可以定义一个解析器端点。ClickHouse 会向
该端点发送一个空的 GET 请求，远程解析器应返回代理主机。
ClickHouse 将使用它按以下模板构造代理 URI： `\{proxy_scheme\}://\{proxy_host\}:{proxy_port}`

**配置模板**

```xml
<proxy>
    <http>
        <resolver>
            <endpoint>http://resolver:8080/hostname</endpoint>
            <proxy_scheme>http</proxy_scheme>
            <proxy_port>80</proxy_port>
            <proxy_cache_time>10</proxy_cache_time>
        </resolver>
    </http>

    <https>
        <resolver>
            <endpoint>http://resolver:8080/hostname</endpoint>
            <proxy_scheme>http</proxy_scheme>
            <proxy_port>3128</proxy_port>
            <proxy_cache_time>10</proxy_cache_time>
        </resolver>
    </https>

</proxy>
```

在下方的选项卡中选择一个父字段，以查看其子字段：

<Tabs>
  <Tab title="<proxy>">
    | Field     | Description |
    | --------- | ----------- |
    | `<http>`  | 一个或多个解析器列表* |
    | `<https>` | 一个或多个解析器列表* |
  </Tab>

  <Tab title="<http> 和 <https>">
    | Field        | Description   |
    | ------------ | ------------- |
    | `<resolver>` | 解析器的端点及其他详细信息 |

    <Note>
      你可以有多个 `<resolver>` 元素，但对于给定协议，只有第一个
      `<resolver>` 会被使用。该协议的其他 `<resolver>`
      元素都会被忽略。这意味着负载均衡
      (如果需要) 应由远程解析器实现。
    </Note>
  </Tab>

  <Tab title="<resolver>">
    | Field                | Description                                                                         |
    | -------------------- | ----------------------------------------------------------------------------------- |
    | `<endpoint>`         | 代理解析器的 URI                                                                          |
    | `<proxy_scheme>`     | 最终代理 URI 的协议，可以是 `http` 或 `https`。                                                  |
    | `<proxy_port>`       | 代理解析器的端口号                                                                           |
    | `<proxy_cache_time>` | ClickHouse 缓存来自解析器的值的时长 (以秒为单位) 。将此值设为 `0` 会使 ClickHouse 对每个 HTTP 或 HTTPS 请求都联系解析器。 |
  </Tab>
</Tabs>

**优先级**

代理设置按以下顺序确定：

| Order | Setting |
| ----- | ------- |
| 1.    | 远程代理解析器 |
| 2.    | 代理列表    |
| 3.    | 环境变量    |

ClickHouse 会检查适用于该请求协议的最高优先级解析器类型。如果未定义，
则会继续检查次高优先级的解析器类型，直到检查到环境解析器为止。
这也允许混合使用多种解析器类型。

<div id="query_masking_rules">
  ## query_masking_rules
</div>

基于正则表达式的规则，在存储到服务器日志、[`system.query_log`](/zh/reference/system-tables/query_log)、[`system.text_log`](/zh/reference/system-tables/text_log)、[`system.processes`](/zh/reference/system-tables/processes) 表以及发送给客户端的日志之前，会应用于查询和所有日志消息。这样可以防止 SQL 查询中的敏感数据 (如姓名、电子邮件、个人身份标识符或信用卡号) 泄露到日志中。

**示例**

```xml
<query_masking_rules>
    <rule>
        <name>hide SSN</name>
        <regexp>(^|\D)\d{3}-\d{2}-\d{4}($|\D)</regexp>
        <replace>000-00-0000</replace>
    </rule>
</query_masking_rules>
```

**配置字段**：

| Setting   | Description                |
| --------- | -------------------------- |
| `name`    | 规则名称 (可选)                  |
| `regexp`  | 与 RE2 兼容的正则表达式 (必填)        |
| `replace` | 用于替换敏感数据的字符串 (可选，默认为六个星号)  |

掩码规则会应用于整个查询 (以防格式错误或无法解析的查询泄露敏感数据) 。

[`system.events`](/zh/reference/system-tables/events) 表中有一个计数器 `QueryMaskingRulesMatch`，用于统计查询脱敏规则匹配的总次数。

对于分布式查询，必须分别配置每台服务器，否则传递到其他
节点的子查询将以未脱敏的状态被存储。

<div id="remote_url_allow_hosts">
  ## remote_url_allow_hosts
</div>

允许在 URL 相关的存储引擎和表函数中使用的主机列表。

添加带有 `\<host\>` XML 标签的主机时：

* 必须与 URL 中的写法完全一致，因为主机名会在 DNS 解析前进行检查。例如：`<host>clickhouse.com</host>`
* 如果在 URL 中显式指定了端口，则会将 host:port 作为一个整体进行检查。例如：`<host>clickhouse.com:80</host>`
* 如果指定主机时未带端口，则允许该主机的任意端口。例如：如果指定了 `<host>clickhouse.com</host>`，则允许 `clickhouse.com:20` (FTP)、`clickhouse.com:80` (HTTP)、`clickhouse.com:443` (HTTPS) 等。
* 如果将主机指定为 IP 地址，则会按 URL 中指定的形式进行检查。例如：`[2a02:6b8:a::a]`。
* 如果存在重定向且已启用重定向支持，则会检查每一次重定向 (location 字段) 。

例如：

```sql
<remote_url_allow_hosts>
    <host>clickhouse.com</host>
</remote_url_allow_hosts>
```

<div id="configuration-of-disks">
  ### 磁盘配置
</div>

`disks` 的配置结构如下：

```xml
<storage_configuration>
    <disks>
        <disk_name_1>
            <path>/mnt/fast_ssd/clickhouse/</path>
        </disk_name_1>
        <disk_name_2>
            <path>/mnt/hdd1/clickhouse/</path>
            <keep_free_space_bytes>10485760</keep_free_space_bytes>
        </disk_name_2>
        <disk_name_3>
            <path>/mnt/hdd2/clickhouse/</path>
            <keep_free_space_bytes>10485760</keep_free_space_bytes>
        </disk_name_3>
        ...
    </disks>
</storage_configuration>
```

上述子标签为 `disks` 定义了以下设置：

| Setting                 | Description                                        |
| ----------------------- | -------------------------------------------------- |
| `<disk_name_N>`         | 磁盘名称，必须唯一。                                         |
| `path`                  | server 数据将存储到的路径 (`data` 和 `shadow` 目录) 。应以 `/` 结尾 |
| `keep_free_space_bytes` | 磁盘上预留的空闲空间大小。                                      |

<Note>
  磁盘的顺序并不重要。
</Note>

<div id="configuration-of-policies">
  ### 策略配置
</div>

上述子标签为 `policies` 定义了以下设置：

| 设置                           | 说明                                                                                                                                                                                 |
| ---------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `policy_name_N`              | 策略名称。策略名称必须唯一。                                                                                                                                                                     |
| `volume_name_N`              | 卷名称。卷名称必须唯一。                                                                                                                                                                       |
| `disk`                       | 该卷中的磁盘。                                                                                                                                                                            |
| `max_data_part_size_bytes`   | 此卷中任意磁盘上可存放的单个数据 part 的最大大小。如果 `merge` 后生成的数据 part 预计会大于 `max_data_part_size_bytes`，则该 part 会被写入下一个卷。该功能本质上允许你将新的 / 较小的 part 存储在热 (SSD) 卷上，并在其变大后移动到冷 (HDD) 卷上。如果策略只有一个卷，请不要使用此选项。 |
| `move_factor`                | 卷上可用空闲空间的占比。如果可用空间低于该值，数据将开始转移到下一个卷 (如果存在) 。转移时，会按大小从大到小 (降序) 对数据 part 排序，并选择总大小足以满足 `move_factor` 条件的 part；如果所有 part 的总大小仍不足，则会移动所有 part。                                         |
| `perform_ttl_move_on_insert` | 禁用在插入时移动生存时间 (TTL) 已过期的数据。默认情况下 (启用时) ，如果插入的一段数据根据 TTL 移动规则已经过期，它会立即被移动到该规则指定的卷 / 磁盘。如果目标卷 / 磁盘较慢 (例如 S3) ，这可能会显著降低插入速度。如果禁用，则已过期的数据部分会先写入默认卷，然后再立即移动到规则中为过期 TTL 指定的卷。             |
| `load_balancing`             | 磁盘均衡策略，`round_robin` 或 `least_used`。                                                                                                                                               |
| `least_used_ttl_ms`          | 设置更新所有磁盘可用空间的超时时间 (以毫秒为单位)  (`0` - 始终更新，`-1` - 从不更新，默认值为 `60000`) 。请注意，如果磁盘仅由 ClickHouse 使用，且不会在运行中动态调整文件系统大小，则可以使用 `-1`。在其他所有情况下都不建议这样做，因为最终会导致错误的空间分配。                           |
| `prefer_not_to_merge`        | 禁止在此卷上合并数据的 parts。注意：这可能带来负面影响并导致性能下降。启用此设置时 (不要这样做) ，将禁止在此卷上合并数据 (这不是好事) 。这可以让你控制 ClickHouse 如何与慢速磁盘交互。我们建议完全不要使用此设置。                                                             |
| `volume_priority`            | 定义卷的填充优先级 (顺序) 。值越小，优先级越高。参数值必须是自然数，并且必须无间断覆盖从 1 到 N 的范围 (N 是指定的最大参数值) 。                                                                                                           |

对于 `volume_priority`：

* 如果所有卷都设置了此参数，则按指定顺序确定优先级。
* 如果只有&#95;部分&#95;卷设置了此参数，则未设置该参数的卷优先级最低。设置了该参数的卷按标签值确定优先级，其余卷之间的优先级则由它们在配置文件中的定义顺序决定。
* 如果&#95;没有&#95;卷设置此参数，则它们的顺序由其在配置文件中的定义顺序决定。
* 各卷的优先级不能相同。

<div id="use_minimalistic_part_header_in_zookeeper">
  ## use_minimalistic_part_header_in_zookeeper
</div>

ZooKeeper 中数据分区片段头信息的存储方式。此设置仅适用于 [`MergeTree`](/zh/reference/engines/table-engines/mergetree-family/index) 家族。可按以下方式指定：

**在 `config.xml` 文件的 [merge&#95;tree](#merge_tree) 部分中全局设置**

ClickHouse 会将此设置应用于服务器上的所有表。你可以随时更改该设置。设置变更后，现有表的行为也会随之改变。

**为每个表单独设置**

创建表时，指定相应的[引擎设置](/zh/reference/engines/table-engines/mergetree-family/mergetree#table_engine-mergetree-creating-a-table)。对于已设置该选项的现有表，即使全局设置发生变化，其行为也不会改变。

* `1` — 功能开启。

<Note>
  启用 `use_minimalistic_part_header_in_zookeeper = 1` 后，你将无法把 ClickHouse server 降级到不支持此设置的版本。在集群中的服务器上升级 ClickHouse 时请务必小心。不要一次升级所有服务器。更稳妥的做法是先在测试环境中，或仅在集群中的少数几台服务器上测试新版本的 ClickHouse。

  已使用此设置存储的数据分区片段头信息，无法恢复为之前的 (非紧凑) 表示形式。
</Note>

<div id="use_shared_merge_tree_log_pipeline">
  ## use_shared_merge_tree_log_pipeline
</div>

<SettingsInfoBlock type="Bool" default_value="0" changeable_without_restart="No" />

仅在 ClickHouse Cloud 中可用。启用后，`system.*_log` 表会通过基于 S3 的管道，以 `SharedMergeTree` 作为底层存储。
对于每个日志 `<log>`，启动时都会自动创建以下对象：

* `system.<log>_s3` — 一个以 `S3` 为后端的表，也是直接的 flush 目标；每次
  `SYSTEM FLUSH LOGS` 调用都会在此写入一个新的分区文件。
* `system.<log>_s3queue` — 一个 `S3Queue` 表 (有序模式) ，用于从
  `<log>_s3` 获取文件并将行流式传输到下游。每个节点仅通过
  基于 `partition_regex` 的分区规则处理自己的文件。
* `system.<log>_mv` — 一个 `MATERIALIZED VIEW`，用于将行从 `<log>_s3queue` 路由到
  最终的 `SharedMergeTree` 表。
* `system.<log>` — 最终的 `SharedMergeTree` 表，行会在此累积并供查询使用。

当 schema 或设置发生变化时，受影响的表会被重命名为 `<log>_0`、`<log>_1` 等，
然后重新创建，这与现有的 `SystemLog` 轮转行为一致。

要求在服务器配置中设置 `<shared_log_pipeline><endpoint>`。
另请参见：`shared_log_pipeline.enable_sync_flush`、`shared_log_pipeline.flush_timeout_seconds`。

`user_directories` 部分可以包含任意数量的条目，条目的顺序表示其优先级 (越靠前的条目优先级越高) 。

```xml
<user_directories>

        <path>/etc/clickhouse-server/users.xml</path>
    </users_xml>
    <local_directory>
        <path>/var/lib/clickhouse/access/</path>
    </local_directory>
</user_directories>
```

用户、角色、行策略、配额和 profile 也可存储在 ZooKeeper 中：

```xml
<user_directories>
    <users_xml>
        <path>/etc/clickhouse-server/users.xml</path>
    </users_xml>
    <replicated>
        <zookeeper_path>/clickhouse/access/</zookeeper_path>
    </replicated>
</user_directories>
```

你还可以定义 `memory` 部分——表示信息仅存储在内存中，不写入磁盘；以及 `ldap` 部分——表示将信息存储在 LDAP 服务器上。

要将 LDAP 服务器添加为未在本地定义的用户的远程用户目录，请定义一个 `ldap` 部分，并包含以下设置：

| Setting  | Description                                                                                                                |
| -------- | -------------------------------------------------------------------------------------------------------------------------- |
| `roles`  | 该部分包含本地定义的 roles 列表，这些角色会分配给从 LDAP 服务器获取的每个用户。如果未指定任何角色，用户在身份验证后将无法执行任何操作。如果在身份验证时，所列出的任一角色未在本地定义，则身份验证尝试将失败，就像提供的密码不正确一样。 |
| `server` | `ldap_servers` 配置部分中定义的 LDAP 服务器名称之一。此参数为必填项，且不能为空。                                                                        |

**示例**

```xml
<ldap>
    <server>my_ldap_server</server>
        <roles>
            <my_local_role1 />
            <my_local_role2 />
        </roles>
</ldap>
```

<div id="wait_dictionaries_load_at_startup">
  ## wait_dictionaries_load_at_startup
</div>

<SettingsInfoBlock type="Bool" default_value="1" changeable_without_restart="No" />

此设置用于指定在 `dictionaries_lazy_load` 为 `false` 时的行为。
(如果 `dictionaries_lazy_load` 为 `true`，则此设置不起任何作用。)

如果 `wait_dictionaries_load_at_startup` 为 `false`，则服务器会在启动时开始加载所有字典，并在加载过程中并行接受连接。
当某个字典首次在查询中使用时，如果该字典尚未加载，查询将等待其加载完成。
将 `wait_dictionaries_load_at_startup` 设为 `false` 可以让 ClickHouse 启动得更快，但某些查询的执行速度可能会变慢
(因为它们需要等待某些字典完成加载) 。

如果 `wait_dictionaries_load_at_startup` 为 `true`，则服务器会在启动时等待，直到所有字典完成加载 (无论是否成功) 后，才接受任何连接。

**示例**

```xml
<wait_dictionaries_load_at_startup>true</wait_dictionaries_load_at_startup>
```

<div id="zookeeper">
  ## zookeeper
</div>

包含允许 ClickHouse 与 [ZooKeeper](http://zookeeper.apache.org/) 集群交互的设置。使用复制表时，ClickHouse 会使用 ZooKeeper 存储副本的元数据。如果不使用复制表，则可省略这一组参数。

以下设置可通过子标签进行配置：

| Setting                                         | Description                                                                                                                                                                                                                                                                   |
| ----------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `node`                                          | ZooKeeper 端点。你可以设置多个端点。例如：`<node index="1"><host>example_host</host><port>2181</port></node>`。`index` 属性指定尝试连接 ZooKeeper 集群时的节点顺序。                                                                                                                                            |
| `operation_timeout_ms`                          | 单次操作的最大超时时间，以毫秒为单位。                                                                                                                                                                                                                                                           |
| `session_timeout_ms`                            | 客户端会话的最大超时时间，以毫秒为单位。                                                                                                                                                                                                                                                          |
| `root` (optional)                               | 作为 ClickHouse server 所用各 znode 根路径的 znode。                                                                                                                                                                                                                                    |
| `fallback_session_lifetime.min` (optional)      | 当主节点不可用时，回退节点上 zookeeper 会话生命周期的最小值 (负载均衡) 。以秒为单位设置。默认值：3 小时。                                                                                                                                                                                                                 |
| `fallback_session_lifetime.max` (optional)      | 当主节点不可用时，回退节点上 zookeeper 会话生命周期的最大值 (负载均衡) 。以秒为单位设置。默认值：6 小时。                                                                                                                                                                                                                 |
| `identity` (optional)                           | 访问请求的 znode 时，ZooKeeper 所需的用户名和密码。                                                                                                                                                                                                                                            |
| `use_compression` (optional)                    | 设为 true 时，在 Keeper 协议中启用压缩。                                                                                                                                                                                                                                                   |
| `use_xid_64` (optional)                         | 启用 64 位事务 ID。设为 `true` 以启用扩展事务 ID 格式。默认值：`false`。                                                                                                                                                                                                                             |
| `pass_opentelemetry_tracing_context` (optional) | 启用将 OpenTelemetry 链路追踪上下文传递到 Keeper 请求。启用后，将为 keeper operations 创建 tracing spans，从而支持跨 ClickHouse 和 Keeper 的分布式链路追踪。更多详情请参见 [Tracing ClickHouse Keeper Requests](/zh/guides/oss/deployment-and-scaling/monitoring/opentelemetry#tracing-clickhouse-keeper-requests)。默认值：`false`。 |

此外，还有一个 `zookeeper_load_balancing` 设置 (可选) ，可用于选择 ZooKeeper 节点的选取算法：

| Algorithm Name                   | Description                                                                 |
| -------------------------------- | --------------------------------------------------------------------------- |
| `random`                         | 随机选择一个 ZooKeeper 节点。                                                        |
| `in_order`                       | 选择第一个 ZooKeeper 节点；如果它不可用，则选择第二个，依此类推。                                      |
| `nearest_hostname`               | 选择主机名与 server 的 hostname 最相似的 ZooKeeper 节点，按名称前缀比较 hostname。                |
| `hostname_levenshtein_distance`  | 与 nearest&#95;hostname 类似，但会以 Levenshtein distance 的方式比较 hostname。          |
| `hostname_longest_common_prefix` | 与 nearest&#95;hostname 类似，但会优先选择其 hostname 与 server 的 hostname 具有最长公共前缀的节点。 |
| `hostname_longest_common_suffix` | 与 nearest&#95;hostname 类似，但会优先选择其 hostname 与 server 的 hostname 具有最长公共后缀的节点。 |
| `first_or_random`                | 选择第一个 ZooKeeper 节点；如果它不可用，则从其余 ZooKeeper 节点中随机选择一个。                         |
| `round_robin`                    | 选择第一个 ZooKeeper 节点；如果发生重新连接，则选择下一个。                                         |

**示例配置**

```xml
<zookeeper>
    <node>
        <host>example1</host>
        <port>2181</port>
    </node>
    <node>
        <host>example2</host>
        <port>2181</port>
    </node>
    <session_timeout_ms>30000</session_timeout_ms>
    <operation_timeout_ms>10000</operation_timeout_ms>
    <!-- Optional. Chroot suffix. Should exist. -->
    <root>/path/to/zookeeper/node</root>
    <!-- Optional. Zookeeper digest ACL string. -->
    <identity>user:password</identity>
    <!--<zookeeper_load_balancing>random / in_order / nearest_hostname / hostname_levenshtein_distance / hostname_longest_common_prefix / hostname_longest_common_suffix / first_or_random / round_robin</zookeeper_load_balancing>-->
    <zookeeper_load_balancing>random</zookeeper_load_balancing>
    <!-- Optional. Enable 64-bit transaction IDs. -->
    <use_xid_64>false</use_xid_64>
    <!-- Optional. Enable OpenTelemetry tracing context propagation. -->
    <pass_opentelemetry_tracing_context>false</pass_opentelemetry_tracing_context>
</zookeeper>
```
